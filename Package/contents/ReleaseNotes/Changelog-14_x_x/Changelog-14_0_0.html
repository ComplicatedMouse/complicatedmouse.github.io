Changes in NintendoSDK NX Addon 14.0.0 (based on Generic 14.0.0)

Specification change that breaks your code

*14-0-0S01: (ASYNC) Made changes to include the async library in the SDK DLL.

Made changes to include the async library, previously provided as libnn_async.a and libnn_async.lib, in the SDK DLL.
Starting with this release, the individual library files are not included in the package.

Application developers who use the NintendoSDK-NPLN package need to delete libnn_async.a and libnn_async.lib from the libraries linked by their applications.

*14-0-0S02: (AUDIO) Changed the specifications for channel mapping of effects available for use with the nn::audio audio renderer.

Changed the specifications for channel mapping of effects available for use with the nn::audio audio renderer.

For the following effects available for use in the audio renderer, you need to pay attention to the channel corresponding to each input/output buffer.

  • Delay
  • Reverb
  • I3dl2Reverb


For these effects, the channel order when set for 6ch is defined as follows in NintendoSDK 13.x.x and earlier.

  • Front Left
  • Front Right
  • Rear Left
  • Rear Right
  • Center
  • LFE


Specifications have been changed in NintendoSDK 14.0.0 and later so that this order matches the order of the nn::audio::ChannelMapping enumerated types. Specifically, the following order is used.

  • Front Left
  • Front Right
  • Center
  • LFE
  • Rear Left
  • Rear Right


Due to this change, you must change the order of the input/output buffers when using the following functions to configure 6ch effect channels.

  • nn::audio::SetDelayInputOutput()
  • nn::audio::SetReverbInputOutput()
  • nn::audio::SetI3dl2ReverbInputOutput()
  • nn::audio::SetAuxDelayInputOutput()
  • nn::audio::SetAuxReverbInputOutput()
  • nn::audio::SetAuxI3dl2ReverbInputOutput()


In addition to noting the change in specifications, also change the order of the channel mapping on the application side.
For more information, see audio Library > Manual > Features > Audio Renderer > Audio Effects.

*14-0-0S03: (DevMenuCommand) Changed the name of the command for setting the number of channels in the audio output target.

Changed the command name for setting the number of channels in the audio output target from audio set_output_mode to audio set-output-mode.
For more information about the feature, see DevMenuCommand > Audio Settings.

While the previous audio set_output_mode command is still available, it is deprecated and will be deleted in the future.

*14-0-0S04: (DevMenuCommand) Fixed a bug where gamecard write failed for NSP files on multi-application cards that contain DLC.

Fixed a bug where the gamecard write DevMenuCommand failed for NSP files on multi-application cards that contain DLC.
This fix allows you to write the file created by the bundleup command in AuthoringTool as is.
In exchange, you can no longer write the NSP files used as the workaround before the fix. These NSP files were the files for multi-application cards that include DLC that were converted to on-card DLC in advance.

For more information, see the NVN Programming Guide.

14-0-0S20: (Compiler) Updated to Clang for NX 1.11.5.

Updated Clang for NX to version 1.11.5.
The main changes are as follows.

  • libc++ can now use the pthread library internally.
  • The std::random_device function can now use more appropriate random sources.
  • The PThread metadata and structure size were updated to be closer to the standard, so more pthread functions and features are now supported.
  • pthread_mutex_t has increased by 4 bytes in 32-bit mode and 8 bytes in 64-bit mode.
  • pthread_cond_t has increased by 4 bytes.
  • pthread_barrier_t has increased by 4 bytes.
  • std::thread now uses pthread_t instead of nn::os::ThreadType.
  • Made many internal functions private.
  • Reduced the size of std::random_device.
  • You can now select whether to have the sanitizer library dynamically or statically allocate memory. If you specify -fsanitize=undefined as normal, the default behavior will be dynamic allocation. If you also specify the -fsanitize-undefined-reduced-dynamic-allocations option, memory will be allocated statically.
  • Added documentation about static analyzer. For more information, see Compiler / Toolchain for NX > LLVM and Clang. . . What are They?

Bug Fixes

14-0-0S54: (DevMenu) Fixed a bug where it was not possible to start an application under some conditions, even though it should be available to start.

Fixed a bug where attempting to start an application under the following conditions caused the "Please turn Airplane Mode off" message to be displayed and the application could not be started.

  • Cannot connect to the Internet
  • The account specified to start the application can use the application even when not connected to the Internet.
  • The account specified to start the application cannot use (at least some) DLC if it cannot connect to the Internet.

Due to this fix, the following verification message is now displayed when you start an application under these conditions.
"You can launch this application, but you don't have some rights of add-on contents. Launch anyway?"
If you select Launch at this point, the application starts, but the applicable DLC is not available.

14-0-0S61: (Graphics) Allocators assigned to nv::SetGraphicsAllocator() will request additional memory.

The allocation functions assigned to nv::SetGraphicsAllocator() encounters an 8-byte increase compared to previous releases when acquiring a buffer for the first time. The time of the increase depends on the graphics API. For example, the memory allocation pattern change is seen with nvn::Window::AcquireTexture().

This affects both 32- and 64-bit applications.

14-0-0S62: (HID) Fixed a bug where some keyboard input could not be obtained correctly in the Windows environment.

The following bugs were fixed.

  • The single-byte/double-byte key was not treated as nn::hid::KeyboardKey::Backquote.
  • When the keypad keys were not locked, they were not treated as input from the keypad.


For more information, see the manual.

Because there was no description in the NintendoSDK movie Library Programming Guide about how to handle ADTS frames that include multiple AAC frames, a description was added stating that the application must pass the whole AAC raw frame, without the ADTS header, to decode using the movie buffer.

14-0-0S66: (MOVIE) Fixed incorrect descriptions in the NintendoSDK API reference for the return values for movie::Buffer::Base() and movie::Buffer::Data().

Fixed errors in the API descriptions of movie::Buffer::Base() and movie::Buffer::Data(). The correct version is that Data() returns the movie buffer storage memory address, including the offset, while Base() returns the movie buffer data storage address.
This issue was incorrectly described in the revision history for 13.3.0. It has been deleted in 14.0.0 and later.
