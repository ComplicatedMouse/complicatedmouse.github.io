Overview

The following sections describe examples of save data design for a few cases.

Case 1: A Single Player RPG Played Independently by Each User

This describes RPG without specifications to play with other users over a network, where each user plays as a single player.

A player character is created for each user, and the story progresses separately for each user independent of other users.

Save Data Structure

The following data is saved to account save data regions for each account.

  • Progress in story
  • Player character
    • Level and status
  • Items and accessories

Case 2: A Multiplayer Sports Game

This case is an application where many users are playing at the same time, and it does not matter whether they have user accounts.

The game has sporting elements where multiple people are playing at the same time. Although the performance of each user is recorded separately, elements within the application are unlocked across multiple users.
For example, where people are playing doubles tennis or running in a racing game, they would play in the same stage or course on the system, no matter who is playing.

Save Data Structure

The following data is saved to account save data regions for each account.

  • Winning percentage of each user
  • Courses and stages cleared by each user
  • Unlocked accessories and characters for each user
  • Accessory combinations and coordinated costumes often used by users
  • Course records

The following information is collated from account save data regions for all accounts on startup, and applied to the application.

  • Rankings of all of the system's users
  • Courses and stages cleared by all of the system's users
  • Unlocked equipment and characters for all of the system's users
