Overview

Overview

Rather than providing an integrated rendering engine when using the 3D runtime library of NintendoWare (called, in short, the g3d library), the SDK supports game development projects in building a rendering engine optimized for each game by providing each feature of a typical rendering engine as a module.

Features

Rendering Features

The g3d library does not include a rendering feature for model particle size. Programmable shader architecture uses several rendering methods. Enabling the library to support all of them invites redundancy of overhead and data structures, and enlarges the library. This practice is not recommended. To make the best use of programmable shader architecture, the g3d library is designed to build a suitable rendering engine, for each project or team, that provides the feature with small particle size.

The rendering feature supports building low layer graphics objects, such as vertex buffers and samplers, and the issuing of render commands. Most simply, you can render models by scanning their materials and shapes, and sending the data in sequence. For better optimization, you can improve performance by embedding a sort process corresponding to the render method or a culling process corresponding to the game.

Multithreading

The g3d library does not use an internal synchronization mechanism for locking, and creating threads. For objects shared by multithreading, you must appropriately lock them when calling (non-const) functions that result in changes to internal states.

Binding by Name

The g3d library resolves all external references for shaders, models, or animations by binding by name. Binding during initialization at run time rather than when converting the binary data eliminates the need to reconvert all files that were bound when each file was updated. You can also dynamically update data by reinitializing at run time. Although the initialization costs increase a little by looking up the name, it can be concealed in the read speed from storage and the convenience in the workflow is emphasized.

We strongly recommend that you not perform name lookups that incur processing costs in each frame, and that you cache results during initialization.

Minimal Options and Callbacks

The g3d library provides minimal options and callbacks. A greater number of options and callbacks leads to greater overhead. High level features, such as a scene graph or a content manager, are often cited as reasons for requiring options or callbacks. The g3d library does not provide many of these features because they duplicate the management structure in the game and are an obstacle to embedding. Because the g3d library provides features with small particle sizes, you can easily select features without providing options and insert them between individual processes without using callbacks.

Double Buffering

The g3d library supports both double buffering and other update processes that do not use double buffering for model buffers that are accessed by the GPU. When instantiating models, materials, skeletons, and shapes, you can specify the number of buffers to use.

When you do not use double buffering, you can reduce the amount of memory used and the cost of managing buffers by updating them when the GPU is not referencing a model.

Memory Allocation

The g3d library does not allocate memory internally, excluding that used for real-time editing functionality. This prevents memory leaks from occurring. Acquiring all needed memory at the same time also reduces the wasteful overhead and administration space required for small, individual memory allocations/deallocations.

Shader Parameters

Information used at run time and by tools can be controlled using command-style annotation when using the g3d shader system. You can edit shaders using any shader authoring tool as long as it is annotated.

You can also use annotations to freely configure the display of shader parameters in 3DEditor, and create animations for any shader parameter.

Real-Time Editing Feature

The g3d library provides a real-time editing feature that communicates with the 3DEditor. Because you can embed the 3DEditor linking feature in the game without any major changes, developers can edit models and animations in real time and preview them using the game's rendering engine. The corresponding classes are defined in the nn::g3d::viewer namespace.
