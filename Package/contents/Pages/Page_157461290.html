HTTPS

Overview

The Nintendo SDK libcurl is configured to provide HTTPS feature backed by Nintendo SDK SSL library. It behaves almost in the same way as the standard libcurl though few additional steps need to be performed to use HTTPS in the Nintendo SDK's libcurl.

Like the off-the-shelf libcurl, non-trivial amount of knowledge of the SSL library is necessary to use HTTPS properly. It is highly recommended to go through the documentation of the SSL library first to understand what the SSL library provides and limitations in it. This documentation refers some entities in the SSL library below.

Exception

  • Prior to Nintendo SDK 0.14.0, 64-bit Windows builds did not support HTTPS. But as of Nintendo SDK 0.14.0, HTTPS is supported in all build types.
  • Please note that CURLOPT_CAINFO and CURLOPT_CAPATH are not supported in Nintendo SDK libcurl to specify alternate certificates
    • Built-in certificates are always utilized in the SSL library without explicit configuration
    • Or necessary certificates need to be imported via nn::ssl::Context

Differences from the normal libcurl

Initialization/Finalization

The Nintendo SDK libcurl automatically initializes/finalizes the SSL library since the user of libcurl is not required to call nn::ssl::Initialize() and nn::ssl::Finalize() explicitly.

The SSL library is initialized automatically only when curl_global_init() or curl_global_init_mem() is called. It is finalized only when curl_global_cleanup() function is called.

It is possible to initialize the SSL library by nn::ssl::Initialize() prior to libcurl initialization. Doing so may be useful especially when the user needs to set concurrency limit manually. Please make sure to call nn::ssl::Finalize() explicitly after curl_global_cleanup() is called if the SSL library is initialized outside libcurl. If nn::ssl::Initialize() is called multiple times before the library is finalized, each successive call after the first will return immediately with nn::Result::ResultLibraryAlreadyInitialized and will not update the concurrency limit.

CURL handle, SSL context object and SSL connection objects

Unlike using the SSL library directly for use of SSL/TLS, it is not required to instantiate a nn::ssl::Context object and a nn::ssl::Connection object explicitly. This is because each CURL handle (which can be obtained by curl_easy_init()) comes with the object of nn::ssl::Connection and nn::ssl::Context inside libcurl.

SSL context

Even though instantiation of such objects is not required, it is still required to "create" a nn::ssl::Context object manually.

In addition to that, libcurl provides a way to import a nn::ssl::Context object instantiated outside libcurl. In this situation, the user of libcurl is responsible to instantiate and create a nn::ssl::Context before importing it to a CURL handle. With this feature, the user of libcurl can share nn::ssl::Context object between CURL handles rather than creating a nn::ssl::Context object every time for each CURL handle.

SSL Connection

Unlike nn::ssl::Context object, nothing is required to deal with nn::ssl::Connection. Instantiation and creation of a nn::ssl::Connection object tied to a CURL handle is automatically handled inside libcurl.

Preparing a SSL context

To use HTTPS, libcurl requires a nn::ssl::Context to be prepared by the user. This context can be prepared by one of the below methods.

  • CURLOPT_SSL_CTX_FUNCTION
  • CURLOPT_SSL_CONTEXT

CURLOPT_SSL_CTX_FUNCTION is the simpler method which uses the nn::ssl::Context object tied to each curl handle. CURLOPT_SSL_CONTEXT requires bit more handling compared to CURLOPT_SSL_CTX_FUNCTION because the user of libcurl needs to import a nn::ssl::Context object to libcurl.

Please find more details below.

Callback Function (CURLOPT_SSL_CTX_FUNCTION)

This is the method which utilizes a nn::ssl::Context object tied to each CURL handle. The user of libcurl registers a callback function by CURLOPT_SSL_CTX_FUNCTION. libcurl calls the registered callback function along with the pointer to the nn::ssl::Context tied to the CURL handle. This is where the user of libcurl has an opportunity to create/configure nn::ssl::Context object.

Note that the CURLOPT_SSL_CTX_FUNCTION option is provided as a standard feature in the public version of libcurl. Please refer to the man page of CURLOPT_SSL_CTX_FUNCTION for details. 

Function prototype

#include <curl/curl.h>
 
CURLcode ssl_ctx_callback(CURL *curl, void *ssl_ctx, void *userptr);

Unlike the normal version of libcurl, the ssl_ctx parameter is actually a pointer to the Nintendo SSL library's nn::ssl::Context. Note that it is the pointer to the nn::ssl::Context object which is maintained inside Libcurl hence the caller is not required to instantiate the object of nn::ssl::Context inside the callback function.

Note that returning a negative value from this callback function will cause CURLE_ABORTED_BY_CALLBACK.

CURLOPT_SSL_CTX_DATA is also supported in the same way as the public version of libcurl. The pointer to the data set by this option will be passed in userptr pointer in ssl_ctx_callback.

Example

When the callback function is called, cast the ssl_ctx variable to a nn::ssl::Context pointer then create/configure the nn::ssl::Context object.

Note that cleanup of the nn::ssl::Context object created in the callback is properly handled by libcurl upon calling curl_easy_cleanup(). Thus the user of libcurl is not required to destroy/free nn::ssl::Context object.


The code example of CURLOPT_SSL_CTX_FUNCTION 
CURLcode CurlSslContextCallback(CURL *curl, void *ssl_ctx, void *userptr)
{
    // Obtain pointer to the SSL context tied to a CURL handle
    nn::ssl::Context* pContext = reinterpret_cast<nn::ssl::Context*>(ssl_ctx);
 
    // Create a nn::ssl::Context
    nn::Result result;
    result = pContext->Create(nn::ssl::Context::SslVersion_Auto);
    if (result.IsFailure())
    {
        return -1; // error out here
    }
 
    // Import CA certificate (optional)
    nn::ssl::CertStoreId serverCertStoreId;
    char*                pCaCert = nullptr; // Pointer to the CA certificate in PEM format need to be set
    uint32_t             caCertSize = 0;    // The size of the CA certificate needs to be set
 
    result = pContext->ImportServerPki(
        &serverCertStoreId,
        pCaCert,                         
        sizeof(pCaCert),                 
        nn::ssl::CertificateFormat_Pem);    // Please pass nn::ssl::CertificateFormat_Der if the CA certificate in DER format is passed in pCaCert
    if(result.IsFailure())
    {
        return -1; // error out here        
    }
 
    // Import client PKI information (optional)
    nn::ssl::CertStoreId clientCertStoreId;
    char*       pPkcs12Data    = nullptr; // Pointer to the data in PKCS12 format needs to be set
    uint32_t    pkcs12DataSize = 0;       // The size of PKCS12 data needs to be set
    char*       pkcs12Password = nullptr; // Appropriate password for the PKCS12 data needs to be set
    uint32_t    passwordSize   = 0;       // The size of the password needs to be set
    result = m_SslContext.ImportClientPki(
        &clientCertStoreId,
        pPkcs12Data,
        pkcs12Password,
        pkcs12DataSize,
        passwordSize);
    if(result.IsFailure())
    {
        return -1; // error out here        
    }
 
    return 0;
}

Importing the SSL context (CURLOPT_SSL_CONTEXT)

This is the method which takes a nn::ssl::Context object from outside so that a nn::ssl::Context tied to a CURL handle is not used.

Instead of creating an SSL context for each CURL handle in the callback, the user of libcurl may want to create only one SSL context and share it among multiple CURL handles. The benefit of doing so is that it will save processing time by not repeating the same operations, such as importing the same certificate multiple times. One example of where this could be helpful is when making many requests for different assets in the same domain.

In order to share the SSL context amongst multiple CURL handles, the application must prepare the SSL context first, then tell each CURL handle to use it via the custom Nintendo SDK libcurl option, CURLOPT_SSL_CONTEXT.

The code for preparing the SSL context is the same as when preparing it in the CURLOPT_SSL_CTX_FUNCTION callback but the timing and scope of the code is different. The SSL context must be prepared before the first CURL handle uses it and it must remain valid until the final CURL handle that uses it has been cleaned up with curl_easy_cleanup().

Note that when sharing the SSL context like this, unlike when using the CURLOPT_SSL_CTX_FUNCTION callback, the application must clean up the SSL context with nn::ssl::Context::Destroy() when it is no longer in use.

Example

The sample code of CURLOPT_SSL_CONTEXT 
void PerformHttps()
{
    CURL*             pCurlHandle = nullptr;
    nn::ssl::Context* pContext = nullptr;
    CURLcode          res = CURLE_OK;
    nn::Result        result;

    // !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
    // Initialize libcurl, create a curl handle and configure the handle
    // !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
        :
        :
    do {
        // Instanciate a nn::ssl::Context object
        pContext = new nn::ssl::Context();
        if (pContext == nullptr)
        {
            break;
        }
 
          // Create the nn::ssl::Context object
        result = pContext->Create(nn::ssl::Context::SslVersion_Auto);
        if (result.IsFailure())
        {
            break;
        }

        // !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
        // Configure the nn::ssl::Context at here if needed
        //     e.g. Importing CA certificates
        // !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
            :
            :
        // Set the nn::ssl::Context object to the curl handle
        res = curl_easy_setopt(pCurlHandle, CURLOPT_SSL_CONTEXT, pContext);
        if (res != CURLE_OK)
        {
            break;
        }
        
        // !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
        // Perform HTTPS with libcurl
        // !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
            :
            :
    } while(0);

    // !!!!!!!!!!!!!!!!!!!!!!
    // Close the curl handle
    // !!!!!!!!!!!!!!!!!!!!!!
        :
        :
    // Cleanup the nn::ssl::Context object
    if (pContext != nullptr)
    {
        pContext->Destroy();
        delete pContext;
    }
    
    // !!!!!!!!!!!!!!!!!!!!!
    // Finalize libcurl
    // !!!!!!!!!!!!!!!!!!!!!
        :
        :
}

Working with a SSL connection

As it is mentioned above, the user of libcurl doesn't need to prepare/cleanup a SSL connection explicitly. Like a nn::ssl::Context object, each curl handle comes with a nn::ssl::Connection object. This object is utilized inside libcurl automatically so the user of libcurl doesn't need to deal with it.

CURLOPT_SSL_CONN_FUNCTION_BEFORE / CURLOPT_SSL_CONN_FUNCTION_AFTER

For users who may need to deal with nn::ssl::Connection object directly, libcurl provides below options.

  • CURLOPT_SSL_CONN_FUNCTION_BEFORE
  • CURLOPT_SSL_CONN_FUNCTION_AFTER

Both options are provided to set a callback function to be called before/after DoHandshake() method of the nn::ssl::Connection class.

The user may set a data pointer to the curl handle by SSL_CONN_DATA option. The pointer to the data is passed to callback functions set by above options. Here're function prototypes.

typedef struct
{
  bool isHandshakeSuccess; /* Set to true when SSL handshake succeeds */
  size_t certDataSize;     /* The size of the certificate(s) data */
  size_t certCount;        /* The number of certificates */ 
} curl_handshake_info;

typedef CURLcode (*curl_ssl_conn_callback_before)(CURL *curl,     /* easy handle */
                                                  void *ssl_conn, /* actually an nn::ssl::Connection */
                                                  void *userptr); /* pointer to the data set by SSL_CONN_DATA */

typedef CURLcode (*curl_ssl_conn_callback_after)(CURL *curl,     /* easy handle */
                                                 void *ssl_conn, /* actually an nn::ssl::Connection */
                                                 void *userptr,  /* pointer to the data set by SSL_CONN_DATA */
                                                 curl_handshake_info handshake_info);

CURLOPT_SSL_CONN_FUNCTION_BEFORE may be used to configure more details in the SSL connection before performing SSL handshake. CURLOPT_SSL_CONN_FUNCTION_AFTER may be used to obtain more information of SSL handshake.

For example, these callback functions need to be used when the user of libcurl needs certificate(s) data obtained during SSL handshake. This is because the user is required to set a pointer to a buffer before performing SSL handshake, then the buffer needs to be referred to get certificate(s) data after SSL handshake.

Certificate Validation

Like normal version of libcurl, verify option can be configured through curl_easy_setopt(). Please refer to the man page of CURLOPT_SSL_VERIFYPEER and CURLOPT_SSL_VERIFYHOST for details.

Both CURLOPT_SSL_VERIFYPEER and CURLOPT_SSL_VERIFYHOST are enabled by default.

In addition to existing verify options in libcurl, the Nintendo SDK libcurl provides few other options related to certificate validation.

  • CURLOPT_SSL_VERIFYDATE
  • CURLOPT_SSL_SKIP_DEFAULT_VERIFY

CURLOPT_SSL_VERIFYDATE

Because many users may not have the date set correctly on their system, the Nintendo SDK libcurl has a custom option, CURLOPT_SSL_VERIFYDATE, for developers who may not want to fail the verification of an SSL certificate based on the date.

When CURLOPT_SSL_VERIFYDATA is 1, the SSL handshake will fail when the certificate sent by the server is expired or not valid yet. The default value of CURLOPT_SSL_VERIFYDATE is 0, it means time validation will not be performed when verifying server certificates.

Example

The sample code to configure verify options 
// Enable VERIFYPEER option
if (curl_easy_setop(curlHandle, CURLOPT_SSL_VERIFYPEER, 1L) != CURLE_OK)
{
    // error out here
}
 
// Enable VERIFYHOST option
if (curl_easy_setop(curlHandle, CURLOPT_SSL_VERIFYHOST, 2L) != CURLE_OK)
{
    // error out here
}
 
// Disable VERIFYDATE option
if (curl_easy_setop(curlHandle, CURLOPT_SSL_VERIFYDATE, 0L) != CURLE_OK)
{
    // error out here
}

CURLOPT_SSL_SKIP_DEFAULT_VERIFY

By default, the SSL library doesn't allow disabling default verify options. This means, below two options cannot be disabled.

  • CURLOPT_SSL_VERIFYPEER 
  • CURLOPT_SSL_VERIFYHOST

It is not recommended to disable these fundamental verify options unless there's a strong reason. For those who may need to disable it (e.g. for testing), the Nintendo SDK libcurl provides CURLOPT_SSL_SKIP_DEFAULT_VERIFY option.

Example

The sample code to disable verify options 
// Enable CURLOPT_SSL_SKIP_DEFAULT_VERIFY option before disabling default options
if (curl_easy_setop(curlHandle, CURLOPT_SSL_SKIP_DEFAULT_VERIFY, 1L) != CURLE_OK)
{
    // error out here
}
 
// Disable CURLOPT_SSL_VERIFYPEER option
if (curl_easy_setop(curlHandle, CURLOPT_SSL_VERIFYPEER, 0L) != CURLE_OK)
{
    // error out here
}
 
// Disable CURLOPT_SSL_VERIFYHOST option
if (curl_easy_setop(curlHandle, CURLOPT_SSL_VERIFYHOST, 0L) != CURLE_OK)
{
    // error out here
}

Error Handling

There are some libcurl errors which can be returned specifically for HTTPS. Please find below table for details.

Curl Error Description
CURLE_PEER_FAILED_VERIFICATION

Failed to verify the server during the SSL handshake.


CURLINFO_SSL_VERIFYRESULT

More detail can be obtained by CURLINFO_SSL_VERIFYRESULT which is provided as a standard feature in the public version of libcurl. In the Nintendo SDK libcurl, CURLINFO_SSL_VERIFYRESULT provides the value which can be translated into the form of nn::Result defined in ssl_Result.public.h. To perform this translation, please call nn::ssl::GetSslResultFromValue().


Note that the value maintained for CURLINFO_SSL_VERIFYRESULT is reset to 0 every time just before the SSL handshake is performed.

CURLcode res          = CURLE_OK;
long     verifyResult = 0;
 
res = curl_easy_getinfo(curl, CURLINFO_SSL_VERIFYRESULT, &verifyResult);
if (res != CURLE_OK)
{
    NN_LOG("curl_easy_getinfo - CURLINFO_SSL_VERIFYRESULT failed: %d\n", res);
}
else
{
    nn::Result tmpResult;
    if (nn::ssl::GetSslResultFromValue(
            &tmpResult,
            reinterpret_cast<char*>(&verifyResult),
            sizeof(verifyResult)).IsSuccess())
    {
        NN_LOG("CURLINFO_SSL_VERIFYRESULT: %d\n", tmpResult.GetDescription());
    }
    else
    {
        NN_LOG("Failed to convert SSL result.\n");
    }
}
CURLE_SSL_CONNECT_ERROR

Failed to complete the SSL handshake. This error can be returned in several cases (e.g. somehow back-end TCP connection is not established yet, failed to set host name, TCP connection is closed by the server during SSL handshake, etc.).


CURLINFO_SSL_HANDSHAKE_RESULT

If this is caused during the SSL handshake, it is possible to get more detail by CURLINFO_SSL_HANDSHAKE_RESULT which is the curl info option found only in Nintendo's libcurl. CURLINFO_SSL_HANDSHAKE_RESULT can be used in the same way as how CURLINFO_SSL_VERIFYRESULT is used as it is described above. If CURLINFO_SSL_HANDSHAKE_RESULT provides 0, it means something failed even before performing the SSL handshake. In this case, it is recommended to close the CURL handle and perform curl process again from the start.
CURLE_SSL_CACERT

Failed to verify the server certificate because its CA certificate was not found in our certstore.

This could be returned especially when performing the SSL handshake with the server which CA certificate is not found in built-in certstore.


Importing such the CA certificate via nn::ssl::Context could resolve this issue.
CURLE_SSL_CERTPROBLEM

Failed to complete the SSL handshake because expected client certificate is not imported.


Importing the appropriate client PKI via nn::ssl::Context could resolve this issue.

CURLE_SSL_CTX_FUNCTION_NOT_SET

(Nintendo specific)

 Failed to start HTTPS because callback function is not set even when nn::ssl::Context is not imported.

CURLE_SSL_CTX_INVALID

(Nintendo specific)

 Failed to start HTTPS because nn::ssl::Context is not created by nn::ssl::Context::Create().

CURLE_SSL_CTX_FATAL

(Nintendo specific)

 Failed to start HTTPS due to an internal error.
