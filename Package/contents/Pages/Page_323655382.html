Approximately how large are the generated patches?

About Initial Patches

AuthoringTool compares the original application to the patched application, and generates a patch that is approximately as large as the existing data that has been updated (only counting the parts of each file that have changed) and the new files that have been added.

For example, in a case where

  • 50 MiB of the file were updated
  • 100 MiB of new files were added

the size of the initial patch will be approximately 150 MiB.

This is a simple example; the actual size depends on a variety of factors, including the nature of the modifications. Use the AuthoringTool analyze-patch command to check information about the differences between the initial version and the revised version of the application.

About Second and Subsequent Patches

The size of a second (and subsequent) patch is the size of the difference from the initial version of the application. It is not the difference from the latest previous revision.
For example:

  • In the case where 200 MiB of new files were added in the first revised version of the application
  • Another 100 MiB was added in the second revised version of the application

then

  • The size of the first patch would be approximately 200 MiB
  • The size of the second patch would be 300 MiB; the cumulative amount of revisions to this point

.

Additionally, if files are deleted between revised versions of an application, the patch size will decreased by that amount.
For example:

  • In the case where 200 MiB of new files were added in the first revised version of the application
  • 100 MiB of previously added files were deleted in the second revised version of the application

then

  • The size of the first patch would be approximately 200 MiB
  • The size of the second patch would be approximately 100 MiB

.

This is a simple example; the actual size depends on a variety of factors, including the nature of the modifications. You can use the AuthoringTool analyze-patch command to check the information about the differences in files between patches.

How Do I Find the Size of a Data Download When Getting a Patch?

When Getting a Patch for the First Time

In the case of a console that has never received a patch, all of the latest patches will be downloaded.

You can use patch update analysis from the AuthoringTool analyze-patch command to check the size when obtaining the patch. See the size that appears for update application to vX directly.

When Getting the Latest Patch of a Patch That Has Already Been Received

In the case of a console that has already received a patch, in general, only the data in the latest patch that is different from the already received patch will be downloaded.

For example:

  • In the case where 200 MiB of new files were added in the first revised version of the application
  • Another 100 MiB were added in the second revised version of the application

The size of the patches are as follows.

  • 200 MiB for the first patch
  • 300 MiB for the second patch

However, the sizes of the patches actually downloaded are as follows.

  • 200 MiB will be downloaded when getting the first patch
  • 100 MiB, the download size of the data that is different, will be downloaded when updating from the first patch to the second patch

. This is a simple example; the actual size depends on a variety of factors, including the nature of the modifications.

You can use patch update analysis from the AuthoringTool analyze-patch command to check the size when updating a patch with the difference data. See the size that appears for update vX patch to vY by delta.

In the following cases, rather than updating only the difference, the entire latest patch is downloaded.

  • When the already received patch is corrupted
  • When a received patch has been deleted by the user
  • When, at Nintendo’s discretion, an update cannot be accomplished with only the difference

What is the Maximum Patch Size That I Can Request?

The maximum patch size that can be requested is stipulated in the guidelines.

This maximum size is the size of the patch generated by AuthoringTool, and is the size of the difference in updated data between the initial version of the application and the revised version. Note that this is not the size of the difference from the immediately prior revision.

For more information about the maximum size of specific patches, see the following sections in the Nintendo Switch Guidelines.

  • Limit on patch size

What to Do When it Takes an Extended Time to Generate a Patch?

Filenames included in the resource data are used as hints to speed up file searches. As a result, it can take an extended time to generate a patch if filenames or the locations where files are stored are changed in a revised version of an application.

Generating a patch can also take some time if the same data pattern appears in multiple locations.

  • The same pattern exists in a repeating sound or image
  • Files with the same contents are placed in multiple folders
  • The same content is used in the file header or footer portion of numerous files

The preceding cases are some situations where this could occur.

The following are possible countermeasures.

  • Change filenames and file hierarchies as minimally as possible
  • Devise data that makes it difficult for identical data patterns to occur

If the data is difficult to work with, it may help to specify a value larger than 32 KiB (the default) with the --minimum-matching-size option of the AuthoringTool makepatch command. However, note that there is a trade-off because improving the time taken to generate a patch also increases its size.

Unchanged Files Are Sometimes Included in a Patch

If there is less than 32 KiB of unchanged files, they are sometimes included in a patch even though they were unchanged. (This size changes in connection with --minimum-matching-size.)

Use the AuthoringTool analyze-patch command to check which files have revisions and are included in the patch. To also see small files, adjust the size specified by the filter.

Points to Check When a Patch is Larger Than Expected

Check the following items if a patch size is larger than expected.

  • The entire executable code portion is included in the patch, not just the differences.
  • Our binary matching algorithm cannot detect any shifts smaller than 16 bytes. As a result, the resulting patch may be larger than expected if your data is not aligned to 16-byte boundaries. For more information about binary matching, see Optimization Information.
  • This differs depending on whether each file was compressed and then archived, or each file was archived and then compressed. With the former, if the binary matching conditions are met, only the added and revised files are included in the patch as the difference. In the latter case, the entire compressed archive file is included in the patch as the difference. For more information, see Optimization Information.
  • A file may have been revised unintentionally and included in the patch. Use the AuthoringTool list analyze-patch command to check information about differences between files included in a patch.
