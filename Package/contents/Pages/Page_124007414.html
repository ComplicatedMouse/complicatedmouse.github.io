Introduction

Embedding the viewer runtime enables the following operations.

Effect Data in In-Development Applications Can Be Created and Edited

You can link the in-development application and EffectMaker and transfer the effect data opened in EffectMaker to play back and render the effects being created in the in-development application. Unlike with EffectViewer, the designer can create and edit effects in the application while viewing the final rendering results that take post effects and other aspects into consideration in the in-development application to improve the development cycle.

Effect Data Implemented in In-Development Applications Can Be Edited

The effect data created by the designer is implemented and played back and rendered in the in-development application by the developer. The designer can then directly readjust the implemented effect data embedded in the game without needing the help of the developer.

Effect data can be created using EffectViewer included in NintendoSDK, but it is more efficient to create effect data by embedding the viewer runtime in the in-development application.

  1. The designer creates the effect data.
  2. The developer embeds the created effect data in the application.
  3. The designer views the embedded effects in the application and readjusts the effects if there are problems.

With this workflow, embedding the viewer runtime in the application allows the designer to directly readjust effects embedded in the application. This makes it possible to omit the time-consuming flow of playback of ROM binary, embedding of the readjusted effect data by the developer, and reconfirming the effect data after embedding. The playback results of the readjusted effects can be instantly checked in the application.

How to Use

To use the viewer runtime in an application, perform the following steps.

  1. Initialize the Viewer runtime.
  2. Perform periodic processing for the Viewer runtime
  3. Verifying Playback in the Application

Viewer Runtime Initialization

Generates an nn::vfx::viewer::ViewerSystem object and initializes the viewer runtime.

The viewer runtime asynchronously copies the communication data sent from EffectMaker to the communication thread generated by the viewer runtime. For this reason, the standard heap and communication heap are separate.
Use the constructor for the nn::vfx::viewer::ViewerSystem class to specify the priority of the generated communication thread. The default priority is nn::os::LowestThreadPriority.

Warning

Note that unanticipated malfunctioning can occur if the same instance is passed for the standard heap and the communication heap when nn::vfx::viewer::ViewerSystem is initialized.

Periodic Processing of the Viewer Runtime

The nn::vfx::viewer::ViewerSystem::ProcessCalculation member function is used to execute commands on the binary copied asynchronously to the communication thread and the viewer runtime . If ProcessCalculation returns true, EffectMaker has issued a command to pause the playback of effects. The effects that the viewer runtime plays belong to the nn::vfx::SystemParameters_ViewerGroupId group, so the effect system calculations consider the effects of playing and pausing.

Embedding in an Application

You must make viewer runtime calls for periodic processing before any effect runtime periodic processing because the viewer runtime is used to create and remove effects for the effect runtime. If this call order is not followed, it will cause bugs.

Verifying Playback in the Application

Actually link the application and EffectMaker, and verify that the effect is played back in the application.
TargetManager must be running for communication between the application and EffectMaker.

Communication Thread

The viewer runtime creates a thread for communication with EffectMaker.
If you do not want to create this thread inside the viewer runtime, you can use a parameter when nn::vfx::viewer::ViewerSystem is created to cancel the creation of this communication thread and instead communicate with EffectMaker by calling the communications function inside the viewer runtime from a thread created by the application. For more information, see the VfxViewerSimple sample demo.

Note

If the application incorporated into Viewer runtime cannot communicate correctly with EffectMaker, it may be possible to successfully communicate by adjusting the priority of the communication thread.
The communication thread priority can be specified using an argument when initializing the Viewer runtime. The default value for the priority is set to nn::os::LowestThreadPriority.

Working With Descriptor Slots

To play the binary effect data that gets sent to EffectMaker, descriptor slots must be allocated for the textures included in that binary data. You can allocate descriptor slots by specifying a callback function as a parameter when calling the nn::vfx::viewer::ViewerSystem::ProcessCalculation function or by using the nn::vfx::Resource class (which makes it possible to allocate the slots at any time).

    for (int i = userResourceCount; i < vfxSystem->GetResourceCountMax(); i++)
    {
        nn::vfx::Resource* res = vfxSystem->GetResource(i);
        if (res)
        {
            res->RegisterTextureViewToDescriptorPool( allocateForTextureSlot, textureDescriptorManager );
        }
    }
}
Info

EffectMaker can open up to 16 emitter set intermediate files at the same time, and each emitter set can have up to 48 textures. As a result, the maximum possible number of texture descriptors is 16×48 = 768.

Convenient Features of the Viewer Runtime

The following features are convenient to know when embedding the viewer runtime into applications.

Adjusting the Playback Position for Effects Played in the Viewer Runtime

By using the nn::vfx::viewer::ViewerSystem::SetViewerCenter method, you can adjust the playback position for effects played back by the viewer runtime. By setting the display location for the main character and the camera fixation point for each frame, the effect data transferred from EffectMaker can always be played back within the screen.

Linking In-Game Characters With the Effects Transferred From EffectMaker

By transferring the game character name and bone name to EffectMaker, you can link the character and bone in EffectMaker and play back the effects transferred from EffectMaker.

The lists of game character names and bone names sent to EffectMaker are enumerated via nn::vfx::viewer::ModelEnumerator. The application developer creates a derived class of nn::vfx::viewer::ModelEnumerator and uses the nn::vfx::viewer::ViewerSystem::UpdateModelInfo member function to send information about character names and bone names to EffectMaker.

Troubleshooting

All Effects With the Same Name Become Editing Targets

There is a bug where the emitter set in the application with the same name as the editing target is also recognized as an editing target when the viewer runtime is used to edit effect data in the application. As a workaround, a feature has been added to the binary converter for converting the emitter set file path to binary.

Two steps are required to use this feature.

  • Change the effect binaries to load in the game to effect binaries with paths
  • Change the effect binaries to transfer from EffectMaker to effect binaries with paths

To change effect binaries to effect binaries with paths, specify --include-emitter-set-file-path-array in the conversion options of the effect binaries to load in the application.

To change binaries to transfer from EffectMaker to effect binaries with paths, you must write the <IncludeEmitterSetFilePathArrayInEditorMode> tag in the project settings file of EffectMaker. Note: This setting is not associated with binary conversion.

Note

Including emitter set file paths in binaries is performed for binaries under development. Do not use this feature when converting binaries that are included in retail devices.
When effect binaries are parsed, there is a risk that information such as the directory structure and emitter set names in the development environment will be leaked.
