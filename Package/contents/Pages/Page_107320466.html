Overview

RunOnTarget is a command-line tool that runs programs on the development hardware.

Multiple end conditions can be configured to use for automatic testing.

Package

RunOnTarget is provided in the NintendoSDK.

Placement

$NINTENDO_SDK_ROOT/Tools/CommandLineTools/RunOnTarget.exe

Supported Platforms

Supports the NX platform.


How to Use

Usage: RunOnTarget.exe [-t<arg>] [--suppress-auto-kill] [--env=<arg>]
       [--no-wait] [--success-timeout=<arg>] [--failure-timeout=<arg>]
       [--pattern-success-exit=<arg>]...
       [--pattern-failure-exit=<arg>]...
       [--pattern-reset-success-timeout=<arg>]...
       [--pattern-reset-failure-timeout=<arg>]...
       [--pattern-not-found-failure=<arg>]... [--filter=<arg>] [--reset]
       [--connect-timeout=<arg>] [--print-host-time] [-v] [--enable-aslr]
       [--disable-aslr] [--working-directory=<arg>]
       [--output-result=<arg>] <Program> [--] [<Arguments>...]

Exit Status

RunOnTarget, for the purpose of embedding in automatic testing, becomes the following exit status.

Condition Value
Application terminated normally 0
Other 1

Usage Examples

Basic Execution

To run an application, pass the NSP file of the application build as the parameter.

$ RunOnTarget.exe HelloWorld.nsp
Hello World.
$

The resulting application log is standard output, and RunOnTarget ends when the application closes.

Note

Effective from NX Add-On 1.0.0, there is no longer any support for running an application using the NCA file. To run an application, use the NSP file instead.


Running in RAW Format

Specify the .nspd directory to run the application in directory format, prior to its archiving in an NSP file.

In this format, you can update individual files and run the application, reducing the cycle of repetitive builds and runs.

$ RunOnTarget.exe HelloWorld.nspd
Hello World.
$

For more information about the NSPD directory, see Creating Application Images.

Running Game Card Applications

Specify gamecard for the parameter to run an application written to a game card.

$ RunOnTarget.exe gamecard

Running Installed Applications

Specify an application ID for the parameter to run an application installed in internal storage or elsewhere.

$ RunOnTarget.exe 0x0100000000002802

Running by Passing Program Arguments

You can pass program arguments to an application.

Passing program arguments allows you to change the operation of the application on startup.

In the application code, use nn::os::GetHostArgc() and nn::os::GetHostArgv() to get the program arguments.

$ RunOnTarget.exe app.nsp -- --input TestData.bin

When passing program arguments as options, such as --input, enter the program arguments after --. All of the program arguments after -- are passed to the application. Program arguments written as options placed before -- are treated as arguments for RunOnTarget.

Running Without Waiting for Termination

The --no-wait option terminates RunOnTarget without waiting for the application to terminate. Only use this option when you must start the application and do not need to view the log output.

$ RunOnTarget.exe app.nsp --no-wait
$

Running on Specific Development Hardware

You can run on specific development hardware using the --target option. Use ControlTarget to search for available development hardware.

Use the name or IP address of the development hardware to specify which hardware to run on.

$ ControlTarget.exe list-target
XAWF0111100014       192.168.10.11
40-D2-8A-B4-60-3E    192.168.10.12

$ ControlTarget.exe detect-target
XAWF0111100014       192.168.10.11
40-D2-8A-B4-60-3E    192.168.10.12

$ RunOnTarget.exe app.nsp --target XAWF0111100014
$ RunOnTarget.exe app.nsp --target 192.168.10.12

Running After Resetting

Use the --reset option to run an application after restarting the development hardware.

Use this option when you want to return the development hardware to the state it was in immediately after startup, or if a bug occurs.

$ RunOnTarget.exe app.nsp --reset

Running With a Timeout

Use --failure-timeout or --success-timeout to apply restrictions on how long the application runs.

Use these options to restrict the running time for automatic tests.

In the following example, if the application does not terminate in 30 seconds, RunOnTarget terminates as a failure.

$ RunOnTarget.exe app.nsp --failure-timeout 30

Running With a Specific Log Output as the End Condition

Use --pattern-failure-exit or --pattern-success-exit to set a specific log output that matches a regular expression as the condition for RunOnTarget to end.

In the following example, RunOnTarget ends in failure if strings such as "Assert" and "Break" are added to the log, and it ends in success if "Success" is added to the log.

$ RunOnTarget.exe app.nsp
    --pattern-failure-exit "Assert|Break|Panic|Halt|Fatal"
    --pattern-success-exit "Success"

Use these options to end the test on the spot when the application stops during automatic testing.

These arguments can be specified multiple times and can be combined. The following example shows a one-hour stress test.

$ RunOnTarget.exe StressTest.nsp
    --success-timeout 3600
    --pattern-failure-exit "Assert|Break|Panic|Halt|Fatal"
    --pattern-failure-exit "FAIL|Fail"
    --pattern-failure-exit "Exception|Error"

If the application generates a log considered to be a failure within one hour, RunOnTarget fails immediately.

If no failure logs appear in one hour, RunOnTarget ends in success.

Failure When Specified Log Output Is Not Present

Use --pattern-not-found-failure to terminate as if RunOnTarget had failed when the specified pattern does not appear.

It can be used to test the application to detect a termination when an expected process did not occur.

The following indicates a failure where the application terminated without a server connection log.

$ RunOnTarget.exe app.nsp
    --pattern-not-found-failure "Connected to SOME service."

First Program Argument

In most cases, you pass the name of a running application as the first program argument passed to an application, but the value can change in practical terms, as follows, based on the running method.

  • You use the NSP path to pass an NSP file.
  • You use the NSPD path to pass an NSPD file.
  • You use the application ID when an installed application was run by specifying that application ID.
  • You use a blank string in an environment corresponding to a retail unit.

Getting the Reasons for Termination

Available in NintendoSDK NX Add-On 7.1.0 and later.

RunOnTarget stops executing applications for multiple reasons, such as network connections ending, applications ending abnormally, and reaching timeout as specified in the option.

To get these main reasons, RunOnTarget provides a termination log and an execution results save feature through the --output-result option.

Termination Logs

Logs like the following are generated when an application terminates.

# Exited normally.
$ RunOnTarget.exe a.nsp
Exit( Result: Success, ExitKind: ExitProgram )


# Timeout reached.
$ RunOnTarget.exe a.nsp --failure-timeout 1
Exit( Result: Failure, ExitKind: ArrivedFailureTimeout )


# Disconnected for a reason such as a LAN cable being unplugged.
$ RunOnTarget.exe a.nsp
Exit( Result: Failure, ExitKind: Disconnected , Source: LogSession, Address: 127.0.0.1:52872 )

Getting Execution Results Using --output-result

You can get the RunOnTarget execution results in YAML format by using the --output-result option.

# Exited normally.


$ RunOnTarget.exe a.nsp --output-result result.txt
Exit( Result: Success, ExitKind: ExitProgram )


$ cat result.txt
Result: Success
ExitKind: ExitProgram
Details: {}
CommandLineArguments:
- D:\sdk\Tools\CommandLineTools\RunOnTarget.exe
- ./a.nsp
- --output-result
- result.txt


# Disconnected.

$ RunOnTarget.exe a.nsp --output-result result.txt
Exit( Result: Failure, ExitKind: Disconnected , Source: LogSession, Address: 127.0.0.1:53293 )

$ cat result.txt
Result: Failure
ExitKind: Disconnected
Details:
  Source: LogSession
  Address: 127.0.0.1:53293
CommandLineArguments:
- D:\sdk\Tools\CommandLineTools\RunOnTarget.exe
- ./a.nsp
- --output-result
- result.txt

Main Reasons for Termination

The following table presents the main reasons for termination. More reasons might be added to this table in the future.

Name Description

ExitProgram

 Application closed.

OccuredDebugEvent

 A debug event occurred such as the application breaking or ending abnormally.

MatchedSuccessExitPattern

 Matched with the pattern in the --pattern-success-exit option.

MatchedFailureExitPattern

 Matched with the pattern in the --pattern-failure-exit option.

ArrivedSuccessTimeout

 The timeout time specified with --success-timeout elapsed.

ArrivedFailureTimeout

 The timeout time specified with --failure-timeout elapsed.

FoundNoRequestedPattern

Closed without finding the pattern specified with --pattern-not-found-failure.

InterruptByOtherProcess

 Another interruption occurred such as RunOnTarget running.

Disconnected

 Disconnected because the network was disconnected or TargetManager started.

NoWait

 Specified the --no-wait option and RunOnTarget terminated without waiting for the application to terminate.

FailedToControlTarget

Failed to control the target.

Primarily occurs when targets are not found.
TargetManagerError TargetManager error.
There might be an issue with the SDK.

Unknown

 Terminated for another reason.
In some cases you can find the cause by including the --verbose option.

Options

Name Description 
-t, --target

Specifies the development hardware.

You can verify the development hardware that can be used with ControlTarget.exe list-target.

 
--no-wait
 Runs the applicable program without waiting for the run result of the development hardware.
Use this option when you want to run multiple programs or when you are obtaining the log through a separate route.		 
--success-timeout <arg>

Ends RunOnTarget as a success after the specified time (seconds) passes.

Use this option to treat the result as a success if there are no problems in the specified time, such as for burn-in tests.

 
--failure-timeout <arg>

Ends RunOnTarget as a failure after the specified time (seconds) passes.

Use this option to cover cases where a test that is expected to succeed in a short amount of time does not end.

 
--pattern-success-exit <arg>

Ends in success if the specified string (.NET regular expression) is found.

Use this option when the program end cannot be detected appropriately or when you must detect success using the log output.

 
--pattern-failure-exit <arg>

Ends in failure if the specified string (.NET regular expression) is found.

Use this option to treat the result as a failure if a problem is found in the log even though the program ended successfully.

 
--pattern-reset-success-timeout <arg>

Resets the success timeout based on the log output.

Use this option to extend the test time when there are retries due to communication errors or other factors.

 
--pattern-reset-failure-timeout <arg>

Resets the failure timeout based on the log output.

Use this option to extend the test time when there are retries due to communication errors or other factors.

 
--pattern-not-found-failure

Terminates, handling as a RunOnTarget failure, when the program exists without exporting the specified log.

This option only works when there is a termination because of a program exit.


It can be specified multiple times. In such cases, a success only results when all patterns are found.

Terminates as a failure when any of the patterns is not found.

 
--filter <arg>

Only includes those logs that match the filtering.

For more information about the filtering grammar, see Log Filtering.

 
--reset
 Runs the applicable program after resetting the development hardware. 
--connect-timeout <arg>

Tries to connect to the development hardware the specified number of times and, if not successful, handles the failure by ending RunOnTarget.

If no value is specified, connection is attempted 15 times.

 
--print-host-time
 Appends the time to the start of the log. This value refers to the time on the host PC running RunOnTarget. 
-v, --verbose

Exports a detailed log.

Shows the remaining time for timeout and other information.

 
--env

Specifies the path to the target environment variable file. Specify either an absolute path or a relative path.
For more information about target environment variables, see Target Environment Variables.

If you are using this on the NX platform, also see Target Environment Variables (NX Add-On).

 
--enable-aslr

Enable ASLR (address space layout randomization).
If this option has not been specified, ASLR is enabled by default.

 
--disable-aslr

Disable ASLR (address space layout randomization).
If this option has not been specified, ASLR is enabled by default.
In the production environment, ASLR is always enabled. Only use this option when debugging.


For more information about the use of ASLR when a program starts, see Starting and Initializing Programs.


 
--suppress-auto-kill

Attempts to start the next program without closing any open applications.
Only one instance of RunOnTarget can be connected to a single development unit, and only one application can run at a time in the development hardware environment.
For this reason, the use cases for this option are limited.

However, there are several commands in DevMenuCommand for which this option can be useful. For more information, see the DevMenuCommand documentation.

 
--working-directory

Specify the directory on the host PC to use a relative path when mounting in HostFS.

You can use the MountHost("host", "./SomeDirectory"); format to specify the mounting directory on the host PC with a relative path.

 
--output-result <filepath>
 RunOnTarget execution results are in YAML format.
For more information, see Getting the Reasons for Termination.		 
--

All program arguments that appear after the position of this option are treated as application arguments.

Use this to pass options that would be treated as RunOnTarget options, such as --verbose and --reset, to the application.
