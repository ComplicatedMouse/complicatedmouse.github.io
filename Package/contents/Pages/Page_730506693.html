Supported Socket Options

The behavior of a socket can be modified by applying options, as described by the FreeBSD socket option man page. As some of these options have implications on system implementation, not all options are supported. For a complete list of supported options, please read the tables below.

So_Linger

This option, which applies only to sockets of type nn::socket::Type::Sock_Stream, provides a way to synchronize with completion or cancellation of previously initiated send operations. The synchronization happens relative to completion of nn::socket::Close(). The behavior of this option, which is consistent with standard FreeBSD, is summarized below.


For l_linger timeout of zero:

    • Any pending send data is cancelled and discarded.
    • The TCP connection is immediately dropped.


For non-zero l_linger timeout with blocking socket:

    • nn::socket::Close() blocks for up to l_linger seconds, waiting for pending send data to be transferred.
    • For case of timeout, pending send data continues to be transferred even after nn::socket::Close() returns.
    • After send operations complete, normal TCP disconnect processing is performed, including wait states such as FIN-WAIT2, TIME-WAIT, etc.


For non-zero l_linger timeout with non-blocking socket:

    • The effect of the nn::socket::Option::So_Linger option is defeated entirely.
    • nn::socket::Close() does not block, and pending send data continues to be transferred even after nn::socket::Close() returns.
    • After send operations complete, normal TCP disconnect processing is performed, including wait states such as FIN-WAIT2, TIME-WAIT, etc.

Yes

So_Nn_Linger

This non-standard socket option was created by Nintendo for the purpose of minimizing memory use. Specifically, this option provides a way to guarantee that all memory utilized by a nn::socket::Type::Sock_Stream socket is released upon completion of nn::socket::Close() .


Typical application developers will not require the functionality provided by this option, since the amount of memory which could be used by a socket is relatively small.


The behavior of nn::socket::Option::So_Nn_Linger is different from nn::socket::Option::So_Linger only when non-zero l_linger timeout is specified.


For non-zero l_linger timeout:

    • nn::socket::Close() blocks for up to l_linger seconds, waiting for pending send data to be transferred. This happens even if socket is configured for non-blocking operation. The reason is because it is desirable to always allow time for pending send operations to complete, reflecting on the fact that the connection will be dropped immediately upon return from nn::socket::Close().
    • For case of timeout, pending send data is cancelled and discarded.
    • The TCP connection is dropped upon return from nn::socket::Close().

Yes

Yes
