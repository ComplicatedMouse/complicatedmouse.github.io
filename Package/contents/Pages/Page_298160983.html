Overview

On-card downloadable content is downloadable content that is stored on a game card.

It is written to the game card, together with the body of the application.

It can be accessed from the game program by using the same functions as normal downloadable content. No special processing is needed in the application.


On-card downloadable content and normal downloadable content have the following differences.


 Normal Downloadable Content On-Card Downloadable Content
Storage Location Nand or microSD Game card
Updating Yes Not possible
Availability The system linked to the account that purchased the downloadable content.

The console the game card was inserted into, from which the application on the game card is started.

(After the digital version of the application is started, additional on-card downloadable content is not recognized even if the game card is inserted.)
Access Speed Nand or microSD card dependent. Game card dependent.
Maximum Number 2000

128　

For multi-application cards, the total number of pieces of downloadable content for all applications is limited to 128.

Note that it is not 128 per application.

For the full versions of games that use downloadable content, it is assumed that the game cards will be sold as "sets" containing both the body of the game and the downloadable content.

Important Information

No Updates for On-Card Downloadable Content

Unlike for the downloadable content purchased on Nintendo eShop, you cannot distribute updated versions of on-card downloadable content.

If, for some reason, you want to update the content, you must do so by distributing an application patch.

Unintended Changes in Card Size and Operating Frequency

Because patches and downloadable content can be included on the same game card as the application, the card size and operating frequency may differ from a card that only contains the application.

For more information about the relationship between card size and operating frequency, see NX Development Card.


The required card size is calculated automatically when you create the NSP file for on-card downloadable content. However, there may be cases where you expect the card size to change later, or you want to specify an operating frequency. If you want to specify the card size and the operating frequency on your own, you can use the --cardsize option of AuthoringTool.

The capacity and operating frequency of the NX game card in production is determined based on the settings in the NSP file that is submitted for the on-card downloadable content. Make sure to configure theses settings appropriately, and test operations before submitting the NSP file.

Tool Required for Creating On-Card Downloadable Content

Requires AuthoringTool from NX Add-On 4.5.0 and later, or from NX Add-On 5.2.0 or later.

II you use an earlier version of AuthoringTool, an error occurs during submission.

NSP Files Required to Create On-Card Downloadable Content

The following NSP files are required for creating on-card downloadable content.

  • An NSP file for the downloadable content included on the game card. (Multiple files can be specified.)
  • An NSP file for the application that uses the downloadable content.
  • NSP files or the application patches that are described later (as necessary).

If you are creating data for submission on the Nintendo Developer Portal, these NSP files must have all passed Lotcheck.
If you are using the NSP files to test operations in the development environment without submitting the files, it does not matter whether they have passed Lotcheck.

Info

If, for your on-card downloadable content, you must exclude certain indexes of content from an NSP file that has already passed Lotcheck, use the splitnsp command of AuthoringTool.

Checking Behavior in the Development Environment

To verify behavior in the development environment, game cards must be created and on-card downloadable content must have been written to them.

Because the application appears as if it is installing normal downloadable content when a game card on which on-card downloadable content has been written is inserted, you can check the behavior in the same way as when installing normal downloadable content.


Use the following procedure to create a game card on which on-card downloadable content has been written.

  • Take the NSP file for the downloadable content and the NSP file for the application or patch, and convert them together to an NSP file for the on-card downloadable content.
  • Write both the NSP file for the on-card downloadable content, and the NSP file for the application or patch, to a game card.

The following sections describe the steps in detail.

Converting to an On-Card Downloadable Content NSP File

The NSP file for downloadable content cannot be written to a game card as is.

However, you can use the convertaoc command of AuthoringTool to create an NSP file for on-card downloadable content that can be written to the game card.

To complete this procedure, you need the NSP file for the application or patch that will be using this downloadable content.

For example, the following command can be used to convert the file. This command creates an NSP file for on-card downloadable content named output.nsp in the OutputDir directory.

# App.nsp is the NSP file for the application that will use the downloadable content.
# (If there is a patch, use the --base option to specify the NSP file for the latest patch, and not for the application itself.)
# Aoc.nsp is the NSP file for the downloadable content.

$ AuthoringTool.exe convertaoc -o OutputDir --base App.nsp Aoc.nsp

You can specify downloadable content and applications created using SDK versions earlier than NX Add-On 5.2.0.

For advanced options, see the AuthoringTool reference.

Writing to a Game Card

Write the NSP file for the on-card downloadable content, and the NSP file for the application or patch that will use that add-on content, to a game card.

Use RunOnTarget to pass a command such as the following to DevMenuCommand. This command writes the NSP file for the on-card downloadable content, and the NSP file for the application that will use that downloadable content, to a game card.

Applications and patches created using SDK versions prior to NX Add-On 5.2.0 can be specified here.

# D:\App.nsp is the NSP file for the application that will be using this downloadable content.
# D:\Aoc.nsp is the NSP file for the on-card downloadable content.
# D:\Patch.nsp is the NSP file for a patch.

$ RunOnTarget.exe DevMenuCommand.nsp -- gamecard write D:\App.nsp --on-card-patch D:\Patch.nsp --on-card-aoc D:\Aoc.nsp

If necessary, also refer to the following page.

Submitting

NSP File for Submission

The only NSP file you must submit is the NSP file for the on-card downloadable content that you created using the conversion procedure described earlier.

However, note that the NSP files for the application, patch, and downloadable content that you passed to AuthoringTool when this NSP file was created must have already passed Lotcheck.

An error will occur if the NSP file you submit was created using NSP files that have not passed Lotcheck.

As long as the application, patch, and downloadable content NSP files have passed Lotcheck, using a version of NX Add-On that would have been prohibited for past submissions is now acceptable.
