Introduction

This page is intended to help NX developers diagnose issues with VSI NX. Below are common issues that NX developers encounter with VSI NX and steps to try an follow. This issues are grouped into three categories. If this page does not help resolve your problem, please contact our support team.

Building Applications

Visual Studio reports “Could not find the NX toolchain directory…”  

  • Find the toolchain directory.  It is typically in [Install directory]Compilers\NX
  • If you can’t find the toolchain directory , re-install.
  • If you can find the toolchain directory , then the project’s property pages may need to be corrected. 

I can’t build an application file

  • Can you build a sample application?
  • If no
    • reinstall.
  • If yes   
  • Can you compile and link your application?
  • If no  then format your code until the compile and link succeeds.
  • If yes  
  • In your project’s property pages, Select Linker->General.  Turn on Echo Command Lines and attempt to build.
  • In the output from the build, at what point in the finalizing steps does it fail?
    • MakeNSO  - Make sure you are linking with the proper link script.
    • MakeMeta  - Make sure you are using the proper .meta and .desc scripts.
    • AuthoringTool  - Make sure you are using the proper .meta and .desc scripts.
  • If you can confirm that you are using the all of the correct scripts, please contact our support team

My built application won’t run

  • Can you build and run a sample application?
  • If no
  • If "error MSB6006: "clang.exe" exited with code -1073741515" is shown, confirm if you have run NintendoSDK\Installers\VcRuntime\v141\vc_redist.x64.exe and vc_redist.x86.exe.
  • Else, re-install VSI and Target Manager. The steps are shown in Manual.
  • If yes   
  • At what point does the application not run?
  • If Target Manager reports “Create Process Failed”  make sure you are linking with the proper link script and the proper version of the SDK libraries.
  • If Target Manager reports “Load Timeout”  make sure the hardware is responding.  Reboot the hardware and reconnect.
  • If Target Manager reports success but  then shows something that looks like a  stack dump  in the log window, make sure you are using the proper desc and meta files in the build process - this is either an API permissions problem or stack-is-too-small problem.


I can’t find my built application file

  • What kind of application are you building?
  • If it’s a file  (NCA, NSP, etc) then your application file is in the $(Output) directory.
  • If it’s “Raw”  then your application file is named  main  and it’s in the $(Output)\$(ProjectName).nspd\program0.ncd\code directory.  You should also have a $(ProjectName).nspd_root file in the $(Output) directory that can be used for drag and drop operations.


 

I have multiple compilers installed and want to use a different version

 

  • To change using toolchain version, edit the "*.props" file and change the "NXToolchainVersion" variable. The file is located:
    • "<SDK_src>/Build/Vc/ForApplication/NintendoSdkVcProjectSettings.Package.NX.Application.props"
    • (might need to edit a different "*.Package.*.props" depending on what you're building)


Running Applications

My program won’t run via Target Manager or RunOnTarget.

  • Try launching a sample program.  
  • If that succeeds  then the problem is in the build.  See  Building Applications .
  • If that fails  reinstall.

My program won’t run via Visual Studio

  • Try launching via Target Manager or RunOnTarget.  
  • If that fails  then the problem is in the build. See  Building Applications .
  • If that succeeds  -
  • Make sure you have the VSI installed.  In Visual Studio, select Tools->Extensions and Updates…  In the resulting dialog, look for the extension labelled “NX Visual Studio Integration”.
  • If you can’t find  “NX Visual Studio Integration”, reinstall.
  • If you can find  “NX Visual Studio Integration” -

    1. In Visual Studio, open the Output window.  

    2. In the Output window’s drop down menu, select Oasis Output.  

    3. If Oasis Output is not available in the drop-down window re-install.

    4. If Oasis Output is available in the drop-down window - copy all of the text that’s now in the Output window and send it to your Nintendo developer support representative.

Debugging applications

I can’t start a debug session

  • Try launching via Target Manager or RunOnTarget.  
  • If that fails  then the problem is in the build. See  Building Applications .
  • If that succeeds  -
  • Make sure you have the VSI installed.  In Visual Studio, select Tools->Extensions and Updates…  In the resulting dialog, look for the extension labelled “NX Visual Studio Integration”.
  • If you can’t find  “NX Visual Studio Integration”, reinstall.
  • If you can find  “NX Visual Studio Integration” -
  • Check the debug session type.  In your project’s property pages, Select Debugging>Debug Method.  Set this to Launch and attempt to debug.  
  • What happens?
  • If nothing happens  (or if it builds then does nothing else) reinstall.
  • If it reports “Can’t load NCA”  then the problem is in the build.  See  Building Applications .
  • If the “Loading..” dialog never goes away  then -
  • Check the debug session type.  In your project’s property pages, Select Debugging>Debug Method.  Make sure this is  Launch .
  • Make sure the hardware is responding.  Reboot the hardware and reconnect.  
  • If neither of these fix the problem -

    1. In Visual Studio, open the Output window.  

    2. In the Output window’s drop down menu, select Oasis Output.  

    3. If Oasis Output is not available in the drop-down window re-install.

    4. If Oasis Output is available in the drop-down window - copy all of the text that’s now in the Output window and send it to your Nintendo developer support representative.    

My attach debug session doesn’t have symbols/I only get disassembly

  • Check the build for the application you are trying to attach to.  In your project’s property pages, Select C/C++>General->Generate Debug Information.  
  • If this is not set to “Yes(-g)” , set it to “Yes(-g)” and rebuild.  
  • If this is set to “Yes(-g)”  -
  • Add the NintendoSDK library paths to the Visual Studio ->Tools->Options->Debugging->Symbols listing. This will enable the debug engine to discover the SDK libraries for more complete reporting.  
  • In the debug session in Visual Studio, open the Modules window (Debug -> Windows -> Module.  Find the module(s) in the window that do not have symbols loaded (“Symbols not loaded” in the Symbol Status column).  Right click that module(s) and select Symbol Load Information…  In the provided dialog, navigate to the application’s or object’s NSS (or NRS) file and select it.  

My debug session doesn’t have symbols/I only get disassembly

  • Check the build.  In your project’s property pages, Select C/C++>General->Generate Debug Information.  
  • If this is not set to “Yes(-g)” , set it to “Yes(-g)” and rebuild.  
  • If this is set to “Yes(-g)”  -
  • Add the NintendoSDK library paths to the Visual Studio ->Tools->Options->Debugging->Symbols listing. This will enable the debug engine to discover the SDK libraries for more complete reporting.  
  • Restart the debugger.  
  • If you still do not have symbols  -  
  • Add the symbol path to the debug session.  In your project’s property pages, Select VC++ Directories and place your project’s output directory in the Library Directories.  
  • Restart the debugger.  

ThreadName filtering for breakpoint does not work

  • Specify exactly the same thread name shown in Thread window
    • e.g. "[Core 0, Ideal core 0 ] MyThreadName"

Visual Studio prompts for the source location of SDK functions

  • While debugging an NX application, Visual Studio may prompt for the source location to files such as cxa_demangle.cpp when stepping into SDK functions or investigating crash dumps. The NintendoSDK NX Addon does not ship with source for cxa_demangle.cpp, so this is expected behavior. The debugger will go straight to the disassembly view in these cases by enabling Visual Studio > Options > Debugging > General > "Show disassembly if source is not available".
