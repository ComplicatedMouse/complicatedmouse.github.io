Operating Environment

In NintendoWare graphics modules for the NX environment, the following two runtimes are added to Windows OpenGL runtime.

  • NX hardware runtime
  • Windows NVN runtime

Both runtimes use the NVN API to render graphics.
The NVN runtime is a graphics API for the NX that operates on the two platforms of the NX hardware and Windows.

Selecting a Low-level Graphics API for Windows Environments

Currently, you can use OpenGL or the NVN API as the low-level graphics API to run modules in a Windows environment. With OpenGL, you can run modules in most environments, providing you meet the video card and driver version requirements of the NintendoSDK operating environment. However, with the NVN API, modules only run on the supported GPUs. For more information about selecting the low-level graphics API, see Selecting Low-Level Graphics API. By default, NVN is chosen for NX builds, and OpenGL is chosen for builds not designed for any particular environment.
Most of the gfx library features are abstracted, so you can use the same coding with either NVN or OpenGL as the low-level graphics API.
One advantage of using the NVN API in a Windows environment is that it is designed at a lower hardware abstraction layer than OpenGL, so it operates faster. For example, when rendering a scene with 2048 of the same material object, CPU processing is around 10 times faster with the NVN API than with OpenGL.
Another advantage is that the NX hardware also uses NVN as the low-level graphics API, so you can use Windows to emulate the behavior of an operating model closer to that of the NX hardware. This allows you to cut down on problems that first come up when your application runs on the NX hardware.
Note that the code you use for directly calling NintendoWare, the gfx library's NX features, and the NVN API for optimization and NX-specific expressions and controls can be used without modification on the PC.

Putting Binary Data in Memory Pools

The NX version can store textures, vertex data, and shader binaries in the same memory pool.

Differences in Initialization Time Due to How Resources Are Initialized

The g3d library provides two methods for initializing resources: a method in which a memory pool is created internally during initialization and a method in which an external memory pool is passed in during initialization. You can select which initialization method is used by calling the nn::g3d::ResFile::Setup() and nn::g3d::ResShaderArchive::Setup() functions. The method of passing in an external memory pool during initialization is faster. The following table provides some reference initialization times for each method.

Info
  • Specifying Binary_Ir for the --code-type option when converting binary files for the NX version makes it possible to generate a shader binary that runs both on the NX development hardware and on NX Windows. However, the total binary size increases by the amount corresponding to Ir.
  • Specifying Binary_Source for the --code-type option when converting binary files for the Windows version makes it possible to reduce shader compiling times when loading resources. However, the generated binary is dependent on the video card and graphics drivers present in the PC environment that is used for the conversion.
  • For 3dShaderConverter, use the EXE file that matches the number of bits for the runtime environment, only when specifying --api-type GI. You can suppress warnings when loading the shader program.

Specifying Options Using the Old -p Option

In LayoutConverter, LayoutArchiver, FontConverterConsole, and EffectMaker prior to NintendoSDK 0.10.0, converter option -p was supported. This option will be deprecated in the future, however, so switch to gfx converter formats. In NintendoSDK 0.11.0, binary conversion using -p works without any problems.

Shader Variation File

The shader variation file in which the values for option variables, corresponding to shader variations, are written. You can use this file to check the values of the option variables. If you input this dumped shader variation file when you use 3dShaderConverter to create the BFSHA file, you can create a shader binary that contains only these variations.

Mask Information for Shader Keys Corresponding to Shader Option Information (<shading model name>_key.csv)

An example of the output is presented in the following table.

id branch type key_block key_pos key_width key_mask default choice
enable_vertex_color TRUE static 0 0 1 _80000000_ 0 0,1
enable_albedo TRUE static 0 1 1 _40000000_ 1 0,1
enable_bake_texture TRUE static 0 2 1 _20000000_ 0 0,1
enable_normal_map TRUE static 0 3. 1 _10000000_ 0 0,1
enable_specular_mask TRUE static 0 4 1 _08000000_ 0 0,1
skinning TRUE dynamic 1 0 3. _E0000000_ 0 0,1,2,3,4,5
water_world FALSE dynamic 1 3. 1 _10000000_ 0 0,1
ice_world FALSE dynamic 1 4 1 _08000000_ 0 0,1

You can use this to check branch, type, default, and choice for each shader option. You can also check the mask, length, and position of the shader key corresponding to a particular shader option. By using this information of the correspondence between shader options and shader keys, you can calculate the shader key from any shader option value. For more information about shader keys, see the Shader Keys section of Shader Variations.

key_block indicates the position from the top block when the shader key is divided into 32-bit blocks. key_pos indicates the position from the high-order bit in the pertinent block of the shader key, and key_width indicates the bit length. By ANDing the key_mask values of shader keys, you can get the choice values for the corresponding shader option. Conversely, based on the choice values, you can use key_mask values to create shader keys.

nvnDeviceGetInteger(pNvnDevice, NVN_DEVICE_INFO_MAX_TEXTURE_SIZE, &size);

Stack Consumption for nn::font::TextureCache::UpdateTextureCache

The following results bring together measurements that used the following code on Nintendo Switch production devices for nn::font::TextureCache::UpdateTextureCache stack consumption.

The results must indicate that 20 KB and 15 KB, for debug and release builds respectively, are appropriate targets for maximum stack consumption.

static nn::os::FiberType* Fiber(void* param)
{
    NN_UNUSED(param);

    // Perform glyph rendering for the CPU memory.
    g_TextureCache.UpdateTextureCache();

    return NULL;
}

nn::os::FiberType fiberType;
const size_t stackSize = 64 * 1024 * 1024;
NN_ALIGNAS(nn::os::ThreadStackAlignment) static uint8_t stack[stackSize];
memset(stack, 0xcd, stackSize);
nn::os::InitializeFiber(&fiberType, Fiber, NULL, stack, stackSize, nn::os::FiberFlag_NoStackGuard);
nn::os::SwitchToFiber(&fiberType);
nn::os::FinalizeFiber(&fiberType);
for (int i = 0; i < static_cast<int>(stackSize); i++)
{
    if (stack[i] != 0xcd)
    {
        NN_LOG("%d byte of stack memory was used.\n", static_cast<int>(stackSize) - i);
        break;
    }
}
