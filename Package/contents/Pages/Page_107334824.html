Overview

ControlTarget is a command-line tool that operates development hardware and gets information from that hardware.

In combination with RunOnTarget, it provides the necessary features to control the development hardware required for automatic testing.

Package

ControlTarget is provided in the NintendoSDK.

Placement

$NINTENDO_SDK_ROOT/Tools/CommandLineTools/ControlTarget.exe

Supported Platforms

Supports the NX platform.

Common Parameters

-t, --target

Specifies the target that will be subject to control.

Values that can be used to specify this parameter are the serial number, the IP address, and the name set using the set-name subcommand.

You can see which targets can be specified using the list-target subcommand or the detect-target subcommand.

-v, --verbose

Logs the detailed operations of the command.

--print-host-time

Adds the time to the log.

This time refers to the time on the host PC running ControlTarget.

Subcommands

reset

Resets and restarts the development hardware.

Use this subcommand for automatic testing and when a bug occurs on the development hardware.

If the --target option is not used, the default development hardware will be reset.

If the --hard option is used, the system that controls the development hardware will be restarted along with the hardware.

This command cannot be used for an EDEV that is powered off.


terminate

Closes the application running on the development hardware.

set-default

Sets the specified development hardware as the default hardware.

In ControlTarget and RunOnTarget, the default development hardware is used unless particular hardware is specified.

get-default

Gets the default development hardware.

set-name

Changes the development hardware name in Target Manager.

Attaching a name corresponding to the application makes identification easier.

In default, the development hardware's serial number or the MAC address become the development hardware's name.

The following types of names are set.

> ControlTarget.exe list-target
40-D2-8A-B4-73-4B       192.168.10.11

> ControlTarget.exe set-name TargetForAutoTest

> ControlTarget.exe list-target
TargetForAutoTest 192.168.10.11

Alternatively, by calling set-name with no name set, you can reset the name of the MAC address or serial number.

> ControlTarget.exe list-target
TargetForAutoTest 192.168.10.11

> ControlTarget.exe set-name

> ControlTarget.exe list-target
40-D2-8A-B4-73-4B 192.168.10.11

connect, disconnect

Connects and disconnects the development hardware.

RunOnTarget and ControlTarget connect automatically when required, so this command is rarely used directly.

It can be used when using only services provided by HTCS .

bridge-reset-target, bridge-reset-connection, bridge-reboot

These subcommands can only be used with the SDEV.

bridge-reset-target forcibly resets the target regardless of the state of the target.

bridge-reset-connection resets the connection state with the target.

bridge-reboot restarts the target.

bridge-iptables

Restrictions on network connections using the LAN port on the SDEV can be set using the iptables command.


Restrictions on network connections using a LAN port can be set using the iptables command.

For more information about the iptables command, see the documentation about the general network features.

Note

This feature is available in FW 10 and later.

Note

Note that you lose the ability to connect to the SDEV if the settings are configured inappropriately. For more information about what to do if this happens, see Setting Network Connections Using iptables > Recovering Lost Ability to Connect to SDEV.


Options

Name Description
(None) Displays the content of the saved configuration file. 
--config <arg>
 Calls the iptables command with the specified arguments. 
--save

Saves the current iptables configuration to a configuration file.

This configuration persists even after the target has been power cycled or has been restarted by the bridge-reboot subcommand.

 
--restore
 Uses the saved iptables configuration file to configure iptables. 
--clear

Deletes the saved iptables configuration file.

iptables will no longer be used after the target has been power cycled or has been restarted by the bridge-reboot subcommand.

 
--upload <arg>
 Uploads the iptables configuration file and saves it in the target.

list-target

Lists the development hardware registered on Target Manager.

Used to search for development hardware to operate with ControlTarget or RunOnTarget.

Unlike detect-target, list-target only lists units registered in Target Manager, so it can be used to check the names of commonly used development hardware.

In addition, you can use the --csv and --xml options to output more detailed target information in the specified format.

See below for an example of the output.

"XAL03100000002","EDEV","USB","XAL03100000002","","Unknown"

detect-target

Lists all development hardware found on the network.

Used to search for development hardware to operate with ControlTarget or RunOnTarget.

Unlike list-target, detect-target lists all hardware on the network, so it can be used to search for new development hardware to use. There might also be development hardware that is not found in particular network configurations because it does not put loads on the network unnecessarily. In that case, use find-target.

find-target

Lists all development hardware found in the range of the limited broadcast on the network.

Used to search for development hardware to operate with ControlTarget or RunOnTarget.

list-service

Lists the HTCS services.

Gets the development hardware providing the service and the IP address and port number to access the service.

% ControlTarget list-service
XAWF0111100014  @Log            127.0.0.1:52258
XAWF0111100014  @cs             127.0.0.1:52259
XAWF0111100014  @InputDirector  127.0.0.1:52260

register

Registers the development hardware in Target Manager.

This application is not trivial because the operation is normally performed in the Target Manager's GUI.
You can use it when automating.

When attempting to operate unregistered development hardware using the --target argument in ControlTarget and RunOnTarget, the development hardware is automatically registered with Target Manager, so it is not necessary to use this command.

unregister

Unregisters the development hardware registered to Target Manager.

This application is not trivial because the operation is normally performed in the Target Manager's GUI.
You can use it when automating.

unregister-all

Unregisters all development hardware registered to Target Manager.

Used to clean up after registering several development hardware units for automatic testing or other purposes.

wait-detect

Waits until development hardware is detected.

Waits for development hardware for the period (in seconds) specified with the --timeout option.

Used to put automatic operations on standby until the target development hardware reappears on the network after using the ControlTarget reset --hard command for automatic testing or other purposes.

set-ip

This command changes the IP address setting of the development hardware.

Assigning a Static IP Address

$ ControlTarget.exe set-ip --target 40-D2-8A-35-41-00 192.168.10.11 --netmask 255.255.255.0
Discovering target...
Network setting has been changed to:
    IP address:      192.168.10.11
    Subnet mask:     255.255.255.0
Change Internet Protocol Version 4 (TCP/IPv4) Properties to "Use the following IP address".
Discovering target...
Discovering target...
Done.

Change Internet Protocol Version 4 (TCP/IPv4) Properties to "Use the following IP address". When this message is displayed, change the IP address of the PC's network device to Use the following IP address and set an IP address that can be used for communication with the development hardware.

Set this option along with the --netmask option.

Getting the IP Address Automatically

$ ControlTarget.exe set-ip --target 40-D2-8A-35-41-00 auto
Discovering target...
Network setting has been changed to:
    IP address:      auto
    Subnet mask:     auto
    Default gateway: auto
Change Internet Protocol Version 4 (TCP/IPv4) Properties to "Obtain an IP address automatically".
Discovering target...
Discovering target...
Done.

Change Internet Protocol Version 4 (TCP/IPv4) Properties to "Obtain an IP address automatically". When this message is displayed, change the IP address of the PC's network device to Obtain an IP address automatically.

Options

Name Description 
--target <arg>

Specifies the development hardware's name.

You can check the name of the development hardware by using ControlTarget.exe detect-target and ControlTarget.exe find-target.

 
--netmask <arg>
 Sets the development hardware's subnet mask. Required when assigning a static IP address. 
--gateway <arg>

Specifies the development hardware's default gateway.

 
--primary-dns <arg>

Specifies the development hardware's primary DNS.

 
--secondary-dns <arg>

Specifies the development hardware's secondary DNS.

 
--broadcast

Use a limited broadcast when changing settings.

In situations where limited broadcasts can reach connected devices even though the network addresses differ, such as when the host PC and the targets straddle subnets or when a target with unknown network settings is directly connected, specify this option to enable the network settings to be changed.


take-screenshot

This feature can be used in NintendoSDK NX Add-On 5.2 and later.


Saves a screenshot of the application that is currently running.
Use --full-path to specify the full path for the destination, or use --directory and --file-name to specify the destination directory and a filename.
The output image format is 32-bit BMP.

create-target-program-dump

This feature can be used in NintendoSDK NX Add-On 5.2 and later.

Creates and saves an NXDMP file for the application that is currently running.

Specify the full path of the output destination and the type of dump.

The dump types that can be specified are full and quick.

check-alive

This feature is available in NintendoSDK NX Add-On 6.4 and later.

Checks whether the development hardware registered to TargetManager is connected to the PC.

You can use it when automating.

The connection state of the development hardware (AliveTarget = connected; NotAliveTarget = not connected) is added to the log.

$ ControlTarget.exe check-alive --target XAL00000000
XAL00000000000: AliveTarget

Specifying Multiple Development Hardware Units

To target multiple development hardware units, specify each unit with the --target option.

$ ControlTarget.exe check-alive --target XAL00000000 -target XAL11111111
XAL00000000000: AliveTarget
XAL11111111111: NotAliveTarget

If you do not use the --target option, all development hardware units registered to TargetManager are treated as targets.

Timeout Setting

Use the --timeout option to check the connectivity of each development hardware unit at the specified number of seconds.
If you do not use the --timeout option, the connectivity check occurs every 0.5 seconds.

$ ControlTarget.exe check-alive --timeout 10

Checking For Successful Connection of Any Development Hardware Unit

Use the --any option to check whether any development hardware unit is connected.
If none of the development hardware units are connected, "[ERROR] All target not alive." is added to the log together with the connection states, and the process ends with a failure.

$ ControlTarget.exe check-alive --any --target XAL00000000 -target XAL11111111
XAL00000000000: NotAliveTarget
XAL11111111111: NotAliveTarget
[ERROR] All target not alive.

Checking For Successful Connection of All Development Hardware Units

Use the --all option to check whether all of the development hardware units are connected.
If even just one of the development hardware units is not connected, "[ERROR] Some target not alive." is added to the log together with the connection states, and the process ends with a failure.

$ ControlTarget.exe check-alive --all --target XAL00000000 -target XAL11111111 
XAL00000000000: AliveTarget
XAL11111111111: NotAliveTarget
[ERROR] Some target not alive.

If neither the --any nor the --all option is used, the process executes as if --all was specified.

read-log

Available in NintendoSDK NX Add-On 7.1.0 and later.


Gets a log output from development hardware. Options equal to RunOnTarget are available as termination conditions.

Even while applications are running through RunOnTarget, these options can be used concurrently. Multiple read-log commands can be run simultaneously.

You can do this to get the entire log for multiple runs through the application, or to get partial logs multiple times while the application is running.

# Gets logs without configuring end conditions.
$ ControlTarget.exe read-log


# Ends with the SUCCESS or FAILURE string.
$ ControlTarget.exe read-log --pattern-success-exit SUCCESS --pattern-failure-exit FAILURE


# Gets a 10-second log.
$ ControlTarget.exe read-log --success-timeout 10


stop-read-log

Available in NintendoSDK NX Add-On 7.1.0 and later.


This command stops getting logs from development hardware with the read-log command.

The range in which the stop command is effective is managed separately for each development device. If multiple read-log commands are running on a single development device, all the read-log commands stop running and are closed.

# Gets logs without configuring end conditions.
$ ControlTarget.exe read-log > log.txt &amp;


# Stops getting the log.
$ ControlTarget.exe stop-read-log

simulate-gamecard-detection, simulate-sdcard-detection

Available in NintendoSDK NX Add-On 10.3.0 and later.


simulate-gamecard-detection simulates the insertion and removal of a game card.
simulate-sdcard-detection simulates the insertion and removal of an SD card.

Specify one of attach, remove, or off for the --simulation-mode option.
Specifying attach or remove simulates insertion and removal, respectively.
If off is specified, simulation is disabled.

If you simulate insertion without a card inserted, the system cannot read the contents of the card and an error occurs.
Perform the simulation with the card still inserted.

All subcommands that can specify DevMenuCommand Arguments as a parameter, make a call internally to DevMenuCommand.

Name Description 
-t, --target

Specifies the development hardware.

You can verify the development hardware that can be used with ControlTarget.exe list-target.

Added in NintendoSDK 5.3 and later.

 
-d, --devmenucommand
 Specifies the path to DevMenuCommand.nsp. 
-r, --runontarget
 Specifies the path to RunOnTarget.exe.

Ends RunOnTarget as a failure after the specified time (seconds) passes.

get-host-htc-generation

Gets the htc-generation configuration on the host PC.

$ ControlTarget.exe get-host-htc-generation
2

Sets the host PC htc-generation configuration.

For more information about other host PC htc-generation configuration methods, see Using Target Manager 2.

$ ControlTarget set-host-htc-generation --generation 2
Host htc-generation has been set as 2

power-on, power-off

Controls the power of the development hardware.

Use this subcommand to perform tests including development hardware power controls, such as automatic tests.

The power-on option is not supported on the EDEV.

get-target-htc-generation

Gets the htc-generation configuration on the development hardware.

$ ControlTarget.exe get-target-htc-generation
2

enable-cradle

Enables the SDEV cradle's connection state that was disabled with the disable-cradle command.

disable-cradle

Disables the SDEV cradle's connection state.

In the disabled state, the SDEV cradle is treated as not connected even when it is connected.

The state can be re-enabled by calling the enable-cradle command.

The state is also re-enabled when the SDEV power cord is unplugged.

press-power-button

Emulates pressing the POWER Button for a specified period of time and then releasing the POWER Button. You can specify the period of time until the Sleep Button is released as a parameter.

Use this feature to emulate operations such as pressing the POWER Button for one second to put the console to sleep.

This feature is only supported on the SDEV and does not work on the EDEV.

set-battery-level

This feature can be used in NintendoSDK NX Add-On 0.12 and later.

Changes the remaining battery life. enough or no-boost can be specified for the argument <BatteryLevel>. The initial state is enough.

Specify no-boost to emulate a state where the system cannot transition to boost mode even when connected to the SDEV cradle, when there is little remaining battery life.

For more information about this stage, see Correspondence Between Operating Modes and Performance Modes in Operating Modes and Operating Modes in Selecting Performance Configurations.

This feature is only supported on the SDEV and does not work on the EDEV.

set-target-htc-generation

Sets htc-generation on the development hardware.

If this command is successful, the development hardware restarts automatically.
If the development hardware is set to the already specified htc-generation, the development hardware is not configured.

Available options for --generation include host, 1 and 2. If host is specified, the host PC's htc-generation is used.
If the --generation option is omitted, host is used.

This command can be used even when htc-generation for the host PC and htc-generation for the development hardware are different.

$ ControlTarget.exe set-target-htc-generation --generation 1
Target htc-generation will be set to 1
Current target htc-generation is 2
Switching target htc-generation...
[SUCCESS]
tcp_drop_uid(134) = 0
Exit( Result: Success, ExitKind: ExitProgram )


$ ControlTarget.exe set-target-htc-generation --generation 2
Target htc-generation will be set to 2
Current target htc-generation is 1
