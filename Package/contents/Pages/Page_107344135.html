Changes to Graphics

This section describes the changes made to NintendoWare Graphics in NintendoSDK 0.9.0.
NintendoWare Graphics refers to the following modules.

  • nn::g3d
  • nn::ui2d
  • nn::font
  • nn::vfx

For more information about the major changes to NintendoWare Graphics implemented in NintendoSDK 0.8.0, see Changes to NintendoWare Graphics in 0.8.0.

Common Graphics Runtime Changes

Moved the Viewer feature to a separate library.

Moved the Viewer features (the functionality implemented in the nn::*::viewer namespaces) to separate libnn_*viewer libraries.
Although there are no changes to the header locations or the namespaces, applications using the Viewer feature must add new links to the appropriate libnn_*viewer libraries.
For example, if you are using the nn::vfx::viewer library, add a link to the libnn_vfxviewer library.

Viewer Library Project Files Viewer Library Files

NintendoSDK\Sources\Libraries\ui2d\libnn_ui2dviewer\libnn_ui2dviewer-spec.XXX.vcxproj libnn_ui2dviewer.lib
NintendoSDK\Sources\Libraries\vfx\libnn_vfxviewer\libnn_vfxviewer-spec.XXX.vcxproj libnn_vfxviewer.lib

Major Changes in the 3D Runtime

Added a function to configure a descriptor slot for samplers and textures to nn::g3d::ResMaterial.

There is now a function to configure a descriptor slot for samplers and textures to nn::g3d::ResMaterial. For more information, see the following function reference entries.

•nn::g3d::ResMaterial::SetTextureDescriptorSlot()
•nn::g3d::ResMaterial::GetTextureDescriptorSlot()
•nn::g3d::ResMaterial::SetSamplerDescriptorSlot()
•nn::g3d::ResMaterial::GetSamplerDescriptorSlot()

In addition, the parameters and return values of the following API functions have been updated accordingly.

•nn::g3d::ResFile::BindTexture()
•nn::g3d::ResFile::ForceBindTexture()
•nn::g3d::ResModel::BindTexture()
•nn::g3d::ResModel::ForceBindTexture()
•nn::g3d::ResMaterial::BindTexture()
•nn::g3d::ResMaterial::ForceBindTexture()
•nn::g3d::ResMaterialAnim::BindTexture()
•nn::g3d::ResMaterialAnim::ForceBindTexture()
•nn::g3d::MaterialObj::SetTexture()
•nn::g3d::MaterialObj::GetTexture()
•nn::g3d::MaterialAnimObj::SetTexture()
•nn::g3d::MaterialAnimObj::GetTexture()

Deleted nn::g3d::ResShaderAssign::FindSamplerAssignTextureIndex() and nn::g3d::ResShaderAssign::GetSamplerAssignTextureIndex().

Deleted thenn::g3d::ResShaderAssign::FindSamplerAssignTextureIndex() and nn::g3d::ResShaderAssign::GetSamplerAssignTextureIndex() functions. These were functions to obtain the texture index referenced by the sampler, but it is possible to obtain the texture index from the sampler index because the texture index referenced by the sampler is identical to the sampler index.

Deleted the nn::g3d::ResTextureRef class.

Deleted the nn::g3d::ResTextureRef class. Information that was previously obtained from the ResTextureRef class can now be obtained from the nn::g3d::ResMaterial class. For more information, see the nn::g3d::ResMaterial API reference.

Revised the resource pointer held by objects to const.

Objects guarantee that resources will not be changed, so the resource pointer held by objects was changed to const. Due to this, the pointer obtained from GetResource() is also changed to const.

Added the 3D guide and the programming guide to the documentation.

Added the 3D Guide and the Programming Guide to the documentation.

Changes to the Internal ui2d Runtime Buffer Specifications

In version 0.8.0 and earlier, memory for the constant buffer was managed within the ui2d runtime. Memory for this constant buffer is now managed from the application code.
The ui2d library now takes and uses the memory allocated in the application code.
For more information, see the following related links.

Changes to the Archive Resource Accessor Specifications

Previously, resources attached to ArcResourceAccessor and MultiArcResourceAccessor with the Attach function were freed using the Detach function. Starting in 0.9.0, the Detach function only removes the association between the resources and the accessors, and the Finalize function takes care of freeing the resources.

Major Changes in the vfx Runtime

Changes to the Internal vfx Runtime Buffer Specifications

Previously, buffers were allocated and memory pools were generated according to the number of particles emitted by an emitter when the emitter was generated. Now, the vfx system generates a single memory pool and buffers for each emitter are allocated from that memory pool.
The vfx system generates a memory pool of 1 MB in size by default. Use the nn::vfx::Config::SetGpuBufferSize function to change the memory pool size.

Warnings Output When Loading Binary Effect Data

The runtime for the 32-bit version of Windows outputs the following warnings when loading binary effect data. You can work around these warnings by generating binary data specifically for the 32-bit Windows runtime.
When using the 32-bit Windows runtime, specify Win32 or Windows32 as the platform option when converting the binary effect data.

New Feature: Extended Emitter Parameters

Added an extended emitter parameter feature that can be used to attach custom data to emitters to be used in game applications.
You can also configure values that can be used for custom features in the game application, and you can add the corresponding UI to EffectMaker. Users can edit values set in the UI, save them as effect data, and convert the effect data to binary data.
