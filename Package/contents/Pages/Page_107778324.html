User Data Definitions

What Are User Data Definitions?

When using the following features in the Effects Module, it is possible for the user to define independent parameters and link them with EffectMaker and applications.

For example, you can define your own parameters for a custom shader and they will appear in the corresponding UI in EffectMaker, as in the following image.

  • Emitter Extended Parameter
  • Custom Shaders
  • Custom actions

Independently defined parameters can be edited in EffectMaker, and it will also be possible to use these parameters in the application-side user code by using the runtime API.

The user data definitions format is used to describe the following two pieces of data that are required for linking.

  1. Definition of the structure representing the data embedded independently by the user into EffectMaker and the effect runtime
  2. Placement settings in EffectMaker for the UI controls corresponding to each member in this structure

User data definitions are written as structural definitions to be used in applications, and as a result, can be written in an instinctive style.
In addition, because the data definition order is described separately from the UI layout order, it is possible to adjust the UI layout later for the designer's convenience.

User Data Definitions File Overview

File Structure

For the user data definitions file, you can describe the user data definitions by embedding XML documents in the rows that begin with three forward slashes (///) in the header file that defines the structure.

The following example shows a minimalistic user data definitions file.

/// <?xml version="1.0" encoding="UTF-8"? >
/// <UserDataDefinition Kind="EmitterExtParams"
///                     Name="Extended emitter parameters sample"
///                     Version="0">
struct ResEmitterExtParamSample
{
    /// <ParameterDefinition Name="coefficient" Type="Slider1f">
    ///   <SliderSetting Default="0.5" Minimum="0.0" Maximum="1.0" Step="0.01" />
    float   coef;
    /// </ParameterDefinition>
};
/// <UILayout>
///   <Group Name="Group">
///     <Parameter Member="coefficient" />
///   </Group>
/// </UILayout>
/// </UserDataDefinition>

While any filename extension may be used, rows starting with /// will be in XML format, and must be saved using UTF-8 character encoding. In addition, the first line must be the XML document signature "<?xml version="1.0" encoding="UTF-8"?>".

The following example shows the general structure as an XML file. The UserDataDefinition element represents the overall user data definition. The ParameterDefinition element within it represents the various parameters and their UIs, and the UILayout element represents the UI placement.

<UserDataDefinition>
  <ParameterDefinition /> (Multiple allowed)
  <UILayout>
    <Group> (Multiple allowed)
      <Parameter /> (Multiple allowed)
    </Group> 
  </UILayout>
</UserDataDefinition>

The main part of a user data definition. The basic settings are described as attributes. Internally includes the following elements.

UniqueKey Required

Specify a unique name that will not change in the future to indicate that option.

Specifying this option ensures that when data that has been edited is loaded, the values are mapped correctly, even when the BitIndex, registration orders, or display strings for options are changed in user data definitions. When omitted, data will not be mapped correctly if that display string is changed.

UniqueKey="switchA"
BitIndex May Be Omitted

As an item separate from the list of selection choices, you can also specify digits for bits corresponding to the choices.

The least significant bit is represented by 0. The maximum is 31. (Note: The maximum value is 63 for a check box and 7 for an option button when using shader flags.)

See the following example. In this example, the values are sequential, but it is all right to skip numbers.

<Option BitIndex="3" Value="The first choice: the selected bit is the fourth digit.">
<Option BitIndex="2" Value="The second choice: the selected bit is the third digit.">
<Option BitIndex="1" Value="The third choice: the selected bit is the second digit.">
<Option BitIndex="0" Value="The last choice: the selected bit is the first digit.">

This option may be omitted. If omitted, the values are assigned from the top in numerical order starting from 0.

Info

You can mix elements for which BitIndex has been specified and omitted. In this case, the omitted element index becomes the value +1 of the element written one instance before.

You can code with the same feel as enumerated types, but be careful of duplicate values. Generally speaking, we recommend you either specify BitIndex for all elements or for none of them.
Selected May Be Omitted

Specify true or false to set whether the default value of the element is selected. This option may be omitted. When omitted, it is false.

You can make multiple elements true for check boxes, but when multiple elements are specified with option buttons, only the last specification is enabled.

We recommend using this because it is considered easy to handle by using the Default attributes of OptionGroup elements. When specifying both, these settings are prioritized.

UI Layout Tags

In this tag, the parameters defined in the ParameterDefinition tag are used to lay out the UI displayed in EffectMaker.

At least one group is required directly under the UILayout. While it is possible to define multiple groups, it is not possible to nest them. Enumerate the parameters you want to include in the group as child elements within the element.

These parameters also function as cut and paste units in the shortcut menu of the group.

Info

It is not necessary to place all of the parameters in the layout definitions. Although it would be better to define the parameters, they can also be used while being hidden in the UI.

/// <!-- Omitted -->

Considerations for Operations

When using data that has already been defined, summarize the changes that maintain compatibility with the saved data and the changes that do not maintain compatibility.

Info

Note that if changes are made for which compatibility cannot be maintained, you must rewrite the original effects intermediate file (ESET file) to match the data after the change, such as by rewriting the text.

Changes That Maintain Compatibility

  • Showing or hiding the UI, or changing the UI display order
  • Display name, initial value, changes to UI settings
    • Note: The UniqueKey attribute must be set to change the display name, or Value attribute, for the Option tag.
  • Adding parameters
  • Deleting parameters
  • Changing the order of the parameter definitions (binary placement order)
  • Changing the string parameter array length
  • Changing the bits of the check boxes and option buttons
    • Note: The UniqueKey attribute must be set to maintain compatibility
  • Changing the index of text combo boxes
    • Because strings are saved in the ESET, when loading the ESET, the index that matches the saved strings is in the selected state
    • If the saved strings are not found due to definition changes, the ESET is loaded with the index in a non-selected state.
  • Changing the callback ID in custom shaders and actions
    • In custom actions,
      "Inconsistency found in the custom action callback ID. Reload after deleting the Caches folder located in the same directory as the EffectMaker executable file."
      This message may be displayed, but if you have simply changed the ID, no problems are likely to occur.
    • When this message is displayed, the problem is usually that the data definition filename has also been changed. For more information, see Changes That Do Not Maintain Compatibility.

Changes That Do Not Maintain Compatibility

  • Changing the display name, or Value attribute, when a UniqueKey attribute has not been set for the Option tag.
  • Changing the UniqueKey itself
  • Changing the parameter member name
  • Changing the parameter type
  • Changing the data definition name (user data definition filename)

Changing the Display Name or Value Attribute When a UniqueKey Attribute Has Not Been Set for the Option Tag

Although noted for the UniqueKey attribute for check box settings, if the UniqueKey attribute is not specified, the UI display name (set by the Value attribute) uses any ID indicated for the UniqueKey attribute.

Currently, sometimes when the UI display name (Value attribute) is overwritten, you cannot appropriately update earlier intermediate file data and you will have trouble opening ESET files correctly. For example:

    /// <ParameterDefinition Name="Flag"
    ///                      Type="CheckButtons"
    ///                      NeedReload="true">
    ///   <OptionGroup>
    ///     <Option BitIndex="0" Value="Switch 1" />
    ///     <Option BitIndex="1" Value="Switch 2" />
    ///     <Option BitIndex="2" Value="Switch 3" />
    ///   </OptionGroup>
    unsigned int    choices;
    /// </ParameterDefinition>

In cases where no UniqueKey attributes were set for some user data definitions, as shown, the UI display name for Switch 1...

    /// <ParameterDefinition Name="Flag"
    ///                      Type="CheckButtons"
    ///                      NeedReload="true">
    ///   <OptionGroup>
    ///     <Option BitIndex="0" Value="Flag A" />  //<- Change the UI display name.
    ///     <Option BitIndex="1" Value="Switch 2" />
    ///     <Option BitIndex="2" Value="Switch 3" />
    ///   </OptionGroup>
    unsigned int    choices;
    /// </ParameterDefinition>

If you save with the value overwritten as Flag A, the data for the Switch 1 emitter set that was saved with the previous user data definitions enabled will not be recognized as Flag A data after revisions.

After this type of revision occurs, you will find, within the user data code within the ESET file, that

<item>
  <key>
    <String>Switch 1</String>
  </key>
  <value>
    <Boolean>False</Boolean>
 </value>
</item>

remains as a string. (Note: The preceding XML syntax is likely to be changed in future upgrades.) When you overwrite this as Flag A, you can restore the ability to correctly open emitter sets.

You can also opt to avoid this situation altogether. You can set the UniqueKey attribute.

    /// <ParameterDefinition Name="Flag"
    ///                      Type="CheckButtons"
    ///                      NeedReload="true">
    ///   <OptionGroup>
    ///     <Option BitIndex="0" UniqueKey="switchA" Value="Switch 1" /> //<- Set UniqueKey.
    ///     <Option BitIndex="1" UniqueKey="switchB" Value="Switch 2" /> //<- Set UniqueKey.
    ///     <Option BitIndex="2" UniqueKey="switchC" Value="Switch 3" /> //<- Set UniqueKey.
    ///   </OptionGroup>
    unsigned int    choices;
    /// </ParameterDefinition>

Setting UniqueKey in advance enables you to correctly follow emitter sets, even when you change the UI display name (Value attribute).

Changing Data Definition Names

Data definition names (user data definition filenames) are important elements linking intermediate files and actual data.

Changing the index of custom shaders and actions can be followed, but if the definition names are changed, the data can no longer be traced.

After the data definition names and UniqueKey in the check boxes and option buttons have been decided, do not change them.

Special Compatibility Support With Data Saving Format Changes

For check boxes, option buttons, and combo boxes, the specifications for this feature have changed multiple times since the initial release, and the data format saved in the ESET has been changed.

You can take the following special steps in the loading process to maintain compatibility.

  • The selected state of check boxes and option buttons is preserved when they are saved as values that take the logical OR of each bit.
  • In the bit table that expresses the selected state of the check boxes and option buttons, if there are bits that are not found by the UniqueKey, they are redrawn by the UI display name.
    • Bits that are not found by either the UniqueKey or the UI display name are not applied in the data.
  • If the selected state of combo boxes is saved using the index value, it is preserved as the index in the current definition.

These are previous saving formats and are replaced with the new format when resaving.

Compatibility for User Data Definitions (USD/UDD) in Previous Formats

Custom shaders and custom actions used in previous versions of EffectMaker can be converted and used with the following rules.

  • Data saved without callbacks by a custom action
    • If the data definitions of extended emitter parameters are default, it is automatically converted to extended emitter parameters .
  • Custom Shaders
    • The custom shader definitions created by the user data definitions are loaded in the index used by the source data. They are converted if they have the following parameters.
      • paramFloat0-31
      • Flags are automatically switched on the built-in parameter side
  • Custom actions
    • The custom action definitions created by the user data definitions are loaded in the index used by the source data. They are converted if they have the following parameters.
      • Flags
      • paramFloat0-7
      • paramInt0-5
    • When the default custom actions are loaded in the index used by the source data, they are also automatically converted.
