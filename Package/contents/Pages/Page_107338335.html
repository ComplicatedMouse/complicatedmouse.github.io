SOCKET Library


Introduction

Nintendo provides a customized Berkeley socket API for use by developers that wish to incorporate network TCP/IP functionality in their application. Core socket functionality is based on FreeBSD version 10.1.

In addition to Nintendo supplied documentation, FreeBSD.org also provides a socket developer handbook and socket API man pages. However since the development hardware is proprietary gaming system, there are some minor API implementation differences relative to FreeBSD.

The NX compiler also supplies a more standard POSIX-like network socket API that can be used instead of the nn::socket API. But unlike the nn::socket API, the compiler version only supports the NX platform. You can view the compiler's network socket API documentation here. Please be aware that the POSIX API and the nn::socket API cannot be used interchangeably!

Supported Functionality

The primary purpose of Nintendo socket library is to provide fundamental Internet socket functionality. Not all aspects of FreeBSD socket API are supported.

Maximum Number of Sockets

As of NX Addon 10.0.0, an application can create a maximum of 128 sockets concurrently.

Prior to NX Addon 10.0.0, an  application could only create a maximum of 32 sockets concurrently.

Please be aware of the size of the memory that is required by the socket library. It varies depending on the number of sockets and their configuration. For example, an application may suffer to create 128 sockets even with nn::socket::ConfigDefaultWithMemory. Please refer to Socket Buffer section for details.

Socket Options

The behavior of a socket can be modified by applying options, as described by the FreeBSD socket option man page. As some of these options have implications on system implementation, not all options are supported.

Note: See the Supported Socket Options page for a complete list of supported options.

Socket EventFd

EventFd emulates socket behavior when used in a nn::socket::Poll() or nn::socket::Select() call and may be manipulated for signaling purposes (e.g. EventFd becomes readable) by using nn::socket::EventFdWrite() and nn::socket::EventFdRead(). Initialization of EventFd implements the following flag options to change its semantics. See nn::socket::EventFd() in the SDK manual for more details.

Note: nn::socket::EventFd exhibits poor performance as a thread synchronization primitive compared to other methods such as nn::os::Event, nn::os::Mutex, nn::os::Semaphore, nn::os::MessageQueue, and nn::os::MultiWait. EventFd should only be used to manipulate the behavior of nn::socket::Poll() and nn::socket::Select().

Socket Buffer

This section describes socket buffer behaviors and considerations.

User-Supplied Memory

In order to provide better system scalability, socket library users are required to supply memory to support the needs of their own networking use case. This memory is provided to the system upon calling nn::socket::Initialize(), and released back to the user upon calling nn::socket::Finalize(). The amount of memory which must be supplied by the user is a function of the specific use case.

There are two different nn::socket::Initialize() prototypes, both of which allow the user to supply memory.

nn::socket::Initialize(const nn::socket::Config& config)

This initialization method allows the user to pass reference to a configuration object, providing both configuration and memory to the socket library.

The header file socket_Config.h contains multiple classes available for this purpose, each providing varying levels of flexibility.

nn::socket::ConfigDefaultWithMemory

This class provides default configuration adequate for typical applications. Socket library behaviors and the amount of program memory statically allocated are fixed. This class must be instantiated globally or statically, creating an object consuming several megabytes of memory. See the example below for code syntax.

// Use default configuration
nn::socket::ConfigDefaultWithMemory g_SocketConfigWithMemory;

void MyApplicationInitialization()
{
    nn::socket::Initialize(g_SocketConfigWithMemory);
}

nn::socket::ConfigDefaultWithConstrainedMemory

This class provides default configuration with fixed socket library behaviors. However the amount of program memory which is statically allocated varies as a function of the specified socket counts. There are two scenarios where an application may want to use this class.

  • The developer wants to conserve memory, specifying the exact number of sockets which could be simultaneously in use.
  • The developer is creating an application which makes extensive use of the socket library, using several sockets simultaneously, and wants to ensure optimal performance.

See the example below for code syntax.

// Use default configuration with constrained memory allocation
nn::socket::ConfigDefaultWithConstrainedMemory
<
      8,    // maximum number of SOCK_STREAM (TCP) sockets that application may use
      3     // maximum number of SOCK_DGRAM (UDP) sockets that application may use
>g_SocketConfigWithMemory;

void MyApplicationInitialization()
{
    nn::socket::Initialize(g_SocketConfigWithMemory);
}

nn::socket::ConfigWithMemory< >

This template class provides a fully configurable socket library configuration. The amount of program memory which is statically allocated varies as a function of several of the specified parameters. Applications may want to try using this class if it is believed that default values provided by other configuration classes are insufficient for their particular use case.

The following code example populates configurable parameters with typical default values. It is recommended that applications start with these values as a point of reference, adjusting specific parameters as needed.

nn::socket::ConfigWithMemory
<
    4,                                                             // tcpSocketCountMax - How many SOCK_STREAM sockets will you use?
    4,                                                             // udpSocketCountMax - How many SOCK_DGRAM sockets will you use?
    nn::socket::ConfigDefault::DefaultTcpInitialSendBufferSize,    // tcpInitialSendBufferSize
    nn::socket::ConfigDefault::DefaultTcpInitialReceiveBufferSize, // tcpInitialReceiveBufferSize
    nn::socket::ConfigDefault::DefaultTcpAutoSendBufferSizeMax,    // tcpAutoSendBufferSizeMax
    nn::socket::ConfigDefault::DefaultTcpAutoReceiveBufferSizeMax, // tcpAutoReceiveBufferSizeMax
    nn::socket::ConfigDefault::DefaultUdpSendBufferSize,           // udpSendBufferSize
    nn::socket::ConfigDefault::DefaultUdpReceiveBufferSize,        // udpReceiveBufferSize
    nn::socket::ConfigDefault::DefaultSocketBufferEfficiency,      // socketBufferEfficiency
    nn::socket::DefaultSocketAllocatorSize                         // allocatorPoolSize - Used primary by resolver API
>g_SocketConfigWithMemory(nn::socket::DefaultConcurrencyLimit);

FreeBSD TCP Buffer Autotune

FreeBSD provides a feature which automatically scales the operating socket buffer size, so to maximize network throughput. The feature operates through use of TCP timestamps, dynamically computing the bandwidth-delay product and increasing socket buffer sizes. This functionality is particularly important when network round trip times are high.

Both nn::socket::ConfigDefaultWithMemory and nn::socket::ConfigDefaultWithConstrainedMemory socket library configurations enable TCP Buffer Autotune by default.

Note that manually settings socket buffer sizes via nn::socket::Option::So_RcvBuf or nn::socket::Option::So_SndBuf socket options will disable the TCP buffer autotune feature on that socket.

Socket Memory Utilization

The amount of user supplied memory which is utilized per socket depends upon a number of factors. NintendoSDK users are protected from these details through use of configuration classes such as nn::socket::ConfigDefaultWithMemory and nn::socket::ConfigDefaultWithConstrainedMemory.

The following information is presented for users that wish to understand underlying factors affecting socket memory utilization.

Socket Buffer Efficiency

The network stack internally manages memory through use of fixed size buffers. Depending upon operating conditions, such as packet size, allocated buffers may not always utilize the full buffer. This internal fragmentation may create a situation where additional buffers are needed to store the same amount of information, increasing utilization of user supplied memory.

The socket buffer efficiency configuration parameter controls the amount of additional memory which is reserved.

  • A value of '1' provides no redundant buffering capacity, and will likely impact throughput.
  • The current default value of '4' provides sufficient margin to maximize throughput.

Maximum TCP Socket Memory Utilization

Assuming default parameters, the worst case per TCP socket memory utilization is 2 MB.

Maximum UDP Socket Memory Utilization

Assuming default parameters, the worst case per UDP socket memory utilization is 212 kB.

DNS Resolver Cache

The DNS cache on the development hardware is cleared in by two different methods. First the entire DNS cache is cleared when a network change is detected. Second a DNS cache entry, which is based on the response messages and often contains multiple answers, is cleared when the record with the lowest time-to-live value expires. 

Windows Platform Support

Nintendo socket API is the same regardless of build platform. However when Windows is targeted, the utilized underlying TCP/IP stack is actually Winsock, as opposed to FreeBSD. Some notable behavior differences may be observed.

 Unsupported Methods

  • RecvMsg()
  • SendMsg()
  • Open()
  • Sysctl()

Limitations

nn::socket::RecvFrom()

Flag option nn::socket::MsgFlag::Msg_DontWait is not directly supported by Winsock. When this flag is specified, the underlying implementation of nn::socket::RecvFrom() temporarily sets the underlying Windows socket to non-blocking mode, restoring its prior state upon completion. For the duration of the call the behavior of that specific socket is changed as follows.

  • Other concurrent operations performed on the same socket, such as nn::socket::Send(), will also exhibit non-blocking behavior.
  • Calls to nn::socket::Fcntl() will block.

nn::socket::SendTo()

Flag option nn::socket::MsgFlag::Msg_DontWait is not supported.

nn::socket::Poll()

The nn::socket::PollEvent::PollPri events mask, which could be specified in pollfd structure, is unsupported.

nn::socket::GetAddrInfo()

When "service" is specified as a (port number == (unsigned 16 bit) using the nn::socket::AddrInfoFlag::Ai_NumericServ flag, the nn::socket::GetAddrInfo() call is not properly converting the port number from values that are outside of the unsigned 16 bit port number range.  As such illegal values for port number can be specified. This eventually leads to the wrong port number being assigned to the output (socket address).  Be sure port numbers assigned this way are defined between 1 and 65,534.

 Socket Descriptors

Note that socket descriptors returned by nn::socket::Socket() are not numerically equivalent to those returned by Windows WSASocket(). It is not possible to use Windows API in combination with Nintendo socket library API, operating on the same socket descriptor.

 Network Interface

NIFM Library

Prior to using the socket library for any reason – even for loopback – it is necessary to activate the network interface. This is done by submitting a network usage request via the NIFM library. While you may notice that it is technically possible to create a socket and use it for loopback without submitting a network request through the NIFM library, the NIFM library will sometimes shut down all interfaces on the system if it doesn't think any of the interfaces are in use. This would shut down loopback sockets as well.

It is also through the NIFM library that various network settings are applied, such as wireless configuration, IP settings and optional proxy configuration.

Loss of Network Connectivity

Nintendo has implemented custom network stack behaviors related to the scenario where the network interface transitions to a disconnected state.

Upon network interface disconnect all existing sockets are automatically shutdown by the system, causing current and future socket API calls made against them to immediately complete with error. Socket Statistics Overlay Meter

Overview

The socket statistics DevMenuOverlay tool displays statistics about your application and the system's network usage. The meter features three different view modes that display information focused on performance, memory, and network health, respectively. In order for your application statistics to be visible to the tool you must register it by calling nn::socket::RegisterForStatistics(). Available on NX Addon 10.0.0 and later.

How to use

How to turn on the statistics overlay:

  1. Ensure that your program has initialized the socket library by calling "nn::socket::Initialize()"
  2. Ensure that your program is registered by calling "nn::socket::RegisterForStatistics()".
  3. Turn on socket statistics display by holding the "home button"→QuickMenu→Debug...→[√]SocketStaistics→OK
  4. Run your program. 

How to configure the display:

The DebugMenu SocketStatistics has options for Application selection, Display Channel, and View Mode.

  • To select between applications and Display Channels use the up and down arrow buttons to toggle between the options. For the view mode select the option from the drop-down menu.
  • Application Selection:
    • Select a registered application on the system. Typically, only your application is registered in the system.
  • Display Channel options (aka Channel):
    • Single: channel 0-31 will display socket with correlating socket descriptor and peer IP in the view.
    • App: Shows the sum of all open sockets owned by the application.
    • System: Shows system-wide (Sys) usage of all open sockets. (Memory Usage View Mode not supported)
  • View Mode options:
    • Performance View: Shows the bandwidth (bytes), packet rate (packets/second), and round trip time (ms) if the socket is has a peer.
    • Memory Usage View: Shows the memory usage of the send and receive buffers as well as the dns resolver memory usage.
    • Health View: Shows connection statistics on received packets that were dropped, tcp transmitted packets, and system load.

Note: See the Socket Statistics Overlay Meters page for more details.
