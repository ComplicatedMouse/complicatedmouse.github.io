When installing the NVIDIA graphics driver, the screen goes black and does not recover.

If you select the Quick (recommended) installation option when installing the NVIDIA graphics driver and then switch to the driver, the screen sometimes goes black and does not recover. If the screen does not recover, in some cases you can still use the computer by logging in via remote desktop.

To avoid this problem, select the Custom (advanced) installation option and make sure to select the Clean Install check box.

The scan-out buffer is not double buffered correctly when a processing slowdown occurs.

Symptoms

When double buffering is enabled using nvnWindowBuilderSetTextures / nn::gfx::SwapChainInfo::SetBufferCount, the command that waits for image output to complete (that is, the command inserted in the nvnQueueAcquireTexture function and executed after PresentInterval has elapsed) does not wait for the next VSync event.
As a result, when you use a function such as nvnQueueFinish to wait for the command to be completed, the function returns at a timing other than when a VSync event occurs. Also, sometimes new data gets rendered into the buffer currently being scanned by the display controller, which causes noisy images to be drawn on the screen.

Workaround

As a temporary workaround, use triple buffering.

Fix Schedule

The cause of this bug is currently under investigation. This bug will be fixed as soon as the root cause is discovered.

GLSLC Known Bugs

  • EOL markers must be present in #include files, else a compilation error will occur.
  • Transform feedback varyings specified in shader source do not work correctly.
  • Compute shader data subsections inside a GPU code section do not start with the magic number GLSLC_GPU_CODE_SECTION_DATA_MAGIC_NUMBER.
  • The TX1 versions of the glslc library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms is likely to be functionally correct though, and will be combined in future releases.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface, and mediump or lowp types on the other side of the interface, could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types qualified with mediump or lowp qualifiers. You can work around this limitation by explicitly constructing float types from the int and uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp in addition to ES shaders regardless of whether the GL_EXT_shader_implicit_conversions extension is used.
    • FP16 math when used with imageStore may cause corruption in some cases. This issue is being investigated and will be fixed in an upcoming release.

Restarting or turning off the development hardware at certain times sometimes causes the hardware to no longer start.

Symptoms

Restarting or turning off the development hardware immediately after initializing it or while the system is booting sometimes causes the system to become unstable and no longer start.

Workaround

Do not reboot or turn off the development hardware immediately after the system boots up. If you experience this bug, the system must be reinitialized (see here for the SDEV procedure and here for the EDEV procedure).

Fix Schedule

This bug will be fixed in the next release.

VSI Bugs

Cannot handle exceptions thrown in SDK libraries.

The NX32 and NX64 SDK libraries do not support C++ exceptions.
For this reason, when exceptions are thrown in the libraries, they are canceled.

If the code is only propagated to the application, you can enable C++ exceptions at build to make it possible to handle exceptions.

Optimized code may not be debugged correctly.

In some cases, optimized code cannot be debugged correctly, such as when it stops on breakpoints placed in functions with inline expansion.

The VSI connection is terminated if the system enters sleep mode while debugging an application.

Symptoms

If you put the target hardware into sleep mode while running the debugger, the connection with VSI is ended and debugging features are disabled until you restart the application.

Workaround

This bug can be avoided by simply not putting the system into sleep mode.

Fix Schedule

This bug will be fixed in the next release.

Setting a breakpoint while the debugger is running sometimes causes the application to crash.

Symptoms

Setting a breakpoint while the debugger is running sometimes causes the application to crash.

Workaround

You can avoid this bug by setting breakpoints before running the application, by pausing the application before setting a new breakpoint, or by changing the location at which the breakpoint is placed.

Fix Schedule

This bug will be fixed in the next release.

fs Bugs

Multi-byte characters cannot be used with save data entry names.

UTF-8 entry names are supposed to be available for use, but specifying entry names that contain non-ASCII characters does not work.

nn::fs::DeleteDirectoryRecursive() fails when deleting directories containing many files.

In the file system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot()), when a directory including around 255 or more files is deleted with nn::fs::DeleteDirectoryRecursive(), it fails with nn::fs::ResultUnexpected.

Cannot open numerous files simultaneously.

The number of files that can be open at any time is restricted as follows.

File System Maximum Number of Open Files
File system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot())

338 (32-bit environment)

182 (64-bit environment)
Resource data (nn::fs::MountRom())

338 (32-bit environment)

182 (64-bit environment)
Save data (nn::fs::MountSaveData()) 30

The function fails with nn::fs::ResultAllocationMemoryFailed if you attempt to open more than the maximum number of files.
We plan to improve the number of files that can be opened at the same time.

In save data, adding entries to directories that have a lot of entries takes a long time.

When creating entries with nn::fs::CreateFile() or nn::fs::CreateDirectory() in directories that include 512 or more entries, it takes much longer than cases with 511 or fewer entries.

nn::fs::ReadFile() or nn::fs::WriteFile() may disconnect from Target Manager.

There is a bug where the connection to the Target Manager may be disconnected (the status in Target Manager becomes "Disconnected") when executing nn::fs::ReadFile() or nn::fs::WriteFile() through MountHost(), MountHostRoot(), or MountRom() in raw mode.

The conditions for this problem occurring are based on the size of the file loaded in nn::fs::ReadFile() or nn::fs::WriteFile() and the size of previously loaded files, but the details are currently being investigated.

If this problem occurs repeatedly, avoid it by changing the buffer size specified in nn::fs::ReadFile() or nn::fs::WriteFile().

Audio Bugs

The nn::audio::SetAudioDeviceOutputVolume() function has several bugs.

Symptoms

The nn::audio::SetAudioDeviceOutputVolume() function exhibits the following bugs.

  1. Setting a volume value greater than 1.0f causes the system to hang.
  2. Settings are not applied to the output if the function is called before the nn::audio::OpenAudioRenderer() or nn::audio::OpenAudioOut() functions.
Workaround
  1. You can avoid the first bug by not setting volume values greater than 1.0f. If you do accidentally set a volume value greater than 1.0f, reboot the system.
  2. You can avoid the second bug by making sure to call the function after one of the open functions.
Fix Schedule

We plan to fix this problem in the next release.

On the development hardware, when sound is being produced with nn::audio and the application stops due to a breakpoint or for other reasons, the sound playback does not stop.

Symptoms

On the development hardware, when an application is paused due to a breakpoint or the like while sound is being produced, playback continues and the sound does not stop.

Workaround

You can avoid this bug by using the following procedure to change the development hardware settings.

  1. Using the settings manager, export the configuration file from the development hardware to a PC.
  2. Open the exported configuration file on the PC, and change the current value for the suspend_for_debugger_enabled element in the audio section from false to true.
  3. Using the settings manager, import the configuration file from the PC to the development hardware.
  4. Reboot the development hardware.

For more information about using the settings manager, see Settings Manager.
Note that after configuring the settings as described, if you connect Target Manager while an application that uses nn::audio to play sound is running, the application will stop responding.
To avoid this problem, connect Target Manager before starting the application.

Fix Schedule

We plan to fix this problem in the next release.

perf Bugs

In the NX Windows version of perf, Tick values measured using nn:perf::GpuMeter do not correspond to the Tick values returned by the nn::os::GetSystemTick() function.

When measuring Tick values using nn:perf::GpuMeter, the Tick values stored in nn:perf::LoadMeterBase::Section::begin and nn:perf::LoadMeterBase::Section::end are slightly different from the Tick values measured using nn::perf::CpuMeter or the nn::os::GetSystemTick() function.
However, the speed at which the counter ticks is the same for both meters, so adjusting the Tick values by the interval (end - begin) (which represents the processing time) makes it possible to use GpuMeter and CpuMeter interchangeably.

This restriction will be removed in a future release.

This bug is limited to the NX Windows version of perf.
On the development hardware, Tick values measured using GpuMeter correctly correspond to Tick values measured using CpuMeter and Tick values returned by nn::os::GetSystemTick().
