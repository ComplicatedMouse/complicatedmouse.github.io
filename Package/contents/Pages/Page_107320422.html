Introduction

NX VSI build and debugging parameters can be configured from the project configuration properties.

The configuration properties include settings that are used for both Windows applications and NX applications, in addition to several settings used only by NX VSI.

This document describes the most important of the NX VSI-specific settings.

Configuration Properties

General

Configuration Type

Select the type of program to build from the following options.

  • Executable (NSP file)
    • Specify this option to generate executable code for the development hardware.
  • Static library (.a)
    • Specify this option to generate a static library.
  • Dynamic library (.nro)
    • Specify this option to generate a dynamic library.
    • For more information about using dynamic libraries, see Using Dynamic Libraries.
  • Makefile
    • Specify this option when using an external build system.
    • For more information, see Using Makefile.

Application Program Format

Raw

Generates executable code, and RomFs uses the file on the host PC directly.
Normally, use this.

NCA

Generates an NCA file.
Because the NCA file includes all files needed to run the app, the app can be run by sharing only this file.
Because archiving and other processes are required, it takes more time to generate the file than Raw mode.
Use this option if you want to include all of the data needed to run the application in a single file.

Note

Effective from NX Add-On 1.0.0, there is no longer any support for running an application using the NCA file. To run an application, use the NSP file instead.

NSP

Generates an NSP file.
NSP files are used to install applications on the development hardware.
Use this option when you want to install an application built for the development hardware.
For more information about debugging the installed application using VSI, see VSI > Debugging a Built Program > Debugging an Already Running Application.


DESC File

Files that specify application permissions and parameters.

These files are signed and so cannot be edited. In most cases, do not change the specified path.

Metadata Source

Files that specify the app permissions and parameters.

These files are written in XML. The following parameters can be modified for each app.

Nintendo is not yet assigning IDs for NX applications, so as of this writing, you do not need to change this value.

Additional MakeNso, MakeMeta, and MakeNrr Options

Specifies additional options to be passed during processing for each tool.

Edit and Continue

Enables or disables the Edit and Continue function. For more information about the Edit and Continue function, see Edit and Continue.

Code Patch Area Size and Data Patch Area Size

When Edit and Continue is enabled, specifies the size of the memory region to be reserved from the application's memory space as the patch area. For more information, see Edit and Continue.

Debugging

Pre-Run Command

Specifies a command to execute before starting the debugging session.
This command is also executed when executing applications without debugging.

Remote Command Arguments

Specifies command-line arguments to pass to the application.
These command-line arguments can be obtained using the nn::os::GetHostArgc() and nn::os::GetHostArgv() functions defined in nn/os/os_Argument.h.

Debug Method

Specifies the method to start debugging.

  • Launch
    • Starts the application, and begins debugging. Use this method for normal development.
  • Attach during launch
    • The state in which the application is not started, but the system is waiting for the application to start. If the application is started up using another method, debugging will be attached automatically and immediately after startup.
    • Used in situations where you want to perform debugging after starting the application from the Home Menu.

Target Environment Variable File

Specifies the path to the target environment variable file. For more information, see htc Library / Manual / Features / Target Environment Variables (NX Add-On).

C / C++

General

Echo Command Lines

Select Yes for this setting to display all commands executed during compilation to the output window.

Multi-Processor Compilation

Select Yes for this setting to enable parallel compilation.

Max Compilations

Sets the maximum number of compilations to run in parallel when parallel compilation is enabled.

Other

Use the following settings to modify the compilation options. NX VSI uses a Clang-based compiler, so the compiler options are based on the Clang compiler options. For more information about the compiler, see Compiler.

  • Generate Debug Information (-g)
  • Warnings (-Wall, -w)
  • Treat Warnings as Errors (-Werror)
  • Extra Warnings (-Wextra)
  • Disable Specific Warnings
  • Extended Diagnostic Messages (-fcaret-diagnostics, -fno-caret-diagnostics)

Optimization

Use the following settings to modify the compilation options. NX VSI uses a Clang-based compiler, so the compiler options are based on the Clang compiler options. For more information about the compiler, see Compiler.

  • Optimization (-O0, -O1, -O2, -O3, -Os)
  • Strict Aliasing (-fstrict-aliasing, -fno-strict-aliasing)
  • Omit Frame Pointer (-fomit-frame-pointer, -fno-omit-frame-pointer)
  • Fast Math (-ffast-math)
  • Unroll Loops (-funroll-loops)
  • Vectorization (-fvectorize, -fno-vectorize)
  • Inline Functions (-fno-inline-functions)
  • Generate Implicit Floating Point Instructions (-mno-implicit-float)

Code Generation

Use the following settings to modify the compilation options. NX VSI uses a Clang-based compiler, so the compiler options are based on the Clang compiler options. For more information about the compiler, see Compiler.

  • Enable C++ exceptions (-fexceptions, -fno-exceptions).
  • Do not use built-in functions (-fno-builtin).
  • Enable stack protection (-fstack-protector, -fstack-protector-strong, -fstack-protector-all).
  • Use safe stack (-fsanitaize=safe-stack).
  • Detect undefined behavior (-fsanitaize=undefined).
  • Enable link time optimization. (-flto-full)
    • Link time optimization can only be applied if you are using the LLD linker.
    • -flto-thin is a beta version.
  • Profile-Guided Optimization
Warning

In principle, conduct profile-guided optimization in every application development sequence. This is because profraw files are expected to be generated in large numbers when you test-play the application, and we do not recommend using Visual Studio to control sequences where these profraw files are systematically converted to profdata format.
Visual Studio supports converting profraw to profdata, but we recommend only doing this on a trial basis.
It would be reasonable to apply the generated profdata in Visual Studio by some means.

For more information, see SDK PGO Supplemental Information.

    • Specifies how to use profile-guided optimization.
    • When applications built with Generate PGO Data selected are executed or ended, the profiled data is saved as default.profraw to the folder specified in Profile Session Data Folder.
    • If a build is run with Apply PGO Data selected, PGO is applied using default.profraw under the Profile Session Data folder. At this point, the intermediate file converted to the PGO input format (.profdata) is added to the file specified as Profile Data.
    • Additional work is required to apply PGO to NRO. See SDK PGO Supplemental Information.
  • Profile Session Data Folder
    • When Generate PGO Data is set, specifies the save location for the default.profraw file generated when the application is executed.
    • When Apply PGO Data is set, it is parsed as the folder with the default.profraw to be applied.
  • Profile Data
    • When Apply PGO Data is set, it is parsed as the output path for the intermediate file.
    • Note that there are no parameters for specifying profdata as input, and the default.profraw under the Profile Session Data folder is treated as the input.

Language

Use the following settings to modify the compilation options. NX VSI uses a Clang-based compiler, so the compiler options are based on the Clang compiler options. For more information about the compiler, see Compiler.

  • Enable runtime type information (-frtti, -fno-rtti).
  • Check ANSI compliance (-ansi).
  • Default char unsigned (-fsigned-char, -funsigned-char).
  • Enable MS extensions (-fms-extensions).

Precompiled Headers

Like with Windows builds, precompiled headers can be used when building for NX.
The specific procedure is as follows.

  1. Prepare the source code and a header file (for this example, prepare files named pch.cpp and pch.h).
  2. Include the headers you want precompiled from pch.h.
  3. Include just pch.h from pch.cpp.
  4. In the project view, select Properties > C/C++ > Precompiled Headers > Precompiled Header File, and specify pch.h.
  5. In the project view, select Properties > C/C++ > Precompiled Headers > Precompiled Header, and then select Use. (This setting is applied to all of the files added to the project.)
  6. For the pch.cpp file, select Properties > C/C++ > Precompiled Headers > Precompiled Header, and then select Create.

Advanced

Use the following settings to modify the compilation options. NX VSI uses a Clang-based compiler, so the compiler options are based on the Clang compiler options. For more information about the compiler, see Compiler.

  • Compile as (-x c, -x c++, -x assembler-with-cpp).
  • Forcibly include files.

Linker

General

Linker Script

Specify a linker script file.

Do not change the specified path.

Link Library Dependencies

Switches between whether the libraries created for a project are automatically linked when such projects are specified as dependent projects.
When this option is enabled, the libraries for a dependent project are automatically linked.

Echo Command Lines

Select Yes for this setting to display all commands executed during linking to the output window.

Linker to Use

ld, gold, and LLD are available.

Generate Map File

Select Yes to create a MAP file when linking.

Map File Name

Specifies the path for the MAP file to create.

Enable Dead Strip

Removes unused symbols from the final output.
Note that nn::diag::GetSymbolName will not work if dead-stripping is used.
For more information, see Troubleshooting (NX Add-On).

If you are not dead-stripping, unnecessary information remains and can cause problems when submitting a ROM. For this reason, dead-strip all retail products.

Retaining Specific Symbols When Dead-Stripping

In the normal build procedure, the NRS files that are the sources of the NRO files used when linking the application are also specified, so all symbols used by the application remain even after dead-stripping.

However, if for some reason you want to keep specific symbols, you can use the following procedure to specify a dynamic list.

  1. Make a copy of $(NINTENDO_SDK_ROOT)/Resources/SpecFiles/ExportDynamicSymbol.lst to have on hand.

  2. Add the mangled names of the functions you want to keep to this copied LST file. For example, if you want to keep void TestFunction(), add that function to the ExportDynamicSymbol.lst file as follows.

    {
  __nnDetailNintendoSdkRuntimeObjectFile;
  nnMain;
  nninitStartup;
  _Z12TestFunctionv;
};
  3. Use the following steps to reference the copied LST file while building.
    1. Open View > Other Windows > Property Manager.
    2. Right-click the project file and select the option to add a new property sheet.
      1. Right-click a specific structure and select the option to add a new property sheet for that structure only.
    3. Open the newly added property sheet and select Common Properties > User Macros > Add Macros.
    4. Specify DynamicSymbolList as the name and the copied LST file path as the value.

Build Identifier Generation

Select how to generate the unique identifier for the output.
This identifier is used to associate debugging information.
For more information, see the Toolchain for NX Specific Features: Using Build IDs section in Compiler.

Input

Additional Dependencies

Specify the dependent static library files (A files).

Additional NSO Files

Specify any dependent dynamic library files (NSO files).

Additional NRO Files

Specify the additional dynamic library files (NRO files) to use.
These files must be manually loaded when executing the application.
For more information, see Using Dynamic Libraries.

NRO Deploy Path

Specify the directory to copy the files specified in Additional NRO Files when copying files during the build.
Normally, a path under the application data directory is specified.
This is because the ROM is included in the files under the application data directory, so these files will be loaded when the application is executed.
For more information, see Using Dynamic Libraries.

NRR File Name

Specify the location of the metadata associated with the NRO file.
The metadata will be generated for the specified path, so this file will be loaded into the application during execution.
For more information, see Using Dynamic Libraries.

Authoring

General

Application Data Directory

The specified directory path is treated as the root directory for all application resource data. For more information, see Accessing Resource Data.
If a relative path is specified, it is treated as a relative path from the VCXPROJ file.

Additional options

Used when manually specifying commands in the AuthoringTool.
In general, do not change these options.

Always Run AuthoringTool

This parameter specifies whether to always run Authoring Tool when building the NSP file.
For the NSP file to reflect the changes made to files in the application data directory, you must run Authoring Tool. However, the current version of Authoring Tool does not check for updates to these files, so you must specify Yes for this parameter to enable Authoring Tool to run when no changes have been made to the program.

Creating a Build Error When a ROM Can Not Be Submitted

Uses a build error when the created ROM cannot be submitted.

Patch

For more information about creating patches, see Creating Patches.

Creating Patch Files

Specify whether to create patch files.

Initial Version NSP File

Specify the NSP file for the initial application.
Note: When creating patch files, the NSP file compatible with the current version will be output as an intermediate file, $(OutDir)$(target name) .nsp, so a correct patch cannot be generated if the same path is specified.

Previous Patch File

Specify the previous patch file.
If the current patch is the first patch, this setting does not need to be specified.

Cache Output Directory

Specifies the directory to which to save the intermediate output when creating a patch.
Having this output available reduces the time required to create the next patch.

Additional Options

Use when specifying additional options when generating a patch.
