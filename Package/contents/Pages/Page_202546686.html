General

You can brick the system if you cut off the power or restart the system repeatedly during startup while the SD card is inserted.

Symptoms
  • If the power is cut off or the system is restarted repeatedly while the SD card is inserted, the screen may remain blank and processing may no longer progress.
  • If the system is restarted in this state, it will still freeze at the same point.
Workaround

Remove the SD card and restart the system.

After restarting, either initialize the SD card with DevMenuCommand or format the SD card using a PC.

The system can be started if a normal SD card is inserted.

Fix Schedule

This problem will be fixed in an upcoming release.

General Graphics

If an application's GPU command is run immediately after a VSync event, the GPU execution timing becomes unstable.

Symptoms

When an application's GPU command is executed immediately after a VSync event, by waiting on a display release fence or other means, it competes with the beginning of the system's GPU time allocation, and the GPU execution timing becomes unstable.

Workaround

Do not issue commands immediately after VSync events; that way, a GPU command is not executed immediately after a VSync event. It is also acceptable to have application commands issued during times when GPU time is allocated to the system. The commands are executed immediately after GPU time is allocated to the application.

Fix Schedule

This bug will be fixed in an upcoming release.

nvn

The display release fence is signaled at times other than when the display updates the framebuffer.

Symptoms

Normally, NVNsync objects (display release fences) returned by the nvnWindowAcquireTexture() function are only signaled when the framebuffer scanned by the display controller is updated on a VSync event. However, these objects sometimes also get signaled at the following times.

  • The focus state transitions between the in-focus state and another focus state.
  • The display is switched due to the system being docked or undocked or due to the HDMI cable being plugged in or unplugged.
  • When the system enters sleep mode or wakes up from sleep mode.
  • When the TV output resolution is changed in system settings.

In these cases, the GPU commands issued by the nvnQueuePresentTexture() function are not assured to be complete when a display release fence is signaled. If an application’s CPU processes determine whether GPU processes are complete based on display release fences, command buffers or other buffers that are still being used by the GPU can be released or overwritten at these times.

Workaround

When waiting for GPU processes to complete, rather than determining when the processes are complete indirectly by using display release fences, determine when the processes are complete directly by using nvnQueueFinish() or nvnQueueFenceSync() and nvnSyncWait().

Fix Schedule

We are currently investigating whether a fix is possible. The exact timing of the fix is to be determined.

PresentTexture() and AcquireTexture() calls may be blocked for several hundred miniseconds during docking and undocking.

Symptoms

The calls to the nvnQueuePresentTexture() and nvnWindowAcquireTexture() functions may be blocked for about 200 milliseconds to 300 milliseconds, depending on the timing at which the display is switched when the console is docked and undocked.
The application's render loop can stop operating at a set interval and, depending on the implementation of the application, this can result in the sound becoming out of sync and also cause other symptoms.

Workaround

In scenes such as when video clips are playing and the screen and audio must be synchronized for a long time, you must implement the application in a way that explicitly forces synchronization between the screen and the audio, without expecting the render loop to operate at a set interval.

Fix Schedule

We are currently investigating whether a fix is possible. The exact timing of the fix is to be determined.

GLSLC Bugs

  • GLSL shaders using NV_fragment_shader_interlock features will consume more scratch memory than similar shaders that do not use the feature. We are investigating whether this requirement can be reduced.
  • Assembly dump support does not currently work when compiling using multi-threaded compilation. If you are compiling with multi-threading enabled, set the GLSLC option outputAssembly to false.
  • EOL markers must be present in #include files, else a compilation error will occur.
  • The TX1 versions of the glslc library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms is likely to be functionally correct though, and will be combined in future releases.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface, and mediump or lowp types on the other side of the interface, could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types qualified with mediump or lowp qualifiers. You can work around this limitation by explicitly constructing float types from the int and uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp in addition to ES shaders regardless of whether the GL_EXT_shader_implicit_conversions extension is used.
    • FP16 math when used with imageStore may cause corruption in some cases. This issue is being investigated and will be fixed in an upcoming release.
