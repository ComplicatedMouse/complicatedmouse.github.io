Build Targets and Build Types

Introduction

The NintendoSDK build system uses the identifiers Build Target and Build Type to differentiate the binary types for the provided SDK library and application. This page describes these identifiers.

Build Targets

Overview

Build targets are identifiers that represent the operating environment mainly for libraries and applications.

For example, because the library for the development hardware environment and the library for the Windows environment are provided as different binary files, some kind of framework is needed to distinguish between the two files. Here, the build target is the identifier expressing which operating environment the binary file is built for. In this example, the former is the development hardware environment's build target library, and the latter is the Windows environment's build target library.

Because when using the build system provided by NintendoSDK, the build target is automatically selected based on Visual Studio's project configuration and property values, the developer usually does not need to pay attention to the build targets. However, when not using the build system provided by NintendoSDK and creating an original build system, the developer must pay attention to the build targets.

In practice, build targets are defined not only by the operating environment but through a combination of various conditions including the ABI, compiler tool chain, and provided API set. The list of defined built targets that follows lists the conditions that each build target expresses. Use this list to select the appropriate build target.

Defined Build Targets

The build targets that can be used by shared packages are as follows. All the build targets generate binary files using the general-purpose API set.

Build Type

Overview

Build types are the identifiers that mainly express the optimization options and debugging options for the library and application.

For example, the NintendoSDK library sets whether internal assertions are enabled or disabled for each build type, so even the same library has a library file with assertion enabled and a library file with assertion disabled. You can use the different build types to link the build type library with assertion enabled when debugging an application, and link the build type library with assertion disabled to increase speed in the full version application.

For projects created using the new project creation wizard in the NintendoSDK, you can select the build type from the project structure.

Defined Build Types

NintendoSDK uses the following three build types.

  • Debug
  • Develop
  • Release

Debug Build Type

Except for the binary for Visual C++ that follows, NintendoSDK does not provide a Debug build type binary. When linking the Debug build type binary using NintendoSDK's property sheet, if there is no Debug build type binary, the Develop build type binary is automatically linked in its place.

If the library and application source code is provided, you can generate the Debug build type binary by building from the source code.

Debug Information in Libraries

The built binaries provided by NintendoSDK do not have debug information added. However, where library and application source code is provided, developers can add the debug information by rebuilding.

Info

Some libraries have debug information added. Presently, debug information is added to static library files for the NX development hardware environment.

Information Unique to Visual C++

The binaries for Visual C++ include binaries for the Debug build type. However, operations are not ensured.
