Important Information When Changing to a Development Hardware Environment

The following compilation of important information is for migrating either a Windows development environment or source code from a prior Nintendo platform to the development hardware environment.

Do Not Use wchar_t

In a Windows environment, the size of the wchar_t type is treated as two bytes. It becomes four bytes in the NX development environment.

Because the SDK libraries for the NX development environment are built with four bytes for the wchar_t type, refrain from using -fshort-wchar in your applications.

To handle multiple-byte characters as the same size in both the Windows environment and the NX development environment, we recommend using the char type and handling it as UTF-8.

Log output for the diag library supports UTF-8, but not all libraries support this type. Some, including the NintendoWare font library, do not yet support UTF-8, but will eventually support it.

Until then, use the nn::util::ConvertStringUtf16NativeToUtf8() function to convert to UTF-8 types.

For information about handling UTF-8 within source code in a Windows environment, see Notes on the Log and Assertion Encoding in the diag Library.

char Type

The default char type in Toolchain for NX is unsigned char, not signed char.

Note that the NX development environment SDK libraries have tested unsigned char for the char type. We cannot recommend the use of --fsigned-char with your application.

This action is in keeping with ARM ABI.

Using NN_BUILD_TARGET_PLATFORM_NX To Identify The NX Environment

If you need to use a macro to determine whether the platform is NX, use NN_BUILD_TARGET_PLATFORM_NX as follows.

#include <nn/TargetConfigs/build_Platform.h>

#ifdef NN_BUILD_TARGET_PLATFORM_NX
...
#endif
