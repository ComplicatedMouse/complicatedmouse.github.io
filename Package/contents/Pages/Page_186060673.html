Initialization for PROD MODE, the Firmware That Approximates Retail Hardware

Introduction

This page describes the procedure to initialize using PROD MODE, the firmware for development hardware that approximates the retail hardware.

Later information will describe procedures for testing applications using this PROD MODE.

What Is PROD MODE, the Firmware That Approximates Retail Hardware?

Differences in the internal structure for the development and retail hardware result in differences in application operations. Using PROD MODE, the firmware that approximates the retail hardware, enables you to create a state that approximates the operating settings for retail hardware, to reduce the differences that could appear when verifying software on development hardware.

Use this PROD MODE as a testing environment before submitting a master ROM for your developed application.

Use of this firmware not only approximates operations on retail hardware but also restricts some of the functionality available for the development hardware.

  • Cannot connect to the console from Target Manager.
  • Cannot use the debugger.
  • Cannot use the htc library.
  • On the EDEV, you must enter Safe Mode manually when reinitializing.

For more information about restricted functions, see PROD MODE Restricted Functions.

You can also use firmware that approximates the retail hardware with the feature to save logs to a microSD memory card. This firmware is PROD MODE (described earlier) with the feature to save logs to a microSD memory card. You can use this to investigate bugs that only occur in PROD MODE. For example, some issues occur when using HostFS or the htc library with PROD MODE.

Initialization using these firmware versions is possible with the various initialization tools: InitializeSdev, InitializeSdevWin, InitializeEdev, and InitializeEdevWin.

Info

In PROD MODE, the development hardware operates only in the HOME Menu environment.
In DEV MODE, which is described later, you can switch the development hardware between the HOME Menu environment and the DevMenu environment.

For more information, see Development Assistance Settings > Initial Boot Menu.

PROD MODE Is the Opposite of DEV MODE

Unlike PROD MODE, which approximates the retail hardware, DEV MODE is firmware specific to the development hardware and is the firmware that is used by default.

If no specific firmware is specified for the initialization tool, the development hardware will be initialized with the DEV MODE firmware.

Initializing

This description assumes that the procedures in the Development Hardware Setup/Firmware Update section of Guide to Setting Up the Environment have been completed. If they have not been completed, see that document.

For the SDEV

Start InitializeSdev.bat as you would normally.

To initialize the development hardware with the PROD MODE firmware that approximates the retail hardware, select the option that includes + Prod in the Select firmware variation combo box, and then click Start initialization. (This example selects the 0.8.16-0 + Prod option to select the proper firmware.)

To use the version of PROD MODE with the feature to save logs to a microSD card, select the option that includes + ProdWithLog in the Select firmware variation combo box, and then click Start initialization.

Info

To initialize the development hardware using the command-line tool (InitializeSdev.exe), specify the --target-version option when running the tool, as follows.

$ InitializeSdev.exe --target XAW0213100XXXX --target-version "0.8.16-0 + Prod"


For the EDEV

Start InitializeEdev.bat.

In the Select firmware variation combo box, select the desired firmware and then click the Start initialization button.

To use the PROD MODE firmware that approximates the retail hardware, select the option that includes + Prod. To use the version of PROD MODE with the feature to save logs to a microSD card, select + ProdWithLog.

You can also specify the firmware, as was the case for SDEV, from InitializeEdev.exe, the command line initialization tool.

There is one area that requires additional considerations for EDEV! You must manually enter Safe Mode to reinitialize an EDEV if you have initialized it using PROD MODE.
For more information about the procedure for manually entering safe mode, see Manually Activating Safe Mode in When EDEV Initialization Fails.

Updating

You can update to PROD MODE using an update tool such as SystemUpdateSdev. However, be aware of the following.

When updating from DEV MODE, the update tool will not result in an appropriate state, even though an update to PROD MODE occurs. This occurs as of NX Add-On 4.3.0.
When using the update tool to change to PROD MODE, you must also switch to PROD MODE before updating, using the update tool after switching.

When updating from DEV MODE, the menu will remain as DevMenu without becoming HomeMenu. You can switch the menu if that occurs by using DevMenuCommand -- debug switch-menu homemenu.

Testing Applications

Certain steps are needed to test applications because you cannot use Target Manager with PROD MODE.

There are two ways to run applications in PROD MODE.

  • Running as an application written to a game card.
  • Running as an Installed Application

Testing Applications Written to Game Cards

The following procedure is needed to test applications written to game cards when using a PROD MODE console.

  1. Initialize the development hardware with the usual initialization method.
  2. Use DevMenuCommand to write an NSP file for an application to a game card.
  3. Initialize the development hardware in PROD MODE.
  4. Insert the game card and start the application.

Note: You cannot run DevMenuCommand on a PROD MODE console because DevMenuCommand uses TargetManager.

This method requires as many game cards as you have hardware to use for testing. If there are not a lot of game cards available, consider this approach: Running as an Installed Application.

Running as an Installed Application

The following procedure is needed to run an application installed using PROD MODE.

  1. Initialize the development hardware with the usual initialization method.
  2. Insert the application's NSP file to the microSD memory card.
  3. Initialize the development hardware in PROD MODE.
  4. Start DevMenuApp from the HOME Menu.
  5. Install the NSP file from DevMenuApp.
  6. Start the application from the console.

Using the Feature to Save Logs to a microSD Memory Card

When you use the SDEV or EDEV initialized with firmware approximating the production hardware with the feature to save logs to a microSD memory card, the logs are saved to a microSD memory card.

A .nxbinlog file is created with the log contents recorded to it under the \NxBinLogs directory on the microSD memory card. You can check the logs in this file by using the LogViewer tool.

For more information about using LogViewer, see its description page in the Document for NX package.
