DIAG

When a log file that is greater than a certain size is opened in Log Viewer, the second half of the log is not displayed.

Symptoms

When Log Viewer opens an NX Binary Log file (.nxbinlog) that is larger than several dozen megabytes in size, the latter part of the log will be omitted and not displayed.

Workaround

To view the entire log, select Convert NX Binary Log from the Log Viewer menu and convert the NX Binary Log file into a text file.

If the log is larger than1 GB in size, this conversion process can take several minutes.

Fix Schedule

This problem will be fixed in an upcoming release.

IRSENSOR

Using the IR Motion Camera for a long period of time while charging the Joy-Con (R) causes the battery level to decrease.

Symptoms

When using the IR Motion Camera for a long period of time while charging the Joy-Con (R), in most cases the controller will continue to charge, but after approximately 10 hours the battery level will start to decrease.
When this happens, continuing to use the IR Motion Camera will cause the error viewer to display a battery level warning if the system is in handheld mode or will cause the controllers to be disconnected if the system is in wireless mode.

Workaround

Reset the controllers manually or stop charging the controllers to reset them.

Fix Schedule

This problem will be fixed in an upcoming release.

If the connection state of the target controller changes when nn::irsensor::StopImageProcessor or nn::irsensor::StopImageProcessorAsync is called, it can take an extended time for the process to end.

Symptoms

If the connection state of the target controller changes when nn::irsensor::StopImageProcessor or nn::irsensor::StopImageProcessorAsync is called, for example because the power has been cut or the controller is suddenly attached to the console, it can take approximately 5 seconds for the process to end.
Furthermore, when nn::irsensor::StopImageProcessor is being used and this bug occurs, the function will block during that time.
Note that tasks such as the retrieval of data and the start of the next process will be made to wait during that time.

Workaround

We recommend implementing a discreet workaround for when this problem occurs, such as showing a message when the state of the controller has changed, or designing your program so that a user cannot seamlessly continue to use the controller.

Fix Schedule

This problem will be fixed in an upcoming release.

If the system transitions to an applet or goes to sleep when nn::irsensor::StopImageProcessor or nn::irsensor::StopImageProcessorAsync is called, the status obtained by nn::irsensor::GetImageProcessorStatus will be invalid.

Symptoms

If the system transitions to an applet or goes to sleep when nn::irsensor::StopImageProcessor or nn::irsensor::StopImageProcessorAsync is called, an inconsistency in the internal state may arise and nn::irsensor::GetImageProcessorStatus will get the wrong processor status. Furthermore, when nn::irsensor::StopImageProcessor is used and this bug occurs, the function will block for approximately 5 seconds.

Workaround

To recover to a valid status, run nn::irsensor::Finalize or call one of the Run functions.
If you are using nn::irsensor::GetImageProcessorStatus to manage the status, set timeouts or otherwise design your program while also considering this bug.

Fix Schedule

This problem will be fixed in an upcoming release.

In Pointing mode, the pointing coordinates that are output are not normalized correctly.

Symptoms

The output values of the pointing coordinates obtained by nn::irsensor::GetPointingProcessorStates should be returned as normalized values with the center at (0.0, 0.0) and the upper-right angle of view at (1.0, 1.0). However, due to a bug, the data are returned in a coordinate system with the upper-left angle of view at (0.0, 0.0) and the lower-right angle of view at (320.0, 240.0).

Workaround

As a temporary workaround, you can try having your application normalize the coordinate values based on this provided data. This bug will be fixed in the future, so if you plan to submit a master for an application that uses Pointing mode, do not use the current version of the SDK, but instead wait for the version in which this bug has been fixed.

Fix Schedule

This problem will be fixed in an upcoming release.

LDN

Communication is unstable on some development hardware.

Symptoms

The following symptoms may occur on some development hardware.

  • Could not find the network when scanning
  • Could not connect to the network
  • High packet loss rate

In addition, if there are two development hardware devices with this symptom, communication between these two devices is even more unstable.

Workaround

The same problem occurs if the surrounding signal environment is poor. Move to an environment with fewer communication devices or use other development hardware and then check to determine whether the symptoms improve. Contact Nintendo support if the symptoms from this issue are particularly prominent.

Fix Schedule

Future handling is under discussion.

Symptoms

After installing and opening the HOME Menu, starting an application in VSI or RunOnTarget puts the system into a special state. The application will start, but this feature cannot be used on some screens and the application will stop responding. The following screens are known to stop responding in this situation.

  • Screen for selecting accounts before starting the application
  • Screen that displays the dialog box that occurs prior to starting the application

Starting the application with VSI or RunOnTarget may result in detailed behavior that is different from the retail version, so we recommend that you either create an NSP and install it or burn the application to a game card for final application verification and checking its behavior when entering or exiting the HOME Menu.

NVN

Under some conditions, noise is sometimes generated in the rendering results when the color buffer and depth buffer share physical memory.

Symptoms

When the color buffer and the depth buffer share a physical memory pool and all the following conditions are met, noise might be generated in the rendering results.

  • Of the virtual memory pools mapped in the physical memory pool, at least one is the16-bit depth (Z16) storage class.
  • Of the virtual memory pools mapped in the physical memory pool, at least one is the compressible color storage class.
  • Of the virtual memory pools mapped in the physical memory pool, at least one is created with NVN_MEMORY_POOL_FLAGS_GPU_CACHED_BIT.
Workaround

You can avoid this issue by not aliasing the virtual memory pools in the same physical memory pool that meets all of these conditions.
You can also avoid this issue by using values not registered as fast clear when clearing aliased color textures.

Fix Schedule

This issue is currently under investigation. The exact timing of the fix is to be determined.

GLSLC Bugs

  • EOL markers must be present in #include files, else a compilation error will occur.
  • The TX1 versions of the GLSLC library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms should be functionally correct though, and convergence of the two will happen in future changes.

FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:

  • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface, and mediump or lowp types on the other side of the interface, could produce linker errors.
  • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages produces a linker error.
  • int and uint types cannot be implicitly converted to float types qualified with mediump or lowp qualifiers. You can work around this limitation by explicitly constructing float types from the int and uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp in addition to ES shaders regardless of whether the GL_EXT_shader_implicit_conversions extension is used.
  • FP16 math when used with imageStore may cause corruption in some cases. This issue is being investigated and will be fixed in an upcoming release.
