Vectors and Matrices

Overview

These features are for calculating vectors and matrices for 3D graphics.

Element Types

Supports single-precision floating-point numbers. Does not support double-precision floating-point numbers.

Coordinate System

Use a right-handed coordinate system.

Vector-Matrix Multiplication

Vectors are treated as row vectors and multiplied by the matrix to the right.

SIMD Support

In environments that use SIMD units, it is assumed that the implementation is switched to one that uses those units.

The regular implementation is provided in environments that do not use SIMD units.

Vectors

Two-dimensional vectors, three-dimensional vectors, and four-dimensional vectors are all supported.

Note

2D vectors are supported in NX Add-On 5.1.x and later.


  • nn::util::Vector2fType
  • nn::util::Vector3fType
  • nn::util::Vector4fType


Note

These types have been optimized for calculations. As a result, the memory layout and the size can vary, depending on the environment. For this reason, direct access to internal elements is prohibited. There are dedicated macros and functions for accessing those elements.

Also, because calculations in the SIMD unit are optimized, this structure requests memory alignment. Ensure that the requested alignment is fulfilled to allocate these types on the memory.

Matrices

3×2 matrices of three rows and two columns, 4×3 matrices of four rows and three columns, and 4×4 matrices of four rows and four columns are all supported.

Note

3×2 matrices of three rows and two columns are supported in NX Add-On 5.1.x and later.

In addition, both row-major order and column-major order are supported as the layout method of matrix elements in memory.

Quaternions

The use of quaternions is supported.

Quaternion calculations can be implemented by using dedicated functions on nn::util::Vector4fType objects.

Expected Uses

nn::util::VectorType and nn::util::MatrixType have been optimized to perform vector and matrix calculations. Memory layout and size can vary with your environment.

In contrast, nn::util::Float has a memory layout that is always predetermined.

Calculations and results using these types are expected to proceed as follows.

  1. Set data using nn::util::VectorType and nn::util::MatrixType.
    • Use an initialization macro.
    • Use functions to set the various elements.
    • Specify nn::util::Float or an array, and load.
  2. Perform calculations using the functions provided by nn::util::VectorType and nn::util::MatrixType.
  3. Get the results stored in nn::util::VectorType and nn::util::MatrixType.
    • Use functions to access the various elements.
    • Specify the address for nn::util::Float or a float array, and write.

Initializing Vectors and Matrices

nn::util::VectorType and nn::util::MatrixType have been optimized to perform vector and matrix calculations. Memory layout and size can vary with your environment.

Consequently, you cannot directly use initializers on nn::util::VectorType and nn::util::MatrixType because an implementation like that would be environment-dependent.

For that reason, macros that can be used as initializers are defined for vectors and matrices.

  • NN_UTIL_VECTOR_2F_INITIALIZER
  • NN_UTIL_VECTOR_3F_INITIALIZER
  • NN_UTIL_VECTOR_4F_INITIALIZER
  • NN_UTIL_FLOAT_ROW_MAJOR_3X2_INITIALIZER
  • NN_UTIL_FLOAT_ROW_MAJOR_4X3_INITIALIZER
  • NN_UTIL_FLOAT_ROW_MAJOR_4X4_INITIALIZER
  • NN_UTIL_FLOAT_COLUMN_MAJOR_3X2_INITIALIZER
  • NN_UTIL_FLOAT_COLUMN_MAJOR_4X3_INITIALIZER
  • NN_UTIL_FLOAT_COLUMN_MAJOR_4X4_INITIALIZER

In addition, aliases have been prepared for these matrices.

  • NN_UTIL_FLOAT_3X2_INITIALIZER
  • NN_UTIL_FLOAT_4X3_INITIALIZER
  • NN_UTIL_FLOAT_4X4_INITIALIZER
  • NN_UTIL_FLOAT_T3X2_INITIALIZER
  • NN_UTIL_FLOAT_T4X3_INITIALIZER
  • NN_UTIL_FLOAT_T4X4_INITIALIZER

These macros form appropriate initialization lists based on the environment. Coding as follows enables you to initialize vectors and matrices.

nn::util::Vector3fType vector = NN_UTIL_VECTOR_3F_INITIALIZER(0.f, 1.f, 2.f);

Differentiating Between Row-Major Order and Column-Major Order

When it comes to deciding which is better to use: row-major order or column-major order, choose the one that is better suited to your implementation.

The following example shows the process for making this decision.

  • Previous versions used column-major matrices, so for convenience when migrating, choose column-major.
  • Otherwise, choose the option that is faster for the matrices with the memory layout you use, based on the order in which you are accessing elements and the calculations you are performing.

Restrictions

The vector and matrix calculations as provided have the following restrictions.

  • Direct access to vector and matrix type internal elements is prohibited.
    • Because members are environment dependent, direct access to internal elements is prohibited.
  • Implementations that depend on vector and matrix type memory layouts are prohibited.
    • Memory layouts are environment dependent, so implementations that depend on them are prohibited.
