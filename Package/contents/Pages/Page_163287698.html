When installing the NVIDIA graphics driver, the screen goes black and does not recover.

If you select the Quick (recommended) installation option when installing the NVIDIA graphics driver and then switch to the driver, the screen sometimes goes black and does not recover. If the screen does not recover, in some cases you can still use the computer by logging in via remote desktop.

To avoid this problem, select the Custom (advanced) installation option and make sure to select the Clean Install check box.

GLSLC Known Bugs

  • Using GLSLC_DEBUG_LEVEL_G2 may cause corruption in rendered results. This bug is currently being investigated with high priority.
  • A 32-bit executable only allows for assembly dumps.
  • EOL markers must be present in #include files, else a compilation error will occur.
  • Transform feedback varyings specified in shader source do not work correctly.
  • Compute shader data subsections inside a GPU code section do not start with the magic number GLSLC_GPU_CODE_SECTION_DATA_MAGIC_NUMBER.
  • The TX1 versions of the glslc library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms is likely to be functionally correct though, and will be combined in future releases.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:


    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface, and mediump or lowp types on the other side of the interface, could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types qualified with mediump or lowp qualifiers. You can work around this limitation by explicitly constructing float types from the int and uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp in addition to ES shaders regardless of whether the GL_EXT_shader_implicit_conversions extension is used.
    • FP16 math when used with imageStore may cause corruption in certain cases. This issue is currently being investigated with high priority.
    • FP16 math when used with mediump textures may result in corruption in some rare cases. This bug will be fixed in the next release.

VSI Bugs

Cannot handle exceptions thrown in SDK libraries.

The NX32 and NX64 SDK libraries do not support C++ exceptions.
For this reason, when exceptions are thrown in the libraries, they are canceled.

If the code is only propagated to the application, you can enable C++ exceptions at build to make it possible to handle exceptions.

Optimized code may not be debugged correctly.

In some cases, optimized code cannot be debugged correctly, such as when it stops on breakpoints placed in functions with inline expansion.

Using precompiled headers and defining string macros for the preprocessor causes a build error.

When using precompiled headers, defining string macros such as STRING_MACRO="foo” for the preprocessor causes a build error.

This bug will be fixed in NX Add-On 0.9.0.

fs Bugs

Multi-byte characters cannot be used with save data entry names.

UTF-8 entry names are supposed to be available for use, but specifying entry names that contain non-ASCII characters does not work.

nn::fs::DeleteDirectoryRecursive() fails when deleting directories containing many files.

In the file system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot()), when a directory including around 255 or more files is deleted with nn::fs::DeleteDirectoryRecursive(), it fails with nn::fs::ResultUnexpected.

Cannot open numerous files simultaneously.

The number of files that can be open at any time is restricted as follows.

File System Maximum Number of Open Files
File system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot())

338 (32-bit environment)

182 (64-bit environment)
Resource data (nn::fs::MountRom())

338 (32-bit environment)

182 (64-bit environment)
Save data (nn::fs::MountSaveData()) 30

The function fails with nn::fs::ResultAllocationMemoryFailed if you attempt to open more than the maximum number of files.
We plan to improve the number of files that can be opened at the same time.

In save data, adding entries to directories that have a lot of entries takes a long time.

When creating entries with nn::fs::CreateFile() or nn::fs::CreateDirectory() in directories that include 512 or more entries, it takes much longer than cases with 511 or fewer entries.

nn::fs::ReadFile() or nn::fs::WriteFile() may disconnect from Target Manager.

There is a bug where the connection to the Target Manager may be disconnected (the status in Target Manager becomes "Disconnected") when executing nn::fs::ReadFile() or nn::fs::WriteFile() through MountHost(), MountHostRoot(), or MountRom() in raw mode.

The conditions for this problem occurring are based on the size of the file loaded in nn::fs::ReadFile() or nn::fs::WriteFile() and the size of previously loaded files, but the details are currently being investigated.

If this problem occurs repeatedly, avoid it by changing the buffer size specified in nn::fs::ReadFile() or nn::fs::WriteFile().

hid Bugs

USB keyboards are occasionally not recognized when connected to the Nintendo Switch dock.

Due to a USB driver issue, USB keyboards are occasionally not recognized when connected to the Nintendo Switch dock.

Resetting the development hardware typically fixes this problem.

LDN Bugs

Immediately after a station connects to the network, in some cases the packets sent by the station and addressed to the station keep getting lost in transit.

In some cases, immediately after a station has called the nn::ldn::Connect() function and connected to an access point, the packets sent by the station and the packets addressed to that station keep getting lost in transit for several seconds. This problem occurs more frequently when the wireless environment is poor, so the workaround is to set a wireless channel that has a strong signal.

This problem will be fixed in an upcoming release.

In rare cases, functions in the ldn library never return.

In rare cases, functions in the ldn library remain blocked and never return. This bug has been confirmed to occur in this version for nn::ldn::Connect(), nn::ldn::Disconnect(), and nn::ldn::Scan(). Other functions may also be affected.

This problem will be fixed in an upcoming release.

The nn::ldn::Scan() function cannot be called while a station is connected to a network.

If the nn::ldn::Scan() function cannot be called while a station is connected to a network, the ldn library stops working normally. Do not call the nn::ldn::Scan() function while connected to a network. The nn::ldn::Scan() function can be called normally when not connected to a network or after building a network via an access point.

This problem will be fixed in an upcoming release.

socket Bugs

If the network interface goes down during TCP communications, the function that is currently running never returns.

If the network interface goes down during TCP communications, the socket library function that is currently running never returns (regardless of whether you are using wireless or wired communications). As a workaround, call nn::socket::Shutdown() on another thread to unblock the library.

This problem will be fixed in an upcoming release.

USB-Ethernet adapters are occasionally not recognized when connected to the Nintendo Switch dock.

Due to an issue with the Nintendo Switch dock hardware, USB-Ethernet adapters are occasionally not recognized when connected to the Nintendo Switch dock. If an adapter is not recognized, disconnect the development hardware from the Nintendo Switch dock, wait approximately one minute, and then reconnect the development hardware.

This problem will be fixed in an upcoming release.

The socket library occasionally returns different errors when used on Windows and when used on the development hardware.

Different errors are sometimes returned when the socket library is used in the Windows environment and when used in the development hardware environment, even when the error happened for the same reason.

This problem will be fixed in an upcoming release.

After a large number of packets are received while using wireless communication, sometimes packets can no longer be sent.

In poor communication environments, receiving more than 2000 packets per second sometimes depletes the communication buffer and makes it impossible to send any more packets. This bug is largely dependent on the communication environment. The 2000 packets number is only an estimation, not an exact figure.

This problem will be fixed in an upcoming release.

vi Bugs

The default visibility setting when creating layers is true.

While the default visibility when creating layers is currently true, the default will be false in future SDK versions.

To display the layer, make sure that you call nn::vi::SetLayerVisibility(pLayer, true) and set it to a displayable state.

RGB values are not displayed within the correct range on the TV.

Currently, the HDMI output via the Nintendo Switch dock is RGB limited range output, with scaling of the RGB components. However, due to a bug in the Nintendo Switch dock firmware, the HDMI metadata signal (AVI InfoFrame) is output as full range data. As a result, if the dock is connected to a TV that supports full-range input, colors may not be displayed correctly.

This bug only occurs when using HDMI output via the Nintendo Switch dock. This bug does not affect the built-in LCD screen because full-range data is always output to the LCD screen.

Even if the HDMI cable is connected, video may remain on the LCD and is not output to the TV.

Various hardware problems (such as the SDEV dock not being turned on) cause the video to remain on the LCD and prevent output to the TV even though the HDMI cable is connected.

For more information, see the Troubleshooting section of the SDEV Operations Manual.

audio Bugs

On the development hardware, when sound is being produced with nn::audio and the application stops due to a breakpoint or for other reasons, the sound playback does not stop.

On development hardware, when an application is paused while sound is being produced, the sound does not stop, and the playback continues.

This problem will be fixed in an upcoming release.

Sometimes, operations become unstable when accessing the area surrounding resource data used by nn::audio on development hardware.

When the memory within a 4-KB range before and after the memory used for the audio renderer is specified as the destination for processing that requires writing, such as file loading, the operation may become unstable.

The following buffers are affected by this problem.

  • Buffers specified for WaveBuffer.buffer and WaveBuffer.pContext.
  • Memory storing the data passed to pParameter by the nn::audio::AcquireVoiceSlot() function
  • The work buffer specified for buffer in the nn::audio::AddDelay() and nn::audio::AddReverb() functions
  • The work buffers specified for pSendBuffer and pReturnBuffer in the nn::audio::AddAux() function

To avoid this problem, switch to using the memory pool (nn::audio::MemoryPoolType).

For more information, also see File System Troubleshooting.

Some TVs do not play audio data that was output to the TV over HDMI via the Nintendo Switch dock.

Depending on the TV being used, occasionally the TV does not play audio data that was output to the TV over HDMI via the Nintendo Switch dock. This problem will be fixed in an upcoming release.

DevMenu Bugs

Applications started from DevMenu or DevMenuCommand cannot be closed with Visual Studio or RunOnTarget.

Applications started from DevMenu or DevMenuCommand cannot be closed with Visual Studio or RunOnTarget. To close these applications, you must disconnect and reinsert the power cable to restart the hardware.

This problem will be fixed in an upcoming release.

Compiler Bugs

When using gold as the linker, using the ld option occasionally results in linking errors not being detected until runtime.

When you specify -fuse-ld=gold.exe to use the gold linker (or select Use gold Linker in VSI), the ld option sometimes causes linking errors to not be detected until runtime.

ui2d Bugs

The NX development hardware version of LayoutViewer cannot display layouts in non-linear gamma mode.

The NX development hardware version of LayoutViewer cannot display layouts in non-linear gamma mode. This version of LayoutViewer always displays layouts in linear gamma mode.

This restriction will be removed in a future release.

perf Bugs

In the NX Windows version of perf, Tick values measured using nn:perf::GpuMeter do not correspond to the Tick values returned by the nn::os::GetSystemTick() function.

When measuring Tick values using nn:perf::GpuMeter, the Tick values stored in nn:perf::LoadMeterBase::Section::begin and nn:perf::LoadMeterBase::Section::end are slightly different from the Tick values measured using nn::perf::CpuMeter or the nn::os::GetSystemTick() function.
However, the speed at which the counter ticks is the same for both meters, so adjusting the Tick values by the interval (end - begin) (which represents the processing time) makes it possible to use GpuMeter and CpuMeter interchangeably.

This bug is limited to the NX Windows version of perf.
On the development hardware, Tick values measured using GpuMeter correctly correspond to Tick values measured using CpuMeter and Tick values returned by nn::os::GetSystemTick().

os Bugs

In general programs that output images, the following error is displayed: “nn::GetArgc() is deprecated. Use nn::os::GetHostArgc().”

As of this SDK release, the functions for working with the command-line arguments used to specify the host have changed as follows.

  • nn::GetArgc -> nn::os::GetHostArgc
  • nn::GetArgv -> nn::os::GetHostArgv

The old functions will remain available during the transition period, but this warning message is now displayed when you use them.
Currently, this warning is only displayed for the samples that output images to the screen because some of these samples use the graphics library.

This behavior will be fixed in a future release.

hid Bugs

If you close a program while the controller is vibrating, it keeps vibrating indefinitely.

If you close a program while the controller is vibrating, the controller keeps vibrating after the program is closed in the SDEV environment.
To avoid this bug, simply make sure to stop vibration before closing a program.

If this bug occurs and the controller starts vibrating indefinitely, restart the program and use a function such as nn::hid::SendVibrationValue() to explicitly stop the vibration.
Resetting the controller also stops the vibration.

3D Bugs

The Edit and Town samples in G3dDemo sometimes freeze.

In the Edit and Town samples in G3dDemo, the main thread sometimes gets blocked by the communication thread, causing the application to freeze.

This behavior will be improved in a future release.

NintendoWare Graphics Bugs

Some of the samples do not work when output to a full HD TV due to a memory shortage.

Some of the samples (such as VfxSimple, VfxCustomShader, VfxManualEmitterSet, VfxViewerSimple, FontSimple, Ui2dSimple, and Ui2dUserShader) do not work when output to a full HD TV due to a memory shortage.

This behavior will be improved in a future release.
