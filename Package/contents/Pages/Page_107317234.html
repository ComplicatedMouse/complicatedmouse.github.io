When the NX add-on is installed, sample resource loading fails.

For samples where resource conversion occurs during the sample project build, an error occurs when attempting to load incompatible resources.
This problem only reproduces in systems where the NX add-on is installed.

As a workaround, build the sample in Visual Studio before executing it.

The demo is dark.

Copying factoring in sRGB format does not occur during processing when the current status is nn::gfx::Queue::CopyToScanBuffer() (to be deleted) or nn::gfx::CommandBuffer::CopyImage(). As a result, the display in the demo specifying sRGB in the scan buffer format is dark. This issue will be fixed in future releases.

When using automatic operations on an application running on a Windows environment, the gamepad connection state default value reverts to a disconnected state.

When using automatic operations by Hid Shell in a Windows environment, the gamepad connection state disconnects.

You can avoid this problem by running the following command in advance.

Tools\CommandLineTools\Hid\HidShell.exe send-event Xpad Basic0 PowerState OnBattery
Tools\CommandLineTools\Hid\HidShell.exe send-event Xpad Basic1 PowerState OnBattery
Tools\CommandLineTools\Hid\HidShell.exe send-event Xpad Basic2 PowerState OnBattery
Tools\CommandLineTools\Hid\HidShell.exe send-event Xpad Basic3 PowerState OnBattery

This problem does not occur for applications running on development hardware.

GamePad disconnection cannot be detected via Input Director.

After the connection with Input Director is established, the value of the nn::hid::BasicXpadAttribute::IsConnected flag of bit collection nn::hid::BasicXpadState::attributes is always true.

Memory heap must be opened when using graphics API.

For the NX32 and NX64 builds, the NVN library allocates memory blocks internally, so you must use nninitStartup() to allocate at least 18 MB. The required size depends on the amount of resources used.

If the open region is insufficient, an error log like the following is output.

NvOsHorizonBlockAlloc: alloc 0x00200000 bytes from heap size 0xcc600000 failed

To avoid this error, leave a sufficient area for NVN as follows.

#include <nn/os.h>
#include <nn/init.h>
#include <nn/util/util_BitUtil.h>
 
// nninitStartup() is invoked before calling nnMain().
//
extern "C" void nninitStartup()
{
    nn::os::MemoryInfo memInfo;
    nn::os::QueryMemoryInfoForDebug(&memInfo);

    size_t size = static_cast<size_t>(memInfo.totalAvailableMemorySize) - memInfo.totalUsedMemorySize;
    size = nn::util::align_down(size , nn::os::MemoryBlockUnitSize);

    nn::Result result = nn::os::SetMemoryHeapSize(size);  // Sets a memory heap using the largest allocatable size.
    NN_ASSERT( result.IsSuccess() );
 
    const size_t MallocMemorySize = size - 18 * 1024 * 1024; // Leave the required region for NVN.
    uintptr_t address;
    result = nn::os::AllocateMemoryBlock( &address, MallocMemorySize );
    NN_ASSERT( result.IsSuccess() );
    nn::init::InitializeAllocator( reinterpret_cast<void*>(address), MallocMemorySize );
 }

In a future release, we will fix this problem so that handling with nninitStartup() is not required, and make it so that an allocator used to allocate dynamic memory can be specified.

Some graphics APIs do not release allocated memory.

For the NX32 and NX64 builds, some graphics APIs do not release memory that was allocated internally with malloc. This memory is allocated when each API is first called and not released until the application ends. If the application has an independent memory leak detection mechanism, this memory may be detected as a memory leak. The memory that does not get released is only allocated when the API is first called, so repeated calling does not cause the available memory to be reduced every time.

The following list includes some of the affected APIs.

  • nn::vi::OpenDefaultDisplay()
  • nvnDeviceInitialize()
  • nn::gfx::Initialize()
  • nn::gfx::Device::Initialize()
  • nn::gfx::SwapChain::Initialize() etc.

We plan to provide a method for releasing the allocated memory internally in a future release.

Some graphics APIs get slower depending on the number of memory pools.

When the NVN library is used to build many memory pools, some graphics APIs get slower depending on the number of memory pools.

  • nvnQueueFlush()
  • nvnQueueFinish()
  • nvnQueuePresentTexture()
  • nvnTextureInitialize()

We plan to fix this problem in a future release.

Regardless of the fix, allocating memory pools entails execution cost and footprint cost, so we recommend minimizing the number of memory pools to optimize.

GLSLC Known Bugs

  • A 32-bit executable only allows for assembly dumps.
  • EOL markers must be present in #include files or else a compilation error will occur.
  • Transform feedback varyings specified in shader source do not work correctly.
  • Compute shader data subsections inside a GPU code section do not start with the magic number GLSLC_GPU_CODE_SECTION_DATA_MAGIC_NUMBER.

Multi-byte characters cannot be used with save data entry names.

UTF-8 entry names are supposed to be available for use, but specifying entry names that contain non-ASCII characters does not work.

Filenames and directory names including "@" are not handled correctly.

When the file system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot()) uses filenames or directory names that include @, it does not work correctly.

nn::fs::DeleteDirectoryRecursive() fails when deleting directories containing many files.

In the file system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot()), when a directory including around 255 or more files is deleted with nn::fs::DeleteDirectoryRecursive(), it fails with nn::fs::ResultUnexpected.

Cannot open numerous files simultaneously.

The number of files that can be open at any time is restricted as follows.

File System Maximum Number of Open Files
File system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot())

338 (32-bit environment)

182 (64-bit environment)
Resource data (nn::fs::MountRom())

338 (32-bit environment)

182 (64-bit environment)
Save data (nn::fs::MountSaveData()) 30

The function fails with nn::fs::ResultAllocationMemoryFailed if you attempt to open more than the maximum number of files.
We plan to improve the number of files that can be opened at the same time.

In save data, adding entries to directories that have a lot of entries takes a long time.

When creating entries with nn::fs::CreateFile() or nn::fs::CreateDirectory() in directories that include 512 or more entries, it takes much longer than cases with 511 or fewer entries.

Multiple instances of the same application can be started at the same time on development hardware.

After running an application on development hardware using RunOnTarget as in RunSdevSample.bat, if you then start that application from VSI, you end up with two instances of the application running at the same time.

This situation is not supported. If the application you started with RunOnTarget has not terminated, perform a target reset from Target Manager.

vsi Bugs

When opening the solution file, a "Microsoft (R) Visual C++ Package Server has stopped working" error message is displayed.

Visual C++ Package Server is a program for IntelliSense, so this bug makes it impossible to use IntelliSense.
You can still run the debug itself.

Cannot handle exceptions that occur in SDK libraries.

The NX32 and NX64 SDK libraries do not support C++ exceptions.
For this reason, when exceptions occur in the libraries, they abort.

If the code is only propagated to the application, you can enable C++ exceptions at build to make it possible to handle exceptions.

Breakpoint and step over features cannot be used unless debug information is generated on development hardware.

There is a bug where the breakpoint and step over features cannot be used unless debug information is generated on development hardware.
For Release builds, this information is not generated by default. If you want to use the breakpoint or step over features, go to Configuration Properties > C/C++ > General > Generate Debug Data in the project to generate debug information.

Optimized code may not be debugged correctly.

In some cases, optimized code cannot be debugged correctly, such as when it stops on breakpoints placed in functions with inline expansion.

Sometimes, the call stack cannot be obtained correctly.

The call stack cannot be obtained when it stops during step over after stopping due to an access violation or debug stop.

Sometimes, items disappear from the project properties.

If the platform is changed frequently, the items under the project Properties > Configuration Properties may disappear.
Reopen the solution file or restart Visual Studio.

Cannot refer to structures with automatic variables.

Currently, it is not possible to refer to a structure correctly in the automatic variable window.

Windows platform may not behave correctly.

Failure sometimes occurs when starting or running debug or generating new projects.
Restart Visual Studio.

Sometimes, items disappear from the project properties.

Sometimes the items under Configuration Properties disappear from the project properties.
Restart Visual Studio.

"Unable to load ***.kip" displayed.

Messages of this type are displayed, but they are the result of a message bug. NintendoSDK does not support files with the *.kip filename extension.

"Unable to create hard link for <SDK root path>/Libraries/NX-NXFP2-a32/Develop/nnSdk.nso" displayed during NX platform build.

NX VSI attempts to hard link the DLL and other files provided by the SDK at build. However, it cannot create hard links to files on the network or in other volumes.
As a result, if the output destination of the build output and the SDK install destination are different volumes, this error log is displayed.
However, even if NX VSI fails to create a hard link and this error log is displayed, the target file is copied instead so that this failure does not cause the build to fail.
We plan to improve the log output in a future update.

Cannot link large files.

Linking files fails if one of the object or library files passed to the linker exceeds 100 MB.

This problem occurs because of a failure to relocate when, during relocation, rounding down occurs to fit to R_AARCH64_CALL26.

For more information, see https://sourceware.org/bugzilla/show_bug.cgi?id=18668.

One of the reasons files become large is debugging information. Specifying -g0 for the compile option allows you to avoid generating debugging information.

Repeatedly running an application on the NX development hardware from VSI can, in rare situations, cause VSI to stop responding during the loading screen.

Repeatedly running an application on the NX development hardware from VSI can, in rare situations, cause VSI to stop responding during the loading screen.
When this happens, restart Visual Studio.

os Library Bugs

Sometimes, an assertion failure occurs when nn::os::InitializeFiber() is called.

When nn::os::InitializeFiber() is called, it sometimes stops with the assertion failure "Internal error: Failed to create the stack area alias."

This problem occurs due to a bug in the os library when several conditions coincide due to relationships such as the core numbers and priorities of the side generating the thread and the generated side. You can work around this problem by specifying nn::os::FiberFlag_NoStackGuard as the sixth argument for nn::os::InitializeFiber(), but in this case no stack guard is created.

This problem is expected to be fixed in the next release.

nn::os::SetThreadCoreMask() before starting a thread may not be applied immediately.

When nn::os::SetThreadCoreMask() is issued before nn::os::StartThread() in a thread that has just been generated, core mask and others settings for that thread are sometimes not applied immediately.

This problem occurs due to a bug in the os library when several conditions coincide due to relationships such as the core numbers and priorities of the side generating the thread and the generated side. This problem can be avoided by running the target thread even a little after issuing nn::os::SetThreadCoreMask(). If urgent handling is required, please contact Nintendo.

We plan to fix this problem in the next release.

socket Bugs

A maximum of 31 threads can perform simultaneous waiting.

When 32 or more threads are simultaneously waiting for an API, the system stops.

If this problem occurs, reset the SDEV.

We plan to fix this problem in the next release.

A data access exception is thrown when using GetAddrInfo() repeatedly.

When you continue to use GetAddrInfo() and do not use FreeAddrInfo(), a data access exception is thrown at around the 2700th time it is called.

We plan to fix this problem to return EAI_MEMORY.

When USB Ethernet is disconnected, TCP APIs being called do not return.

When a USB Ethernet device is disconnected, blocked TCP APIs do not return.

We plan to fix this problem in the next release.

When an application is closed without calling nn::socket::Finalize(), resources are not released.

In the socket library, when you close an application without calling nn::socket::Finalize(), such as by stopping the application from Target Manager or performing a forced exit with Abort, the following problems occur when you next run the application.

If this problem occurs, reset the SDEV.

  • Bound port numbers cannot be reused.
  • Memory and internal resources leak without being released.

We plan to fix this problem in the next release.

Target Manager Bugs

Right-clicking the target and selecting Power On in Target View causes a problem if the target is already turned on.

When development hardware is connected, Target Manager freezes.
When this symptom occurs, restart Target Manager and run Reconnect to Target.
When no development hardware is connected, it becomes impossible to connect to development hardware.
Right-click the target in Target View and select Reset to restart the target.

These problems do not occur if the target is turned off.

Targets are enumerated with the same name in the name column in the Add Target window.

Use the IP address to identify them.

Some NCA files cannot be run with Load Executable File.

Currently, the following bugs have been found in Target Manager's Load Executable File feature, so it does not support NX Add-On 0.4.0.
Official support is planned with future releases that fix the bugs.

In NX Add-on 0.4.0, use VSI or RunOnTarget.

vi Bugs

The Windows environment is not supported.

nn::vi is currently only available on NX development hardware. The Windows environment is not supported.

We plan to support the Windows environment in the next release.

When outputting to the LCD, downscaling with nn::vi::SetLayerSize does not work correctly and displays a black screen.

When the system is outputting to the LCD and the application uses nn::vi::SetLayerSize to downscale, the following error is output to the log from nvnflinger and the LCD displays a black screen.

hwcomposer (42): Failed to flip on display 0: 52 (No error information)

We plan to fix this problem in the next release.

Cannot change whether Layer is displayed.

The Layer display settings cannot be changed, even if nn::vi::SetLayerVisibillity or nn::vi::SetDisplayEnabled is called. The created Layer is permanently stuck with display enabled.

We plan to fix this bug in the next release so that the display can be enabled or disabled.

Alpha settings cannot be configured correctly.

The Alpha settings cannot be configured, even if nn::vi::SetLayerAlpha is called.

We plan to fix this problem in the next release.

It is possible to create a Layer using a pixel format that is not supported on NX.

It is possible to create a Layer using a pixel format that is not supported by nvn::Window.

Use either nn::vi::PixelFormat_Rgba8888 or nn::vi::PixelFormat_Rgbx8888.

We plan to make it so that pixel formats other than these two cannot be used in the next release.

audio Bugs

On the development hardware, when sound is being produced with nn::audio and the application stops due to a breakpoint or for other reasons, the sound playback does not stop.

On development hardware, when an application is paused while sound is being produced, the sound does not stop, and the playback continues.

This problem will be fixed in an upcoming release.

nn::audio HDMI output does not work on development hardware.

On development hardware, audio produced with nn::audio cannot be output with HDMI. Currently only system speaker and headphone outputs are supported.

This problem will be fixed in an upcoming release.

Sometimes, operation becomes unstable when accessing the area surrounding resource data used by nn::audio on development hardware.

When the memory within a 4-KB range before and after the memory used for the audio renderer is specified as the destination for processing that requires writing, such as file loading, the operation may become unstable.

The following buffers are affected by this problem.

  • Buffers specified by nn::audio::WaveBuffer.buffer
  • Buffers using various effects
  • Buffers allocated as the area passed to pParameter of nn::audio::AcquireVoiceSlot()

To avoid this problem, verify the following prior to access.

  • For nn::audio::WaveBuffer.buffer and the pParameter of nn::audio::AcquireVoiceSlot(), the Voice using their buffers is in Stop status or released.
  • For the buffers used in various effects, those effects are disabled.

For additional information, see the file system troubleshooting.

We plan to provide a countermeasure for this problem in the next release.

Sometimes noise is mixed into the Aux effect of nn::audio on development hardware.

In some cases, noise is mixed in with the results of effect processing by the Aux effect.

This problem will be fixed in an upcoming release.

nn::audio::AudioRenderer cannot be used after using nn::audio::AudioOut.

nn::audio::AudioRenderer cannot be used after using nn::audio::AudioOut.

This problem will be fixed in an upcoming release.

The system volume goes to maximum when nn::audio::OpenAudioOut() and nn::audio::OpenAudioRenderer() are called.

When nn::audio::OpenAudioOut() and nn::audio::OpenAudioRenderer() are called, the volume on the development hardware goes to maximum regardless of the volume button settings.

This problem will be fixed in an upcoming release.

Restrictions When Using nn::audio::AudioOut on the NX64 Environment

The following restrictions apply when using nn::audio::AudioOut on the NX64 environment.

  • There is a limit to the maximum number of buffers that can be registered.
  • One instance of AudioRenderer is implicitly opened internally.

This problem will be fixed in an upcoming release.

Invalid memory access occurs sometimes with nn::atk::SoundSystem::AppendEffect() and ClearEffect().

When adding or deleting nn::atk::EffectDelay or nn::atk::EffectReverb with the APIs in the heading, unintended memory rewriting can occur on rare occasions. The likelihood of occurrence increases if adding and deleting is performed frequently.

This problem will be fixed in an upcoming release.

User effects using nn::atk::EffectBase are not supported.

User effects using nn::atk::EffectBase cannot currently be used. The API is also temporary, and we plan to fix it later.

This problem will be fixed in an upcoming release.

The program sometimes stops on NN_ABORT when in-game editing with an application using nn::atk::viewer on development hardware.

When a development hardware application with the nn::atk::viewer library embedded is connected to SoundMaker and in-game editing is performed using wave sounds or sequence sounds, the program sometimes stops with NN_ABORT.

This problem will be fixed in an upcoming release.

codec Bugs

Restrictions for the OpusEncoder output path specification option (-o)

When specifying the -o option to output to the current directory with OpusEncoder, the current directory must be clearly indicated as follows.

$ OpusEncoder.exe -o ./output.opus input.wav

In the future, we plan to make it so that you can output to the current directory as follows.

$ OpusEncoder.exe -o output.opus input.wav

gfx Bugs

glew is required for build and execution on a PC.

When nvn is used as a low-level graphics API on PC, the glew library, which is normally not required, becomes required.
Link glew32.lib when building applications that use gfx, and place glew32.dll in the search path when executing.
This problem will be fixed in an upcoming release.

3D Bugs

When executing and closing the G3dDemo Town demo repeatedly, memory becomes insufficient and a forced exit occurs.

When repeatedly executing and closing the G3dDemo Town demo built from Visual Studio solutions, memory becomes insufficient and a forced exit occurs.

This problem will be fixed in an upcoming release.

Aborts when a file of 4 MB or larger is loaded to an application with nn::g3d::viewer embedded from 3DEditor

When an application with nn::g3d::viewer embedded is built on NX32 or NX64, 3DEditor is connected, and a binary of 4 MB or larger is sent from 3DEditor, an abort occurs in nn::g3d::viewer.

For example, this problem occurs when assigning a texture of 4 MB or larger to a model.

This problem will be fixed in an upcoming release.

When the A Button is pressed repeatedly while a model is selected in the G3dDemo Edit demo, the attachment state between 3DEditor and G3dDemo becomes inconsistent.

When the A Button is pressed repeatedly while a model is selected in the G3dDemo Edit demo, sometimes it becomes attached in 3DEditor but unattached in the application.

This problem will be fixed in an upcoming release.


vfx Bugs

The NX32 development hardware effects viewer does not start.

The NX32 development hardware effects viewer (EffectViewerGfx.nca) does not start. If you want to use a development hardware effects viewer, use the one that was built for NX64.

This problem will be fixed in an upcoming release.

Standard Library Bugs

The strtod function does not work correctly for the NX64 build.

When running the NX64 build, when a string is passed to the strtod function, the string is converted to a value other than the value of the expected negative number.

Conversion is correct, however, when passing a string for a conversion result to a positive number.
