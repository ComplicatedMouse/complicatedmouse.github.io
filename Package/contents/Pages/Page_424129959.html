Changing the System Settings

Use the settings or factoryreset command to configure the following settings.

  • Language
  • Region setting
  • Platform region setting
  • TV Settings
  • Sleep settings
  • Security Settings

Language

Info

Available in NintendoSDK Firmware for NX 10.0.0-3.0 and later.

$ RunOnTarget.exe DevMenuCommand.nsp -- settings
usage: DevMenuCommand settings set-language <language> [--no-confirmation]
       DevMenuCommand settings get-language

Language Settings

Use the set-language and get-language subcommands to configure the language settings and get the language status.
The following languages are available. You can specify any of the following language options for your configuration. For example, to set American English, specify AmericanEnglish or es-US.

  • AmericanEnglish (en-US)
  • BritishEnglish (en-GB)
  • Japanese (ja)
  • French (fr)
  • German (de)
  • LatinAmericanSpanish (es-419)
  • Spanish (es)
  • Italian (it)
  • Dutch (nl)
  • CanadianFrench (fr-CA)
  • Portuguese (pt)
  • Russian (ru)
  • SimplifiedChinese (zh-Hans)
  • TraditionalChinese (zh-Hant)
  • Korean (ko)
  • BrazilianPortuguese (pt-BR) (NintendoSDK Firmware for NX 11.0.0-6.0 and later)

After the language is changed, the development hardware automatically restarts.
Use the --no-confirmation option to process the language setting without button input from the controller.

# Language settings
$ RunOnTarget.exe DevMenuCommand.nsp -- settings set-language <AmericanEnglish|BritishEnglish|Japanese|French|German|LatinAmericanSpanish|Spanish|Italian|Dutch|CanadianFrench|Portuguese|Russian|SimplifiedChinese|TraditionalChinese|Korean|BrazilianPortuguese>
*** Push A (Left) button to switch language.
*** Device will be rebooted after switching language.
*** If you want to cancel, push B (Up) button.
[SUCCESS]
 
# Get language settings
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-language
AmericanEnglish
[SUCCESS]

Platform Region Setting

Info

Available in NintendoSDK Firmware for NX 9.1.0-1.0 and later.

$ RunOnTarget.exe DevMenuCommand.nsp -- factoryreset --help
usage: DevMenuCommand factoryreset do-switch-platform-region <global|china> --no-confirmation

$ RunOnTarget.exe DevMenuCommand.nsp -- settings
usage: DevMenuCommand settings get-platform-region

Switching the Platform Region

Use the factoryreset command. When the command succeeds, the development hardware automatically restarts and is reverted to its default state.
Use the --no-confirmation option to switch without button input from the controller.

$ RunOnTarget.exe DevMenuCommand.nsp -- factoryreset do-switch-platform-region <global|china>
*** Push L (SL) and R (SR) buttons for 3 seconds to switch platform region.
*** If you want to cancel, push B (Up) button.
*** Switch the platform region to '<global|china>'.
*** Initializing the system... Done.
*** The device will restart.
[SUCCESS]

When the platform region is changed to China, the two other settings are also changed and fixed to the following values.

Option Setting Value
Language Simplified Chinese
Region China
Time Zone Asia/Shanghai
Note

The execution result of this command may be [Failure], as shown, but it is a consequence of restarting the development hardware.
The platform region has completed switching, so you can ignore these displays of [Failure].

  • Exit( Result: Failure, ExitKind: OccuredDebugEvent )
  • Exit( Result: Failure, ExitKind: Disconnected , Source: LogSession, Address: 127.0.0.1:60142 )

When ExitKind: OccuredDebugEvent occurs, it has been verified that development hardware shuts down without restarting. When this happens, use the POWER Button to turn on the power.

Getting the Platform Region

# Getting the Platform Region
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-platform-region
global
[SUCCESS]

Sleep Settings

Auto-Sleep Settings

Use the set-auto-sleep and get-auto-sleep subcommands for sleep settings and getting the status of the settings.

$ RunOnTarget.exe DevMenuCommand.nsp -- settings
usage: DevMenuCommand settings set-auto-sleep [--handheld <1m|3m|5m|10m|30m|never>] [--console <1h|2h|3h|6h|12h|never>] [--mediaplay <enabled|disabled>]
       DevMenuCommand settings get-auto-sleep --<handheld|console|mediaplay|all>
       DevMenuCommand settings set-wakes-at-power-state-change <enabled|disabled>
       DevMenuCommand settings get-wakes-at-power-state-change

Handheld Mode Auto-Sleep Setting

Use the --handheld option for auto-sleep settings when the unit is operating as a handheld device. The following trigger times are available.

  • 1 minutes
  • 3 minutes
  • 5 minutes
  • 10 minutes
  • 30 minutes
  • No Auto-Sleep
# Sets auto-sleep setting for handheld mode.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings set-auto-sleep --handheld <1m|3m|5m|10m|30m|never>
[SUCCESS]
 
# Gets auto-sleep settings for handheld mode.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-auto-sleep --handheld
10m
[SUCCESS]

TV Mode Auto-Sleep Setting

Use the --console option to set the auto-sleep setting to use when the unit is operating as a console device. The following trigger times are available.

  • 1 hours
  • 2 hours
  • 3 hours
  • 6 hours
  • 12 hours
  • No Auto-Sleep
# Sets auto-sleep setting for console mode.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings set-auto-sleep --console <1h|2h|3h|6h|12h|never>
[SUCCESS]
 
# Gets auto-sleep settings for console mode.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-auto-sleep --console

[SUCCESS]

Content Playback Auto-Sleep Setting

Use the --mediaplay option for auto-sleep settings during video playback. When this setting is enabled, the system will not sleep until four hours have elapsed.

# Sets the auto-sleep setting during content playback.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings set-auto-sleep --mediaplay <enabled|disabled>
[SUCCESS]

# Gets the auto-sleep setting during content playback.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-auto-sleep --mediaplay
enabled
[SUCCESS]

Setting to Wake Up From Sleep Mode When the Power State Changes

Use the set-wakes-at-power-state-change subcommand to set whether the system wakes up from sleep mode when the power state changes, and the get-wakes-at-power-state-change subcommand to get the setting.

If disabled, the system no longer wakes from sleep mode when it is undocked, or if the AC adapter is connected or disconnected, while the system sleeps.

Info

Some of the behavior during the disabled state for SDEV (such as displaying the icon when entering sleep mode again) differs from the retail device. Use the EDEV to check the behavior equivalent to the retail device.

# Sets waking up from sleep mode when the power state changes.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings set-wakes-at-power-state-change <enabled|disabled>
[SUCCESS]
 
# Gets settings for waking up from sleep mode when the power state changes.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-wakes-at-power-state-change
enabled
[SUCCESS]

Device Lock Setting

Info

Available in NintendoSDK Firmware for NX 11.0.0-3.0 and later.

By enabling device lock, DevMenu and the HOME Menu display a lock screen and do not accept operations when the hardware is started or awakened from sleep mode.
The lock screen is displayed until the controller buttons configured for unlocking are pressed simultaneously. If the screen is not unlocked within a certain amount of time, the system shuts down.

Info

The device may be unlocked at the following times.

  • For DevMenu, the device displays "Locked" on startup and when waking from sleep mode, and accepts controller button input at this time. When the device is unlocked, "Unlocking" is displayed and it then transitions to DevMenu.
  • For the HOME Menu, controller button input is accepted after the Nintendo Switch logo disappears after startup, or during the 10 seconds when the screen goes dark after pressing the POWER Button when waking from sleep mode. When the device is unlocked, it transitions to the entrance.
Note
  • Device lock only restricts operations in DevMenu and the HOME Menu. To protect application data more securely, we also recommend following the procedure in AuthoringTool > Creating NSP Files for Secure Startup.
  • If you forgot the controller button that was configured for disabling the lock, disable the device lock by initializing the firmware.

Specify two or more of the following controller buttons to press simultaneously when unlocking.

--key String Description
A A Button
B B Button
X X Button
Y Y Button
Left Directional buttons
Up
Right
Down
Plus + Button
Minus - Button
L L Button
R R Button
ZL ZL Button
ZR ZR Button
L Stick Press the L Stick
R Stick Press the R Stick 
# Gets the enabled/disabled status of device lock.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings get-device-lock-enabled
disabled
[SUCCESS]

# Enables device lock and specifies the controller buttons to press simultaneously when unlocking with the --key option.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings enable-device-lock --key A,B,L,Down
[SUCCESS]

# Disables device lock and specifies the controller buttons to press simultaneously when enabling device lock with the --key option.
$ RunOnTarget.exe DevMenuCommand.nsp -- settings disable-device-lock --key A,B,L,Down
[SUCCESS]
