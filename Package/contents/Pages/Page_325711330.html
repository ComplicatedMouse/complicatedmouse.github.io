Controlling the Operation Mode

You can change the control policy for the operation mode for development purposes.

Getting the Operation Mode Control Policy Setting

$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-operation-mode-policy
auto
[SUCCESS]

Setting the Operation Mode Control Policy

$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-operation-mode-policy auto
[SUCCESS]
  • If the operation mode policy is auto, the operation mode will change based on whether the system is docked or undocked.
  • If the policy is handheld, the operating mode will be fixed to handheld mode.
  • If the policy is console, the operation mode will be fixed to TV mode.

To update the settings after changing them, restart the development hardware.

Info

Use the suppress-auto-kill option in RunOnTarget to switch the operating mode without closing the currently running application.
For this option to operate, the application must originally be started from Target Manager or from RunOnTarget, and with the no-wait option.

Example
Start with the no-wait option
RunOnTarget.exe --no-wait Application.nsp
Start with the suppress-auto-kill option
RunOnTarget.exe --suppress-auto-kill DevMenuCommand.nsp -- debug set-operation-mode-policy handheld

For more information about the no-wait and suppress-auto-kill options, see Options in RunOnTarget.

[SUCCESS]

    • However, if the operation mode policy is handheld, the performance mode is fixed to normal mode.

  • If the policy is boost, the performance mode will be fixed to boost mode.
Note
  • The boost performance mode can be used only with the SDEV. It cannot be used with the EDEV.

Enabling and Disabling Feature with High Power Consumption

Note
  • The features that can be controlled using the following commands have had no impact on the development hardware since the time they were initially prepared.
  • The following commands cannot be used in NintendoSDK Firmware for NX 8.0.0-1.0 and later.

While features that consume a lot of power cannot be used in the retail device, it is possible to enable or disable them using the following commands on the development hardware.

# Determine whether features that consume a lot of power are enabled or disabled.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-battery-draining-performance-enabled
enabled
[SUCCESS]

# Enable features that consume a lot of power
$ RunOnTarget.exe DevMenuCommand.nsp -- debug enable-battery-draining-performance
[SUCCESS]

# Disable features that consume a lot of power
$ RunOnTarget.exe DevMenuCommand.nsp -- debug disable-battery-draining-performance
[SUCCESS]

Enabling and Disabling CPU Overclocking

Note
  • The following commands cannot be used in NintendoSDK Firmware for NX 10.0.0-1.0 environments and later.
  • To enable CPU overlocking in environments with NintendoSDK Firmware for NX 10.0.0-1.0 and later, use the nn: :oe::SetCpuOverclockEnabled() function.

The CPU clock can be set to a rate that is higher than the clock rate that can be used in the retail device.

The CPU clock rate available in retail devices is 1020 MHz. When CPU overclocking is enabled, the clock rate becomes 1785 MHz.

This option is designed for the debug build environment and to handle the higher processing load that can result from features unique to development.

# Determines whether CPU overclocking is enabled.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-cpu-overclock-enabled
disabled
[SUCCESS]
 
# Enables CPU overclocking.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug enable-cpu-overclock
[SUCCESS]
 
# Disables CPU overclocking.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug disable-cpu-overclock
[SUCCESS]
Note
  • Only available on the SDEV. Even if CPU overclocking is enabled in the EDEV, it has no effect. The CPU clock rate operates the same as when the feature is disabled.
  • When CPU overclocking is enabled, the surface temperature of the SDEV will be higher, compared to when CPU overclocking is disabled. Be aware of this during use.
  • When CPU overclocking is enabled, the result of power consumption measurement by the pcm library, and the result of the CPU profiler, will be significantly different. Disable CPU overclocking when using these values as references in the production environment.
  • When CPU overclock is enabled, USB devices cannot be run from the system's power supply, due to the limits on the overall power consumption for the board as a whole.

CPU Boost Debug Feature

With the CPU boost feature, when the CPU boost is active there are periods where CPU processing capacity is actually increased and interval periods where the system automatically throttles the CPU back to its normal processing capacity. The timing of this switching and the length of each period are defined by the system, and cannot be freely controlled by the program. CPU boost debug mode is provided as a way to verify that the program is working properly during interval periods.

You can use the following commands to enable or disable CPU boost debug mode and to check the current state of the mode. If you enable CPU boost debug mode, the system always operates with processing capacity equivalent to the preceding interval period while the program is requesting use of the CPU boost feature. The setting configured to enable or disable the mode remains in effect even if the console is restarted.

# Check whether the current CPU boost debug mode is enabled (true) or disabled (false).
$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-cpu-boost-debug-mode-enabled
cpu-boost-debug-mode-enabled = false
[SUCCESS]
Exit( Result: Success, ExitKind: ExitProgram )

# Change the boost debug mode to enabled (true) or disabled (false).
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-cpu-boost-debug-mode-enabled true
set cpu-boost-debug-mode-enabled = true
To apply this change, restart the running application (if it is running).
[SUCCESS]
Exit( Result: Success, ExitKind: ExitProgram )

The enabled or disabled state of CPU boost debug mode is determined when the program starts and is not applied to the running program. To enable or disable the feature while the program is running, first exit the program and then restart it.

Controlling the Sleep Mode

You can change the system sleep mode for development purposes.

Getting the Sleep Mode Setting

$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-sleep-mode
default
[SUCCESS]

Setting the Sleep Mode

$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-sleep-mode pseudo
[SUCCESS]
  • When the sleep mode is set to default, the development hardware operates like the retail device.
    • The system is put to sleep, so the connection between the host PC and the target will be lost.
  • When sleep mode is set to pseudo, the development hardware does not enter actual sleep even if the POWER Button is pressed, and it will wake from sleep after about one second. Note that the behavior of the retail product in sleep mode differs in many ways.
    • The connection between the host PC and the target can be maintained before and after sleep.
      • Because sleep notifications and other aspects of the interface with the application operate the same way as with the retail product, this feature is useful for debugging the application straddling the sleep state.
    • The system will not disconnect from the network nor the session to which the controller is connected.
    • The system will also wake after about 1 second from the sleep it automatically enters when left idle, but the Screen Burn-In Reduction feature is not deactivated.

The setting becomes enabled after a change from the sleep state after the command is executed. Restarting the development hardware is not required.

Controlling the Idle State Detection Mode

For development purposes, you can enable and disable the Auto-Sleep and Screen Burn-In Reduction features that are triggered when the idle state is detected.

  • When this idle time detect mode setting is enabled, the Auto-Sleep and Screen Burn-In Reduction features are triggered as configured in the system settings. (The settings, which can be changed in the system settings screen in the HomeMenu environment, define factors like how long to wait before automatically transitioning to sleep.)
  • When this setting is disabled, the system settings are ignored and the Auto-Sleep and Screen Burn-In Reduction features are entirely disabled.

Currently, this setting is disabled by default. This feature is always enabled in retail devices.
For purposes such as burn-in tests, you must use this command to set the automatic sleep mode feature to disabled.

Note

In the state with the screen burn-In reduction feature disabled, prolonged display of a screen pattern where the RGB values change in every frame can cause ghosting and flickering.
If this happens, stop changing the RGB values in every frame and the problem will gradually resolve itself.

Getting the Idle State Detection Mode Setting

$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-idle-time-detect-mode
on
[SUCCESS]

Setting the Idle State Detection Mode

$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-idle-time-detect-mode off
[SUCCESS]
  • Specify on to enable the idle state detection feature.
  • Specify off to disable the idle state detection feature.

To update the settings after changing them, restart the development hardware.

Disabling Screen Transition When Fatal Errors Occur

A fatal error indicates a problem with the system.
Typically, the system stops when a problem with the system occurs because subsequent system operations cannot be ensured. However, this can be inconvenient during development.

Use this feature to skip the system shutdown process when a fatal error occurs.
Limit the use of this feature to times like when there is a bug in the firmware and the system frequently stops for reasons unrelated to the application.

# Disables the process of shutdown on a fatal error.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug disable-fatal-screen
[SUCCESS]
 
# Enables the process of shutdown on a fatal error.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug enable-fatal-screen
[SUCCESS]

Controlling the Memory Mode

Switching the Memory Mode

$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-memory-mode auto
[SUCCESS]
  • When the memory mode is set to auto, the target uses the maximum available size of memory.
  • When the memory mode is set to 4 GB, the target uses a fixed 4 GB of memory.
  • When the memory mode is set to 6GB, the target uses a fixed 6 GB of memory.
    However, if the target is not equipped with 6 GB or 8 GB of memory, the size is fixed to 4 GB.

The following values are available.

  • 4GB: The target uses a fixed 4 GB of memory.

Controlling GPU Resource Limitations

Use this during development to enable or disable the settings that control the GPU resources assigned to the application.
When the settings are enabled, several hundred GPU resources that are used by the system are reserved, accordingly limiting the GPU resources that can be used.
GPU resource limitations are enabled by default.

For more information about GPU resource limitations, see Graphics > Graphics (NX Add-On) > GPU Resource Allocation. 

# Gets the state of GPU resource limitations.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-gpu-resource-limitation-enabled
enabled
[SUCCESS]
 
# Enables GPU resource limitations.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug enable-gpu-resource-limitation
[SUCCESS]
 
# Disables GPU resource limitations.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug disable-gpu-resource-limitation
[SUCCESS]

To update the settings after changing them, restart the development hardware.

Info

In environments with firmware up to 10.0.0-1.0 recorded in NX Add-On 10.0.1, there is a restriction that disables the CPU Boost Functionality if the preceding GPU resource constraints are disabled. In this state, the CPU boost is not activated even if nn::oe::SetCpuBoostMode() is called, and these features cannot be used simultaneously.

With firmware 10.0.0-2.0 or later recorded in NX Add-On 10.1.0, this restriction is deprecated, and you can disable the GPU resource restriction and simultaneously use CPU Boost Functionality.

JIT Debugging Settings

When this feature is enabled and the application unexpectedly exits, the debugger connection continues to wait.
If this feature is disabled, no dump will be generated when the application unexpectedly exits.
To update the settings after changing them, restart the development hardware.

Disabling the Timer for Forcibly Closing Applications

Normally, when a user closes an application from the menu screen, the application is forcibly closed if the handling for closing the application is not complete within 15 seconds. A feature for disabling this forcible closing behavior has been prepared for use in debugging processes executed when closing an application. This setting takes effect immediately, so the system does not need to be restarted. By default, the (force-terminate) timer for forcibly closing applications is enabled the same it is on retail systems.

# Check the current value of the setting for disabling the application force-terminate timer (starting from the default settings).
$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-disable-force-terminate-application
disable-force-terminate-application = false
[SUCCESS]

Exit( Result: Success, ExitKind: ExitProgram )

# Disable the application force-terminate timer.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-disable-force-terminate-application true
set disable-force-terminate-application = true
[SUCCESS]
Exit( Result: Success, ExitKind: ExitProgram )

# Re-enable the application force-terminate timer.
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-disable-force-terminate-application false
set disable-force-terminate-application = false
[SUCCESS]
Exit( Result: Success, ExitKind: ExitProgram )

Note that when this feature is used to disable the timer to forcibly close the application, menu screens such as DevMenu wait indefinitely for an application to close if it does not close correctly. If this happens, forcibly close the application from TargetManager.

Info

Setting to Enable Displaying the Health and Safety Warning Message

Info

Available in NintendoSDK Firmware for NX 9.1.0-1.0 and later.

This setting is enabled only when the platform region is set to China.
You can switch whether to show or hide the Health and Safety warning message when an application is started.

# Show the Health and Safety warning message
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-health-warning-enabled true
set health-warning-enabled = true
[SUCCESS]

# Hide the Health and Safety warning message
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-health-warning-enabled false
set health-warning-enabled = false
[SUCCESS]

# Check whether the Health and Safety warning message is shown
$ RunOnTarget.exe DevMenuCommand.nsp -- debug get-health-warning-enabled
health-warning-enabled = true
[SUCCESS]

For more information about displaying the Health and Safety warning message, see Important Implementation Information for Consoles for China.

Emulation of Pressed System Buttons

Info

Available in DevMenuCommand included in NintendoSDK NX 11.1.0 and later.

This feature is for emulating the following presses of system buttons. However, this emulation only works when connected to TargetManager and either DevMenu or the HOME Menu are running. The emulation does not work correctly if the development hardware is in sleep mode, or the preceding conditions are not satisfied.

  • Short press of the HOME Button
  • Long press of the HOME Button (approximately 0.5 seconds)
  • Short press of the POWER Button
  • Medium press of the POWER Button (approximately 3 seconds)
  • Short press of the Capture Button
  • Long press of the Capture Button (approximately 0.38 seconds)


Info

Another way to emulate a short press of the POWER Button is to use the press-power-button subcommand of ControlTarget, as described in ControlTarget. If you use that command, you can also wake up the development hardware if it is in sleep mode.

# Short press of the HOME Button.
$ DevMenuCommand debug perform-home-button-short-pressing
[SUCCESS]

# Long press of the HOME Button.
$ DevMenuCommand debug perform-home-button-long-pressing
[SUCCESS]

# Short press of the POWER Button.
$ DevMenuCommand debug perform-power-button-short-pressing
[SUCCESS]

# Medium press of the POWER Button.
$ DevMenuCommand debug perform-power-button-middle-pressing
[SUCCESS]

# Short press of the Capture Button.
$ DevMenuCommand debug perform-capture-button-short-pressing
[SUCCESS]

# Long press of the Capture Button.
$ DevMenuCommand debug perform-capture-button-long-pressing
[SUCCESS]

Managing DLC Availability

Info

Available in NintendoSDK Firmware for NX 13.0.0-3.0 and later.

This feature for development and debugging is based on the assumption that DLC will be available in the production environment.

In the development environment, all installed DLC is available to all accounts. On the other hand, in the production environment, the availability of DLC differs for each user.
You can use this feature to set whether DLC is available to individual users in the development environment.
The settings are retained even if the application or the system is restarted. Before starting the application, reconfigure the settings as needed to get the desired initial state.

The configured settings are applied to all applications.

Enabling and Disabling the Debug Feature for Controlling DLC Availability

# Enable the debug feature for controlling DLC availability.
$ DevMenuCommand debug enable-content-availability-settings-for-debug

# Disable the debug feature for controlling DLC availability.
$ DevMenuCommand debug disable-content-availability-settings-for-debug

When the debug feature for controlling DLC availability is enabled, the configured settings for the availability of individual pieces of DLC are applied.

When these settings are configured and a user account is selected at startup, the status of that user account might be Unqualified, depending on the settings.

For more information about the Unqualified status, see User Account > Restrictions on Opening User Accounts.

Setting the Availability of Individual Pieces of DLC

# Make the DLC represented by index 1 available to the user account represented by index 0.
$ DevMenuCommand debug set-available-content-for-debug 0 --addoncontent-index 1

# Make the DLC represented by indexes 2 and 3 available to the user account represented by index 1.
$ DevMenuCommand debug set-available-content-for-debug 1 --addoncontent-index 2,3

# Make all DLC unavailable to the user account represented by index 2.
$ DevMenuCommand debug set-available-content-for-debug 2

Use --addoncontent-index to specify available DLC. Specify the indexes for the DLC, delimited with commas (",").

Clearing the Setting for the Availability of Individual Pieces of DLC

# Clear the setting for the user account represented by index 0.
$ DevMenuCommand debug clear-available-content-for-debug 0

# Clear the settings for all users.
$ DevMenuCommand debug clear-available-content-for-debug --all

Clear the settings that were configured by set-available-content-for-debug.

Forcibly Disabling the Ability to Display a Library Applet Started From the Application in Remote Video

Info

Available in DevMenuCommand included in NintendoSDK NX 14.1.0 and later.

Specifies whether to force a library applet started from the application to be shown in Target Manager's remote video.
This specification is applied from the next started library applet. It does not apply to library applets that are already running.

# Hide in remote video (force-disable)
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-force-disable-app-chain-la-recording true

# View in remote video (force-disable removed)
$ RunOnTarget.exe DevMenuCommand.nsp -- debug set-force-disable-app-chain-la-recording false
