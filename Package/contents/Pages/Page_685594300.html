Limitations and Resource Requirements

Please review the limitations and resource requirements to see if they apply to you.

Limitations when using Full Integration

When you ship your game, it is required that you do not ship the profiler code/library in the executable. Prior to submission, please disable all profiler related code and do not link against the profiler library.

The  In-Process profiling mode cannot collect data from game threads running with a priority of nn::os::HighestThreadPriority, however  Out-of-Process profiling does not have this limitation.

The  Out-of-Process profiling mode does not work on EDEV units or on SDEVs in 4GB mode.

Resource Requirements when using Full Integration

Memory

The profiler requires that you allocate a buffer of at least 64MB in your game.

Warning

It is strongly recommended that the buffer come from the heap rather than being a static buffer. If a static buffer is used, it inflates the size of the .bss section of your application and may make it less likely that features like function Instrumentation and Thread State tracing will be available.

Threads

The profiler creates up to 7 threads. Of these, 6 will always start upon calling nn::profiler::Initialize(). The final thread is started if the Memory Tracking API is used.

All threads created by the profiler have "NX CPU Profiler:" prefixed to their name.

Warning

The profiler's sampling threads are started with a priority of nn::os::HighestThreadPriority. For proper operation of the profiler do not create threads with this priority. 

Thread Local Storage

To enable Thread State tracing with the  Threads button, the profiler needs access to thread local storage. During its initialization call, the profiler will call nn::os::AllocateTlsSlot() to get access to a slot. If this call fails an error message is output, but initialization can still succeed. If there are no other initialization problems, nn::ResultSuccess() will be returned even when allocating a TLS slot fails.
