movie

Contains symbols aside from the movie namespace.

Symptoms

API symbols that are not from the movie library are included in multimedia.nso.

Workaround

No workaround is currently available.

Fix Schedule

This bug will be fixed in the next version.

A video with a nonstandard resolution played in CPU mode is sometimes not played correctly.

Symptoms

When movie::DecoderMode_Cpu is used to play a video with a resolution different from general standards like 720p or 1080p, it may not be played correctly.

Workaround

Use movie::DecoderMode_NativeTexture or change the resolution to one that meets standards, such as 720p, and re-encode.

Fix Schedule

This bug will be fixed in a future release.

In rare cases, the Windows version of MoviePlayerSimple sometimes crashes on startup.

Symptoms

In rare cases, the Windows version of MoviePlayerSimple may crash after closing the previously started instance of the sample.

Workaround

No workaround is currently available.

Fix Schedule

This bug will be fixed in a future release.

In the Windows version of Decoder, audio might not be played correctly while playing videos with a resolution of 1080p or higher.

Symptoms

In the Windows version of Decoder, audio may be corrupted when playing videos with a resolution of 1080p or higher.

This bug is more likely to occur on low-performance PCs.

This bug does not occur on the Switch platform.

Workaround

No workaround is currently available.

Fix Schedule

This bug will be fixed in a future release.

tool

Cannot update PROD MODE using the development hardware update tools.

Symptoms

PROD MODE cannot be updated using the development hardware update tools (SystemUpdateSdev and DevKitUpdater.nsp).

Workaround

Use the initialization tools to overwrite the PROD MODE firmware.

Fix Schedule

Future handling is under discussion.

Log output is slow when multiple devices are connected using Target Manager 2.

Symptoms

Log output can become slow when multiple devices are connected at the same time using Target Manager 2.
The number of connected devices when this bug occurs can vary depending on PC performance.

This bug can also occur in the tm library when configuring the settings for using Target Manager 2.

Workaround

No workaround is currently available.

Fix Schedule

This bug will be fixed in a future release.

When development hardware configured to use Target Manager 2 is connected via USB, its state sometimes becomes Disconnected when entering sleep mode.

Symptoms

When development hardware configured to use Target Manager 2 is connected via USB, the connection status changes to Disconnected if it enters sleep mode.
After this problem occurs, you may not be able to reconnect to the development hardware.

This issue occurs based on the USB port of the connection target host PC.

Workaround

You can avoid this issue by changing the USB port on the host PC that you are connecting to.
If this issue occurs, restart the development hardware.

Fix Schedule

This bug will be fixed in a future release.

If the SDEV settings are changed using Target Manager, the tool can no longer connect to that SDEV.

Symptoms

If you right-click the SDEV in Target Manager and click SDEV Settings to change the SDEV settings, that SDEV is automatically restarted but is no longer connected to Target Manager.

Workaround

Right-click the SDEV and perform a hard reset.

Fix Schedule

Because development has completed for Target Manager, there are no plans to fix this bug. Consider switching to Target Manager 2.

Recording the previous 30 seconds of video in Target Manager sometimes causes the feature to break.

Symptoms

In rare cases, unintended videos are created if you select Save a video clip of the last 30 seconds after completing the Start recording > End recording process in the Target Manager remote video window.

Workaround

The target video is the last 30 seconds of the video that is created.

Fix Schedule

Because development has completed for Target Manager, there are no plans to fix this bug. Consider switching to Target Manager 2.


Long runtime arguments cannot be passed.

Symptoms

If you pass runtime arguments that exceed 512 bytes from Target Manager or VSI, only the first 512 bytes are applied.

Workaround

If you use RunOnTarget or Target Manager 2, you can pass arguments that are as long as 32 KiB.
For debugging, start the application using the preceding procedure and, if you are debugging with VSI, combine it with the procedure described in Debugging a Built Program > Debugging Immediately After Starting Applications Outside of Visual Studio.

Fix Schedule

This bug will be fixed in a future release.

When device lock is enabled the first time the HOME Menu starts, unlocking is not accepted during initial settings, and the device shuts down.

Symptoms

When device lock is enabled the first time the HOME Menu starts, unlocking is not accepted during initial settings and the device shuts down.

Workaround

Use the -- settings disable-device-lock DevMenuCommand to disable device lock, configure initial settings, and then re-enable device lock.

Fix Schedule

This bug will be fixed in a future release.

OpenGL / Vulkan

Memory allocation occurs from the default heap and not the custom memory allocator in the OpenGL and Vulkan applications.

Symptoms

In the OpenGL and Vulkan applications, dynamic memory allocation takes place within the library.

The memory for this is essentially supposed to be allocated using the memory allocator specified using the nv::SetGraphicsAllocator() function. However, memory ends up being allocated from the default heap without using this allocator.

Workaround

No workaround is currently available.

Fix Schedule

The exact timing of the fix is to be determined.

Workaround

No workaround is currently available.

Fix Schedule

This bug will be fixed in a future release.

GLSLC Bugs

  • The TX1 versions of the glslc library may produce slightly different machine code from the Windows version, resulting in potential performance differences. The code produced on both platforms is likely to be functionally correct, and the two will converge in future releases.
  • 16-bit floating point values are supported in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, the following known issues occur.
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface and mediump/lowp types on the other side of the interface may produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types with mediump/lowp qualifiers. You can work around this problem by explicitly constructing float types from the int or uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp and ES shaders regardless of whether the extension GL_EXT_shader_implicit_conversions is used.
    • Using FP16 math to compute the data argument for imageStore() GLSL function calls may result in corruption in the stored data. This issue is being investigated and will be fixed in an upcoming release.

RenderDoc

  • While RenderDoc is saving a large capture to disk, there is no progress indicator. Worse, if the captured application is terminated on the Nintendo Switch while the save is still in progress, the saved capture will be corrupted. Large saves can sometimes take a few seconds to finish. The only indication that the thumbnail save is complete is a subtle one: before the save is finished the thumbnail label is followed by "(Remote)," but after the save is finished the "(Remote)" disappears.
  • The RenderDoc layer adds considerable overhead to Vulkan API calls, so programs that make many such calls run slower when the layer is enabled. Slowdowns have been as high as 10x with some large complicated programs.
  • RenderDoc sometimes crashes when running Python scripts.
  • When capturing from an OpenGL project, there is a significant performance penalty for using persistent mapped buffers.
