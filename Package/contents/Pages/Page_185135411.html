Overview

This section describes patch development procedures as envisioned by Nintendo.

It is intended for reference purposes, because compliance with these procedures is not absolutely necessary.

Building and Debugging the Revised Application

The first step is to build and debug the revised application, paying no heed to the patch. You can use VSI or another tool to create the revised application.

With the exception of storage access speed, you can assume that the revised application is the same as the initial application plus the patch. For this reason, you can use the revised application for testing except where speed is affected.

It is possible to verify the compatibility of save data from the initial application here. In some cases, starting the application might be restricted by required version settings when switching between the initial application and revised application to verify compatibility. If a restriction is encountered, reset the required version using DevMenuCommand. When switching between application versions again, the save data created by the revised application will be loaded into the initial application. If this is undesirable, also delete the save data.

Building and Debugging the Patch

To test the effect on speed and to conduct QA testing on the product that will be ultimately submitted, you must build and debug the patch. In addition, the size of a patch is limited, so you must build the patch to make sure it fits within the limit. For more information about patch size, see the Patch Guide.

For more information about building a patch and debugging using the created patch, see Building and Running a Patch. The patch affects the load speed, so if there is a previous patch make sure that you specify that patch.

A patch cannot currently be started from VSI after it has been optimized. However, after you start the patch using the method described in Building and Running a Patch, you can attach the patch to VSI for debugging. For more information about attaching a patch after it has been started, see Debugging a Built Program > Debugging an Already Running Application.

If there are plans to produce NX cards with on-card patches, make sure to verify the operation of the NX development card with an on-card patch.
