Workaround

When playing a movie with a non-standard resolution in CPU mode, it may not be played correctly.

Symptoms

When movie::DecoderMode_Cpu is used to play a video with a resolution different from general standards like 720p or 1080p, it may not be played correctly.

Workaround

Use movie::DecoderMode_NativeTexture or change the resolution to one that meets standards, such as 720p, and re-encode.

Fix Schedule

This bug will be fixed in a future release.

Some WebM format videos cannot be fully played back.

Symptoms

An error can occur and the video fails to play through to the end when memory allocation from the global heap fails while playing back certain WebM videos.

Workaround

You can avoid this bug by increasing the global heap capacity.

Fix Schedule

This bug will be fixed in a future release.

OpenGL / Vulkan

Memory allocation occurs from the default heap and not the custom memory allocator in the OpenGL and Vulkan applications.

Symptoms

In the OpenGL and Vulkan applications, dynamic memory allocation takes place within the library.

The memory for this is essentially supposed to be allocated using the memory allocator specified using the nv::SetGraphicsAllocator() function. However, memory ends up being allocated from the default heap without using this allocator.

Workaround

No workaround is currently available.

Fix Schedule

The exact timing of the fix is to be determined.

GLSLC Bugs

  • The TX1 versions of the glslc library may produce slightly different machine code from the Windows version, resulting in potential performance differences. The code produced on both platforms is likely to be functionally correct, and the two will converge in future releases.
  • 16-bit floating point values are supported in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface and mediump/lowp types on the other side of the interface may produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types with mediump/lowp qualifiers. You can work around this problem by explicitly constructing float types from the int or uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp and ES shaders regardless of whether the extension GL_EXT_shader_implicit_conversions is used.
    • Using FP16 math to compute the data argument for imageStore() GLSL function calls may result in corruption in the stored data. This issue is being investigated and will be fixed in an upcoming release.

RenderDoc

  • While RenderDoc is saving a large capture to disk, there is no progress indicator. Worse, if the captured application is terminated on the Nintendo Switch while the save is still in progress, the saved capture will be corrupted. Large saves can sometimes take a few seconds to finish. The only indication that the thumbnail save is complete is a subtle one: before the save is finished the thumbnail label will have "(Remote)" after it, but after the save is finished the "(Remote)" disappears.
  • The RenderDoc layer adds considerable overhead to Vulkan API calls, so programs that make many such calls run slower when the layer is enabled. Slowdowns have been as high as 10x with some large complicated programs.
  • RenderDoc sometimes crashes when running Python scripts.
  • Custom visualization shaders do not compile correctly.
  • When capturing from an OpenGL project, there is a significant performance penalty for using persistent mapped buffers.
  • Resetting the layout to the default while a capture is loading causes the application to crash.

web

Symptoms

In builds where optimization has been disabled (-O0), a link error is generated when you reference a static member constant of a class in the nn::web namespace.

Workaround

hid

An error occurs when mapping buttons for a gamepad transferred using InputDirector.

Workaround

Fix Schedule

The controller does not work on the Strap Usage screen when nn::hid::ShowControllerStrapGuide() is called before nn::hid::SetSupportedNpadStyleSet().

Symptoms

When an application is started and nn::hid::ShowControllerStrapGuide() is run without nn::hid::SetSupprotedNpadStyleSet() ever being called, all wireless controllers that had been connected after transitioning to the Strap Usage screen are disconnected and can no longer be connected.
You can also no longer operate Joy-Con controllers connected to the console rail or buttons on the HDEV unit, and cannot exit the Strap Usage screen without using touch operations.

Workaround

Call nn::hid::SetSupprotedNpadStyleSet() before nn::hid::ShowControllerStrapGuide().
Install the application to the development hardware and start it from the HOME Menu.

Fix Schedule

This bug will be fixed in a future release.
