Using Symbol Server

Overview

The NX Symbol Store provides an automated system for archiving and retrieving appropriate debug information via a shared network resource directly within Visual Studio.

Definitions

NX Symbol File

An "NX Symbol File" is a binary file generated by the NX build toolchain.  It contains:

  1. Debug symbols that facilitate debug sessions, and

  2. A unique build identifier.

These files are output as *.NSS (for applications) or *.NRS (for DLLs).  These files are generated when creating executable files for the NX platform.

The unique build identifier (or build ID) is heuristically generated each time an application or library is linked.  The available heuristics are: random, MD5 (VSI default), and explicit.  This can be directly specified via the build-id linker command line option.

Note:  The debugger and the symbol store use the build ID to associate debug information with corresponding executable code.  We strongly advise against attempting to explicitly specify the build ID.

NX Symbol Store

An "NX Symbol Store" is a collection of NX symbol files, arranged in a formalized manner for automated access during debugging.  See NX Symbol Store Format (below) for more details.

NXSymStore

The NXSymStore utility application (NXSymStore.exe) is a command-line tool for creating, managing, and querying an NX Symbol Store.

NXSymStore archives symbols in a format that enables the NX debugger to retrieve symbols based on the build ID contained within the executable image.

Once an NX Symbol Store is established and properly configured, the archival and retrieval of symbols is fully automatic.

The NXSymStore utility and supporting files can be found in the %PROGRAMFILES%\Nintendo\Oasis\bin folder.  

Use

The NXSymStore utility is invoked on the command line with one valid operation to perform.  Valid operations are Help, Add, Query, Delete, or Extract:

Help ( ? )

Outputs help text to the console.  No arguments are required.

Add ( add )

Saves the symbol file to the symbol store.  Syntax:

NXSymStore add -f path [-f path …] -s symbol_store [-d log_file] [-r] [-o] [-compress] [-force]

-f path

Local or UNC path containing files to be uploaded.  If a directory is specified, all supported file types within that directory will be uploaded to the symbol store.

If path starts with '@' it is treated as a response file.  It is expected to contain a list of files, separated by newlines, each of which will be added.

-s symbol_store

Local or UNC path to the symbol store. If default path is already set by "config -dstore" option, this option is not necessary.

-r

Recursively scan all subdirectories within the specified path for supported files to add to the symbol store.

-d log_file

Redirect messages to the specified log file rather than stdout.  If the given log_file already exists, then an error message will be generated, the -d option will be ignored, and messages will go to stdout.

-compress

Compress the files before uploading.

-force

Force the symbol upload.  Overwrite any existing symbol file that already exists in the the symbol store.

-o

Enable verbose output.

-userdata file_path

Associate this file as a user data with the symbol file. This user data can be queried by "userdata" subcommand later.
See Using Symbol Server also.

Example:

NXSymStore add -f MyGame.nss -s \\server1\symbols -compress

Uploads and compresses file MyGame.nss to the network-based symbol store at \\server1\symbols.

Query ( query )

Queries the symbol store regarding one or more symbol files.  Use this to determine if a symbol file is within the symbol store and, if so, retrieve information regarding the file.  Syntax:

NXSymStore query -f path [-f path …] -s symbol_store [-d log_file] [-r] [-o]

-f path

Local or UNC path identifying a file to be queried.

If path starts with '@' it is treated as a response file.  It is expected to contain a list of files, separated by newlines, each of which will be queried.

-s symbol_store

Local or UNC path to the symbol store. If default path is already set by "config -dstore" option, this option is not necessary.

-d log_file

Redirect messages to the specified log file rather than stdout.  If the given log_file already exists, then an error message will be generated, the -d option will be ignored, and messages will go to stdout.

-r

Recursively scan the path specified by the -f argument for supported files to query on the symbol store.  The -r flag does not apply to symbol stores, but to symbol files that could potentially be included in the symbol store.

-o

Verbose output.

Example:

NXSymStore query -f MyGame.nss -s \\server1\symbols

Output:

Querying for MyGame.nss in store at \\server1\symbols 
  Path:  84\84883f959a24ab439910169f0076d5bc00000000.nxsym 
  Is compressed 
  Added 05/30/2017 09:30:19 
  Created 05/22/2017 15:59:29 
  Initiated from JOHNSCOMPUTER (user johndoe)

Recursive Examples:

NxSymStore query -f C:\MyProjects -s D:\Server\A

This will cause NxSymStore to search the C:\MyProjects folder to find all symbols that are also in the symbol store at D:\Server\A, then report on the found entries.

NxSymStore query -f C:\MyProjects -s D:\Server\A -r 

This will cause NxSymStore to perform a recursive search starting in the C:\MyProjects folder to find all symbols that are also in the symbol store at D:\Server\A, then report on the found entries.

Delete ( del )

Deletes symbol files from the symbol store.  There are multiple options for specifying the files to be removed from the symbol store.  Syntax:

NXSymStore del -b Id                 -s symbol_store  [-d log_file]   [-o] [-y] [-list] 

NXSymStore del -days NumberOfDays    -s symbol_store [-d log_file] [-o] [-y] [-list] 

NXSymStore del -date Month-Day-Year  -s symbol_store [-d log_file] [-o] [-y] [-list]

NXSymStore del -size NumberOfGBs     -s symbol_store [-d log_file] [-o] [-y] [-list] 

NXSymStore del -user UserName        -s symbol_store [-d log_file] [-o] [-y] [-list]

NXSymStore del -machine MachineName  -s symbol_store [-d log_file] [-o] [-y] [-list]

Note that the options available to the del command are mutually exclusive.  If, for example, -size is specified, then none of the other options ( -b, -days, -date, -user, or -machine) can be used within the same command line invocation.

-b Id

Identifier of the symbol file to delete.

-days NumberOfDays

Specifies the cutoff date and time for deleting files.  All files that were last accessed prior to the specified number of days ago will be deleted.  NumberOfDays specifies an integer number of 24-hour days.  For example, if the specifier -days=3 is used at 6:00 PM on February 17, 2008, all files last accessed prior to 6:00 PM on February 14, 2008 will be deleted.

-date Month-Day-Year

Specifies the cutoff date for deleting files.  All files that were last accessed prior to the specified date will be deleted.  The date must be specified in the format Month-Day-Year with hyphens between the month, day, and year.  Leading zeros in Month and Day are optional.  The year can be specified with two or four digits.  Thus you can indicate the date of January 5, 2008 as 01-05-2008 or as 1-5-08.

-size SizeInBytes

Specifies the total size of the files that should remain after the deletion, in bytes.  When this switch is used, NXSymStore deletes files in the store, beginning with the files accessed least recently, until the total size of the remaining files is less than or equal to SizeInBytes.

-user UserName

Specifies the user name to use for deleting files.  All files that were added to the store by the user will be deleted.  For example, if -user JohnSmith is specified, all files that were added to the symbol store by user JohnSmith will be deleted.

-machine MachineName

Specifies the machine name to use for deleting files.  All files that were added to the store from the machine will be deleted.  For example, if the -machine BUILDBOX is specified, all files that were added to the symbol store from the BUILDBOX machine will be deleted.

-s symbol_store

Local or UNC path to the symbol store.

-d log_file

Redirect messages to the specified log file rather than stdout.  If the given log_file already exists, then an error message will be generated, the -d option will be ignored, and messages will go to stdout.

-o

Enable verbose output.

-y

Suppresses the (y/n) prompt.  If this option is not used, NXSymStore prompts you with "Are you sure?" before deleting any files.

-list

Causes NXSymStore not to delete any files, but merely to list all the files that would be deleted if this same command were run without the -list option.

Examples:

NXSymStore del -b 84883f959a24ab439910169f0076d5bc00000000 -s \\server1\symbols -y

Deletes the symbol files associated with the build Id 84883f959a24ab439910169f0076d5bc00000000 from the symbol store at \\server1\symbols without prompting.

NXSymStore del -user JohnSmith -s \\server1\symbols

Deletes the symbol files added by user JohnSmith from the symbol store at \\server1\symbols, prompting before doing so.

NXSymStore del -user JohnSmith -s \\server1\symbols /list

Lists the symbol files added by user JohnSmith to the symbol store at \\server1\symbols.

NXSymStore del -days 365 -s \\server1\symbols

Deletes the symbol files not used in the last 365 days from the symbol store at \\server1\symbols, prompting before doing so.

NXSymStore del -date 01-5-2018 -s \\server1\symbols

Deletes the symbol files not accessed after January 5, 2018 from the symbol store at \\server1\symbols, prompting before doing so.

NXSymStore del -size 300000 -s \\server1\symbols

Deletes the least-used symbol files, until the total size of the symbol store is not more than 300,000 bytes, from the symbol store at \\server1\symbols, prompting before doing so.

NXSymStore del -machine BUILDBOX -s \\server1\symbols

Deletes the symbol files added from the BUILDBOX machine from the symbol store at \\server1\symbols, prompting before doing so.

Extract ( extract )


Extracts a symbol file from a symbol store, decompressing if required.  Syntax:

NXSymStore extract -b Id -s symbol_store [-out output] [-d log_file] [-o]

-b Id

Identifier of the symbol file to extract.

-s symbol_store

Local or UNC path to the symbol store.

-out output

Local or UNC path identifying the file name to output

-d log_file

Redirect messages to the specified log file rather than stdout.

-o

Verbose output.

Example:

NXSymStore extract /b 84883f959a24ab439910169f0076d5bc00000000 /s \\server1\symbols /out MyGame.nss

Extracts the symbol file with the Id of 84883f959a24ab439910169f0076d5bc00000000 from the symbol store at \\server1\\symbols and copies it into a file named MyGame.nss.

User data ( userdata )

Handles a user data in a symbol store. Any format can be used as user data.
If it is a simple text file, the content can be referred by -read option easily.
Even if it is a binary file, you can leverage it by -dump option.

Syntax:

NXSymStore userdata {-create|-read|-update|-delete|-dump} -b Id -s symbol_store

Sub options

-create

Create a user data with "-userdata" option and associate it with the symbol file.
-userdata option is essential.

-read

Output contents of the user data to stdout.

-update

Update the user data with a file specified by "-userdata" option.
-userdata option is essential.

-delete

Delete a user data associated with the symbol file.

-dump

Copy the user data to the path specified by "-out" option.
-out option is essential.

Common options

-b Id

Identifier of the symbol file which is associated with the user data to handle.

-s symbol_store

Local or UNC path to the symbol store.

Specific options

-userdata file_path

Local or UNC path identifying the file name of the user data.
Only valid for "create" or "update" sub option.

-out output

Local or UNC path identifying the file path to dump the user data.
Only valid for "dump" sub option.

Configuration ( config )

Configure settings.

-dstore symbol_store

Local or UNC path to the default symbol store. This path will be used when "-s" is not specified to add/query/del/extract.

Response files ( @ response_file  )

Provides command line switches separated by whitespace.

Examples:

NXSymStore @response_file.txt

This parses the response_file.txt file for commands.

NXSymStore add @default_options.txt -f MyGame.nss

This performs the add operation with the MyGame.nss file, using the additional arguments listed in the default_options.txt file.

NXSymStore @command_in_file.txt -o -force -compress

This performs the operations found in the command_in_file.txt file, and uses the verbose, compress, and force parameters.

If a given switch can be specified multiple times on the command line, then response files which are provided as an argument to such a switch are processed slightly different.  Each item in the response file is processed as a separate instance of the given switch.

For example, given a file named response_file containing the text:

file1
file2

A command line of:

NXSymStore -f @response_file

Is interpreted as:

NXSymStore -f file1 -f file2

Instantiation

Notice that NXSymStore performs only one of these operations (Help, Add, or Query) at a time.  Invoking it with more than one operation results in an error.

For example (notice the add and query operations):

NXSymStore add -f C:\MyProject\MyProject.nss -s \\MyServer\MySymbolServer -force query -f C:\MyProject\MyProject.nss -s \\MyServer\MySymbolServer 

Results in NXSymStore reporting:

ERROR - Only one operation allowed per instantiation

Output

NXSymStore saves (and optionally compresses) symbol files within a structured format on a local file path or shared network for access during debugging sessions.

NX Symbol Store Format

A file named NintendoSymbolStore is automatically created in the selected directory when the first symbol file is uploaded.  The presence of this file identifies the directory as a symbol store:

Uploaded files are organized and named by the build ID contained within the symbol file.  The build ID is also present in stripped executable files and core dump files generated while the executable is running.  The debugger and the symbol store use the build ID to associate, identify, and download the relevant files from the symbol store.

Additional information about each symbol file is recorded in an XML file using the same base file name.

Use in Development

Building

Configure your build system to invoke NXSymStore.exe on the target output as a post build step:

[Oasis Install Folder]\bin\NXSymStore.exe add -f $(TargetDir)$(TargetName).nss -s [Symbol Store Path] [Additional Arguments]

For example:

"C:\Program Files\Nintendo\Oasis\bin\NXSymStore.exe" add -f $(TargetDir)$(TargetName).nss -s \\Network\ProjectSymbolsStore\ -force -compress -o

You can use a response file to enforce uniform operation between users and builds.

For example, place this text in the AddToSymStoreArgs.txt file:

add -s \\Network\ProjectSymbolsStore\ -force -compress -o

Then invoke NXSymStore thus:

"C:\Program Files\Nintendo\Oasis\bin\NXSymStore.exe" @AddToSymStoreArgs.txt -f $(TargetDir)$(TargetName).nss

Debugging

Add the path to the symbol store into the Visual Studio / Debugging / Symbols settings:

During debugging, the appropriate symbol files are automatically downloaded to the user's computer, decompressed if necessary, and stored in the %TEMP%\Nintendo\SymbolCache folder.
