In some cases, the memory size available for nn::os::SetMemoryHeapSize() shrinks if debugging is executed without resetting.

If debugging is executed multiple times without resetting the development hardware, the memory available for nn::os::SetMemoryHeapSize() may shrink. If this problem occurs, memory allocation fails, or an exception is thrown when starting the program.

If this problem occurs, reset the development hardware.

The demo is dark.

Copying factoring in sRGB format does not occur during processing when the current status is nn::gfx::Queue::CopyToScanBuffer() (to be deleted) or nn::gfx::CommandBuffer::CopyImage(). As a result, the display in the demo specifying sRGB in the scan buffer format is dark. This issue will be fixed in future releases.


GLSLC Known Bugs

  • A 32-bit executable only allows for assembly dumps.
  • EOL markers must be present in #include files, else a compilation error will occur.
  • Transform feedback varyings specified in shader source do not work correctly.
  • Compute shader data subsections inside a GPU code section do not start with the magic number GLSLC_GPU_CODE_SECTION_DATA_MAGIC_NUMBER.
  • The TX1 versions of the glslc library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms is likely to be functionally correct though, and will be combined in future releases.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, the following known issues occur.
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface, and mediump or lowp types on the other side of the interface, could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types qualified with mediump or lowp qualifiers. You can work around this limitation by explicitly constructing float types from the int and uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp in addition to ES shaders regardless of whether the GL_EXT_shader_implicit_conversions extension is used.

Some sample programs do not work correctly when installed on development hardware.

Many sample programs cannot be run when installed on development hardware.
To check the procedures for installing applications to development hardware using sample programs, use the following sample programs.

  • GfxShaderCompile
  • GfxSimple
  • NvnTutorial01GettingStartedWithNVN
  • HidSimple

Multiple instances of the same application can be started at the same time on development hardware.

After running an application on development hardware using RunOnTarget as in RunSdevSample.bat, if you then start that application from VSI, you end up with two instances of the application running at the same time.

This situation is not supported. If the application you started with RunOnTarget has not terminated, perform a target reset from Target Manager.

When a Visual Studio project file is opened, you are prompted to upgrade the compiler and libraries.

When a project file not attached to a solution file, such as a Visual Studio project for an SDK library whose source code is publicly available, is opened in Visual Studio, a dialog box might be displayed prompting you to upgrade the compiler and libraries.

If you upgrade them, you will no longer be able to build in earlier versions of Visual Studio. Click Cancel and close the dialog box.

HID Bugs

GamePad disconnection cannot be detected via Input Director.

After the connection with Input Director is established, the value of the nn::hid::BasicXpadAttribute::IsConnected flag of bit collection nn::hid::BasicXpadState::attributes is always true.

VSI Bugs

Cannot handle exceptions thrown in SDK libraries.

The NX32 and NX64 SDK libraries do not support C++ exceptions.
For this reason, when exceptions are thrown in the libraries, they are canceled.

If the code is only propagated to the application, you can enable C++ exceptions at build to make it possible to handle exceptions.

Optimized code may not be debugged correctly.

In some cases, optimized code cannot be debugged correctly, such as when it stops on breakpoints placed in functions with inline expansion.

Windows platform may not behave correctly.

Failure sometimes occurs when starting or running debug or generating new projects.
Restart Visual Studio.

Sometimes, items disappear from the project properties.

Sometimes the items under Configuration Properties disappear from the project properties.
Restart Visual Studio.

When double-clicking the call history during a breakpoint, it sometimes jumps to an incorrect file or line number.

Double-clicking the call history during a breakpoint allows you to jump to the applicable file or line number.

However, if the call history contains no-return functions, it jumps to a different location than the file or line number that it would originally have jumped to.

In this case, the content in the call history is correct, but it jumps to the incorrect file or line number.

Target Manager

When an application is started, “Unable to find debug data for module id ...” is output to the log.

This bug occurs when the debugging information file (NSS file) is not present in any of the directories included in the search.
A bug exists where directories specified in Project Properties > Linker > General > Additional Library Directories are not correctly included in the search path. As a result, any NSO files in the directories are linked correctly, but the debugging information itself cannot be found, which is why this message is output to the log.
To correct this problem, manually copy the NSS files corresponding to the required libraries to the directory in which your Application_name.nca and Application_name.nspd files are located.

fs Bugs

Multi-byte characters cannot be used with save data entry names.

UTF-8 entry names are supposed to be available for use, but specifying entry names that contain non-ASCII characters does not work.

Filenames and directory names including "@" are not handled correctly.

When the file system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot()) uses filenames or directory names that include @, it does not work correctly.

nn::fs::DeleteDirectoryRecursive() fails when deleting directories containing many files.

In the file system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot()), when a directory including around 255 or more files is deleted with nn::fs::DeleteDirectoryRecursive(), it fails with nn::fs::ResultUnexpected.

Cannot open numerous files simultaneously.

The number of files that can be open at any time is restricted as follows.

File System Maximum Number of Open Files
File system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot())

338 (32-bit environment)

182 (64-bit environment)
Resource data (nn::fs::MountRom())

338 (32-bit environment)

182 (64-bit environment)
Save data (nn::fs::MountSaveData()) 30

The function fails with nn::fs::ResultAllocationMemoryFailed if you attempt to open more than the maximum number of files.
We plan to improve the number of files that can be opened at the same time.

In save data, adding entries to directories that have a lot of entries takes a long time.

When creating entries with nn::fs::CreateFile() or nn::fs::CreateDirectory() in directories that include 512 or more entries, it takes much longer than cases with 511 or fewer entries.

nn::fs::ReadFile() may disconnect from Target Manager.

There is a bug where the connection to the Target Manager may be disconnected (the status in Target Manager becomes "Disconnected") when executing nn::fs::ReadFile() through MountHost() and MountHostRoot().

The conditions for this problem occurring are based on the size of the file loaded in nn::fs::ReadFile() and the size of previously loaded files, but the details are currently being investigated.

If this problem occurs repeatedly, avoid it by changing the buffer size specified in nn::fs::ReadFile().

LDN Bugs

Immediately after a station connects to the network, in some cases the packets sent by the station and addressed to the station keep getting lost in transit.

In some cases, immediately after a station has called the nn::ldn::Connect function and connected to an access point, the packets sent by the station and the packets addressed to that station keep getting lost lost in transit for several seconds. This problem occurs more frequently when the wireless environment is poor, so the workaround is to set a wireless channel that has a strong signal.

This problem will be fixed in an upcoming release.

Sometimes UDP packets are not sent at all when you attempt to send them faster than a particular speed.

In some cases with ldn, if packets are sent via UDP at a transfer speed faster than about 20 Mbps after the network has been built, at some point packets can no longer be sent at all. This problem does not occur at low transfer speeds or when packets are sent via TCP.

This problem will be fixed in an upcoming release.

On rare occasions, control does not return from the nn::ldn::Connect and nn::ldn::Disconnect functions.

On rare occasions, the nn::ldn::Connect and nn::ldn::Disconnect functions remain blocked and control does not return.

This problem will be fixed in an upcoming release.

socket Bugs

When USB Ethernet is disconnected, TCP functions that are called do not return.

When a USB Ethernet device is disconnected, blocked TCP APIs do not return.

We plan to fix this problem in the next release.

When an application is closed without calling nn::socket::Finalize(), resources are not released.

In the socket library, when you close an application without calling nn::socket::Finalize(), such as by stopping the application from Target Manager or performing a forced exit with Abort, the following problems occur when you next run the application.

  • Bound port numbers cannot be reused.
  • Memory and internal resources leak without being released.

If this problem occurs, reset the SDEV.

We plan to fix this problem in the next release.

Cannot use 255.255.255.255 as the broadcast recipient.

When performing broadcast using UDP, the broadcast will not be transmitted correctly if you specify the limited broadcast address (255.255.255.255) as the recipient.

Use a directed broadcast address if broadcast is required.

We plan to fix this problem in the next release.

vi Bugs

The default visibility setting when creating layers is true.

While the default visibility when creating layers is currently true, the default will be false in future SDK versions.

To display the layer, make sure to call nn::vi::SetLayerVisibility(pLayer, true) and set it to a displayable state.

The SetLayerScalingMode settings are not applied.

There is a bug where the API call succeeds if nn::vi::SetLayerScalingMode is called, but nn::vi::ScalingMode_FitToLayer is always applied regardless of the specified mode.

We plan to fix this problem in the next release.

Scaling is not applied correctly when dynamically switching between HDMI and LCD.

If the HDMI cable is inserted or removed when the application is running, switching between HDMI and LCD will occur but the scaling will not be applied correctly.

For example, if a TV with a maximum resolution of 1080p is connected, the following behavior will occur.


 Application started with HDMI cable connected to dock Application started with HDMI cable disconnected from dock
TV

◯ Displays correctly on the TV.

☓ Does not display correctly if the HDMI cable is inserted while the application is running.
LCD

☓ Does not display correctly if the HDMI cable is removed while the application is running.

◯ Displays correctly on the LCD.

This bug will not surface when using a TV with a maximum resolution of 720p. When using a TV with a maximum resolution of 480p, the scaling will not be applied correctly as with a 1080p TV.

Switching between HDMI and LCD output (in either direction) is not supported. Planned to be supported in a future release.

After resetting the development hardware, the output does not go the TV the first time the HDMI cable is connected.

There is a bug where output does not go the TV when the HDMI cable is first connected after resetting the development hardware.

As a workaround, disconnect the HDMI cable, confirm that the image displays on the LCD screen, and then reconnect the HDMI cable.

Even if the HDMI cable is connected, video may remain on the LCD and is not output to the TV.

Various hardware problems (such as the SDEV dock not being turned on) cause the video to remain on the LCD and prevent output to the TV even though the HDMI cable is connected.

For more information, see the Troubleshooting section of the SDEV Operations Manual.

audio Bugs

On the development hardware, when sound is being produced with nn::audio and the application stops due to a breakpoint or for other reasons, the sound playback does not stop.

On development hardware, when an application is paused while sound is being produced, the sound does not stop, and the playback continues.

This problem will be fixed in an upcoming release.

nn::audio HDMI output does not work on development hardware.

On development hardware, audio produced with nn::audio cannot be output with HDMI. Currently only system speaker and headphone outputs are supported.

This problem will be fixed in an upcoming release.

Sometimes, operations become unstable when accessing the area surrounding resource data used by nn::audio on development hardware.

When the memory within a 4-KB range before and after the memory used for the audio renderer is specified as the destination for processing that requires writing, such as file loading, the operation may become unstable.

The following buffers are affected by this problem.

  • Buffers specified by nn::audio::WaveBuffer.buffer
  • Buffers using various effects
  • Buffers allocated as the area passed to pParameter of nn::audio::AcquireVoiceSlot()

To avoid this problem, switch to using the memory pool (nn::audio::MemoryPoolType).

For more information, see File System Troubleshooting.

The system volume goes to maximum when nn::audio::OpenAudioOut() and nn::audio::OpenAudioRenderer() are called.

When nn::audio::OpenAudioOut() and nn::audio::OpenAudioRenderer() are called, the volume on the development hardware goes to maximum regardless of the volume button settings.

This problem will be fixed in an upcoming release.

Restrictions When Using nn::audio::AudioOut on the NX64 Environment

The following restrictions apply when using nn::audio::AudioOut on the NX64 environment.

  • There is a limit to the maximum number of buffers that can be registered.
  • One instance of AudioRenderer is implicitly opened internally.

This problem will be fixed in an upcoming release.

User effects using nn::atk::EffectBase are not supported.

User effects using nn::atk::EffectBase cannot currently be used. The API is also temporary, and we plan to fix it later.

This problem will be fixed in an upcoming release.

When the NintendoWare AtkPlayer starts on the development hardware and several sounds are played, the application will stop.

When the NintendoWare AtkPlayer starts on the development hardware and several sounds are played, the application might stop. The problem can be avoided by only playing back the target sound after starting AtkPlayer. In addition, you can start AtkPlayer again by resetting the development hardware from Target Manager.

This problem will be fixed in an upcoming release.

Stopping the hardware version of AtkPlayer in NintendoWare SoundMaker does not stop the hardware version of AtkPlayer.

Even if you stop the hardware version of AtkPlayer from SoundMaker's Preview menu, the hardware version of AtkPlayer does not stop.

This problem will be fixed in an upcoming release.

3D Bugs

When executing and closing the G3dDemo Town demo repeatedly, memory becomes insufficient and a forced exit occurs.

When repeatedly executing and closing the G3dDemo Town demo built from Visual Studio solutions, memory becomes insufficient and a forced exit occurs.

This problem will be fixed in an upcoming release.

When the A Button is pressed repeatedly while a model is selected in the G3dDemo Edit demo, the attachment state between 3DEditor and G3dDemo becomes inconsistent.

When the A Button is pressed repeatedly while a model is selected in the G3dDemo Edit demo, sometimes it becomes attached in 3DEditor but unattached in the application.

This problem will be fixed in an upcoming release.

Japanese messages sent using nn::g3d::viewer::ViewerServer::QueueSendUserMessageCommand are garbled in the 3DEditor.

Japanese messages sent using nn::g3d::viewer::ViewerServer::QueueSendUserMessageCommand on the development hardware are garbled in 3DEditor.

This problem will be fixed in an upcoming release.

If you connect the development hardware and 3DEditor and edit shaders in real time, the GPU may stop responding.

If you connect the development hardware and 3DEditor and edit shaders in real time, the GPU may become unresponsive and processing may stop running.

This problem will be fixed in an upcoming release.

DevMenu Bugs

Applications started from DevMenu or DevMenuCommand cannot be closed with Visual Studio or RunOnTarget.

Applications started from DevMenu or DevMenuCommand cannot be closed with Visual Studio or RunOnTarget. To close these applications, you must disconnect and reinsert the power cable to restart the hardware.

This problem will be fixed in an upcoming release.

gfx Bugs

Cannot initialize multi-sample textures in the Windows version of nvn.

When using nvn as a low-level graphics API and running on Win32 or Win64, trying to initialize a texture with multiple multi-samples will fail.

This problem will be fixed in an upcoming release.

Compiler Bugs

Using gold for the linker causes the link to not complete or take a very long time.

If you make it possible to use gold linker by specifying the -fuse-ld=gold.exe option (or, in VSI, if you enable the Use gold linker option), depending on the linked program, the link may not complete or may take a very long time.
Currently this problem has only been verified to occur for NX64 targets.

For programs that have this problem, use the default ID linker instead of the gold linker.

This problem will be fixed in an upcoming release.

vfx Bugs

EffectViewer stops if the background image transferred to the viewer is deleted.

In the Windows/NX DEV-KIT version of EffectViewer for NX, a bug causes EffectViewer to stop if the background image transferred to the viewer is deleted.

This problem will be fixed in an upcoming release.
