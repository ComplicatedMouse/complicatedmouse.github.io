diag

When a log file that is greater than a certain size is opened in Log Viewer, the second half of the log is not displayed.

Symptoms

When Log Viewer opens an NX Binary Log file (.nxbinlog) that is larger than several dozen megabytes, the latter part of the log is omitted and not displayed.

Workaround

To view the entire log, on the Log Viewer menu, select Convert NX Binary Log and convert the NX Binary Log file to a text file.

If the log is larger than1 GB, this conversion process can take several minutes.

Fix Schedule

There are no plans to fix this issue. Use Target Manager, selecting File > Open log file.

Other Log Viewer Bugs

Symptoms
  • When you attempt to select a range within the log, you get a broader selection of the log.
  • The indentation is broken for any metadata displayed in tool tips when you use Windows 7.
Workaround

No workaround is currently available.

Fix Schedule

There are no plans to fix this issue. Use the Log view in Target Manager.

hid

nn::hid::GetSixAxisSensorState Sometimes gets an invalid attitude rotation.

Symptoms

The SixAxisSensorState::direction value obtained by nn::hid::GetSixAxisSensorState is sometimes an invalid attitude rotation (a zero matrix or an identify matrix).

A zero matrix can be obtained in the following situations.

  • No sampling is performed after the application starts.
  • An Npad play format based on SixAxisSensorHandle is not enabled.
  • Immediately after control has returned to the application from an applet or HOME.

An identify matrix can be obtained in the following situations.

  • Immediately after the controller has established a wireless connection
  • When sampling does not start (nn::hid::StartSixAxisSensor has not been called or nn::hid::StopSixAxisSensor has been called)

In use cases where a time-series variation of attitude rotation is used, it is possible when getting the attitude rotation amount in one of these cases for the values to be deemed as abrupt rotating positions or discontinuous movement.

When getting a zero matrix or identity matrix, the quaternion obtained by nn::hid::SixAxisSensorState::GetQuaternion() is (Q, X, Y, Z) = (1.0f, 0.f, 0.f, 0.f).

Workaround

If SixAxisSensorState::direction returns a zero matrix or an identity matrix, assume the value is invalid and discard it.

Fix Schedule

Future handling is under discussion.

System Features

Restrictions on where the application can be started in VSI and RunOnTarget with the HOME Menu installed.

Symptoms

After installing and opening the HOME Menu, starting an application in VSI or RunOnTarget puts the system into a special state. The application starts, but this feature cannot be used on some screens and the application stops responding. The following screens are known to stop responding in this situation.

  • Screen for selecting accounts before starting the application
  • Screen that displays the dialog box that occurs prior to starting the application
Workaround

Do not start applications from these screens.

Starting the application with VSI or RunOnTarget may result in detailed behavior that is different from the retail version, so we recommend that you either create an NSP and install it or burn the application to a game card for final application verification and checking its behavior when entering or exiting the HOME Menu.

Fix Schedule

Not yet determined.

Graphics

The system may enter an invalid state if a texture is presented to the display before the GPU is unblocked.

Symptoms

The system can enter an invalid state after performing the following procedure.

  1. The application submits a texture to the display with GPU processing blocked.
  2. After the data has been submitted, an operation (such as a transition to the HOME Menu or sleep mode) stops the application while the GPU is still blocked.

The system entering an invalid state is specifically defined as the following states.

  • The system does not correctly transition to the HOME Menu.
  • The system does not correctly enter sleep mode.

These symptoms can occur for all currently released versions.

For example, they can occur in the following code.

nvnCommandBufferBeginRecording(&m_CommandBuffer);
{
    ... omitted ...
    // Wait for events.
    nvnCommandBufferWaitEvent(&m_CommandBuffer, &m_Event, NVN_EVENT_WAIT_MODE_GEQUAL_WRAP, nvnEventGetValue(&m_Event) + 1);
    ... omitted ...
}
m_CommandHandle = nvnCommandBufferEndRecording(&m_CommandBuffer);
// Submit commands.
nvnQueueSubmitCommands(&m_Queue, 1, &m_CommandHandle);
// Flush the submitted command, and start GPU execution.
nvnQueueFlush();
// Submit textures.
nvnQueuePresentTexture(&m_Queue, m_pWindow, index);
// Signal waiting events in the command buffer.
nvnEventSignal(&m_Event, NVN_EVENT_SIGNAL_MODE_WRITE, nvnEventGetValue(&m_Event) + 1);


In this example, nvnEventSignal() unblocks the GPU after nvnQueuePresentTexture() has been called.

As a result, an invalid state can occur if the system performs an operation such as transitioning to the HOME Menu or entering sleep mode immediately after the nvnQueuePresentTexture() call.

A similar state can also be created in other ways, such as when blocking GPU processing between multiple NVNqueues using NVNsync.


Workaround

Do not submit a texture to the display while GPU processing is blocked.

The following implementation works around this issue.

nvnCommandBufferBeginRecording(&m_CommandBuffer);
{
    ... omitted ...
    // Wait for events.
    nvnCommandBufferWaitEvent(&m_CommandBuffer, &m_Event, NVN_EVENT_WAIT_MODE_GEQUAL_WRAP, nvnEventGetValue(&m_Event) + 1);
    ... omitted ...
}
m_CommandHandle = nvnCommandBufferEndRecording(&m_CommandBuffer);
// Submit commands.
nvnQueueSubmitCommands(&m_Queue, 1, &m_CommandHandle);
// Flush the submitted command, and ensure an execution start by the GPU.
nvnQueueFlush();
// Signal the waiting events in the command buffer and unblock the GPU before submitting textures.
nvnEventSignal(&m_Event, NVN_EVENT_SIGNAL_MODE_WRITE, nvnEventGetValue(&m_Event) + 1);
// Submit textures.
nvnQueuePresentTexture(&m_Queue, m_pWindow, index);


Calling nvnEventSignal() before nvnQueuePresentTexture() assures that GPU processing will be unblocked before the texture is submitted.

As follows, there are three ways to create a state of blocked GPU processing in NVN.

  • nvnCommandBufferWaitSync
  • nvnCommandBufferWaitEvent
  • nvnQueueWaitSync

Similarly, if you are using a non-NVN graphics API, do not submit a texture to the display while GPU processing is blocked.

Fix Schedule

This behavior is by design, so there are no plans to fix it.

GLSLC Bugs

  • The TX1 versions of the glslc library may produce slightly different machine code from the Windows version, resulting in potential performance differences. The code produced on both platforms is likely to be functionally correct, and the two will converge in future releases.
  • 16-bit floating point values are supported in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface and mediump/lowp types on the other side of the interface may produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types with mediump/lowp qualifiers. You can work around this problem by explicitly constructing float types from the int or uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp and ES shaders regardless of whether the extension GL_EXT_shader_implicit_conversions is used.
    • Using FP16 math to compute the data argument for imageStore() GLSL function calls may result in corruption in the stored data. This issue is being investigated and will be fixed in an upcoming release.

RenderDoc

  • While RenderDoc is saving a large capture to disk, there is no progress indicator. Worse, if the captured application is terminated on the Nintendo Switch while the save is still in progress, the saved capture will be corrupted. Large saves can sometimes take a few seconds to finish. The only indication that the thumbnail save is complete is a subtle one: before the save is finished the thumbnail label will have "(Remote)" after it, but after the save is finished the "(Remote)" disappears.
  • The RenderDoc layer adds considerable overhead to Vulkan API calls, so programs that make many such calls run slower when the layer is enabled. Slowdowns have been as high as 10x with some large complicated programs.
  • Annotations such as bookmarks in the Event Browser or text in the Capture Comments tab do not seem to persist in captures.
  • RenderDoc crashes when opening the Event Browser in a layout that does not have it. That is, when loading a layout that only contained the Texture Viewer and Timeline and attempting to reopen the Event Browser, RenderDoc sometimes crashes. This bug is probably a general RenderDoc bug rather than an NX bug.
