Audio Effects

I/O Buffers for Effects

The nn::audio effects must be added to either SubMix or FinalMix before use. All added effects are processed through operations on the mix buffers managed by SubMix and FinalMix In other words, the mix buffers managed by SubMix and FinalMix are the input and output for the added effects.

For more information about the relationship between the mix buffers and SubMix and FinalMix, see Mix Buffers. This section describes the relationship between the various nn::audio effects and the effects managed by SubMix and FinalMix.

The code example adds the effect BufferMixer to a SubMix object that has four mix buffers. Use nn::audio::AddBufferMixer() to add the BufferMixer effect to SubMix. The instance of the BufferMixer effect is initialized at the same time it is added.

nn::audio::SubMixType subMix;
// Initialize SubMixType as having four mix buffers.
// ...

// Add the BufferMixer effect to subMix.
nn::audio::BufferMixerType bufferMixer;
nn::Result result = nn::audio::AddBufferMixer(pConfig, &bufferMixer, &subMix);
if (result.IsFailure())
{
    // Error handle
}

BufferMixer is an effect that performs mixing among different mix buffers. You can use the four mix buffers managed by subMix as I/O for bufferMixer. The following code is used to mix the content of subMix mix buffer index 0 to 2, and the content of mix buffer index 1 to 3, for example.

// Of the four mix buffers managed by subMix, mix the content of mix buffer 0 to 2 and the content of 1 to 3.
const int inOutCount = 2;
int8_t input[inOutCount] = [0, 1];
int8_t output[inOutCount] = [2, 3];
nn::audio::SetBufferMixerInputOutput(&bufferMixer, input, output, inOutCount);

The following image shows the results from the code to this point, together with the I/O relationship.

Additional nn::audio effects, such as Delay and Reverb also undergo I/O effect processing with the mix buffers that manage the added SubMix and FinalMix.

Info

You cannot add any initialized effects that were added to specific instances of SubMix or FinalMix to different instances of SubMix and FinalMix before deleting those effects from the original instances of SubMix and FinalMix. If you add effects that have already been added to specific instances of SubMix or FinalMix to different instances of SubMix or FinalMix, the following warning message appears.

[nn::audio] Warning: Passed effect is already initialized. Please confirm if (the effect address you are attempting to add) is valid effect object.

Make sure that the effect for the address included in the warning message has been added correctly.

Effect Processing Sequence

In the audio renderer, all effects are processed after all Voice processes have finished.

All effects are processed in the order in which they were appended. For example, if effects are added to FinalMix in the order of AddDelay() followed by AddAux(), the effects are processed in the order of Delay followed by Aux.

The SDK provides the following kinds of audio effects.

  • BufferMixer
  • Delay
  • Reverb
  • I3dl2Reverb
  • Aux
  • Capture
  • BiquadFilter
  • LightLimiter

The features for each audio effect are described in the following sections.

BufferMixer

Applies the volume to the mix buffer that is the input, and mixes it with the mix buffer that is the output. This effect is intended for purposes like bus routing.

Mix routing by BufferMix uses the nn::audio::SetBufferMixerInputOutput() function. The volume applied during mixing can be set using the nn::audio::SetBufferMixerVolume() function. The relationship between the input mix buffer, the output mix buffer, and the volume is as follows.

(Output mix buffer) += (Input mix buffer) * (Volume)

Delay Parameters

Low Pass Filter

LPF stands for low-pass filter. The behavior of the low-pass filter is determined by the Low Pass Amount parameter shown in the following formula.
The larger the value for the Low Pass Amount parameter, the lower the cutoff frequency and the "hollower" the sound.

\[ LPF(z) = \frac{b_{0}}{1-a_{1}z^{-1}} \]
\[ a_{1} = 0.95 * (LowPassAmount) \]
\[ b_{0} = 1.0 - a_{1} \]


The relationship between the coefficients and the cutoff frequency f can be expressed as follows.

\[ a_{1} = 2 - cos(\omega_{c}) - \sqrt{(2 - cos(\omega_{c}))^2 - 1} \]

The following code shows how the variables are defined.

\[ \omega_{c} = 2{\pi}f_{c} \]
\[ f_{c} = \frac{f}{f_{s}} \]
\[ f_{s} : sampling frequency \]

You can use these relational expressions to calculate the value of Low Pass Amount for any individual frequency.

In addition, by specifying the following parameters you can operate nn::audio::DelayType like a simple low pass filter.
For descriptions of these parameters, see the following sections.

dryGain = 0;
inGain = 1;
feedbackGain = 0;
channelSpread = 0;
delayTime = 0;

Channel Spread

The Channel Spread Matrix process is conducted only when there are multiple channels. It works to adjust the effective delay among the channels. To be specific, it performs the following matrix calculations.

For Two Channels

The channel spread assumes that the input channels are laid out in the order of FrontLeft and FrontRight.

\[ \begin{bmatrix} FeedbackDirect & FeedbackCross \\ FeedbackCross & FeedbackDirect \end{bmatrix} \]

If Input is the input sample data and Output is the output sample data, the result is the following expression.

\[ \begin{bmatrix} Output_{1ch} \\ Output_{2ch} \end{bmatrix} = \begin{bmatrix} FeedbackDirect & FeedbackCross \\ FeedbackCross & FeedbackDirect \end{bmatrix} \begin{bmatrix} Input_{1ch} \\ Input_{2ch} \end{bmatrix} = \begin{bmatrix} FeedbackDirect * Input_{1ch} + FeedbackCross * Input_{2ch} \\ FeedbackDirect * Input_{2ch} + FeedbackCross * Input_{1ch} \\ \end{bmatrix} \]
\[ Output_{1ch} = FeedbackDirect * Input_{1ch} + FeedbackCross * Input_{2ch} \]
\[ Output_{2ch} = FeedbackCross * Input_{1ch} + FeedbackDirect * Input_{2ch} \]
For Four Channels

The channel spread assumes that the input channels are laid out in the order of FrontLeft, FrontRight, RearLeft, and RearRight.

\[ \begin{bmatrix} FeedbackDirect & FeedbackCross & FeedbackCross & 0 \\ FeedbackCross & FeedbackDirect & 0 & FeedbackCross \\ FeedbackCross & 0 & FeedbackDirect & FeedbackCross \\ 0 & FeedbackCross & FeedbackCross & FeedbackDirect \end{bmatrix} \]

The various values of Feedback Direct and Feedback Cross are computed using the following expression.

\[ (FeedbackDirect) = (1 - (ChannelSpread)) * (FeedbackGain) \]
\[ (FeedbackCross) = (ChannelSpread) * (FeedbackGain) \]

Accordingly, the larger the value for Channel Spread, the greater the effect on other channels.

For Six Channels

The input channels in the channel spread differ, depending on the NintendoSDK version.

For NintendoSDK 14.0.0 and later, the input channels are assumed to be laid out in the order of FrontLeft, FrontRight, FrontCenter, LowFrequency, RearLeft, and RearRight.

\[ \begin{bmatrix} FeedbackDirect & 0 & FeedbackCross & 0 & FeedbackCross & 0 \\ 0 & FeedbackDirect & FeedbackCross & 0 & 0 & FeedbackCross \\ FeedbackCross & FeedbackCross & FeedbackDirect & 0 & 0 & 0 \\ 0 & 0 & 0 & Feedback & 0 & 0 \\ FeedbackCross & 0 & 0 & 0 & FeedbackDirect & FeedbackCross \\ 0 & FeedbackCross & 0 & 0 & FeedbackCross & FeedbackDirect \end{bmatrix} \]

For NintendoSDK 13.x.x and earlier, the input channels are assumed to be laid out in the order of FrontLeft, FrontRight, RearLeft, RearRight, FrontCenter, and LowFrequency.

\[ \begin{bmatrix} FeedbackDirect & 0 & FeedbackCross & 0 & FeedbackCross & 0 \\ 0 & FeedbackDirect & 0 & FeedbackCross & FeedbackCross & 0 \\ FeedbackCross & 0 & FeedbackDirect & FeedbackCross & 0 & 0 \\ 0 & FeedbackCross & FeedbackCross & FeedbackDirect & 0 & 0 \\ FeedbackCross & FeedbackCross & 0 & 0 & FeedbackDirect & 0 \\ 0 & 0 & 0 & 0 & 0 & Feedback \end{bmatrix} \]

The various values of Feedback, Feedback Direct, and Feedback Cross are computed using the following expression.

\[ Feedback = 0.98 * (FeedbackGain) \]
\[ FeedbackDirect = (1 - ChannelSpread) * Feedback \]
\[ FeedbackCross = 0.5 * ChannelSpread * Feedback \]

Reverb

Reverb is an effect that emulates the echoing of sound in a closed space.

When a sound occurs in an enclosed space, such as a room or a cave, the sound waves travel through the air and reflect off the walls repeatedly. This reflected sound continues to reflect repeatedly against the other walls, forming additional echoes. At each reflection some energy is absorbed by the wall, making the echo slightly quieter. As this process continues, the size of each echo decreases gradually over time. At the same time, the overall effect is to create an increasing number of echoes. Sounds formed in this way are known as reverberation.

The reverb effect (the artificial reverberations) is designed to emulate the sound of reverberation created by enclosed spaces. It can be used to create a sense of space, and the reverb parameter settings can be changed to create different types of space. For example, imagine a game character inside a large stone room. You would want a reverb where the first reflections are spread far apart (due to the time it takes the sound to travel across the large room), and where the sounds take a long time to decay (due to the high reflectivity of the stone walls and floor.) If the character then enters a small carpeted room with lots of furniture, you would want to use a reverb where the reflections would be quicker and would decay more quickly (due to the sound being absorbed by the walls.)

Reverb Models

The reverb effect separates these effects into three sounds for modeling.

  • Direct Sound
    • This sound travels directly from the sound source to the listener and arrives first. The direct sound is created by the signal that travels through the mixer without going through any effects.
  • Early Reflection
    • Sounds begin to arrive a short time after the direct sound. The first reflections are few and relatively spread out in time.
  • Late Reverberation
    • As the reflections multiply they eventually become so close together that they are indistinguishable. At that point, it becomes late reverberation. The levels of the reflections decay until they are inaudible.

Aux

This provides functionality for users to implement their own audio effect processes and incorporate them into the audio renderer. It is comprised of the following two features.

  1. A feature to get rendering process sample data from the audio renderer (ReadAuxSendBuffer()).
  2. A feature to write back any sample data to the rendering process from which data was obtained (WriteAuxReturnBuffer()).

The user can realize any effect by conducting some process on the sample data obtained using feature 1, and then writing the data back to the audio renderer using feature 2. Like other effects, Aux effects are processed in the order in which they were added. Accordingly, the user can use Aux to get sample data from any rendering process.

Aux uses two buffers at this time. (The following figure is an example of Aux added to FinalMix.)

  • Send Buffer holds the sample data taken from the buffer specified by Final Mix.
  • Return Buffer holds the sample data passed from the user. Data is read from this buffer by Aux in every frame, so the required amount of sample data must always be written to it.
    The user must take responsibility for writing the data.

For more information about the format of the sample data held in Send Buffer and Return Buffer and how to handle the data, see the API reference content for the ReadAuxSendBuffer() function and the WriteAuxReturnBuffer() function.

Info

The sizes of Send Buffer and Return Buffer correspond to the delay time before user-specified effects are applied. Note that you cannot specify the same buffer for Send Buffer and Return Buffer.

Note

The user is responsible for reading data from Send Buffer and writing data back to Return Buffer. If not enough data is written back to Return Buffer, and the audio renderer cannot load the required sample data, this might produce dropouts and other audio glitches. The user must control the sizes of Send Buffer and Return Buffer, striking a balance between the computational costs and processing time of user-defined signal processing.

Info

In past versions of the SDK supplied by Nintendo, this same functionality was realized by registering callback functions to the audio renderer. That method assumed that the callback functions would be called in every audio frame, requiring users to consume CPU time at a fixed frequency.

With Aux, the user can control the sizes of Send Buffer and Return Buffer. This enables control over the processing load balance.

Capture

Provides the functionality to acquire and process sample data without affecting the output after effect processing.
The user can periodically call the nn::audio::ReadCaptureBuffer() function to get the sample data during the rendering process.

Aux exists as a similar effect, but the following figure shows the difference in behavior between Aux and Capture. (The following figure shows an example where Aux and Capture are added to FinalMix.)



Specifically, the following points differ when compared to Aux.

  • It does not have the functionality to write back arbitrary sampling data to the rendering process.
    • It does not have a feature equivalent to nn::audio::WriteAuxReturnBuffer().
  • It does not affect the output after effect processing.
    • For this reason, the output delay after effect processing, which occurs with Aux, does not occur.

Due to these characteristics, Capture is expected to be used for purposes such as the user's independent audio analysis processing.

For more information about the sample data that can be obtained and how to handle it, see the API reference for nn::audio::ReadCaptureBuffer().

Info

The user must call the nn::audio::ReadCaptureBuffer() function periodically to get the sample data.
If sample data is not periodically loaded, old sample data will be overwritten and the recorded sample data will be lost.

Use either of the following methods to make adjustments and prevent the loss of recorded sample data.

  1. Increase the size of the buffer used, and increase the interval between loading sample data.
  2. Increase the frequency of calling nn::audio::ReadCaptureBuffer().

LightLimiter

The nn::audio library provides a limiter effect that has a relatively light processing load.

The purpose of the limiter effect is to limit signals to levels below the threshold.
By limiting the amplitude or the RMS (root mean square) below the threshold, you can prevent audio from distorting and cutting out, as can happen when signal levels exceed the threshold.

The library also provides nn::audio::AuxLimiterType as a limiter, but this effect has a lighter processing load than that limiter.

Behavior of LightLimiter

The following figure depicts how LightLimiter operates.

Using the look-ahead feature, the input signals are pre-loaded and the decision whether to limit the signals is made so the amplitude is kept below the threshold.
The limiting is conducted progressively beforehand to reduce the amplitude of signals that exceed the threshold and keep them below the threshold.
In this way, even when signals with a large amplitude are suddenly input, the amplitude is limited so the signals do not exceed the threshold.

Info

The limiting on the pre-loaded signal is stropped at the release stage, so there is a possibility that the signal will exceed the threshold at that time.
However, if the release stage lasts a sufficient time and is handled like a normal sound waveform, for practical purposes there are few situations where signals exceeding the threshold at the release phase is a problem.

Features of LightLimiter

The LightLimiter effect provided by the nn::audio library has the following characteristics.

  1. Limits the signal amplitude.
    1. The effect works to keep the amplitude of the signal below the threshold.
  2. Limiting is performed on the input signals using look-ahead.
    1. Even signals with large amplitudes that are suddenly input can be limited and kept below the threshold.
    2. The delay only amounts to the time spent looking ahead at the signals.
      1. Generally speaking, the delay is not an issue because the amount of the delay is set below the discernible level for humans.
  3. The limiter is applied independently to each channel.
    1. LightLimiter can be used with multichannel audio, and the signal limiting is conducted on each channel independently.

In addition, compared to nn::audio::AuxLimiterType, this effect has the following characteristics.

  1. The processing load is lighter.
  2. Signals are not necessarily kept below the threshold.
    1. Under the following conditions, the signal amplitude may exceed the threshold.
      1. If the look-ahead time is not long enough compared to the attack phase time.
      2. During the release phase.

Info

This is a detailed explanation of the relationship between the attack time and the release time.

In the table of LightLimiter parameters, the attack time is described as "the amount of time before the amplitude is limited" and the release time is described as "the amount of time before the limiting is released." However, in actuality, limiting and releasing are not performed completely with this timing.
If the amplitude limiting rate is represented by a range of 0 to 1, 0 is the situation where the limiter is not limiting the amplitude at all, and 1 is where limiting is applied completely and the amplitude is held below the threshold. The transition of the amplitude limiting rate as it changes from 0 to 1 can be expressed in the following way:

\[ 1 - e^{- t / T_{attack}} \]

The attack time is the time in the preceding expression where

\[ t = T_{attack} \]

In other words, it is the time until

\[ 1 - 1 / e \fallingdotseq 0.63 \]

becomes the value of the amplitude limiting rate.

In this same way, the transition of the amplitude limiting rate as it changes from 1 to 0 can be expressed as follows.

\[ e^{- t / T_{release}} \]

The release time is the time in the preceding expression where

\[ t = T_{release} \]

In other words, it is the time until

\[ 1 / e \fallingdotseq 0.37 \]

becomes the value of the amplitude limiting rate.

This transition in the amplitude limiting rate is represented by exponential decay, and the amplitude limiting rate itself is calculated using floating-point, resulting in arithmetic errors.
For this reason, in the attack phase, the amplitude limiting rate approaches a value of 1 but does not necessarily become 1.
And in the same way, in the release phase, the amplitude limiting rate approaches a value of 0 but does not necessarily become 0.
