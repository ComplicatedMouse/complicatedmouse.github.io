General Graphics

Non-60-fps applications interrupt system GPU processing at each VSync event.

Symptoms

When a PresentInterval greater than 1 is specified, due to GPU resource limitations, the application interrupts each VSync event when it is suppoed to synchronize with PresentInterval and interrupt the system GPU processing. As a result, it has been observed that the GPU processing, for the portion that interrupted the system, takes longer than usual. For more information about the typical behavior of GPU resource limitations, see GPU Resource Allocation.

Workaround

If this bug leads to issues, use DevMenuCommand to disable the GPU resource limitations. For more information about DevMenuCommand, see DevMenuCommand.

DevMenuCommand debug disable-gpu-resource-limitation
Fix Schedule

This bug will be fixed in an upcoming release.

GPU resource limitations do not work if the HOME Menu is not enabled.

Symptoms

If the HOME Menu is not enabled, GPU resource limitations do not operate and the application can use 100% of the GPU time. For more information about the typical behavior of GPU resource limitations, see GPU Resource Allocation.

Workaround

If this bug leads to issues, use DevMenuCommand to enable the HOME Menu. This feature is still under development so it is only available to some users. For more information about DevMenuCommand, see DevMenuCommand.

DevMenuCommand debug switch-menu homemenu
Fix Schedule

This bug will be fixed in an upcoming release.

If an application's GPU command is run immediately after a VSync event, the GPU execution timing becomes unstable.

Symptoms

When an application's GPU command is executed immediately after a VSync event, by waiting on a display release fence or other means, it competes with the beginning of the system's GPU time allocation, and the GPU execution timing becomes unstable.

Workaround

Do not issue commands immediately after VSync events; that way, a GPU command is not executed immediately after a VSync event. It is also acceptable to have application commands issued during times when GPU time is allocated to the system. The commands are executed immediately after GPU time is allocated to the application.

Fix Schedule

This bug will be fixed in an upcoming release.

If you select the Quick (recommended) installation option when installing the NVIDIA graphics driver and then switch to the driver, the screen sometimes goes black and does not recover. If the screen does not recover, in some cases you can still use the computer by logging in via remote desktop.

GLSLC Bugs

  • Multi-threading does not work with the GLSLC compiler. If you want applications to use multiple compile objects in parallel, they must use parallel processes instead. This bug is being worked on for the next release.
  • EOL markers must be present in #include files, else a compilation error will occur.
  • Transform feedback varyings specified in shader source do not work correctly.
  • Compute shader data subsections inside a GPU code section do not start with the magic number GLSLC_GPU_CODE_SECTION_DATA_MAGIC_NUMBER.
  • The TX1 versions of the glslc library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms is likely to be functionally correct though, and will be combined in future releases.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension. When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface, and mediump or lowp types on the other side of the interface, could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages produces a linker error.
    • int and uint types cannot be implicitly converted to float types qualified with mediump or lowp qualifiers. You can work around this limitation by explicitly constructing float types from the int and uint variables (similar to explicit casting in C/C++). This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp in addition to ES shaders regardless of whether the GL_EXT_shader_implicit_conversions extension is used.
    • FP16 math when used with imageStore may cause corruption in some cases. This issue is being investigated and will be fixed in an upcoming release.

VSI Bugs

Cannot handle exceptions thrown in SDK libraries.

The NX32 and NX64 SDK libraries do not support C++ exceptions.
For this reason, when exceptions are thrown in the libraries, they are canceled.

If the code is only propagated to the application, you can enable C++ exceptions at build to make it possible to handle exceptions.

Link errors can occur when using libnn_multimedia.a for applications that enable RTTI.

Symptoms

The SDK libraries for NX32 and NX64 have been built with RTTI enabled.
RTTI is disabled for libnn_multimedia.a, however. As a result, link errors can occur in applications that use this library.

Workaround

Use -fno-rtti when compiling applications if link errors are occurring.
In Configuration Properties, set Enable runtime type information to No.

Fix Schedule

This bug will be fixed in an upcoming release.

The number of files that can be open at any time is restricted as follows.

File System Maximum Number of Open Files
File system on the host PC (nn::fs::MountHost() or nn::fs::MountHostRoot())

338 (32-bit environment)

182 (64-bit environment)
Resource data (nn::fs::MountRom())

338 (32-bit environment)

The function fails with nn::fs::ResultAllocationMemoryFailed if you attempt to open more than the maximum number of files.
We plan to improve the number of files that can be opened at the same time.

nn::fs::ReadFile() or nn::fs::WriteFile() may disconnect from Target Manager.

A bug may cause the connection to the Target Manager to be disconnected (the status in Target Manager becomes "Disconnected") when executing nn::fs::ReadFile() or nn::fs::WriteFile() through MountHost(), MountHostRoot(), or MountRom() in raw mode.

The conditions for this problem occurring are based on the size of the file loaded in nn::fs::ReadFile() or nn::fs::WriteFile() and the size of previously loaded files, but the details are currently being investigated.

If this problem occurs repeatedly, avoid it by changing the buffer size specified in nn::fs::ReadFile() or nn::fs::WriteFile().

The FsSaveData sample stops.

Symptoms

The FsSaveData sample stops with the message, "[fs] Error: Specified target is not found."

Workaround

For NX32:

  • Remove the comment tags on UserAccountSaveDataSize and UserAccountSaveDataJournalSize of the Application.arm.ilp32.meta file.

            <UserAccountSaveDataSize>0x0000000000400000</UserAccountSaveDataSize>
        <UserAccountSaveDataJournalSize>0x0000000000100000</UserAccountSaveDataJournalSize>

  • From VSI, rewrite the following under Configuration Properties / Build Event / Post-build event:

    $(NintendoSdkRoot)\Tools\CommandLineTools\AuthoringTool\AuthoringTool.exe createnspmeta --type Application --meta $(NintendoSdkSpecFilesDirectory)Application.arm.ilp32.meta -o $(OutDir)\$(TargetName).nspd

For NX64:

  • From VSI, rewrite the following under Configuration Properties / Build Event / Post-build event:

    $(NintendoSdkRoot)\Tools\CommandLineTools\AuthoringTool\AuthoringTool.exe createnspmeta --type Application --meta $(NintendoSdkSpecFilesDirectory)Application.aarch64.lp64.meta -o $(OutDir)\$(TargetName).nspd

perf Bugs

In the NX Windows version of perf, Tick values measured using nn:perf::GpuMeter do not correspond to the Tick values returned by the nn::os::GetSystemTick() function.

When measuring Tick values using nn:perf::GpuMeter, the Tick values stored in nn:perf::LoadMeterBase::Section::begin and nn:perf::LoadMeterBase::Section::end are slightly different from the Tick values measured using nn::perf::CpuMeter or the nn::os::GetSystemTick() function.
However, the speed at which the counter ticks is the same for both meters, so adjusting the Tick values by the interval (end - begin) (which represents the processing time) makes it possible to use GpuMeter and CpuMeter interchangeably.

This restriction will be removed in a future release.

This bug is limited to the NX Windows version of perf.
On the development hardware, Tick values measured using GpuMeter correctly correspond to Tick values measured using CpuMeter and Tick values returned by nn::os::GetSystemTick().

Time Bugs

Time advances more slowly during sleep mode.

Symptoms

There is a bug where the various clocks that use the time library become slower during sleep mode.

As a result, those clocks return values that are further in the past than anticipated when waking from sleep mode. The affected clocks are as follows.

  • nn::time::StandardSteadyClock
  • nn::time::StandardUserSystemClock
  • nn::time::StandardNetworkSystemClock
Workaround

None.

These clocks will start returning the correct values after restarting the system.

Fix Schedule

This problem will be fixed in an upcoming release.
