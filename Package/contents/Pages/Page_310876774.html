Overview

This part of DevMenu provides various features essential for running applications.

  • Installing and deleting
  • Listing, starting, and ending
  • Writing to game cards
Background Application: No Background Application: Yes

How to Use

List

This list displays the applications installed on the development hardware console.

To display the version or application name, you must specify the display version and title name in the application NMETA file.
For more information about setting the title, see Writing the NMETA File.

Info

The application control data is cached by the system in system memory. This cache is loaded, rather than the application control data on the game card. If you do not want the HOME Menu or DevMenu launcher to load this cache, use DevMenuCommand to invalidate the cache. For more information, see Features > Controlling the Development Hardware > DevMenuCommand > Running and Controlling Applications > Invalidating Application Control Data Cache.

Installing

You can install NSP files (for applications, downloadable content, and patches) that have been saved to a microSD card.
Clicking the Install from SD button updates the screen to display the list of NSP files on the microSD card.

Note

The DevMenu cannot be operated during installation.

Installation Devices

Choose where to install the NSP file from among the following devices.

Device Process
Auto (Default) Preferentially tries to install to a microSD card.
If installation to a microSD card is not possible, installation to system memory is attempted.
System Memory Tries to install to system memory.
SD Card Tries to install to a microSD card.

Individual Installation

Select the NSP file to install, and press the A Button. A confirmation message is displayed. Click the Start button to begin installation.
Returns to the screen showing the list of NSP files after installation completes.

Batch Installation

Installing All NSP Files on an SD Card

Click the Install All button to batch-install all of the NSP files on the microSD card.
Click the Start button to begin installation. Returns to the screen showing the list of NSP files after installation completes.

Installing All NSP Files Directly Under a Directory

Click Select Directory to display a screen listing the directories on the SD card that meet the following conditions.

  • There is at least one NSP file directly under the directory.
  • It is the root directory of the SD card, or any subdirectory under that path.

Select the directory and click Start to start installing all NSP files in the selected directory.
Click Install All to start installing the NSP files in all directories that meet these conditions.
After installation is complete, the directory listing screen is displayed again.

Click Select File to return to the NSP file listing screen.

Installing HTTP Server NSP Files

Info

This feature is available in NintendoSDK Firmware for NX 10.0.0-6.0 and later.

You can install NSP files located on the HTTP server.
Click Install via HTTP to transition to the screen for selecting the NSP files.

The HTTP server redirect setting is supported.

Info

The HTTP server redirect setting is supported in NintendoSDK Firmware for NX 13.0.0-3.0 and later.

The following conditions must be met for an NSP file to appear.

  • There is an HTTP server within the network that the system is connected to.
  • The URL is configured correctly. (If the HTTP server URL is not configured, click Edit URL and enter the URL.)
  • The URL being referenced is configured on the HTTP server to have no access authentication, and to show the list of files.
  • A microSD card is inserted.
Info

The microSD card is used to save the management file that stores the URL, and to temporarily store the downloaded NSP file.
The file path for the management file where the URL is stored is /NintendoSDK/InstallationData/InstallViaHttp.txt.

The target installation devices for the selected NSP file are the same as the devices for NSP files installed from a microSD card.

Edit URL

You can use the software keyboard to enter a URL to reference.

Automatic

Info

This feature is available in NintendoSDK Firmware for 13.0.0-5.0 and later.

Gets information in 1-minute intervals from the HTTP server, and automatically downloads and installs the discovered NSP files.
The NSP files that were processed automatically and succeeded or failed are recorded in the history and are not downloaded the next time Automatic is run.
The downloaded NSP files are installed to an microSD card as the fixed storage location.

To stop the automatic process, click Cancel and suspend automatic trigger. The in-process download and installation will be cancelled.
To resume the automatic process, click Restart automatic trigger.

When downloading and installation are not in progress, the Close button is enabled and you can use this to close the screen.
Also, if history exists at this point, the Delete all history button is enabled, allowing all history to be deleted at once.

Info

Cancel and suspend automatic trigger is available in NintendoSDK Firmware for NX 13.2.0-1.0, 14.0.0-1.0 and later.
Delete all history is available in NintendoSDK Firmware for NX 13.2.0-1.0, 14.0.0-2.0 and later.

Info

microSD cards can be used in the following ways with Automatic.

  • Management files storing URLs
  • Temporally storing downloaded NSP files
  • Storing download and installation histories
NSP File Status

The NSP file status shows as one of the following.

Status Description
Waiting Waiting for download.
Downloading Downloading.
Download failed

The download failed.

This can happen if there was a network failure or there was insufficient space on the microSD card.
Installing Downloading succeeded and the file is now installing.
Install failed

The installation failed.

This can happen if the NSP file is corrupted or not valid.
Installed Installed.
Not target
 The file is not a download target because there are more than 100 items in the history and the file has an old last-updated date and time.

Use one of the following methods to install a file for which installation failed.

  • Use the feature described in Installing HTTP Server NSP Files and manually select the failed NSP file.
    • Note that even if manual installation succeeds, the file's status remains as failed in the Automatic history.
  • Use Delete all history to delete all history, then use this feature to redo the download and install.
    • Items that are already installed are also re-downloaded.
  • Change the last updated date and time for the failed NSP file and use this feature to download and install the file again.
Download and Installation History

The download and installation results for up to 100 NSP files are stored in the history.
If 100 entries are exceeded, the earliest entries in the history are deleted.
NSP files downloaded and installed without the use of the Automatic feature are not recorded in this history.

Download Order and Conditions for Downloading

NSP files are evaluated in order to decide whether to download them, starting with the file with the earliest last updated date and time. If a file satisfies the following conditions, it is downloaded.

  • The target filename does not exist in the history.
  • The last updated date and time of the target file is different from the date and time in the history.

If there are more than 100 entries in the history, only 100 NSP files with new last updated dates are evaluated for possible download.

Writing to Game Cards

Use this feature to install an NSP file (for the application itself and patches) that has been saved to a microSD card.
Click the Write to Game Card button to open the screen for selecting the NSP file on the microSD card to write to the game card.
To write to a game card with settings for China, select the Game Card for China check box using NintendoSDK Firmware for NX 9.1.0-1.0 or later.

Starting and Exiting

Press the A Button to start the application. After starting the application, DevMenu resides in the background without closing.
Press the HOME Button in the application to restore control to DevMenu.

If the application is running in the background, the following operations are possible with DevMenu. The application state can be checked by checking whether the application is displayed at the bottom of the screen.

Operation Processing Executed
Pressing the HOME Button Restores control to the application.

Pressing the X Button

 Exits the application. (This works even outside of the Application page).
Simultaneously Pressing the L, R, and X Buttons

Forcibly quits the application. (This works even outside of the Application page).
Starting another application
(Pressing the A Button → Launch)		 Closes the application that is running, and starts the selected application.
Note

Startup fails if downloadable content or patches are installed before the application is installed.

Installing and Uninstalling Launch Parameters

Files installed on microSD cards can be retrieved by applications as launch parameters.
For more information about application launch parameters, see Getting Launch Parameters.

Using DevMenuCommand

See Installing and Uninstalling Application Launch Parameters in the DevMenuCommand documentation.

Using PCs

  1. Open the microSD card on a PC and create a directory named NintendoSDK\ApplicationLaunchParameters.
  2. Create a file for storing the parameters, and save it with a name of the following format in the directory created in step 1.
    • The data in this file is passed, unmodified, to the application.
(application_ID).nxargs

.nxargs is the filename extension. Use a sixteen-digit single-width alphanumeric hexadecimal string as the application ID.

01004B9000490000.nxargs
Note

Installed parameters are only available when DevMenu is selected from the Initial Boot Menu.
Note: You cannot pass parameters to applications when the HOME Menu is selected, even when parameters are installed.

For more information about switching menus, see Initial Boot Menu in Development Assistance Settings.

Note

The maximum size of the launch parameter is 16 KiB.

Secure Startup

Info

The following information describes the types of scenes and content for which secure startup is available in each NintendoSDK firmware version.

  • Earlier than NintendoSDK Firmware for NX 10.0.0-6.0
    • Secure startup can be used only in exhibition mode. (For more information about exhibition mode, see Exhibition Features.)
  • NintendoSDK Firmware for NX 10.0.0-6.0 and later (★)
    • Secure startup can be used even if the mode is not exhibition mode. In DevMenu, select the application and perform the secure startup.
  • NintendoSDK Firmware for NX 10.1.0-3.0 and later (★)
    • Secure startup can be used with the NSP files for patches and downloadable content.
  • NintendoSDK Firmware for NX 11.0.0-3.0 and later
    • A passphrase for secure startup can be entered in DevMenuApp, which also makes secure startup possible in the HOME Menu environment (although only in Dev mode).
  • NintendoSDK Firmware for NX 11x (11.0.1-1.2 and later), 12x (12.0.0-3.0 and later) (★)
    • Secure startup is also possible in the HOME Menu in Prod mode. For more information, see Using Secure Startup for Applications later on this page.
  • NintendoSDK Firmware for NX 13.0.0-3.0 and later
    • You can write an NSP file for secure startup to a game card and start it from there.

If the following conditions all apply, use the SystemUpdater package to initialize the development hardware and update the firmware, and use the versions of DevMenuApp and DevMenuCommand that are included in the package.

  • The secure startup feature is being used in the current environment and will continue to be used after the firmware is updated.
  • Either of the following conditions apply.
    • For the preceding versions marked with (★), the development hardware firmware must be updated.
    • The existing firmware version is NintendoSDK Firmware for NX 11.0.1-1.2.

If these conditions apply and you update the firmware using the DevKitVersionUpdater package, there will be a break in compatibility of the system data for secure startup. The following features might not work anymore, and the save data for secure startup in the old format linked to an unknown application ID might be displayed on the Save Data Management page.

  • Installing and uninstalling content for secure startup
  • Starting and loading content for secure startup

Secure startup is a method of startup that reduces the risk of your application data being leaked if the development hardware is stolen.

If you install an NSP file for secure startup on the development hardware, you can require a passphrase to be entered when the application starts from DevMenu or the HOME Menu.
The data contained in the application is decryptable and the application can be started (in the secure startup state) only if the correct passphrase is entered.
The secure startup state is maintained until the development hardware is powered off. If the development hardware is shut down and restarted, the passphrase must be entered again.

The secure startup state has the following characteristics. These characteristics help prevent content from being stolen in a decryptable state.

  • The development hardware does not sleep or restart.
    • If a normal sleep operation such as tapping the POWER Button is performed on the development hardware, it shuts down instead of going to sleep.
    • If you try operations that restart the device, such as clicking the restart button in the upper right of the DevMenu screen, it shuts down instead of restarting.
Info

In NintendoSDK Firmware for NX 11.0.0-6.0 and later, you can temporarily allow sleep functionality.
This feature is expected to be used for testing conformance with guidelines.

Procedure

  1. Enter the appropriate passphrase and put the system in the secure startup state.
  2. In the screen for DevMenu or DevMenuApp, simultaneously press the ZL Button, ZR Button, + Button, and Down Button.
  3. Select the duration for which to temporarily allow sleep, then select the OK button.
    • 0 min(disabled), 5 min, 10 min, 30 min, 60 min, 90 min

Specification

  • When the temporary sleep allowed period ends in a non-sleep state (when the development hardware is operable), it returns to the sleep prohibited state.
  • If you set sleep mode to allowed again during the temporary sleep allowed period, the newer setting takes effect.
  • If you use DevMenu to temporarily allow sleep, the development hardware automatically shuts down if the temporary period ends while the system is in sleep mode.
  • If you use DevMenuApp to temporarily allow sleep, the development hardware shuts down if it wakes after the temporary period has ended.

The following behavior is added while exhibition mode is enabled in anticipation of the device being stolen in the exhibition environment.

  • The development hardware shuts down if it shifts from being charged with a genuine AC adapter to not being charged.
Info

The maximum amount of content that can be installed to the development hardware is as follows.

  • Maximum of 10 (earlier than NintendoSDK Firmware for NX 10.1.0-3.0)
  • Maximum of 96 (NintendoSDK Firmware for NX 10.1.0-3.0 and later)
    • Downloadable content is counted by index.
Note
  • The information contained in the application control data is what is passphrase-encrypted in an NSP file for secure startup.
    • Elements under the Application tag in NMETA files such as the title or icon correspond to application control data.
    • To protect information in the application control data, use the AuthoringTool replacenspmeta command or AuthoringEditor to change values in the original NSP file to an extent that does not impact the behavior of the application before you make the secure startup version of the NSP file.
  • Save data created by an application is stored in the same way as during normal startup. It is not related to the encryption for secure startup.

Creating NSP Files for Secure Startup

For more information about creating NSP files for secure startup, see the following documentation.

  • Features > Building > Creating Application Images > Creating NSP Files for Secure Startup.
  • Features > Building > Creating Application Images > Tool Reference > AuthoringTool > Creating NSP Files for Secure Startup.

Installing an NSP File for Secure Startup

To install an NSP file for secure startup, you can use the same procedure used for installing a regular NSP file.
When you install an NSP file for secure startup, save data settings for secure startup are created with the name Setting data for Secure Startup.

Note

If you intend to delete the save data settings for secure startup, confirm that all content for secure startup has been uninstalled.
Secure startup is not possible after this has been deleted.

Writing an NSP File for Secure Startup to a Game Card

If you plan to write an NSP file for secure startup to a game card and start it from there, use this procedure instead of installing the NSP file.
To write an NSP file for secure startup to a game card, you can use the same procedure used for writing a regular NSP file.
For more information about writing to a game card, see the following documentation.

  • NX Development Card Features > Writing the Application

Using Secure Startup for Applications

Either install the NSP file for secure startup, or write it to a game card that is inserted in the device, and then perform the following procedure.

DevMenu Environment
  1. From the list of applications, select the one you want to start and then press the A Button.
  2. Enter the correct passphrase on the passphrase input screen.
  3. The application is started securely.
HOME Menu Environment
  1. Start DevMenuApp.
  2. From the list of applications, select the one you want to start securely and then press the A Button.
  3. Enter the correct passphrase on the passphrase input screen.
    • At this point, the application is in the secure startup state and can be started.
  4. Press the HOME Button and return to the HOME Menu.
  5. Run the application that you want to start securely.
Note

The following restriction applies to NintendoSDK Firmware for NX 12.0.0-3.0 and later of the HOME Menu environment.

  • If Initialize Console Without Deleting Save is used in maintenance mode, all of the installed content for secure startup must be uninstalled.

If the content is not uninstalled, the following kinds of problems can arise. To resolve these problems, use the SystemUpdater package to initialize the system.

  • New content for secure startup cannot be installed as though the limit has been reached, even though it has not.
  • Content for secure startup cannot be started.

Viewing Details

Press the + Button or - Button to display the application details.

Manage

Provides features for deleting and resetting the required version for application-related content (including the application, downloadable content, and patches).
For more information, see Managing Content.

Content Availability

This is a debug feature for controlling the availability of downloadable content (DLC).
Click the Content Availability button in any application details screen to display the settings screen. The settings screen is the same no matter where Content Availability is clicked.
For more information, see Managing DLC Availability.

Unqualified User

This debug feature is for setting a user to a status with restrictions (Unqualified).
For more information, see Managing User Restriction Status.

Metadata Information

Shows the information set in the NMETA file. Display elements will be added later.

Application Information

Shows the total size of the application, downloadable content, and patches.

Managing Content

It provides the following features.

  • Resetting the required version
  • Deleting the application, downloadable content, and patches
  • Displaying the ROM ID hash
    • For more information, see Identifying the Build Output.

Reset RequiredVersion

Resets the application's required version.

Delete

Deletes the selected content. In the following cases, the content is not deleted and only the system management data is deleted.

  • The game card was removed after insertion.
  • The microSD card was removed after the content was installed on it.
Info

Content written to a game card cannot be individually deleted.

Delete All

Deletes all listed content.

Info

Deleting the content of a game card still does not erase data written to the game card. Only the system's management data will be deleted.

Show Hash

Displays the ROM ID hash for the selected content.

Managing DLC Availability

Info

This feature is available in NintendoSDK Firmware for 13.0.0-3.0 and later.

This feature for development and debugging is based on the assumption that DLC will be available in the production environment.

In the development environment, all installed DLC is available to all accounts. On the other hand, in the production environment, the availability of DLC differs for each user.
You can use this feature to set whether DLC is available to individual users in the development environment.
The settings are retained even if the application or the system is restarted. Before starting the application, reconfigure the settings as needed to get the desired initial state.

The settings configured in this screen are applied to all applications.

Note

If you configure the following settings on this screen, the account selection screen appears when the application starts.

  • Select the Content Availability Settings For Debug check box.
  • Click Set to set the availability of the DLC.

Content Availability Settings For Debugging

Selecting this check box enables the debugging feature for controlling what DLC is available.
Even if this check box is cleared, you can still set the availability of DLC using the Set button.

Show All

Shows the indexes of the available DLC configured for the user.

Set

Use the software keyboard to enter the indexes of the DLC to make available.
You can specify multiple indexes by separating them with commas (",") and you can specify a range with the hyphen ("-").
You can specify indexes in the range from 1 to 2000.

  • Specifying multiple indexes
1,2,3,4,5...
  • Specifying ranges
1-10,13-20,40-200,500-2000...
Note

For users in the following specific states, the settings might not be applied immediately.

  • The user account selected at startup
    • Restart the application to apply the settings.
  • The user in the open state
    • Close the user account to apply the settings.

Clear

Clears the information about the available DLC set for the user account.

Managing User Restriction Status

Info

This feature is available in NintendoSDK Firmware for 13.0.0-3.0 and later.

This feature is provided for development and debugging based on the assumption that there will be restricted (Unqualified) users.

In the development environment, all users are treated as unrestricted (Qualified) users. On the other hand, in the production environment, restrictions vary from user to user.
You can also use this feature to set individual users as restricted (Unqualified) users in the development environment.
For more information about restricted users, see User Account.

Target Application Is Running

Target Application Is Not Running

Enable All

Changes the Force Unqualified For Debug option to Enabled for all users.
For more information, see Force Unqualified For Debug.

Disable All

Changes the Force Unqualified For Debug option to Disabled for all users.
For more information, see Force Unqualified For Debug.

Force Unqualified For Debug

Select Enabled to set the user to the status with restrictions (Unqualified).
The default is Disabled.

You cannot select Enabled in the following situations.

  • The user in the open state
    • Close the user account to apply the setting.
  • The Force Unqualified For Debug option has already been set to Enabled for two other applications .
    • To configure this setting for a new application, change the setting to Disabled for all users in an application that has already been configured.
    • If you restart the development hardware, you can revert all settings to Disabled.
Note

This feature can be set for a maximum of two applications. Trying to change the setting to Enabled for a third application fails.

Storage Capacity

Displays the system memory built into the device and the capacity of the microSD card currently inserted.
To display the capacity of the card, you must start the system with the microSD card inserted.
