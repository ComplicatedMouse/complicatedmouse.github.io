Introduction

Purpose of This Documentation

This documentation describes the features provided by the nifm library for managing network connections included in NintendoSDK.

Intended Audience

This documentation is intended for developers who are using NintendoSDK to develop applications that require the console to communicate using Internet communication or local communication.

Related Documentation

Also see the nifm API Reference.

nifm Library Features

The nifm library provides an API for managing network connections in the NintendoSDK environment.

The library for managing network connections provides a variety of features, including functionality to mediate between the demands of the application and the system, to select which network interface to use, to establish network connections, and to determine whether networks can be used.
If the application is designed to be used with a network connection, this library can be used to make the request to use the network and to establish the appropriate connection. If the application cannot use the network, the library can be used to get the reason.

Considerations and Restrictions

Common

Importing the Network Connection Settings

Upon installation, the default network connection settings are for a wired connection, automatically acquiring the DHCP and DNS, with no proxy.

To use a different network connection, you must first register the appropriate network connection settings using the Settings Manager. Default settings are overwritten at registration.

The network connection settings file and the batch file for importing the connection settings used in the following example are stored in the same directory as the source code for the NifmNetworkConnectionSimple sample program.

  • Importing the Network Connection Settings File
$> ImportNetworkSettings-spec.*.bat

When you run this batch file, you are presented with a selection of targets you can import the settings to (such as Windows) and a selection of configuration files (such as wired or wireless) to choose from. Select the combination you want and import the settings. If necessary, you can restart the development hardware at the same time and display the export results of the settings. If no error appears, importing was successful.

This batch file uses the Settings Manager internally to import settings. As a result, for example, you can import from the command line in Windows, as follows.

$> Tools\CommandLineTools\Settings\SettingsManager.exe Import c:\\absolute\\path\\to\\networkSettings_ForEthernet.txt

When importing for development hardware using this method, you must manually restart the system.


The settings file is coded as follows.

In this example, only one group of settings is configured. However, you can code multiple <dict> elements in <array>, and by doing this you can have multiple sets of connection settings in a single file. In the current version, you must use dummy settings for those settings that would normally be unnecessary for a wired connection, such as SSID and security.

  • Example of Network Settings to Import (Wire-Line, With IP Address Obtained Automatically)
{
    "head": {
        "version": "1.0"
    },
    "body": {
        "network_settings": [
            {
                "name": "ethernet-profile", // The name of the network connection settings.
                "id": "93188440-b6d5-14b7-056b-08cd57910964", // The ID of the network connection settings.
                "auto_connection": true,    // Flag for whether to connect automatically using the network connection settings.
                "large_capacity" : true,    // Flag for whether to allow large-volume communications.
                "nic_type": "Ethernet", // NIC type [Ieee80211, Ethernet]
                "ssid_config": {
                    "name": "", // SSID
                    "non_broadcast": false  // Flag for whether to try connecting to a stealth access point using these settings.
                },
                "security": {
                    "authentication": "Open",   // Authentication method [Open, WpaPsk, Wpa2Psk]
                    "encryption": "None",   // Encryption method [None, Wep, Aes]
                    "key_material": "x"    // Password or passphrase.
                },
                "ip_settings": {
                    "auto_ip": true,    //  Flag for whether to get the IP address automatically.
                    "address": "0.0.0.0",   // IP address.
                    "subnet_mask": "0.0.0.0",   // Subnet mask.
                    "default_gateway": "0.0.0.0",   // Default gateway.
                    "auto_dns": true,   // Flag for whether to get the DNS server address automatically.
                    "preferred_dns": "0.0.0.0", // Address of the preferred DNS server.
                    "alternate_dns": "0.0.0.0"  // Address of the alternate DNS server.
                },
                "proxy": {
                    "enables": false,   // Flag for whether to use a proxy.
                    "hostname": "", // FQDN proxy (does not include scheme or port).
                    "port": 0,  // The proxy's port number.
                    "authenticates": false, // Flag for whether to conduct proxy user authentication.
                    "username": "", // The user name for proxy user authentication.
                    "password": ""  // The password for proxy user authentication.
                },
                "mtu": 1400 // MTU
            }
        ]
    }
}
Key Description
name The name for the network connection settings.
id The ID for the network connection settings. Use the UUID format (hexadecimal string of 8-4-4-4-12 characters).
If this is being imported by the Settings Manager from somewhere external to the system, any unique value in the file will do.
large_capacity Indicates whether large capacity communication is permitted. In general, we recommend that developers specify true for this key.
auto_connection Flag to determine whether to use the network settings for automatic connection.
nic_type

The NIC type. [Ieee80211, Ethernet]
ssid_config.hex SSID (hexadecimal representation) name and hex can be used exclusively.
ssid_config.name SSID (ASCII string) name and hex can be used exclusively.
ssid_config.non_broadcast Flag that determines whether to perform a connection test with these settings for stealth APs.
security.authentication The authentication method [Open, Shared, WpaPsk, Wpa2Psk].
security.encryption The encryption format [None, Wep, Aes]
key_material Password or passphrase
ip_settings.auto_ip Flag that determines whether to automatically get an IP address.
ip_settings.address The IP address.
ip_settings.subset_mask The subnet mask.
ip_settings.default_gateway The default gateway.
ip_settings.auto_dns Flag that determines whether to automatically get a DNS server.
ip_settings.preferred_dns The address of the preferred DNS server.
ip_settings.alternate_dns The address of the alternate DNS server.
proxy.enables Flag that determines whether a proxy can be used.
proxy.hostname The FQDN proxy. Does not include scheme (such as http://) or port (such as :8080).
proxy.port Port number of the proxy.
proxy.authenticates Flag that determines whether to perform proxy user authentication. The only supported authentication method is basic authentication.
proxy.username The user name for proxy user authentication.
proxy.password The password for proxy user authentication.

Exporting Network Connection Settings

As described previously, use the Settings Manager to export the connection settings.

However, the access point and proxy passwords or passphrases are sent as plain text, so be careful when handling the output files.

$> Tools\CommandLineTools\Settings\SettingsManager.exe Export c:\\absolute\\path\\to\\networkSettings.txt

localhost

If communications only take place within the console, you do not need to submit a network use request.

Socket Shutdown

The socket is shut down in the following situations, regardless of whether a network usage request is used (which is the case when communication is only with the localhost.) After the socket is shut down, create the socket again if necessary.

  • The IP-configured network gets disconnected, due to a factor such as moving away from the wireless AP, detaching the LAN cable, or turning on airplane mode.
  • The system entered sleep mode while connected wirelessly.
  • The system was shut down.

NX Version

Restrictions When Using Local Network Node

For applications using SDK 10.0.0 and later, the communication local network mode after calling nn::nifm::SetLocalNetworkMode(true) can be restricted by Nintendo when it is seen as an invitation to hackers.

Socket Usage Before the Network Use Request Is Accepted

The evaluation of the network use request might be delayed if the system's CPU resources are consumed by frequent socket sending and receiving during the time after the request has been submitted but before the request has been accepted.

The occurrence of this problem is not limited to times when sockets are being used, but you need to be careful because it is common for the thread submitting the network use request to be different from the thread executing socket sending and receiving. Do not conduct external communication using sockets before the network use request has been accepted.

Windows Version

Importing the Network Connection Settings

In the Windows version, the settings imported by the Settings Manager tool can be deleted by user operations, file deletion utilities, and similar operations. If necessary, re-import the settings before connecting.

Examples of Network Connection Settings

The Windows version of the network connection settings must conform to the Windows network environment. The following examples show the network connection settings for several typical environments.

Example 1: Wired Connection

{
    "head": {
        "version": "1.0"
    },
    "body": {
        "network_settings": [
            {
                "name": "ethernet-profile",
                "id": "93188440-b6d5-14b7-056b-08cd57910964",
                "auto_connection": true,
                "large_capacity": true,
                "nic_type": "Ethernet", // Specify a wired connection just like for a Windows PC.
                "ssid_config": {
                    "name": "",

                },
                "security": {
                    "authentication": "Open", 
                    "encryption": "None",
                    "key_material": ""
                },
                "ip_settings": { // Not used with the Windows version, so it can be left as is.
                    "auto_ip": true, 
                    "address": "0.0.0.0",
                    "subnet_mask": "0.0.0.0",
                    "default_gateway": "0.0.0.0",
                    "auto_dns": true, 
                    "preferred_dns": "0.0.0.0",
                    "alternate_dns": "0.0.0.0"
                },
                "proxy": {
                    "enables": false, // The setting when a proxy is not used. If a proxy is used, see example 3.
                    "hostname": "",
                    "port": 0,
                    "authenticates": false,
                    "username": "",
                    "password": ""
                },
                "mtu": 1400 // Not used with the Windows version, so it can be left as is.
            }

}

Behavior When Network Usage Request Is Submitted

Maintains the connection state of Windows when the network usage request is made. No actual attempt to connect to the network is made.

In a Windows environment with a wireless connection, if the request to use the network is issued with settings only for a wired NIFM connection, the network will not be available to the application regardless of the current connection settings.
