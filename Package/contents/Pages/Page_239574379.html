3D Texture Converter

This section describes how to use the 3D texture converter and important considerations when using it.

  • Overview
  • Command-Line Options
  • Example Use
  • Order of Internal Processing

Overview

The 3D texture converter is a converter like the GFX texture converter with the added functionality of handling texture intermediate files.

It creates texture intermediate (FTXA and FTXB) files from image (TGA, PNG, DDS, and EXR) files.
Also converts one or many texture intermediate files into a texture binary (BNTX) file.

You can also import texture intermediate files, change the format, and reconvert.
In addition, you can convert texture intermediate files to image files.

You can convert image files to DDS files in each format, and convert one or more DDS files to texture binary files.

Additional Information

Hint information, user data,

tool data, and more.

User data only.

Texture binary files store data in a format that is optimized for the target platform.

The origin is always at the upper-left for images in texture intermediate files and texture binary files.
Cube map texture data is stored using a right-hand coordinate system, in +x, -x, +y, -y, -z, +z order. (A left-hand coordinate system is used when referenced in the shader.)
In the following description, the coordinate axes of cube maps use a right-hand coordinate system.
In the following description, formats that use 9 bits or more for one element, as well as HDR-compatible formats, are real-number types, and all other formats are integer types.

Input File

Specifies the input file. Specifies an image file (TGA, PNG, DDS, or EXR) or a texture intermediate file.
If a texture binary file is being exported, you can specify multiple texture intermediate files or DDS files.
(You can also directly convert a single image file to a texture binary file, but we do not recommend it because a large amount of data is generated for alignment purposes.)
The input file becomes the front face of the cube map (+Z direction) if you specify the options --cubemap-back, --cubemap-left, --cubemap-right, --cubemap-top, and --cubemap-bottom.
16 bit/channel PNG files are handled as real-number type files.
DDS files support the following formats: DXTC, BC, ASTC, and all RGBA formats. If the input file is a DDS file, the output file is not re-encoded if you do not specify an option such as -f or -i. (If multiple DDS files are being converted to texture binary files, the files are never re-encoded). (If you specify -i 1 without specifying -f, the mipmap is deleted, and the file is not re-encoded.)
The filename must end with a numeric value when using the -t option to export 3D textures or array textures. For more information, see the description of the -t option.
Specify only one texture binary file as the input file if the --extract-image-files option has been specified.

--args-file ""

You can use the contents in the parameter file to specify arguments.
The parameter file allows you to specify each option over several lines.
Lines starting with a number sign (#) are treated as comments.
Newline characters cannot be included in escape sequences used in option specifications that appear between quotation marks (").
Save the parameter file using an ANSI encoding (Shift-JIS in Japanese environments).

--version

Displays version information.

-h --help

Displays Help.

-s --silent

Suppresses the display of messages during processing.

--job-list ""

Specifies the job list file.
When you use the job list, multiple conversion processes can be executed with one instance of process startup.
The job list file writes each process argument in a newline.
When specifying arguments other than --job-list at the same time, it results in the same operation as if you added arguments specified in each line of the job list file after the arguments other than --job-list.
Lines starting with a number sign (#) are treated as comments.
In the job list file, the --version, -h, -s, --job-list, and --continue-on-error options cannot be specified.
Save the job list file using an ANSI encoding (Shift-JIS in Japanese environments).

--continue-on-error

When the --job-list option is being used, processing continues even if errors occur in some jobs.

-o "" --output ""

Specifies the output file. Specifies a texture intermediate file, a texture binary file, or an image file (TGA, PNG, DDS, or EXR).
If this option is not specified and the input file is an image file, the output file will be named with the .ftxb extension.
If the input file is a texture intermediate file, that file gets overwritten.

When you have specified a texture binary file, -i, -c, and any options not related to texture intermediate files are ignored when the input file is either a texture intermediate file or a DDS file. (For the -i option, only deleting mipmaps by specifying -i 1 is effective.) All options will be used, however, when the input includes only one DDS file. This case is an exception.

Specify an image file to export the pre-encoded image data of the texture intermediate file to a TGA, PNG, DDS, or EXR file.
Specify the -f / --format-with-a / --format-translucent option to encode image files and export them to the various different DDS format files.
You can output TGA files only when the input files are integer type files.
You can output EXR files only when the input files are real-number type files.
When you output a PNG file, the pixel value is clamped to the range of [0,1] if the input file is a real-number type.
When exporting to a TGA, PNG, or EXR file, cube maps are exported in the state of a horizontal cross-shaped cube map.
If you specify the -i option when exporting a DDS file, the file will have mipmaps.
The -l and --mip-gen-filter options are also applied.
If you specify an srgb_* format using the -f / --format-with-a / --format-translucent option, mipmaps are calculated in linear color space and then reverted to sRGB color space for export.
The quantize_type, linear, and mip_gen_filter settings are used if the input file is a texture intermediate file and the -f, --format-with-a, --format-translucent, -l, and --mip-gen-filter options are not explicitly specified.

--project-root ""

Specifies the project's root folder.
When specified, a relative path from the root folder is generated based on the following attributes of the texture intermediate file.

  • src_path for <create>
  • src_path_r / g / b / a for <texture_info>
  • original_path for <original_image>

--disable-file-info

<file_info> is not exported to the texture intermediate file if the input file is anything other than a texture intermediate file.
If the input file is a texture intermediate file, the file's <file_info> is not modified.
This option has no effect if something other than a texture intermediate file is being exported.

--disable-original-image

This option omits the original image data from before encoding from the texture intermediate file.
Although the file size is smaller, the image quality deteriorates if the output file is reconverted.
If the input file is a texture intermediate file that does not include the original image, the output file does not include the original image data even without specifying this option.
This option has no effect if something other than a texture intermediate file is being exported.

--disable-memory-pool

When a texture binary file is exported, a region for a memory pool object is not allocated, regardless of the memory pool alignment restriction.

Specify this option when using an external memory pool.
This option has no effect if something other than a texture binary file is being exported.

--tile-mode ""

Specifies the tiling mode when a texture binary file is being exported.
Specify linear to export linear tiling data.
When a value other than linear is specified, the add-on for the target platform must be installed.
If this option is not specified, linear tiling data is exported.
When using linear tiling, the address and size of the data for each mipmap level are aligned to bytes per pixel (the number of bytes in one block in compressed format). The exception is when the data is in an uncompressed format and the number of bytes per pixel is less than 4 or not a power of 2, in which case the alignment is 4 bytes.

--tile-optimize ""

Specifies the tiling optimization mode when a texture binary file is being exported.
Specify performance for optimization for higher performance, specify size for optimization to make the file smaller, and specify size_auto to automatically select size optimization according to the data reduction rate.
When size_auto is specified, specify the threshold value of the data reduction rate using the --tile-size-threshold option.
If you do not specify this option, optimization for performance is executed.
This option only has an effect when a specific mode is specified using the --tile-mode option.

--tile-size-threshold ""

If you specified size_auto using the --tile-optimize option, specify an integer from 0 to 100 for the threshold value [%] to judge whether to execute optimization for size.
Textures for which (size for size priority / size for performance priority) x 100 is less than the threshold value are optimized for size. All other textures are optimized for performance.
The larger the threshold value, the higher the probability that size priority is selected.
If you do not specify this option, the threshold value is 75.

--sparse-tiled ""

When exporting texture binary files, specify true if you are performing tiling conversion for sparse textures, and specify false if you are performing tiling conversion for ordinary textures.
If this option is not specified, tiling conversion for ordinary textures is performed.
Currently, sparse textures can only be used for specific platforms.

--abgr

Specify this option if textures in 16 bit/pixel RGB(A) format will be generated in order from the lowest bit (in (A)BGR order) when the texture binary files are generated. (Available in NintendoSDK 11.1.0 and later.)
This option affects textures in unorm_5_6_5, unorm_5_5_5_1, and unorm_4_4_4_4 format. This option is specified when these formats are used in OpenGL or Vulkan.
If this option is not specified, textures are generated in RGB(A) order.

--resource-options ""

Stores the options, for converting to a texture binary file, in the texture intermediate file. (This feature is available in NintendoSDK 13.0.0 and later.)
Use this option when you want to use the texture intermediate file to control the texture binary file status.
Note that texture intermediate files created with the --resource-options option will be platform-dependent. Normally, when exporting a texture binary file, we recommend specifying options related to texture binary files.

The texture binary file related options that you can specify are --disable-memory-pool, --tile-mode, --tile-optimize, --tile-size-threshold, --sparse-tiled, and --abgr.
For example, when converting image1.ftxb created with --resource-options "--sparse-tiled true" into a texture binary file, specify 3dTextureConverter.exe image1.ftxb -o image1.bntx to apply a tiling conversion for space textures.


Use single quotation marks when enclosing each option value in quotation marks.
(Example: --resource-options "--tile-mode 'linear'")

When converting multiple textures into a single texture binary file, an error will occur if the --disable-memory-pool and --tile-mode settings differ between textures. You can specify different values for each texture for the other options.
The options specified by the argument when converting to texture binary files take priority over the options stored in the texture intermediate file.
　3dTextureConverter.exe image1.ftxb image2.ftxb -o images.bntx --tile-mode linear
If you specify the above, all files will be processed as --tile-mode linear regardless of the --tile-mode settings in image1.ftxb and image2.ftxb.

The --resource-options options are enabled even when converting one image file directly into a texture binary file.

-m "" --merge ""

Specifies the texture intermediate file to merge.
This option creates a texture intermediate file from the <texture_info> settings of the specified texture intermediate file (hint, linear, dimension, quantize_type, mip_level, mip_gen_filter, comp_sel, weighted_compress, and resource_options) and the <comment>, <user_data_array>, <tool_data> and <user_tool_data> settings.


If the mip_level of the file being merged is two or higher, a mipmap that includes levels where the height and width of the smallest level in the output file is less than the height and width of the smallest level in the file being merged is created.
If the input file is a texture intermediate file and the text attribute for the <comment> element is not blank, text is not merged.
If you specify this option, and at the same time, specify the -n, -l, -d, -f, --format-with-a, --format-translucent, -w, -i, --mip-gen-filter, -c, or --comment option, specification of those other options takes priority.

--disable-merge-error

Suppresses the error that occurs when the file specified by the --merge option does not exist (a merge is not performed if the file does not exist).
Specify this option when the merge option is used only for merging user data or tool data.

--dcc-preset ""

Specifies the preset name for the DCC plug-in. This option has no effect if something other than a texture intermediate file is being output.

-n "" --hint ""

Specifies hint information.
If you do not specify this option, it is treated as normal if the format is snorm_bc5 or snorm_eac_11_11. If the format is something else, the output is an empty string.
If the hint information is set to normal and the input file is an integer type, the data is normalized so the z-component of the normal is 0 or more and the length is 1.0, after which the data is encoded in the output format and output. However, if normalization of the normal map has been specified by either the --replace-r, --replace-g, --replace-b, or --replace-a option, the data is not normalized before encoding.

-l "" --linear ""

To convert a particular RGBA color component from sRGB to linear, specify 1 for that component. Specify 0 if you do want that component to be converted.
For example, is you specify the option as -l 1110, the R, G, and B components are converted to linear, but the A component is not.
If you do not specify this option, no components are converted to linear.
The values set for the RGB components in this option will be ignored if the format is srgb_* (the RGB components are always converted to linear), and only the setting for the A component is applied in the output.

-d "" --dimension ""

Specifies the dimension (1d / 2d / 3d / cube / 1d_array / 2d_array / cube_array).
For 3d, 1d_array, and 2d_array, you must specify the depth using the -t option.
For cube_array, you must specify the number of cubes using the -t option.
For 1d and 1d_array, this option exports the top line of the input file.
If this option is not specified, and the input file is TGA , PNG, or EXR, 2d is used as the dimension for the output file.
If the input file is a DDS file or a texture intermediate file, the original dimension is maintained.
If the --cubemap-back, --cubemap-left, --cubemap-right, --cubemap-top, or --cubemap-bottom options have been specified, cube is used as the dimension for the output file even if -d cube has not been specified.

Using a Horizontal Cross or a Vertical Cross Cube Map

Specify -d cube without specifying the --cubemap-back, --cubemap-left, --cubemap-right, --cubemap-top, or --cubemap-bottom options.
Horizontal cross and vertical cross cube maps are distinguished based on the height and width of the input file.

For a horizontal cross cube map, the square-shaped images for each view direction must be arranged as follows, without any gaps. (Axes are oriented in a right-handed coordinate system.)

+Y

-X

-Z

+X

+Z

-Y

For a vertical cross cube map, the square image for each view direction must be arranged as follows, without any gaps.

+Y

-X

-Z

+X

-Y

+Z

-f "" --format ""

Specifies the format.
When this option is not specified, the output format is unorm_8_8_8_8 when the input file is a TGA file; either unorm_8_8_8_8 or float_16_16_16_16, depending on the original format, when the input file is a PNG file, and float_16_16_16_16 when it is an EXR file. (If the output file is DDS, it is exported in 32-bit float format instead of float_16_16_16_16.) The original format is maintained when the input or texture intermediate file is a DDS file.
The following formats are currently supported.

If unorm_4_4, unorm_8_8, unorm_bc5, or unorm_eac_11_11 has been specified as the format and something other than normal has been specified for the hint information, the A component of the input file is applied to the G component of the texture (if the input file has an alpha channel component).

If you specify either the snorm or sint format when the input file is an integer type, a value equal to the unsigned 8-bit value, ranging from 0 to 255, minus 128 is encoded. As a result, when the unsigned 8-bit value is 0, it will be encoded as -127.
If the ASTC format has been specified and normal has been specified for the hint information, the R component of the input file is applied to the RGB components of the texture, and the G component of the input file is applied to the A component of the texture.
When an input file that contains real number pixel values is converted to an ASTC file, pixel values in the input file that are outside of the [0,1] range are clamped to [0,1].
You can specify the ETC, EAC, and PVRTC file formats only when the output file is a texture intermediate file or a texture binary file.

Info

The unorm_4_4 format may only be used for specific platforms.
An error occurs if you convert unorm_4_4 format texture intermediate files into Generic texture binary files.

Info

If the format has been specified as BC1, BC2, or BC3 in an environment where no special add-on has been installed, use dithering to cut down on padding when the image contains broad areas where the color changes gradually.
Use dithering when all of the following conditions are satisfied.

  • Input File Is Integer Type
  • The image width or height is at least 256 pixels, and the smaller of the two is at least 8 pixels.
  • The maximum difference in RGB values with pixels around the periphery is 1 for at least 50 percent of the pixels.
  • The maximum difference in RGB values with pixels around the periphery is 64 or more for fewer than 20 percent of the pixels.

The RGB values for the pixels around the periphery in all eight directions (top, bottom, left, right, and the diagonals) are calculated as 8-bit (256-level) color.

--format-with-a ""

Specifies the format to use when the input file contains the A component. If the input file contains the A component, it is converted to the format specified by this option. If the file does not contain the A component, it is converted to the format specified by the -f option.
If the input file is in a format that has the A component (the alpha channel), the A component is considered to exist in the file even if the alpha value for all pixels is 255 (1.0).
If this option is not specified, the file is converted to the format specified by the -f option regardless of whether it contains the A component.
When there is a component A, you can specify selection of another component in the format "format, component selection" (available with NintendoSDK 9.0.0 and later).
Specify the component selection in the same format as the -c option. If a component selection is omitted, the component selection specified with the -c option is applied, regardless of whether there is a component A.

--format-translucent ""

Specifies the format to use when the input file contains a translucent alpha component. If the input file contains a translucent alpha component, it is converted to the format specified by this option. If the file does not contain a translucent alpha component, it is converted to the format specified by the -f or --format-with-a option.
When this option is not specified, the -f or --format-with-a option is used to covert to the specified format.
The --format-translucent option takes precedence over the --format-with-a option. When both are specified, the file is converted to the format specified by the --format-with-a option if none of the pixels have a translucent A component (that is, if the pixels are all opaque or all transparent or a combination of both).
If component selection has been specified by the --format-with-a option, the component selection specified by that option is applied even if the format specified by this option is being used.

-w "" --weighted-compress ""

Specify true to apply a weight according to the human eye to RGB errors when using texture compression method bc1, bc2, bc3, and so on; otherwise, specify false.
If this option is not specified, weighting is not applied if the input file is a TGA, PNG, DDS, or EXR file.
If the input file is a texture intermediate file, the original setting is maintained.

-t "" --depth ""

Specifies the depth of a 3D texture and array texture.
For cube array textures, specifies the number of cubes.
At the same time, specifies -d 1d_array for a 1D array texture, -d 2d_array for a 2D array texture, and -d cube_array for a horizontal cross or vertical cross cube map.

If you specify this option and omit the -d option, the result is a 3D texture if the input file is 2D, or a cube array texture if the input file is a cube map.

If there are input files with names ending in incrementing numeric values, the number specified in this option defines how many of the input files to use.
For example, if there is an input file named image1.tga and the option has been specified as -t 3, the input files image1.tga, image2.tga, and image3.tga are used.
If there is an input file named image010.tga and the option has been specified as -t 4, the input files image010.tga, image011.tga, image012.tga, and image013.tga are used.
If individual images are being specified for each face of a cube map, files with names that end in incrementing numeric values specified by the --cubemap-back, --cubemap-left, --cubemap-right, --cubemap-top, and --cubemap-bottom options are also used as input files.
To convert a group of files with filenames that end in numbers that are not in numerical sequence, set --discrete-number true.

--discrete-number ""

Specify true to convert a group of files whose filenames end in numbers that are not in sequential order to a 3D texture or an array texture. Specify false to convert a group of files whose filenames end in numbers that are sequential. (This feature is in Nintendo SDK 9.1.0 and later.)
If you specify true, a number of files specified by the -t option are used as input files, starting in order from the file with the filename ending with the lowest number.
If this option is not specified, conversion involves the group of files whose filenames end in numerical sequence.
For example, if you have the files image03.tga, image05.tga, and image08.tga, and you specify image05.tga as the input file and -t 2 --discrete-number true as the options, the conversion process involves image05.tga and image08.tga.

--no-encoding

Specified when converting without re-encoding.
Specify this option when creating an array texture if you are combining the encoded data from either DDS files or texture intermediate files.
Encoded texture intermediate file data is exported to a DDS file when specified while converting texture intermediate files to DDS files. (The mipmap images are also applied.)
When this option is used, all options not related to texture binary files are ignored, as are --version, -h, -s, -o, -t, -i, and -c. (For the -i option, only deleting mipmaps by specifying -i 1 is effective.)

When creating texture arrays, note the following important considerations.
The input files must be DDS files or texture intermediate files. The input file mipmap data is also applied.
Specify this option at the same time as the -t option.
The input file dimensions must be 1d, 2d, or cube.
The hint information, linear conversion flag, dimension, width, height, format, number of mipmaps, and component selections must be the same for each input file.
The preset name, initial swizzle value, user data, editing comments, tool data, and user tool data in the DCC plug-in output a leading input file.
For example, if the dimension is 2d for image1.dds, image2.dds, and image3.dds and you specify image1.dds -o array1.dds -t 3 --no-encoding, encoded data will be combined and output to the file array1.dds (with the dimension 2d_array).

--gpu-encoding ""

Sets whether to encode in the GPU when converting to a particular format.
In the current environment, if you specify true, encoding takes place in the GPU if possible, or otherewsie an error is generated.
Specify false to use CPU encoding.
If you specify auto, encoding takes place in the GPU if possible, or in the CPU otherwise.
When this option is not specified, the behavior will be the same as if true was specified.
Currently, this option only works when converting to the BC1, BC2, BC3, BC6, BC7, and ASTC formats.
The encoding results might differ between encoding on the GPU and encoding on the CPU.
The encoding results might differ even when both instances occur on the CPU if the CPU vendors are different.
In some cases, GPU encoding might not be possible if the graphics driver version is out of date.
If the specific add-ons are not installed, this option is ignored, and encoding always takes place in the CPU.

--check-gpu-encoding

Determines whether GPU encoding is possible in the current environment.
If GPU encoding is possible, the 3dTextureConverter.exe exit code (%ERRORLEVEL%) is 0. If it is not possible, the exit code is some nonzero value.
Even if the exit code is 0, encoding takes place in the CPU if the format does not support GPU encoding.
GPU encoding is possible if the specific add-on is installed, and the GPU and graphics driver satisfy the conditions.
If this option has been specified, the other options are ignored, and conversion does not take place.

-i "" --mip-level ""

Specifies the maximum number of mipmap levels.
If the height and width of the input file are too small to create the specified number of mipmap levels, mipmaps are created up to the level where the height and width are 1.
If this option is not specified and the input file is in TGA, PNG, or EXR format, mipmaps are generated down to the level where the height and width are 1. (However, if the output file format is DDS, mipmaps are not generated unless the -i, -f, --format-with-a, and --format-translucent options have also been specified.)
If this option is not specified, and the input file is a DDS file or a texture intermediate file, the original number of levels is maintained.
If the input file is a DDS file and neither the -i, -m, --crop, -e, -x nor -y option is specified, the mipmap images of the input file will be used.
If the input file is a texture intermediate file and the --no-encoding option is specified without the -i option, the mipmap images of the encoded data are used in the output file.

--mip-gen-filter ""

Specify the filter to use when creating the mipmaps.

point Point sampling.
linear Linear interpolation.
cubic Standard cubic interpolation.
cubic_sharp Sharp cubic interpolation. (Available in NintendoSDK 9.0.0 and later.)

If this option is not specified, linear interpolation is used if the input file is a TGA, PNG, DDS, or EXR file. The original filter is maintained if the input file is a texture intermediate file.
Depending on the image, the cubic value might yield sharper results than the cubic_sharp value.

-c "" --comp-sel ""

You can specify which color component to select (r, g, b, a, 0, or 1) for each RGBA channel.
Enabled when the export file is either a texture intermediate file or a texture binary file.
For example, by specifying -c rrr1, you can select the R component for all three RGB channels and the constant 1 for the A channel.
If this option is not specified, you can select any of the following depending on the format.

rrr1 (number of format components is 1) 
rrrg (number of format components is 2, unsigned) 
rg01 (number of format components is 2, signed, or either unorm_16_16 or uint_*_*) 
rgb1 (number of format components is 3)
rgba (number of format components is 4)
ra01 (ASTC format, with "normal" specified for the hint information)

As an exception, the default component selection is rg01 when the input file is a DDS file with unsigned red and green components.
If the input file is a texture intermediate file and this option is not specified, the original comp_sel will be maintained, providing there is no change in the number of format components or whether they are signed or unsigned.

--cubemap-back ""

Specifies the back face of the cube map (-z direction).

--cubemap-left ""

Specifies the left face of the cube map (-x direction).

--cubemap-right ""

Specifies the right face of the cube map (+x direction).

--cubemap-top ""

Specifies the top face of the cube map (+y direction).

--cubemap-bottom ""

Specifies the bottom face of the cube map (-y direction).

--mip-imageM ""
(M is an integer between 1 and 14 that indicates the mipmap level.)

When specifying mipmap images individually, specify the image file for each level.
Levels not specified in this option are compressed versions of high-resolution level images.

For example, say that you specify the command-line options base.tga -i 5 --mip-image2 sub.tga.
level 0 is a base.tga image,
level 1 is an image with base.tga compressed to 1/2,
level 2 is a sub.tga image,
level 3 is an image with sub.tga compressed to 1/2, and
level 4 is an image with sub.tga compressed to 1/4.
If the width and height of sub.tga is not one-fourth of the width and height of base.tga, an error occurs.

--cubemap-frontM ""
 --cubemap-backM ""
 --cubemap-leftM ""
 --cubemap-rightM ""
 --cubemap-topM ""
 --cubemap-bottomM ""
(M is an integer between 1 and 14 that indicates the mipmap level.)

Specifies the image file for each level when you are individually specifying the images for each direction of a cube map using the --cubemap-back, --cubemap-left, --cubemap-right, --cubemap-top and --cubemap-bottom options, and individually specifies the mipmap images.
--cubemap-frontM, --cubemap-backM, --cubemap-leftM, --cubemap-rightM, --cubemap-topM, and --cubemap-bottomM correspond to level M for the front, back, left, right, top, and bottom faces of the cube.
Levels not specified in this option are compressed versions of high-resolution level images.

--comment ""

Represents comment text for editing the texture intermediate file.

--crop ""

When cutting out a part of an image for export, specify a rectangle to cut out with four integer values that are separated by commas (in X, Y, W, H format).
X and Y specify an integer value of 0 or greater for the upper-left coordinate of the rectangle.
W and H specify an integer value of 1 or greater for the width and height of the rectangle.
If you are using the -e, -x, and -y options together, the size of the rectangle is changed after it has been cut out.
If the input image is a horizontal or vertical cross-shaped cube, specify a rectangle for an image with a single view direction.

-e "" --reduce-level ""

Specifying an integer N that is greater than or equal to 1 reduces the size of the output image to 1/(2^N).
(By half if N = 1, by one-quarter if N = 2, and by one-eighth if N = 3.)
If you do not specify this option, the size of the image is not reduced.
If you are using the -x and -y options together, this option is disabled.

-x "" --resize-w ""
 -y "" --resize-h ""

Specifies the resized width (-x) and height (-y) when images are resized and output, either in terms of a number of pixels (an integer of 1 or more) or in terms of a percentage of the original size (real number + %).
If only either the -x or the -y option is specified, the aspect ratio is fixed when the size is changed.
If -x 100% -y 50% is specified, the height is reduced by half without changing the width of the image.
To specify a percentage value in a Windows BAT file, you must write two percentage symbols, as in 50%%.
If the input image is a horizontal or vertical cross-shaped cubemap, specify the height and width of the image in a single view direction.

--resize-single-color ""

Resizes the height and width of single-color images to the specified sizes (available in NintendoSDK 9.2.0 and later).
For example, --resize-single-color 4 changes the size of an image to 4 × 4 if the image is a single-color image.
When used along with the -e, -x, or -y option, this option first determines whether the resized result is a single-color image, and then if so resizes to the size specified in this option.
When used along with the -r, -g, -b, or -a option, this option determines whether the result of replacing the R/G/B/A channel is a single-color image.

--resize-filter ""

Specifies the filter when changing the image size with the -e, -x, and -y options.

point Point sampling.
linear Linear interpolation.
cubic Standard cubic interpolation.
cubic_smooth Smooth cubic interpolation.
cubic_sharp Sharp cubic interpolation.

If this option is not specified, the size change is performed with linear interpolation.

--resize-gamma ""

If specifying the -e, -x or -y options and converting the gamma from sRGB to linear using either the -l option or the input image settings, specify true to calculate the size changes in linear space, or false to calculate the size changes without converting the color space.
If this option is not specified, the resizing is calculated in linear space.

--reduction-only ""

Specify true if you only want to resize if the size specified with the -x or -y option yields an image smaller than the original size, or specify false if you want to resize regardless of the relationship to the original size (available in NintendoSDK 9.2.0 and later).
If this option is not specified, the image is resized regardless of the relationship to the original size.

--ignore-alpha ""

Specify true to ignore the alpha component of the input image file and process it as opaque. Specify false to take the alpha into consideration.
For example, specify true if converting an image file unnecessarily configured with an alpha component, or if the format is something such as unorm_bc5 and you do not want the alpha to be applied to the G component.
If this option is not specified, the alpha is used.
The -a option takes precedence over this option. The replaced alpha component is not ignored when -a replacement alpha component image file is specified and --ignore-alpha-true is also specified.

-u "" --adjust-transparent-rgb ""

Specify true if you are adjusting the RGB components of transparent pixels to prevent transparent pixel color bleeding due to linear interpolation. Specify false if you are not making such adjustments.
If you specify true, the RGB components of transparent pixels (those with an A component of 0) are set with the average of the RGB components of adjacent non-transparent pixels (those with an A component of greater than 0).
Also, when true is specified, the RGB components of transparent pixels are not affected when the size changes due to the -e, -x, or -y options.
If the sampling range is made entirely of transparent pixels, however, the RGB components of the transparent pixels are averaged.
If this option is not specified, the RGB components of transparent pixels are not adjusted.

-r "" --replace-r ""
-g "" --replace-g ""
-b "" --replace-b ""
-a "" --replace-a ""

Specifies the other image file to use when replacing the R, G, B, and A channels of the input file with the values of another image file.
To use the specified channel from that other image file, enter a colon followed by a color channel (:r, :g, :b, or :a) after its name.
The R channel is used if the color channel is omitted.
You can specify -a even if the input file does not have the A component. (This adds the A component, rather than replacing it.)
For example, if you specify -g image1.tga -b image2.tga:a, the G channel of the input file is swapped with the R channel of image1.tga and the B channel of the input file is swapped with the A channel of image2.tga.
If the other image is an FTX file or DDS file, the linear conversion flag is also replaced.

If you specify ,normal after the channel name, the RGB components of the other image file are normalized as the normal map (adjusted so that z-component is 0 or more and the length is 1.0) and then replaced. (This feature is available in NintendoSDK 13.0.0 and later.)
For example, if you want the RG channels to be the xy-components of the normal map and the B channel to be the specular map, specify as -r normal1.tga:r,normal -g normal1.tga:g,normal -b specular1.tga:r.
If the normals have been normalized at the time of replacement, the original image data of the FTX file will consist of the normalized values.

If the resolution of the other image file is different from the input file, this other image file is resized to match the input file. This operation uses the filter specified by the --resize-filter option.
There is no support for using this option together with the -t, --cubemap-back, --cubemap-left, --cubemap-right, --cubemap-top, --cubemap-bottom, and --mip-imageM options.

--no-tga-rle

Specify this option to leave TGA files uncompressed when exporting them.
TGA files are exported with RLE compression if this option is not specified.

--extract-image-files ""

Converts the textures in the texture binary file to image files. (This feature is available in NintendoSDK 12.2.0 and later.)
Specifies the path to the output folder for the image files.
As the input file, only one texture binary file is specified.
By default, the images are PNG files. To change the output file type, use the --extract-ldr-type and --extract-hdr-type options.
In the case of TGA, PNG, and EXR files, the data is converted and the output applies the selected components. (Cubemaps are flattened into the shape of a cross. 3D textures and array textures appear vertically as a stack of faces.)
In the case of DDS files, encoded data is output if possible. (Mipmaps are also output. The selected components are not applied.)
If this option has been specified, all other options except for --version, -h, -s, --extract-ldr-type, --extract-hdr-type, and --no-tga-rle are ignored.

--extract-ldr-type ""

Specifies the output file type (TGA, PNG, or DDS) when extracting LDR images from the texture binary file. (This feature is available in NintendoSDK 12.2.0 and later.)
This option is used together with the --extract-image-files option.
If this option is not specified, the output files are PNG files.

--extract-hdr-type ""

Specifies the output file type (TGA, PNG, DDS, or EXR) when extracting HDR images from the texture binary file. (This feature is available in NintendoSDK 12.2.0 and later.)
This option is used together with the --extract-image-files option.
Textures in real-number formats and in formats with 9 bits or more per component are output in the file type specified by this option.
If this option is not specified, the output files are PNG files.
In the case of TGA and PNG files, the output images have 8 bits per channel, with the RGBA values clamped to the range of [0,1].

Example Use

Converting Image Files to Texture Intermediate Files

The following command converts the file named image.tga and creates a texture intermediate file named image.ftxb in srgb_bc1 format with eight mipmap levels.

3dTextureConverter.exe image.tga -o image.ftxb -f srgb_bc1 -i 8

The following command specifies merging image.tga with merge.ftxb using the settings (hint information, linear conversion, dimension, format, mipmap, component selection, initial swizzle values, user data, tool data, and user tool data) in merge.ftxb, and then creates a texture intermediate file named image.ftxb.

3dTextureConverter.exe image.tga -o image.ftxb -m merge.ftxb

Converts image1.tga, image2.tga, and image3.tga, and creates a texture intermediate file array1.ftxb for a two-dimensional array.

3dTextureConverter.exe image1.tga -o array1.ftxb -d 2d_array -t 3

Batch conversion is possible using the job list file.
The image1.ftxb and image2.ftxb hint information is albedo, and the image3.ftxb hint information is specular.

list.txt

image1.tga -o image1.ftxb -f srgb_bc1
image2.tga -o image2.ftxb -f srgb_bc3
image3.tga -o image3.ftxb -f unorm_8 --hint specular
 
3dTextureConverter.exe --job-list list.txt --hint albedo

Converting Texture Intermediate Files to a Texture Binary File

Converts imageA.ftxb, imageB.ftxb, and imageC.ftxb to the texture binary file textures.bntx, using linear tiling.

3dTextureConverter.exe imageA.ftxb imageB.ftxb imageC.ftxb -o textures.bntx --tile-mode linear

Reconverting Texture Intermediate Files

The following command changes the format of image.ftxb to unorm_5_6_5 and saves it.
There is no degradation of image quality due to re-encoding because the texture intermediate file includes image data before encoding.

3dTextureConverter.exe image.ftxb -o image.ftxb -f unorm_5_6_5

Converting Texture Intermediate Files to TGA Files

Before encoding, the image data for image.ftxb is output to original.tga.

3dTextureConverter.exe image.ftxb -o original.tga

Converting Image Files to the Various DDS Format Files

The following command converts the file named image.tga and creates a DDS file named image.DDS in srgb_bc1 format with eight mipmap levels.

3dTextureConverter.exe image.tga -o image.dds -f srgb_bc1 -i 8

Converts image1.tga, image2.tga, and image3.tga, and creates a DDS file called array1.dds for a two-dimensional array.

3dTextureConverter.exe image1.tga -o array1.dds -f srgb_bc1 -d 2d_array -t 3

Batch conversion is possible using the job list file.
The format for image1.dds and image2.dds is srgb_bc1, and the format for image3.dds is unorm_5_6_5.

list.txt

image1.tga -o image1.dds
image2.tga -o image2.dds
image3.tga -o image3.dds -f unorm_5_6_5
 
3dTextureConverter.exe --job-list list.txt -f srgb_bc1

Converting DDS Files to a Texture Binary File

Converts image_dxt1.dds, image_bc3.dds, and image_r5g6b5.dds into the texture binary file named textures.bntx, using linear tiling.
DDS files are used as is during the conversion process. No re-encoding occurs. 

3dTextureConverter.exe image_dxt1.dds.dds image_bc3.dds image_r5g6b5.dds -o textures.bntx --tile-mode linear

Order of Internal Processing

  1. Read the input file.
  2. Delete the A component if --ignore-alpha true is specified.
  3. Replace the channels if --replace-r/g/b/a is specified and any of the following conditions are met.
    • --reduce-level and --resize-w/h are not specified.
    • --crop is specified.
    • Horizontal (vertical) cross cube map.
    • The input file and the replacement file are the same size and --replace-r/g/b/a does not specify normalization of the normals.
  4. Crop, if --crop is specified.
  5. Resize, if --reduce-level or --resize-w/h is specified.
    (Affected by the specification of --resize-filter, --resize-gamma, --adjust-transparent-rgb, and --linear.)
  6. If the channels were not replaced in step 3, replace them in this step according to --replace-r/g/b/a.
  7. Adjust the RGB components of transparent pixels if --adjust-transparent-rgb true has been specified.
  8. Resize if --resize-single-color is specified and the color is monochrome.
  9. (↑ The result of the processing to this point becomes the original image data of the FTX file.)
  10. Create mipmaps.
  11. Copy the A component to the G component if the output format is either unorm_4_4, unorm_8_8, unorm_bc5, or unorm_eac_11_11, the hint information is not specified as normal, and the image has an A component.
  12. Normalize the normals, if the hint information is normal and the input file is an integer type and --replace-r/g/b/a does not specify normalization of the normals.
  13. Encode the data in the output format.
