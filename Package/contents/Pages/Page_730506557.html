Socket Statistics Overlay Meters

Overview

The socket statistics DevMenuOverlay tool displays statistics about your application and the system's network usage. The meter features three different view modes that display information focused on performance, memory, and network health, respectively. In order for your application statistics to be visible to the tool you must register it by calling nn::socket::RegisterForStatistics(). Available on NX Addon 10.0.0 and later.

How to use

How to turn on the statistics overlay:

  1. Ensure that your program has initialized the socket library by calling "nn::socket::Initialize()"
  2. Ensure that your program is registered by calling "nn::socket::RegisterForStatistics()".
  3. Turn on socket statistics display by holding the "home button"→QuickMenu→Debug...→[√]SocketStaistics→OK
  4. Run your program. 

How to configure the display:

The DebugMenu SocketStatistics has options for Application selection, Display Channel, and View Mode.

  • To select between applications and Display Channels use the up and down arrow buttons to toggle between the options. For the view mode select the option from the drop-down menu.
  • Application Selection:
    • Select a registered application on the system. Typically, only your application is registered in the system.
  • Display Channel options (aka Channel):
    • Single: channel 0-31 will display socket with correlating socket descriptor and peer IP in the view.
    • App: Shows the sum of all open sockets owned by the application.
    • System: Shows system-wide (Sys) usage of all open sockets. (Memory Usage View Mode not supported)
  • View Mode options:
    • Performance View: Shows the bandwidth (bytes), packet rate (packets/second), and round trip time (ms) if the socket is has a peer.
    • Memory Usage View: Shows the memory usage of the send and receive buffers as well as the dns resolver memory usage.
    • Health View: Shows connection statistics on received udp packets that were dropped and tcp packets retransmitted.

Socket Statistics Overlay Meter Layout

Network performance meter view mode details

Legend:

  • RTT: Round Trip Time

Title: Shows the selected application name and the current download and upload rates for the application. (updated at 2 fps).

Graphs 1,2,3:

  • Bandwidth (Updated at 30 fps):
    • Channel 0-31: Graphs the number of udp or tcp bytes sent to and received by the socket over ~33ms.
    • Channel App: Graphs the number of udp and tcp bytes sent to and received by the application over ~33ms.
    • Channel Sys: Graphs the number of udp and tcp bytes sent to and received by the system over ~33ms.
  • Packet Rate (Updated at 1 fps)
    • Channel 0-31: Graphs the number of udp or tcp packets sent to and received by the socket over 1 second.
    • Channel App: Graphs the number of udp and tcp packets sent to and received by the application over 1 second.
    • Channel Sys: Graphs the number of udp and tcp packets sent to and received by the system over 1 second.
  • Round Trip Time (RTT, Updated at ~2 fps)
    • Channel 0-31, App, System: Graphs the RTT of packets to the socket's peer address.
    • Channel App, Sys: App and Sys modes default to Channel 0's connection when selected.
    • RTT is calculated via ICMP echo requests and may update at a rate <2 fps depending on the connection to the peer.
    • RTT cannot be calculated if the socket is not connected to a peer. The graph will display "RTT NA" above the graph if the selected socket is not connected.

Text:

  • Channel Display Information: Text which indicates what Channel Display is active and the current state.
  • "Peak[rcv,snd]: [PeakBytesReceivedOver33ms, PeakBytesSentOver33ms] KB/MB [PeakPacketsReceivedOver1Second, PeakPacketsSentOver1Second] pkt/s"
    • PeakBytesReceivedOver33ms: Peak bytes received over ~33ms
    • PeakBytesSentOver33ms: Peak bytes sent over ~33ms
    • PeakPacketsReceivedOver1Second: Peak packets received over 1 second
    • PeakPacketsSentOver1Second: Peak packets sent over 1 second
  • "RTT[min,pk]: [MinimumRTT, PeakRTT] ms."
    • MinimumRTT: The minimum RTT on the selected display since the display channel became active.
    • PeakRTT: The peak RTT on the selected display since the display channel became active.

Socket library memory meter view mode details

Legend:

  • Memory Overhead: The socket library uses fixed-sized buffers. Network messages that may not efficiently use the fixed-sized buffers.
  • Efficiency Factor: When you call "nn::socket::Initialize(config)" the efficiency factor is one of the parameters that affects memory usage of the socket buffers.
  • So_SndBuf: Directly correlates to sb_size and defaults to config type passed in Initialize. This size may be changed dynamically using "nn::socket::SetSockOpt".
  • So_RcvBuf: Directly correlates to sb_size and defaults to config type passed in Initialize. This size may be changed dynamically using "nn::socket::SetSockOpt".
  • mb_pool: memory buffer pool
    • Send, Receive = Efficiency Factor * sb_size.
  • mb_used: memory buffer used
    • Send, Receive = sb_data + Memory Overhead
    • mb_used may never exceed mp_pool.
    • Note: (UDP ONLY) If the receive buffer is full (ie. mb_used is nearing mb_pool) all subsequent packets will be dropped.
  • sb_size: socket buffer size
    • Send = So_SndBuf
    • Receive = So_RcvBuf
  • sb_data: socket buffer data
    • Send, Receive = The count of actual data bytes in the socket buffer, not accounting memory buffer overhead.
    • sb_data  may never exceed sb_size.
    • Note: (UDP ONLY) If the receive buffer is full (ie. sb_data is nearing sb_size) all subsequent packets will be dropped.

Title: Shows the selected application name and the current (sum of mb_used) memory out of all memory given to the socket library (updated at 2 fps).

Graph 1,2:

  • Socket buffer Send and Receive (Updated at 30 fps)
    • Channel 0-31: Graphs the mb_pool, mb_used, sb_size, and sb_data for the udp or tcp socket over ~33 ms intervals.
    • Channel App: Graphs the mb_pool, mb_used, sb_size, and sb_data as a sum of all live udp and tcp socket over ~33 ms intervals.
    • Channel Sys: Not supported

Text:

  • Channel Display Information: Text which indicates what Channel Display is active and the current state.
  • "Live fd[pk]: LiveSockets[LiveSocketsPeak]. Sum mb_pool pk: SocketMbPoolPeak KB"
    • LiveSockets: The current number of live sockets opened by the application.
    • LiveSocketsPeak: The peak number of sockets open since initialize of the socket library.
    • SocketMbPoolPeak: The peak number of KBs allocated to socket buffers opened by the application since initialize of the socket library.
  • "Mem over [rcv,snd]: [ReceiveSocketBufferMemoryOverhead, SendSocketBufferMemoryOverhead KB]"
    • ReceiveSocketBufferMemoryOverhead: The current number of KB in the sockets receive buffer's mb_pool lost to internal overhead.
    • SendSocketBufferMemoryOverhead: The current number of KB in the sockets send buffer's mb_pool lost to internal overhead.

Graph 3:

  • DNS Resolver Memory (Updated at 2 fps)
    • Channel 0-31, App: Shows the registered applications' memory used
      • avail: Total memory available to the dns resolver.
      • used(pk): used: Current memory used by the dns resolver. pk: Peak memory used since initialize of the socket library.
    • Channel Sys: Not supported

Additional Notes:

  • (TCP ONLY) If you have very small send and/or receive buffer and the connection is very poor the tcp window size will also be small which can inhibit data transfer dramatically as the sender can only attempt to send a window size's worth of packets before receiving acknowledgment packets to continue.

Legend:

  • Channel Display Information: Text which indicates what Channel Display is active and the current state.
  • "Live fd[pk]: LiveSockets[LiveSocketsPeak]. RTT[min,ave,pk]: [MinimumRTT, AverageRTT, PeakRTT] ms"

    • LiveSocketsPeak: The peak number of sockets open since initialize of the socket library.

Additional Notes:

  • (TCP ONLY): A poor connection in which a sizable number of packets may be lost in transmitting may benefit from a large send and receive buffers. This improved the window size and allows the sender to send up to the last known window size allowing for wider stream reconstruction by the peer.

Socket Statistics Overlay Sample Code

Overview

The Socket Statistics Overlay sample is a simple server-client connection which transfers data from the server to the client. While the sample runs you may look at the socket statistics tools to see how certain types of unhealthy and normal behavior appear in the graphs.

Building the sample

The sample program may be compiled for windows or NX. The socket statistics are only visible on NX side of the connection. If you only have 1 device you may run the client or server on the NX device to see its statistics and then swap to see the other's behavior. If you want to see the behavior of both the server and client at the same time you will need 2 NX devices; running one as a server and the other as a client.

Running the sample (SocketStatisticsOverlay)

Both server and client use the same sample program and it chooses to be the server or client based on its command line arguments.

Run server: Startup a single server to listen for both tcp and udp client requests. Client requests include: Burst Send request (both udp and tcp), and Hello Server request (udp). 

  • Server command-line arguments (both tcp and udp): "-server"
  • Server function description:
    • Burst Send Request: Requests the server to send back 1 KB messages at a specified rate.
    • Hello Server: Reply back with Hello Client message to client.

Run client: Startup a udp or tcp client to connect to a server.

  • Udp Client command line arguments: "-udpClient server=192.168.1.20:22819"
  • Tcp Client command line arguments: "-tcpClient server=192.168.1.20:15215"
  • Note: You can find the IpAddress and port to pass the client as a command line argument from server program's print output.

    • Example Log: "Udp Server: Listening for incoming messages on fd 1 at: 192.168.1.20:22819"

    • Example Log: "Tcp Server: Listening for incoming messages on fd 0 at: 192.168.1.20:15215"

  • Client function description:
    • Changes send and receive socket buffer size before each burst send requests.
    • Connects to the server and sends out 2 burst send requests and receives the messages in sequence.
      • The first burst send request shows a build-up on the client side whereby the client is not reading data quickly enough. This causes a buildup of information on the client side's receive buffer.
        • TCP only: The sender will only send if there is widow size can fit the transmission so any sends will backup and fill the sender's send buffer once the receiver's receive buffer is full. When the sender's buffer is full, Send() will block if it is blocking I/O. This effect is visible in the Performance and Memory Usage View Modes.
        • UDP only: The build-up in the client's receive buffer will cause packets to drop in software. This effect is visible in the Performance, Memory Usage, and Health View Mode.
      • The second burst send request shows a healthy connection whereby the client is reading slightly faster than the server is sending. Because the client is reading slightly faster then the sender, the client doesn't have any buildup which avoids the issues 

Open the socket statistics DevMenuOverlay by holding the "Home button"→QuickMenu→Debug...→SocketStaistics→ OK to enable the display.

Note: There are is also some printed logs from the client and server to indicate what is being seen in the socket statistics graphs that should give you a head start in understanding the behavior of the sample and how it relates to the meter graphs.

Frequently Asked Questions (FAQ)

What are the expected memory and run-time costs of using the socket statistics display?

  • StatisticsMemory struct: 8 KB of memory.
  • Registration resources: 1 Thread added
  • Runtime overhead while statistics display is not open: None
  • Runtime overhead while statistics display is running: 2-5%. 
  • When the system is under high load you might notice that the statistics graphs update at lower than 30 fps or appear to stutter/freeze for a few hundred milliseconds. This is because other aspects of the system take priority over the devOverlayDisplay and may prevent it from updating when busy.

How accurate are the socket statistics display round trip time (RTT, Ping) readings?

  • The latency readings are gotten by sending an ICMP echo request (ping) 2 times per second.
  • ICMP requests may be processed more quickly or slowly than UDP, TCP, or SSL/TSL traffic depending on the peer connection. As a result, the latency readings may not match your protocol-specific latency. 
  • If you want more accurate RTT and are using TCP, please see "nn::socket::GetSockOpt(socketDescriptor, nn::socket::Level::Sol_Tcp, nn::socket::Option::Tcp_Info, output, len)" for a TCP specific RTT.
