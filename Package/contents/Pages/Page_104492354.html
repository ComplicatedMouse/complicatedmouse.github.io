tool

The SDEV serial number does not appear in the SDEV connection list for InitializeSdevWin.

Some serial numbers for SDEV MP devices do not appear when you run InitializeSdevWin, a GUI version of the setup tool.

You can specify the SDEV you want to initialize with the following procedure if it does not appear in the list.

  1. In the connection list for a serial number that is not appearing, right-click one of the displayed rows.
  2. On the menu that appears, click Blink LEDs. The LEDs on the back of the SDEV console will blink.
  3. Try step 2 again after selecting another row in the connection list if the LEDs for a different SDEV light up.

The SDEV IP address cannot be identified.

If the SDEV IP address is fixed as an address that cannot be accessed, it cannot be detected by means such as Target Manager.
The IP address is displayed in the Firmware tab of the DevMenu.
If the system falls to a state in which DevMenu is not started, connect the HDMI directly to a monitor and put the target into sleep mode.
The IP address will be displayed on the monitor.

For SDEV units that have been configured to have an inaccessible IP address, create a network that can be accessed by the SDEV or initialize the PC connection settings on the SDEV.
For more information about resetting connection settings, see SDEV Operations Manual > SDEV Not Recognized by the PC.

Cannot add development hardware to Target Manager.

Before connecting to development hardware, the target must be added (Menu > Tools > Add Target).

If the development hardware is not displayed on the Add Target screen, check the following items.

  • Is the development hardware turned on?

If you are using the SDEV devkit, verify the following items.

  • Is the SDEV connected to the PC via a LAN cable?
  • Is the SDEV on the same subnet as the PC? (Hardware on different subnets cannot be detected.)
  • Are the PC firewall settings correct?

If you are using the EDEV devkit, verify the following items.

  • Is the EDEV connected to the PC via a USB cable?
  • Is the EDEV USB device driver installed?
    • Use the Device Manager tool in Windows to check whether the device driver is installed. If the EDEV is connected, NintendoSdkDebugger or NintendoSDK DevKit is displayed in Device Manager.
    • To manually install the device driver, run the Installers\DevkitDriver\dpinst.exe file located in the NintendoSDK installation directory.
    • You must run the USB driver installation script with administrator privileges. In addition, make sure that you check that execution is not blocked by security software, UAC, or anything else.

Cannot connect to development hardware in Target Manager.

In some cases, the status does not become Connected when selecting Connect to Target for a target that has been added.

When that happens, try the following.

  1. Delete the entry for the development hardware you want to run using Delete Target.
  2. Add the development hardware you want to run using Add Target.
  3. Select Connect to Target for the development hardware added in step 2.

In some cases, the target added in step 3 is assigned an IP address different from the IP address that the target deleted in step 1.
A different IP address may be assigned if Connect to Target is selected immediately after the development hardware is connected to an AC adapter. To prevent a different IP address from being assigned, wait about 60 seconds after connecting the AC adapter before selecting Connect to Target.

It is also possible to avoid having a different IP address assigned by setting a static IP for the development hardware.
For more information, see Setting a Static IP Address for SDEV for the full procedure.

Cannot run programs on development hardware. ("Unable to load..." is displayed in the Target Manager log window.)

Reset the development hardware.

How do I manually install the EDEV USB device driver?

In most cases, the EDEV USB driver is installed automatically when you use NDI to install NintendoSDK.

However, if the automatic install fails or the USB device driver gets deleted for some reason, you must install the device driver manually.

To install the device driver manually, run the Installers\DevkitDriver\dpinst.exe file located in the NintendoSDK installation directory.

A dump file is not created when the application crashes.

If the dump file save location is set to C:\Windows or a similar system directory, and Target Manager is executed without administrator rights, the dump file will not be generated.
This bug is caused by Windows imposing file write restrictions on Target Manager.
Specify a directory that Target Manager can write to as the save location.

Entering and waking from sleep mode takes time when connected to Target Manager.

Because the development environment has a different system structure than the retail device, entering and waking from sleep mode tends to take more time than with the retail device.

The time it takes for each transition becomes longer, especially when connected to Target Manager. It may take 10 seconds to wake from sleep mode, but this is not a bug.

This does not occur in Firmware Approximating the Production Hardware (PROD MODE).

Even though Auto-Sleep is set in the system settings, the system does not sleep when left unused.

In the development environment, the Auto-Sleep and Screen Burn-In Reduction features are disabled by default, regardless of the system settings. (Control Over Idle State Actions)

The current system settings are displayed in the HOME Menu. Note that in some cases, even though a function appears to be enabled in this menu, it may actually be disabled.

You can check and edit the settings for this dedicated development environment in the DevMenu by using Device > Idle Time Detection.

When attempting to update the SDEV to a previously released SDK, the process stops midway.

After updating the SDEV to the latest SDK, if you attempt to update to a previously released SDK and the process stops midway, do the following to initialize the SDEV.

  1. Start the command prompt.
  2. Configure the environment variables.
    1. Set the SDK_NX_LATEST environment variable to the path to the latest SDK.
      　Example: SET SDK_NX_LATEST=D:\NintendoSDK9
    2. Set the SDK_NX environment variable to the path of the SDK you want to update to.
      　Example: SET SDK_NX=D:\NintendoSDK3
    3. Set the SDEV environment variable to the IP address of the SDEV.
      　Example: SET SDEV=192.168.0.10
    4. Execute the following command.
      　%SDK_NX_LATEST%\Tools\CommandLineTools\UpdateHostBridge.exe update --force --image %SDK_NX%\Resources\Firmwares\NX\HostBridgeFirmware.nhf --target %SDEV%
  3. After the command from in 2d is complete, wait approximately 30 seconds while the SDEV restarts.
  4. Initialize with InitializeSdev.

If the SDEV screen remains black and nothing is displayed even after waiting for several minutes, see Initializing the Connection Settings Between the SDEV and PC.

Note that you can also revert to a previous SDK by following the instructions in Initializing the Connection Settings Between the SDEV and PC from the beginning, although that procedure initializes the SDEV network settings.

VSI

When building, I receive the error "Warning: The platform 'NX32' / 'NX64' referenced in the XX project file is not found."

VSI may not have been installed correctly.

To check whether the correct version of Visual Studio Integration for NX is installed, select VS > Tools > Extensions and Updates.

If it is not installed, reinstall VSI from NDI.

For more information about reinstalling, see Manual.

When a previous version of Visual Studio is installed, I cannot build a project from MSBuild.

If an earlier version of Visual Studio is installed, and it is Visual Studio 2010 or later, MSBuild attempts to build projects using that version of Visual Studio.

However, you can only build NintendoSDK projects in the supported versions of Visual Studio (Visual Studio 2012, Visual Studio 2013, and Visual Studio 2015 in NX Add-On 0.5.0).

As a result, in an environment where a previous version of Visual Studio is installed, you cannot use MSBuild to build NintendoSDK projects.

In this situation, use VisualStudioVersion properties or environment variables to properly configure the settings for the Visual Studio version used by MSBuild.

For example, if you are using Visual Studio 2015, set VisualStudioVersion to 14.0.

NX code is not generated, even if NX32/NX64 is added to the solution platforms for a project that was created with Generic selected as the platform.

If either of the following methods was used to create the project, adding NX32/NX64 to the solution platforms generates code for the PC environment instead of for NX.

  • The project was created in an SDK environment where the NX Add-On package was not installed.
  • The NX Add-On package was installed, but Generic was selected in the screen in the wizard for selecting the platform.

To avoid this problem, try the following procedure.

  1. Open the applicable project in Visual Studio.
  2. In the Property Manager window, replace the imported property sheet NintendoSdkSpec_Generic with NintendoSdkSpec_NX.

For more information about using the Property Manager, see the following page.

Adding NX Development Hardware Platforms to an Existing Visual C++ Project

The generated binary data is not the same, even though the build options and input files were not changed.

When linking applications or dynamic modules, the linker attaches the build ID, that is used when debugging in VSI, to the binary data. If the UUID is used for this build ID, the generated binary data will not always be the same, even if there were no changes to the build options or input files, because a different UUID is generated for each build.

To work around this problem, you can set a hash value as the build ID. To make this change, open the project properties page in Visual Studio, and then select Linker > General > Generate Build Identifier. The time it takes for the hash value to be calculated depends on the size of the binary data that is generated, and as a result, making this change generally increases linking time.

The build fails with an unsupported option '--build-id=md5' error.

If VSI for NX with a version earlier than 6.1 is installed after VSI for NX 6.1 or later was already installed, building applications would fail with the error message unsupported option '--build-id=md5'.

To resolve this issue, reinstall using the procedures described inManual.

Visual Studio sometimes stops responding for several minutes.

If Visual Studio sometimes stops responding, the cause may be antivirus software. Try one of the following fixes for the problem.

  • Delete the C:\Users\<YOUR_USER_NAME>\AppData\Local\Microsoft\VSCommon\12.0\SQM directory.
    • The "12.0" part of the name depends on which Visual Studio version you are using. Read it as "14.0" if you are using Visual Studio 2015, and read it as "15.0" if you are using Visual Studio 2017.
  • Remove devenv.exe from the WindowsDefender search targets.

Cannot handle exceptions thrown in SDK libraries.

The NX32 and NX64 SDK libraries do not support C++ exceptions.
For this reason, when exceptions are thrown in the libraries, they are canceled.

If the code is only propagated to the application, you can enable C++ exceptions at build to make it possible to handle exceptions.

Debugging stops when the system enters sleep mode.

The specification calls for debugging with VSI to stop when the system enters sleep mode.
If you want resume debugging after the system wakes, reattach as described in Debugging an Already Running Application in Debugging a Built Program.
By setting a breakpoint prior to the transition to sleep, you can attach immediately after waking.

Installation of the VSI for NX package fails in environments without Internet connectivity.

In environments where you cannot connect to the Internet, installation of the VSI for NX package sometimes fails.

If you cannot connect to the expiration server for the digital certificate, you cannot check the validity of the signature, and VSI cannot be installed.

To avoid this problem, try the following procedure.

  1. Connect to the Internet
    or
    install an intermediate certificate issued by an authority higher than the digital certificate (CN = DigiCert SHA2 Assured ID Code Signing CA).
  2. Re-install the VSI for NX package.

VSI for NX will fail to install if both Build Tools for Visual Studio 2017 and Visual Studio 2017 are installed.

Installing VSI for NX can fail when both Build Tools for Visual Studio 2017 and Visual Studio 2017 are installed.

This failure is caused by a bug in the VSIXInstaller for Visual Studio 2017 (for version 15.4) where the Visual Studio extensions are not installed when Build Tools for Visual Studio 2017 has already been installed. For more information, see the following page: vsixinstaller.exe throws NullReferenceException when Visual Studio Build Tools 2017 are installed.

To avoid this problem, try the following procedure.

  1. Uninstall Build Tools for Visual Studio 2017.

Execution From VSI Fails

The VS version may be out of date.
Update to a supported version listed in NintendoSDK Overview > Operating Environment.

diag

When the connection with Target Manager is unstable, sometimes logs will be omitted or the log API will run slowly.

When the connection with Target Manager is unstable, sometimes logs will be omitted or the log API will run slowly. If you encounter these symptoms, improve the connection status with Target Manager.

The SD card log will work without an issue, even if the connection with Target Manager is unstable. If improving the connection status is problematic, consider using the SD card log.

A "(too far)" message is added to the stack trace log, while the expected function names are not included.

When using something like the default user exception handler, a "(too far)" message is added to the stack trace log together with unexpected function names that are not running.

This happens when there are instructions in local functions or in functions where symbol information is deleted due to dead-stripping. This behavior is by design. The stack trace output implementation works as follows.

  • If no symbol information is present for a function that includes instructions, output the name of the closest function that has symbol information and output an offset from that function as the stack trace.
  • If the offset is greater than the size used for the closest function distance, display the "(too far)" message.

You can enable dead-stripping in Visual Studio project properties. To do so, select Linker > General > Enable Dead Strip.

Similarly, when the nn::diag::GetSymbolName() function cannot find symbol information, it returns the name of the closest symbol.

fs

The message "Passed buffer is not usable for fs library" appears, and nn::fs::ReadFile(), nn::fs::WriteFile(), and nn::fs::ReadDirectory() stop.

The following buffers cannot be passed to nn::fs::ReadFile(), nn::fs::WriteFile(), and nn::fs::ReadDirectory().

  • Unallocated buffers
    • To use the nn::fs::ReadFile() function, you must allocate a memory region that is at least size, starting from the address set in buffer and regardless of the actual file size for the target file.
    • To use the nn::fs::ReadDirectory() function you must allocate a memory region that is at least entryBufferCount × sizeof(nn::fs::DirectoryEntry), starting from the address set in entryBuffer, irrespective of the actual number of entries for the target directory.
  • nn::gfx
    • Memory pool buffer with the nn::gfx::MemoryPoolProperty_CpuUncached attribute specified
  • nvn
    • Memory pool buffer with the NVN_MEMORY_POOL_FLAGS_CPU_UNCACHED_BIT or the NVN_MEMORY_POOL_FLAGS_CPU_NO_ACCESS_BIT attribute specified

For the nn::fs::ReadDirectory() function, in addition to the preceding buffers, the following buffers cannot be passed.

  • nn::audio
    • Buffers specified in nn::audio::WaveBuffer.buffer and nn::audio::WaveBuffer.pContext
    • Memory storing the data passed to pParameter by the nn::audio::AcquireVoiceSlot() function
    • The work buffer specified for buffer in the nn::audio::AddDelay() and nn::audio::AddReverb() functions
    • The work buffers specified for pSendBuffer and pReturnBuffer in the nn::audio::AddAux() function
  • nn::gfx, nvn
    • Memory pool buffer, irrespective of attributes


When these buffers are passed to nn::fs::ReadFile(), nn::fs::WriteFile(), and nn::fs::ReadDirectory(), the programs stop along with the following type of message.


Passed buffer is not usable for fs library. For more information, see the fs library manual.


Do not run nn::fs::ReadFile(), nn::fs::WriteFile(), or nn::fs::ReadDirectory() directly using these buffers, and instead run them using different buffers.

When calling an fs function, the message "Data access to an invalid memory region" is displayed and the application stops.

If the allocator set by the nn::fs::SetAllocator() function returns an address that is not 8-byte aligned, calls to fs functions throw an exception, a message like the following is displayed, and the application stops.

The allocator you set with the nn::fs::SetAllocator() function normally returns addresses that are 8-byte aligned.

UserException handler is called at pid=XXX
==============================================================================
  Data access to an invalid memory region. (ExceptionType=0x0101)
  An alignment violation memory access occurred. (DFSC=0x21)
  Accessed to: 0xXXXXXXXXXXXXXXXX
==============================================================================

nn::fs::MountRom() fails for a NAND-booted application and returns 0x000AF002 (Module 2 = fs, Description 1400).

If the NMETA file specified in the AuthoringTool subcommand creatensp disagrees with the NMETA file specified when the code was built (that is, the NMETA file specified in VSI), the nn::fs::MountRom() function will fail and return 0x000AF002 when the application is installed and run in NAND memory.

Make sure that you specify the same NMETA file for creatensp that was specified at build time.

In the future, a feature will be added to check whether the correct NMETA file has been specified for creatensp.

The "Specified target is not found" message appears and the nn::fs::MountSaveData() function stops.

No save data is created.

You must perform one of the following: Either denote the save data information in the NMETA file and create it automatically using the option to select a user before starting an application, or create the save data using the nn::fs::EnsureSaveData() function and then call the nn::fs::MountSaveData() function.

For more information, see Save Data (NX Add-On).

Changes to save data size are not updated.

If save data already exists, it will not be recreated, even when you update the NMETA file. Delete the save data before proceeding.

Along with the message "Invalid path was specified," nn::fs::CreateFile() and nn::fs::CreateDirectory() on save data stop.

The maximum length of the save data file and directory names is 64 bytes, not including the terminating null character.

For more information about save data restrictions, see Save Data.

Creating files and directories on save data fails with nn::fs::ResultUsableSpaceNotEnough.

If the size of the data save memory for save data is insufficient, nn::fs::ResultUsableSpaceNotEnough is returned.
Check whether a sufficient size has been written in the NMETA file.

For more information, see Save Data (NX Add-On).

nn::fs::MountSdCardForDebug() fails.

When nn::fs::MountSdCardForDebug() fails, the following conditions can be considered.

  • The SD card is not inserted.
  • The SD card format is invalid.
    • FAT formatting was not supported in versions of NintendoSDK Firmware for NX earlier than 2.0.0-3. Note that SD cards larger than 32 GB are often formatted in exFAT.
    • FAT formatting is supported starting with version 2.0.0-3 of NintendoSDK Firmware for NX.
  • The contact between the SD card and the SD card connector is bad.
    • You can resolve this problem by removing and then reinserting the SD card.
  • Either the SD card or its connector are damaged.
    • A label or sticker on the SD card can damage the connector when the card is inserted that way.
    • It is highly possible that the SD card connector is damaged if you can access the SD card from another device, like a PC.

The Last Updated or Last Accessed timestamps in the SD card file appear to revert to a time earlier than the creation time.

Because the timestamp precision for files in the SD card is different, in some cases the Last Updated or Last Accessed timestamps appear to be at a time or date earlier than the time when the file was created.

For more information, see SD Card.

The access log is not available.

  • Common
    • You must start the application to be checked after changing FS Access Log Mode to Log, SdCard, or Log+SdCard.
  • FS Access Log Mode set to either Log or Log+SdCard:
    • The application build type must be Develop.
  • FS Access Log Mode set to either SdCard or Log+SdCard:
    • You must insert an SD card into the development hardware before starting the application.
    • For more information, see the earlier bug nn::fs::MountSdCardForDebug() fails.

The access log is not saved to the end on the SD card.

  • You must remove the SD card after either shutting down the development hardware or putting it into sleep mode.
    Access logs are written to an SD card when the internal fs buffer is about to overflow, or when either a shutdown or sleep mode is about to occur, because consecutive writing of the access log to an SD card causes the application behavior to lag.
  • FS_ACCESS: { start_tag: true } is written at the start of an access log exported to SD cards. FS_ACCESS: { end_tag: true } is written at the end of an access log when a shutdown occurs or when entering sleep mode.

[INFO] Access log is incomplete. is exported from FsAccessLogChecker.

  • The access log is not saved to the end on the SD card. For more information, see the earlier bug The access log is not saved to the end on the SD card.

[WARNING] There are multiple fs access logs is exported from FsAccessLogChecker.

  • Included in the access log when an application is started multiple times.
    Check all the results, and not just those at the end of the log, because FS Access Log Check Result is exported every time the application is started.
  • Logs will also be added if a FsAccessLog.TXT file exists at the root level of an SD card, if the access log is exported to an SD card.

[FAIL] TotalTime (expect: More than 10 minutes) appears in FsAccessLogChecker.

FsAccessLogChecker needs at least 10 minutes of access log data because even when there are sporadic bouts of high-frequency access, the values are averaged out and checked.

For applications that almost never access the file system, the access log should be at least 10 minutes long, and it should include operations that involve access to the file system, such as saving data or changing scenes.


[WARNING] application_info_tag was not detected is output in FsAccessLogChecker

Enter an access log obtained from SD card output by development hardware initialized with NintendoSDK Firmware for NX 11.0.0-6 or later.

Access logs obtained from development hardware initialized with versions earlier than NintendoSDK Firmware for NX 11.0.0-6 and access logs obtained from log output do not include logs for Switching Between Programs, and technically cannot be analyzed.

The DirectoryHandle opened for the host PC file system is no longer valid after waking up from sleep mode.


Symptoms

Run nn::fs::OpenDirectory() for the host PC file system and let the system enter sleep mode. When the system wakes up, the obtained DirectoryHandle is no longer valid, and nn::fs::ReadDirectory() and nn::fs::GetDirectoryEntryCount() abort.

Workaround

In Target Manager, do not let the system enter sleep mode while a directory is open for the host PC file system.
Use Target Manager 2, as this issue does not occur in Target Manager 2.

movie

When the user transitions to the HOME Menu during video playback and then returns to the application, the video playback has progressed.

The behavior of an application in background focus state, after the user transitions from the application to the HOME Menu, differs depending on the focus handling mode. If the focus handling mode is in Notification Mode, video playback processing continues as is even in the background focus state.

To avoid having the video continue playing on its own, stop the video decoding process when a notification of a change to the application focus state (notification of transitioning to background focus state) is received.

Info

The performance of the HOME Menu and other system features is slightly affected, so we recommend stopping video decoding processing under these conditions.

For more information about the focus handling modes, see the following page.

Notifications For Programs

The program sometimes does not close immediately after video playback finishes in MovieDecoderPlayer.

After playback of the specified video finishes (when the loop playback option is not used), the MovieDecoderPlayer sample may continue to display the last frame of the video for a short time.

This has no effect on the video and audio playback processing itself, so it is the specification of the sample.

NVN

I’m experiencing a large number of link errors.

When you include nvn/nvn_FuncPtrImpl.h from multiple files, a large number of link errors may occur, and the linking time may become extremely long.
One type of link error you might see is duplicate definition errors for symbols such as nvnLoadCProcs.

nvn/nvn_FuncPtrImpl.h defines global functions and global variables. We assume that it is included in only one source code in a project. For this reason, if you include this header file from multiple C or CPP files, you may experience these types of issues and errors.

Include this header file with a single C or CPP file.

Pressing the HOME Button when an application that uses multiple GPU queues is running causes the system to freeze at the application screen.

The application must ensure that the GPU rendering command completes for the framebuffer submitted by functions such as nvn::Queue::PresentTexture(). Normally, functions such as nvn::Queue::PresentTexture() flush the GPU queue, so the render command always finishes without any involvement from the application developer.

For applications that use multiple GPU queues, a certain queue (A) may wait for the GPU commands collected in another queue (B) to complete. In this case, if the application CPU processing first calls PresentTexture() in queue A and then queue B implements flushing, depending on when the HOME Button is pressed, the commands in queue B may be suspended by the application before flushing, resulting in the rendering for the framebuffer being unable to complete.

Use one of the following methods to avoid this problem.

  • Flush all other dependent queue commands before calling PresentTexture().
    • In the preceding example, this means flushing queue B before calling PresentTexture() in queue A.
  • The application always flushes all queues before suspension.
    • Change the focus handling mode to notification mode to prevent the application from suspending automatically, then flush all of the queues required for completing rendering before suspending the application. The focus handling mode is described in Notifications For Programs .

The flushed GPU commands continue running even if the application CPU processing was suspended, so as long as the queue has been flushed, the application can be suspended without waiting for GPU processing to complete.

TV Output and LCD Issues

The CEC features do not work on some TVs.

The one-touch play and system standby features may not work when using some TVs. Although the connectivity of the CEC feature will be improved in the future, it might not work on all TVs.

With some televisions, if the CEC feature for controlling power with the TV is enabled and the console is undocked and then quickly docked, in some cases the video feed will not play on the TV.

When using some televisions, video to the television might no longer be displayed if the console is docked immediately after being undocked. Although connectivity will be improved in the future, it might not work on all TVs.

The colors appearing on the screen are dark.

The NX only supports the sRGB format for display output buffer formats. Buffer values are parsed as is as sRGB values, without any gamma correction, when the buffer format is linear RGB. As a result, the screen is darker overall when you create the display output buffer using the linear RGB format.

Use the following procedure to convert the display output buffer format to sRGB.

  • Set the swap chain color format to nn::gfx::ImageFormat_R8_G8_B8_A8_UnormSrgb when using nn::gfx.
  • Create the textures you configure in NVNwindow using the NVN_FORMAT_RGBA8_SRGB format when using NVN.

To determine the buffer format for the graphics API you use, you do not need to specify this setting from nn::vi.

new Is Called Internally in the SDK (only in a PC environment)

In the PC environment only, the SDK library calls new internally and allocates a maximum of approximately 200 MB of application memory.
If the user has overloaded the new function, the SDK calls the overloaded version of the function.

The SDK library sometimes calls new internally even if you use one of the API functions (such as nn::fs::SetAllocator) to explicitly set an allocator.

There is no way to prevent the new function from being called. Make sure to always leave at least approximately 200 MB of free space in memory.

This limitation is not present in the NX development hardware environments (SDEV/EDEV).

Charging/Power

The EDEV system cannot be charged to 100 percent.

When running in TV mode, the EDEV system sometimes cannot be charged all of the way to 100%.

This occurs because the charging settings are updated for safety purposes if the system temperature reaches a certain level.

To charge the system fully to 100%, you must perform one of the following to allow the system temperature to decrease.

  • Put the system into sleep mode.
  • Remove the system from the dock and directly connect the AC adapter.

Cannot connect to the network.

nn::nifm::IsNetworkAvailable() does not become true.

Use the following procedure for both wired and wireless networks.

Confirm that the settings are imported using the settings manager for the development hardware.

Importing the settings using the Windows tool will not affect the development hardware, and vice versa. Also ensure that the system is always restarted after importing settings.

Check that the settings are as you intended.

To check the imported content, see nifm Library > Registering Network Connection Settings to export and verify the settings.

The settings for authentication and encryption must be a combination supported by the SDK.

To include targets used automatically by the system, you must enable the settings used for automatic connections.

To connect to wireless access points with the stealth settings, you must set the non_broadcast option in the connection settings to true.

Wired

Confirm that the USB Ethernet adapter is connected via the dock.

Operations are not ensured when directly connected to the SDEV.

Confirm that there are not multiple connected USB Ethernet adapters.

Only one USB Ethernet adapter can be connected at a time. If multiple USB Ethernet adapters are connected, detach them all and then reconnect just the one you intend to use.

Confirm that you are using a correct combination of dock and SDEV hardware.

The USB Ethernet adapter is not recognized when different versions of the dock and SDEV are used in combination.

Wireless

Check that the DIP switches numbered 1 and 2 on the lower left of the back of the SDEV are turned on.

If they are turned off, the wireless connection will be unstable.

Confirm that the security type is supported.

Check that you are using a line that can connect to the external network.

As of NintendoSDK 0.11.0, proxy settings configured in the network settings are automatically applied. Also, any proxy settings configured using curl_easy_setopt() are ignored. To disable proxy settings, follow the procedure in nifm Library > Registering Network Connection Settings when configuring your connection settings or specify 0 for CURLOPT_PROXYAUTOCONFIG in curl_easy_setopt().
