Important Information for Starting a Project

This page summarizes important information for developers to consider when starting a project.

Project Settings

The settings that are applied to an overall project are known as project settings and are stored in a project configuration file.

The project configuration file is a single file located directly beneath the parts root folder and allows all developers and designers working on the project to use the same shared settings.

For more information about the settings contained in the project configuration file, see Project Configuration File Format.

Items to Consider First

During the very initial stages of a project, developers must consider the following list of settings.

Many of these settings would require a large amount of work to fix the project if they are changed later .

  • Color space (linear or sRGB)
    • We recommend using linear color space.
    • More specifically, we recommend the Binary Converter settings --degamma-texture=true and --degamma-parameter=false.
      • Reference: Layout Binary Converter
  • Screen size
    • 1080p, 720p, etc.
    • Specifies the size of the newly created layout using the DefaultInitialLayoutSize element.
  • Supported languages
    • Affects font type selection.
  • Animation segment tag behavior (PreviewConverterOptionOmitNoKeyAllEnabled)
    • We recommend setting --omit-nokey-all to ON.
      • Reference: Animation Segment Tag Animation Conflicts and Workaround
  • Search path for the PSD source file for texture assets
    • Configure according to the folder structure rules.
  • Capture texture
    • Disabled by default, but in general we recommend enabling it in most projects.
  • Default values in LayoutEditor
    • Default pane names
    • Default texture format

Settings to Add Later

The following list includes settings that developers can add as a project progresses.

  • ui2d Extended User Information
    • Extended User Data Presets
  • ASTC Textures
  • Commands to use when layouts are saved and loaded in LayoutEditor
    • Can be used to link up assets and resources with version control systems (Perforce, git, svt, and the like).
  • Preview Resolution Settings
    • Set according to the screen resolution assumed in the project.

Data Conversion Workflow

In general, we recommend the following workflow.
LayoutEditor also supports a feature for exporting binary data directly, but in most workflows use of this feature is unnecessary.

  • Designers edit FLYT files in LayoutEditor and submit the results.
  • Developers create batch processes for starting LayoutConverter and performing binary conversion. At this point, layouts are archived in LayoutArchiver at a level of granularity appropriate for the game.
    • Fonts are generally archived and loaded as shared resources separate from layout archives.
      • In LayoutConverter, you can specify the --no-copy-font option to remove fonts from exported binary layout data.

Fonts

Supported font formats include scalable fonts (where TTF and OTF fonts are used as is) and bitmap fonts (where glyphs are converted to images).

For scalable fonts, we recommend creating and using FCPX files, and for bitmap fonts, we recommended creating and using FFNT files.

Each file type has different characteristics. Use them accordingly.

In particular, when supporting Chinese, Korean, and Taiwanese versions of an application, the number of required characters is very large. This can result in very large font file sizes when using bitmap fonts, so there is a more pronounced need to use scalable fonts.


For an overview of the font library, see Overview, and for a detailed workflow, see Quick Start.
