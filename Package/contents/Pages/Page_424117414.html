Installing Applications

For more information about installing applications, see Installing and Running an NX Application on the Development Hardware.
For more information about NSP files, see Creating Application Images and Guide to Setting Up the Environment.

$ RunOnTarget.exe DevMenuCommand.nsp -- application install "C:\application.nsp"
Preparing to install C:\application.nsp...
[SUCCESS]

Use the --force option to install the application using the same existing program ID.
(The original application is deleted.) 

$ RunOnTarget.exe DevMenuCommand.nsp -- application install "C:\application.nsp" --force
Preparing to install C:\application.nsp...
[SUCCESS]

Use the -s option to specify the storage where the application will be installed.
The following values can be specified with this option.

  • builtin: Install to the system memory.
  • sdcard: Install to a microSD card.
  • auto: Automatically selects where to install the application.

If the -s option is not specified, the application is installed to system memory.

$ RunOnTarget.exe DevMenuCommand.nsp -- application install "C:\application.nsp" -s sdcard
Preparing to install C:\application.nsp...
[SUCCESS]

To install from a microSD card, specify sdcard:\ at the start of the path of the NSP file being installed.

$ RunOnTarget.exe DevMenuCommand.nsp -- application install "sdcard:\application.nsp"
Preparing to install sdcard:\application.nsp...
[SUCCESS]
Note

After installing content, namely the application, patch, and downloadable content, to the SD card, inserting and removing that SD card might cause the content on the SD card to become unavailable depending on how the SD card was removed.

For example, in the development environment, this occurs if you perform some sort of installation while the SD card is removed. (This operation does not make the content unavailable in the production environment.)

If this issue occurs, re-install the content.


Installing and Uninstalling Application Launch Parameters

For more information about application launch parameters, see Getting Launch Parameters.

Files installed on microSD cards can be retrieved by applications as launch parameters.

Note

Installed parameters are only available when DevMenu is selected from the menu.
Note: You cannot pass parameters to applications when the HOME Menu is selected, even when parameters are installed.

For more information about switching menus, see Switching Menus on the home page.

Insert a microSD card in the development hardware and then run the command. The data in this file is passed, unmodified, to the application.
If parameters are already installed, you must use the --force option to overwrite them.

$ RunOnTarget.exe DevMenuCommand.nsp -- application install-nxargs "C:\parameter.nxargs" --application-id 0x01004b9000490000
[SUCCESS]

The following command uninstalls the launch parameters installed on a microSD card.

$ RunOnTarget.exe DevMenuCommand.nsp -- application uninstall-nxargs 0x01004b9000490000
[SUCCESS]

Starting an Application

This command specifies an application ID and starts that application.

$ RunOnTarget.exe DevMenuCommand.nsp -- application launch 0x01004b9000490000
[SUCCESS]

Use the --user-index option to specify the index of the user account that is selected when the application starts.

$ RunOnTarget.exe DevMenuCommand.nsp -- application launch 0x01004b9000490000 --user-index 0 
[SUCCESS]
Note

In NintendoSDK Firmware for NX 12.0.0-3.0 and later versions, the following operation is unnecessary.

Normally, the feature to select a user account at startup is disabled after the application has started, but it might not be disabled if application startup fails and is canceled.
If the feature to select a user account when starting is not disabled, a user account is always selected when the application is started from the host PC (including via TargetManager or VSI).
To avoid this happening, run the application disable-auto-user-selection command to disable user account selection at startup.

Info

The --timeout option is available in NintendoSDK Firmware for NX 12 .0.0-3.0 and later.

In NintendoSDK Firmware for NX 12 .0.0-3.0 and later, the launch command behavior is just like selecting and starting the application from the menu screen.

When the application is selected and started from the menu screen, the menu screen might wait for some kind of user operation, like selecting a user or responding to a displayed error.

Use the --timeout option to set how much time, in seconds, for the launch command to wait for the menu-driven startup sequence.

The launch command ends if the time set with the --timeout option has elapsed, but the menu-driven startup sequence continues.

If the --timeout option is not specified, the timeout time is set to 5 seconds by default.

$ RunOnTarget.exe DevMenuCommand.nsp -- application launch 0x01004b9000490000 --timeout 10
[SUCCESS]
Info

The --path option is available in NintendoSDK Firmware for NX 13.0.0-1.0 and later.

In NintendoSDK Firmware for NX 13.0.0-1.0 and later, you can use the launch command to start NSP and NSPD files.

For the --path option, specify the absolute path of the NSP or NSPD file you want to start.

$ RunOnTarget.exe DevMenuCommand.nsp -- application launch --path "C:\application.nsp"
[SUCCESS]
Info

The -i option is available in NintendoSDK Firmware for NX 14.0.0-1.0 and later.

For the -i option, specify an absolute path to the file that stores the launch parameters.

For more information about the launch parameters, see Getting Launch Parameters.

$ RunOnTarget.exe DevMenuCommand.nsp -- application launch 0x01004b9000490000 -i "C:\01004B9000490000.nxargs"
[SUCCESS]

Sizes of System Memory and microSD Cards

Use this command to display the total space and free space of system memory or a microSD card.

$ RunOnTarget.exe DevMenuCommand.nsp -- application storage-size sdcard
Free space      3998973952 B
Total space     4000317440 B
[SUCCESS]

Use the -b option to change the units in which the sizes are displayed.

  • g: GigaByte
  • m: MegaByte
  • k: KiloByte
  • b: Byte
  • a: Auto

Application Size

Displays the application size (the program, updates, and downloadable content).

$ RunOnTarget.exe DevMenuCommand.nsp -- application occupied-size 0x0100000000012801
----------------------------------------------------------
[Built-in]
Application     5159936 B
Patch           0.0 B
Downloadable Content  0.0 B
----------------------------------------------------------
[SD Card]
Application     0.0 B
Patch           0.0 B
Downloadable Content  0.0 B
[SUCCESS]

Use the -b option to change the units for the sizes that are displayed.

  • g: Gigabyte
  • m: Megabyte
  • k: Kilobyte
  • b: Byte
  • a: Auto

Invalidating Application Control Data Cache

Use this command to invalidate the cache for the application control data, specified in the NMETA file.

$ RunOnTarget.exe DevMenuCommand.nsp -- application invalidate-cache application-control
[SUCCESS]

The application control data is cached by the system in system memory. In the following cases, this cache is loaded instead of the application control data on the game card. Run this command if you do not want the HOME Menu or DevMenu launcher to load this cache in these cases.

Case Solution
When the game card is switched out for another game card with the same version of the application. Run this command and then insert the game card. If using the HOME Menu, take the additional step of restarting the development hardware.
When the same version of the application is reinstalled using DevMenu or DevMenuCommand.

Restart the development hardware, but only if using the HOME Menu.

The cache is refreshed when the application starts, so in such cases, the application control data that is included in the NSP file will be used even if you do not run this command.

Resetting the Required Version

Resets the required version information. The required version mechanism is used to prevent out of date versions of an application from starting.

$ RunOnTarget.exe DevMenuCommand.nsp -- application reset-required-version 0x01004b9000490000
Reset: 0x01004b9000490000
[SUCCESS]

To remove all, you can use the --all option instead of specifying an application ID.

$ RunOnTarget.exe DevMenuCommand.nsp -- application reset-required-version --all
Reset: 0x01004b9000490000
Reset: 0x01004b9000490001
[SUCCESS]

Disabling the Required Version Check

Info

This feature is available in NintendoSDK Firmware for NX 13.0.0-5.0 and later.


Disable the process of checking for the required version at startup.

$ RunOnTarget.exe DevMenuCommand.nsp -- application disable-required-version-check
[SUCCESS]

The check for the required version is performed to alleviate concern about the compatibility of save data in the application.
Note that if you disable this check for the required version at startup, earlier versions of the application will be able to read save data written by the new version of the application.


Enable the process of checking for the required version at startup.

$ RunOnTarget.exe DevMenuCommand.nsp -- application enable-required-version-check
[SUCCESS]

Check whether the process of checking for the required version is currently enabled.

$ RunOnTarget.exe DevMenuCommand.nsp -- application get-required-version-check-enabled
enabled
[SUCCESS]


Disabling User Account Selection at Application Startup

Use this command to disable the feature to select a user account at application startup, which was enabled by the --user-index option when the application launch command was run.

$ RunOnTarget.exe DevMenuCommand.nsp -- application disable-auto-user-selection
[SUCCESS]

Application Functionality Restrictions

Info

Available in NintendoSDK Firmware for NX 10.0.0-1.0 and later.

Some of the functions provided by the system can become a security risk if used incorrectly.
If security risks related to these functions are discovered from an application, the system may restrict the use of those functions applicable with regard to that application.
When developing an application that uses these functions, use the following command to confirm that the restrictions work as expected.
DevMenu may also be used for configuration instead of the following commands.

Restricting Specified Applications

The application register-function-authorization-error-simulation command restricts the function of the specified application.
If this command is executed continuously, the previously registered content is erased and new function restrictions are registered.
Restrictions cannot be registered for multiple applications simultaneously.

$ RunOnTarget.exe DevMenuCommand.nsp -- application register-function-authorization-error-simulation 0x0100eee00197e000 --function LocalNetworkMode
[SUCCESS]

Specifies the function to be restricted with the --function option. The following functions can be specified.

Feature Name Overview Reference
LocalNetworkMode Internal LAN communication mode achieved by calling nn::nifm::SetLocalNetworkMode()

nifm Library

The release version of the application to be restricted can be specified using --version.
If the application release version is less than or equal to the release version specified via this option, the function specified by the --function option is restricted.
Functionality is restricted for all versions if the --version option is not specified.
For more information about specifying the application release version, see Writing the NMETA File > Release Version.

Removing Application Functionality Restrictions

The application unregister-function-authorization-error-simulation command removes function restrictions on the currently active application.

$ RunOnTarget.exe DevMenuCommand.nsp -- application unregister-function-authorization-error-simulation
[SUCCESS]

Checking the Application Functionality Restriction Settings

The application get-function-authorization-error-simulation-settings command displays restriction information on the currently active application.

$ RunOnTarget.exe DevMenuCommand.nsp -- application get-function-authorization-error-simulation-settings
ApplicationId: 0x0100eee00197e000
Function: LocalNetworkMode
Version: 65535
[SUCCESS]
