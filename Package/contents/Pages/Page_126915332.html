Display Process

Overview

The display process's primary purpose is managing the physical displays and all content submitted to them.  This ultimately leads to a few areas that concerns applications.  Applications interface with the display process through use of the VI Library.

System Policies

For a given platform, there may be strict limitations on the content that may be submitted to physical displays without affecting performance.  If an application attempts to perform an action that might cause performance degradation or is otherwise impossible due to hardware limitations, that action will be rejected.  From the application's point of view, the function will return an appropriate error result.

The display process provides information about the various policies with the use of logical displays.  These displays may map to one or more physical displays.  For more information, please check this section.

Composition

Multiple processes can submit content to physical displays at the same time.  The display process accepts content from all of these sources and composites them into a single image that is finally presented to the physical display.

Info

Composition functionality utilized by the display process is not available to applications and is reserved for system use.

The display process will manage the details of composition on behalf of the application.  This will ensure that the output appears correct for users when accounting for all system features.

Framebuffer Management

All content appearing on the display passes through the display process at some point.  Applications create layer objects that map to final render targets.  The render targets and the backing texture memory is owned by the application.  However, the display process manages the layer objects and will provide a texture not in use by either the physical display or the GPU upon request.

The entire sequence of events is described here and forms the basis of the various buffering modes available on the platform.
