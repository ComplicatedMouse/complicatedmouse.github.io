Overview

You can use DevMenu to change the settings for the development assistance features.

If the settings on this page have been changed from outside of DevMenu (such as with DevMenuCommand), leave the Debug page and then return to see the changes applied on the page.

Note

Some of these items are not displayed in PROD MODE. They can be set only in DEV MODE.

How to Use

Launch Mode Settings

Initial Boot Menu

Configure the menu that starts initially on the development hardware. The default is DevMenu.

  • Specify DevMenu to start DevMenu when the system starts.
  • Specify HomeMenu to start the HOME Menu when the system starts.

You must restart the development hardware to apply the new setting.

To switch between starting the HOME Menu and starting DevMenu, use a DevMenuCommand such as debug switch-menu.

Select HomeMenu to configure the development hardware identically to retail units.

Exhibition Mode

Use this setting to enable or disable operations in exhibition mode. The default is Disabled.

You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

In-Store Demo Mode

This feature enables and disables RID mode. The default is Disabled.

For more information, see Setting Up a Testing Environment for Retail Interactive Demos.

Select Disabled to configure the development hardware identically to retail units.

Account Restricted Mode

Enables the feature for setting restrictions on opening user accounts. The default is Disabled.

For more information, see User Account.

Performance Settings

Operation Mode Policy

You can change the control policy for the operation mode for development purposes.

  • When Operation Mode Policy is set to Auto, the operation mode changes depending on whether the system is docked.
  • When Operation Mode Policy is set to Handheld, the operation mode is fixed to handheld mode.
  • When Operation Mode Policy is set to Console, the operation mode is fixed to TV mode.

The default is Auto.
You must restart the development hardware to apply the new setting.

Select Auto to configure the development hardware identically to retail units.

Performance Mode Policy

You can change the control policy for the performance mode for development.

  • When Performance Mode Policy is set to Auto, the performance mode changes depending on whether the system is docked.
    • However, if Operation Mode Policy is set to Handheld, the performance mode is fixed to normal mode.
  • When Performance Mode Policy is set to Normal, the performance mode is fixed to normal mode.
  • When Performance Mode Policy is set to Boost, the performance mode is fixed to boost mode.

The default is Auto.

Note
  • Boost can only be used for the Performance Mode Policy on the SDEV. It cannot be used with the EDEV.
  • When Performance Mode Policy is set to Boost, USB devices cannot be run from the system's power supply due to the limits on the overall power consumption by the board.

Select Auto to configure the development hardware identically to retail units.

CPU Overclock

Note
  • Not available in NintendoSDK Firmware for NX 10.0.0-1.0 and later environments.
  • In environments with NintendoSDK Firmware for NX 10.0.0-1.0 and later, use the nn::oe::SetCpuOverclockEnabled() function to enable CPU overclocking.

The CPU clock can be set to a rate that is higher than the clock rate that can be used in the retail device.

  • If CPU Overclock is set to Disabled, the CPU clock rate is 1020 MHz, which is the clock rate that can be used by the retail device.
  • If CPU Overclock is set to Enabled, the CPU clock rate becomes 1785 MHz. This setting is designed for the debug build environment and to handle the higher processing load that can result from features unique to development.

The default is Disabled.

Select Disabled to configure the development hardware identically to retail units.

Note
  • It is only available with the SDEV. It cannot be used with the EDEV.
  • Setting CPU Overclock to Enabled on the EDEV has no effect. It operates with the same CPU clock rate as when Disabled is set.
  • When CPU Overclock is set to Enabled, the SDEV case heats up. Be aware of this during use.
  • The results of the power consumption measurements by the pcm library and the results of the CPU profiler both differ greatly depending on whether CPU Overclock is enabled. If you will be using these values as references for the production environment, set CPU Overclock to Disabled.
  • When CPU Overclock is set to Enabled, USB devices cannot be run from the system's power supply due to the limits on the power consumption for the board as a whole.

USB 3.0 SuperSpeed for HTC Connection

Only available on the EDEV.

You can increase the communication speed between the development hardware and the host PC by enabling USB 3.0 SuperSpeed communication.
Depending on the environment, enabling this feature can increase maximum communication speed by as much as 3.3 times.

Note
  • There is no benefit, however, for enabling USB 3.0 SuperSpeed on the SDEV because it uses an Ethernet connection and not USB.
  • This setting only works for the connection between the development hardware and the host PC. There is no effect on communication speeds between the development hardware and USB peripherals, where the development hardware is the USB communication host.
  • There is the potential for wireless communication conditions to worsen when using USB 3.0 SuperSpeed together with 2.4 GHz wireless communication. Only enable this feature after understanding the potential impact on system Wi-Fi communications and on communications for Joy-Con controllers and other Bluetooth devices connected to the hardware.
  • The USB charging cable (HAC-010) bundled with the EDEV does not support USB 3.0. To use this feature, prepare a separate USB charging cable that supports USB 3.0.
  • When using USB 3.0 SuperSpeed for EDEV-D, connect to the PC directly or use it in combination with RelayBox-D. Communication is unstable if you use it in combination with the RelayBox bundled with the EDEV.

The default is Disabled.

You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Error, Exception, and Crash Settings

User Exception Handler

Enables or disables the application's user exception handler. The default is Enabled.
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Red Screen at System Error

Specifies whether to jump to the error screen (a red screen) when there is a system error. The default is Disabled.
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Terminate Application on GPU Error

Use this setting to enable or disable the system-enacted forced termination of applications that cause GPU errors. The default is Enabled.
You must restart the development hardware to apply the new setting.

Select Enabled to configure the development hardware identically to retail units.

GPU Crash Report

Info

Available in NintendoSDK Firmware for NX 9.0.0-3.0 and later.

The system creates a crash report containing the GPU crash dump and closes the application when a GPU error occurs, and sends the report if possible. The default is Enabled.
You must restart the development hardware to apply the new setting.

Select Enabled to configure the development hardware identically to retail units.

Saves Movie On Exception

This feature enables and disables the automatic saving of video clips when an exception is thrown. The default is Disabled.

For more information, see Video Saving Feature for Debugging.

Select Disabled to configure the development hardware identically to retail units.

JIT Debug

This option enables and disables JIT debugging. The default is Enabled.
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Crash Dump

Info

In NX Add-On 11.1.0 and earlier, when sending crash reports is enabled, the feature to automatically generate CPU dumps when the application crashes is disabled.
In NX Add-On 11.2.0 and later, you can use both the feature to send crash reports and the feature to automatically generate CPU dumps at the same time.

CPU Crash Automatic Dump

When this option is selected, a CPU crash dump file is saved automatically when the application ends on NN_ABORT or otherwise crashes. This is enabled by default.
For more information, see Using the Dump Feature.
You must restart the development hardware to apply the new setting.

Clear this setting to configure the development hardware identically to retail units.

CPU Crash Full Dump

When this option is selected, the CPU crash dump file includes the state of user memory. This is disabled by default.
The dump takes longer, so select this option on a need-to-use basis.
You must restart the development hardware to apply the new setting.

Output All Thread Information to Log

When this option is selected, all thread information is output to the log when the application ends on NN_ABORT or otherwise crashes. This is disabled by default.

GPU Crash Automatic Dump

When this option is selected, a GPU crash dump file is automatically saved when the application generates a GPU error. This is enabled by default.
For more information, see Dump Feature During GPU Crash.
You must restart the development hardware to apply the new setting.

Clear this setting to configure the development hardware identically to retail units.

Memory Settings

Memory Mode

Emulates a memory size for the development hardware. If the configured size is larger than the actual system memory, the setting is ignored.
The following memory sizes can be set.

  • Auto (default)
  • 4 GB
  • 6 GB

You must restart the development hardware to apply the new setting.

Select 4 GB to configure the development hardware identically to retail units.
The following table shows the relationship between the setting and the memory size.


 EDEV(-D) SDEV SDEV-D ADEV
AUTO 4 GB 6 GB 8 GB 8 GB
4 GB 4 GB 4 GB 4 GB 4 GB
6 GB 4 GB 6 GB 6 GB 6 GB
Note

These are the valid settings for the SDEV, SDEV-D, and ADEV. The EDEV and EDEV-D always run with a memory size of 4 GB, regardless of the setting.

When the SDEV Memory Information Needs Repairing

Info

The red text (Needs Memory Repair) only appears for DevMenuApp (the application version of DevMenu). Nothing is displayed for DevMenu.

There is a known issue noted in an article in the Nintendo Developer Portal that SDEV systems manufactured and sold between February and March 2017 can only use 4 GB of memory. For these SDEV units, "Needs Memory Repair" id displayed in red lettering as follows if the memory information has not been repaired.

The memory information must be repaired for SDEVs that display this message. Use the tool introduced in the Nintendo Developer Portal article to make those repairs.

Note that this message does not appear if Memory Mode has been set to 4 GB. To determine whether your SDEV requires a memory information repair, set Memory Mode to either Auto or 6 GB.

Fill Memory With Non-Zero Value

For the development hardware, sets whether memory regions whose contents are undefined, like memory regions immediately after allocation, are filled with a nonzero value.
However, it does not apply to all undefined memory regions.
The contents of such memory regions are undefined for production hardware. Initialize and use memory regions as needed.

The default is Enabled. You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Graphics Settings

No Vsync Capability

Sets whether the graphics API is permitted to turn off Vsync on the development hardware. By default, this feature is set to Disabled and the graphics API cannot turn off Vsync. In retail units, the graphics API is not permitted to disable Vsync. For more information, see the description of the nvnWindowSetPresentInterval() function.

Select Disabled to configure the development hardware identically to retail units.

GPU Timeout

Configures the setting for GPU timeouts. The default is Enabled.
For more information about GPU timeouts, see Graphics (NX Add-On) > Time Limits That Can Be Applied to Each Draw Call. You must restart the development hardware to apply the new setting.

Select Enabled to configure the development hardware identically to retail units.

Graphics Firmware Memory Margin

Switches the graphics firmware memory margin setting. The default is Enabled.
For more information about the graphics firmware memory margin, see Memory Used by the Graphics Firmware in Graphics (NX Add-On).
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

File System Settings

FS Access Log Mode

Use this setting to specify whether to save the FS access log for the application. The default is Off.
For more information about using this log, see fs Library > Debugging Features > Checking the Frequency of Access.

Select Off to configure the development hardware identically to retail units.

Info

Set this before starting the application. When you restart the development hardware, this setting reverts to the default (Off).

FS Speed Emulation Mode

Use this setting to configure the FS access speed emulation feature for the application installed on the development hardware. The default is Off.
For more information about using this feature, see fs Library > Debugging Features > Checking Access Speed Dependency.

Info

Set this before starting the application. When you restart the development hardware, this setting reverts to the default (Off).

Resource Data Redirect Feature Settings

Info

Available in NintendoSDK Firmware for NX 7.0.0-4.0 and later.

You can switch the location where you access resource data (ROM) when running the application on the PC. The default is Off.
In addition to Off, you can select System Memory, SD Card, and Game Card.
You must restart the development hardware to apply the new setting.

Select Off to configure the development hardware identically to retail units.

Fill NAND Free Space

You can fill the Free Space in System Memory. Fill the free space using Fill, and free the filled space using Empty.

Mii Settings

Mii Testing Mode

Mii Testing Mode allows you to enable and disable the development support features in the Face Library. The default is Disabled.

Select Disabled to configure the development hardware identically to retail units.

Import Database

Use this setting to import a Mii database from and microSD card.

Create a directory named mii on the microSD card and put the Mii database file (DAT file) in this directory.

To use this feature, you must set Mii Testing Mode to Enabled.

Note

The exported file can only be used on the development hardware.

Export Database

Use this setting to export a Mii database to a microSD card.

The exported database file is saved to the directory named mii on the microSD card as a file named with a timestamp (yyyyMMddHHmmss.data).

To use this feature, you must set Mii Testing Mode to Enabled.

Format Database

Use this setting to format a Mii database.

To use this feature, you must set Mii Testing Mode to Enabled.

Export CharInfo

Use this setting to export a Mii database in the format of a set of nn::mii::CharInfo files.

When the database is exported, a directory named with a timestamp (yyyyMMddHHmmss) is created in the mii directory on the microSD card.
The nn::mii::CharInfo files (DATA files) are saved to this directory along with a file named nicknames.txt.
The nicknames.txt file contains the nickname of each nn::mii::CharInfo object.

To use this feature, you must set Mii Testing Mode to Enabled.

Miscellaneous Settings

SD Card Logging Mode

Use this setting to enable or disable the feature to save the log to a microSD card. The default is Disabled.
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Save

In the enabled state, the following procedure saves the log to a microSD card.

  1. Start the system with a microSD card inserted in the slot.
  2. The log is saved to the NxBinLogs/(devkit serial number)_YYYYMMDDhhmmss.nxbinlog file in the root directory of the microSD card.
  3. Remove the microSD card after you have saved a log for the necessary amount of time.
Info

If a file with the same name already exists, a number is added to the end of the filename and incremented in a range from 2 to 99 (for example, NxBinLogs/(devkit serial number)_YYYYMMDDhhmmss_(2-99).nxbinlog).
The number in the filename continues to be incremented until 99, after which no additional logs are saved.

Viewing

  1. Start Target Manager.
  2. Select File > Open log file.
  3. Select the saved nxbinlog file.

A window like the following opens.

Movie Recording Debug Mode

This feature enables and disables video capture debug mode. The default is Disabled.

For more information, see Video Saving Feature for Debugging.

Select Disabled to configure the development hardware identically to retail units.

Force Disable Game Recording

This setting forcibly disables the video capture feature. If set to On, the video capture feature is forcibly disabled. This is disabled by default.

The specified setting is applied to the next application that is started. The setting is not applied to an application that is already running, so restart that application to apply the setting to it.

Performance Monitoring Unit

Set whether to use the CPU's PMU. The default is Enabled.
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

Initial Tick Value

The initial tick value can be configured. The default is 0. In addition to 0, other available options are 7 days, 14 days, 30 days, 90 days, and <Custom>.
When <Custom> is selected, you can enter a hexadecimal value starting with 0x to set any tick value as the default.
You must restart the development hardware to apply the new setting.

Select 0 to configure the development hardware identically to retail units.

Application Extra Thread

You can configure whether to ease restrictions on the upper limit for the number of threads that an application can use. The default is Disabled.
You can ease the upper limit for the number of threads, so this scenario only applies when 6 GB of memory is available in SDEV. This option cannot be used on retail units or the EDEV.
You must restart the development hardware to apply the new setting.

Select Disabled to configure the development hardware identically to retail units.

HTC Disconnection Emulation Mode

Use this setting to enable or disable the feature for emulating the disconnection of communication with the host PC. The default is Disabled.
You must restart the development hardware to apply the new setting.
Setting this to Enabled results in the following behavior.

  • Even when Target Manager and the development hardware are connected, the behavior of the HTC and HTCS libraries and the file system operations on the host PC are the same as if they were disconnected.
  • With the exception of some features, the host PC cannot control the development hardware.
    • If you want to perform these types of operations, you must configure this setting to Disabled.
  • Debugging and log output are functional.
    • The application cannot be run from the host PC when debugging, so you must install the application ahead of time and use the procedure described in Debugging a Built Program.

Ease Nro Restriction

There are normally restrictions that the NRR file must be stored in ROM-FS in the /.nrr directory, where / is the ROM-FS root directory, and that VSI's RAW format cannot be used.
However, this restriction can be removed and the settings changed to allow the use of VSI's RAW format and the use of the host PC's NRR and NRO files.
The default value is enabled with the restriction removed.

Select Disabled to configure the development hardware identically to retail units.

Thermal Burn Prevention Mode

This feature enables and disables the feature for avoiding low-temperature burns.

When Thermal Burn Prevention Mode is set to Enabled to activate the feature for avoiding low-temperature burns, the system is forced to sleep when the all of the following conditions apply for a prolonged period of time.

  • The mode is handheld mode.
  • There are no button presses or other obvious user operations.
  • The case exceeds a specific temperature.

This feature is Disabled by default on the development hardware, except in the case of Initialization for PROD MODE, the Firmware That Approximates Retail Hardware, in which case it is Enabled by default.

Other system settings like those for idle time detection and the auto-sleep activation time do not affect the behavior of this feature.

Select Enabled to configure the development hardware identically to retail units.

Reset Pseudo Device ID

This feature resets the pseudo device ID.
After the value has been reset, applications returns a different pseudo device ID than the one they obtained before it was reset.

Temporary Profanity Word Pass-Through for the China Region

The profanity detection functionality for systems where Platform Region is China enables the feature to temporarily pass profanity through.
Use this feature for purposes like guideline support checks. The default is Disabled.

When enabled, unless the device is restarted or it is explicitly disabled, profanity is passed through.

Background Download Stress Test

Download tasks will continue to be executed in the background. The default is Disabled.
You must be connected to a network to use this feature.
This feature wears out system memory, so only enable it when necessary.

You must restart the development hardware to apply the new setting.

This feature is available when an application fails during a background download, after the application has been released to the product environment.
Use it to reproduce issues in the development environment.

HTC Generation

Info

Available in NintendoSDK Firmware for NX 10.0.0-6.0 and later.

You can change the communication protocol used with the host. The default is gen2.
For more information about changing the communication protocol with the host, see Guide to Setting Up the Environment > Using Target Manager 2.
You must restart the development hardware to apply the new setting.

Note

This feature is available only in DEV MODE. It cannot be used in PROD MODE.

FunctionAuthorization Testing Mode

Info

Available in NintendoSDK Firmware for NX 10.0.0-6.0 and later.

Note

There was a bug in versions earlier than NintendoSDK Firmware for NX 11.0.0-2.0 where the settings would reset on restart.
There are also some usage differences from versions earlier than NintendoSDK Firmware for NX 11.0.0-2.0.


  • ApplicationId
    • From the drop-down menu, select the application to restrict functionality for.
      • Selecting Off disables the entire Function Authorization Testing Mode setting.
      • The first five applications listed in the Application tab may appear in the drop-down menu.
      • If the application you are looking for does not appear in the drop-down menu, select <Custom> and enter the application ID.
  • Version
    • From the drop-down menu, select the release version to configure.
      • Release versions less than or equal to the specified version are subject to functionality restrictions.
      • In general, select Max.
      • If you patch an application and want to verify that the functionality restrictions have been lifted, select Custom and set a value smaller than the release version of the patch.
  • Function
    • From the drop-down menu, select the function to restrict and click the Add button to add it.
      • Functions listed below this function will be restricted.
