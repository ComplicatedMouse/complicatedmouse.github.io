Confirmation of Compliance with the Nintendo Switch Guidelines

The Nintendo Switch Guidelines, provided separately, define items that applications should comply with. Check that all applications are in compliance with the guidelines. If an application does not comply with all applicable items in the guidelines, the Master ROM will not be accepted.

Verify That No Development Performance Configurations Are Used

Among the performance configurations in nn::oe::PerformanceConfiguration that can be set, there are several configurations that can only be used for development purposes. Applications that use any of the development-only performance configurations will not run on retail NX systems, and as a result, you must verify that your application is not using any of these performance configurations. We recommend running your application on an EDEV that is set to PROD MODE and then verifying that the application continues to work correctly in scenes in which the performance configuration set for nn::oe::SetPerformanceConfiguration is updated.

See the following page for more information about the performance configurations available in nn::oe::PerformanceConfiguration.

  • Selecting Performance Configurations

Verify That All Check Items Are Configured Correctly During Application Startup

If executing the application through VSI or RunOnTarget, several checks will be run during application startup such as for account selection, creating save data, and Parental Controls restrictions. However, not all checks or processing will be performed. Starting the application from VSI and RunOnTarget are features for development purposes, and there are differences from the retail version.

To check whether the various metadata for the application have been correctly configured, start the application by selecting its icon in the HOME Menu. It does not matter whether the application is installed on the system or written to an NX development card.

For more information, see Writing the NMETA File.


Enabling the File System Speed Emulation Feature

The access speeds at which an application can access storage media vary according to both the type of storage media used and deterioration in performance over time. For this reason, developers must ensure that applications are implemented in a way that does not depend on storage media access speed so that users can progress in the application regardless of the current access speed. To test this capability, you must enable the speed emulation feature while testing an application. This testing procedure is also required during development of packaged versions of applications.

After installing the application to the system’s NAND memory, test the application in all of the following speed emulation modes: Slower Mode, Faster Mode, and Random Mode. The speed emulation feature can be enabled from DevMenu in Debug > FS Speed Emulation Mode. For more information about testing applications and using the speed emulation feature, see the following page.

  • Checking Access Speed Dependency
  • Development Assistance Settings
Info

The Slower Mode in the file system speed emulation feature can be used in combination with the background download stress test.

However, do not use Faster Mode or Random Mode in combination with the background download stress test. The background download stress test causes a decrease in file system access speed, which prevents the fast access speed and randomly-varying access speed modes from emulating the intended performance configurations.

In addition, the Slower Mode setting is only intended to test for serious bugs, such as when the application crashes or stops responding. Do not use it to optimize application performance for slower SD cards.

Note

This test item can be tested from the development phase. You can enable these settings in both DEV MODE and PROD MODE. However, do not enable them in the PROD MODE environment used for final application testing.

Testing Support for Combinations of Operation Modes and Performance Modes


Confirm that the application supports all combinations of operation modes and performance modes. (Excludes applications that do not support certain play styles.)
The combinations of operation modes and performance modes are shown in the following table.

Handheld Mode

Console Mode
Normal mode The combination when the system is not docked. The combination when the system is docked and there is insufficient remaining battery life.
Boost mode Does not occur The combination when the system is docked and there is sufficient remaining battery life.

In most cases, the combination is "handheld mode and normal mode" or "tabletop mode and boost mode," but "tabletop mode and normal mode" is also a possibility.
To play the application in the expected way in all combinations, you must properly implement a process that supports all of the transitions, without confusing the operation mode transitions and the performance mode transitions.

It is difficult to sustain the combination of "tabletop mode and normal mode" with normal operations. To debug this combination, use the DevMenuCommand tool's features for setting the operation mode control policy, and the performance mode control policy, as described in Development Support Features.

In NX Add-On 3.2.0 and later, these settings are now easier to configure because features to set the operation mode control policy and the performance mode control policy have been added to the DevMenu page to assist development (debugging). For more information, see Controlling the Development Hardware > DevMenuCommand.

Testing the Video Capture Feature Activated by Long-Pressing the Capture Button

Starting in NX Add-On 4.2.0 (or in 3.6.0 or later for NX Add-On 3.x), the Switch system features include a video capture feature activated by long-pressing the Capture Button.

This feature is enabled by default in applications, but it can be completely disabled using the setting in the NMETA file, and it can be temporarily disabled by using a function call.

When the application is in focus, a long press of the Capture Button activates the process of video capture.
Check that you can disable and re-enable this feature, and that the increased processing load, due to video capture, does not impair your application.

For more information about the processing load of the video capture feature, see the following page.

  • Important Implementation Information (NX Add-On)

For more information about the NMETA file settings and the function for video capture, see the following pages.

Testing With the System Tick Default Value Changed

Even though calling tick.GetInt64Value() on the tick value that was obtained returns an int64_t type immediate system tick value, do not convert this value to a float, int32_t or another small or low-precision type value for the purposes of equality or size comparisons.
System ticks continue to advance, even during sleep mode, without the Switch frequently restarting. As a result, the system tick values can become very large.

When you convert the values to types that are smaller than int64_t and lower in precision, information in the original system tick values can get lost and comparison operations can be incorrect.
As a result, the application cannot start, or excessive processing slowdowns or stalls may occur.

To avoid this issue, check the behavior using the following procedure for applications that use the nn::os::Tick class.

  1. Set the values for Initial Tick Value in Development Assistance Settings.
  2. Restart the system.
  3. Start the application and check the behavior.
  4. Step 1. Set all values except for Custom in Initial Tick Value, and perform steps 1 to 3.

The test procedure uses setting values except for Custom for the sake of convenience, but it is not a problem to check behavior using the Custom setting as necessary.
Because occurrence of this problem is dependent on the application's implementation, test the Initial Tick Value using small values through large values.
