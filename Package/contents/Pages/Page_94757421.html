Creating an Independent Build System (NX Development Hardware Environment)

Introduction

About This Page

This section is for developers who intend to create their own build system without using the VSI provided by NintendoSDK. It presents the information needed to create applications for the NX hardware environment.

Related Items

For more information about debugging applications that have been created using a proprietary build system, see Debugging a Built Program.

Notation

The following command syntax is used in the descriptions of build procedures.

  • Angle brackets (< >) indicate that you must substitute the appropriate option value.
  • Square brackets ([ ]) indicate optional items.
  • ... An ellipsis (...) indicates that multiple options are possible in the location where the ellipsis was appended.

To improve the readability of the examples of the command-line options, line breaks have been added in places. However, the actual commands do not include these line breaks.

Important Information

This section contains information that is highly likely to change in future NintendoSDK releases. Whenever there is a new NintendoSDK release, check the change log and confirm whether related items are present.

Build Tools

Compiler tool chains

The compiler tool chains for the NX development hardware environment are installed in the following directory.

  • %NINTENDO_SDK_ROOT%\Compilers\NX

There are several tool chains. Use one of the following target-specific tool chains to suit the application you will create.


 Target Name
32-bit applications armv7l-nintendo-nx-eabihf
64-bit applications aarch64-nintendo-nx-elf

Other Tools

The other build tools (the tools other than these compiler tool chains) are installed in the following directory.

  • %NINTENDO_SDK_ROOT%\Tools

Build Procedures

Overview

The following describes the overall build procedure, depicting the sequence from the source files to the creation of the final image of the application.

  1. Creating the Object File
  2. Creating a Static Library
  3. Creating a Dynamic Module
  4. Creating an Application Program
  5. Creating an Application Image

Creating a static library and creating a dynamic module are not required for creating application images.

The legend for the diagrams used in the descriptions of the build procedures is as follows.

Blue square: input file in the procedure. The developer must create this file.

Orange square: output file in the procedure. This file is required when executing other build procedures and executing the application.

Green square with hollow center: intermediate output in the procedure.

Large green square: indicates a build tool.

Creating the Object File

Compile the source file and create the object file.

For more information about the procedure, see Build Procedure: Creating the Object File.

Creating a Static Library

Archive object files and other static library files and create a new static library.

The developer can omit this procedure if the static library does not need to be created.

For more information about the procedure, see Build Procedure: Creating a Static Library.

Creating a Dynamic Module

Create dynamic modules (common libraries) from object files and static library files that can be used by the NintendoSDK ro library.

The developer can omit this procedure if the ro library feature is not required.

For more information about the procedure, see Build Procedure: Creating a Dynamic Module.

Creating an Application Program

Link object files, static library files, and the like and create an application program.

For more information about the procedure, see Build Procedure: Creating an Application Program.

Creating Application Images

Create an application image that can run on the NX development hardware from the application program, the dynamic module files, and so on. By using VSI, you can create the image as an NSP file and in a RAW format.

For more information about the procedure, see Build Procedure: Creating an Application Image.

Compiler and Linker Options

It is assumed that several compiler options and linker options must either be specified or not specified for NintendoSDK. Not following the rules for specifying these options causes build errors and results in applications that do not run correctly. The rules for specifying options are listed on the pages describing the detailed procedures of the build flow.

Compiler options and linker options may change in future releases of the NintendoSDK. If you want to implement features that are dependent on certain compilation or linker options, consult with NintendoSDK support.
