Application Development

This tutorial describes the process of NX application development.

By changing the platform that is the build target for your NX application, you can use basically the same code to debug and run the application in both the Windows environment and on the NX development hardware.

Development on the Windows Platform

Application Development

The code for the application is written using the NX API set.
For more information about the available features and APIs, see the section that describes the features and the API Reference.

Info

Use the following procedure to browse the API Reference from the Visual Studio editor via the Microsoft Help Viewer.

In Visual Studio, select Help > Set Help Preference > Launch in Help Viewer.

In the editor, rest the mouse pointer on the API you want to look up.

Press the F1 key. The Microsoft Help Viewer starts and displays the reference documentation for that API.

Note

Unless you install the Help Viewer, the feature is not available in Visual Studio 2017.
If not already installed, you can run the InstallVsHelp.ps1 script file, included in the SDK, to install the Help Viewer together with the SDK Help.

Building and Debugging

To build and debug on the Windows platform, select a solution structure (Debug, Develop, or Release) and set the solution platform to Win32 (for a 32-bit application) or x64 (for a 64-bit application).
For more information about the differences between solution structures, see Build Targets and Build Types.

For debugging, you can use the features of the standard Visual Studio debugger.

Running your application

Run the exported EXE file to run your application.

Development on the NX Platform

Application Development

For the most part, the code written for the Windows platform can be used without modification.
However, some APIs may work differently on the Windows platform and the NX platform. For more information, see the feature descriptions in the NintendoSDK documentation and the API Reference.

Preparing the Application.desc File

The Application.desc file describes the scope of permission for the features that a regular application can use.

You must specify and use the following files, which are provided by the various versions of NintendoSDK used to create applications.

Type of Application

 DESC File
32-bit applications %NINTENDO_SDK_ROOT%\Resources\SpecFiles\Application.desc
64-bit applications

Performing the following operations prevents the application from running on the development hardware, and creates unstable operations.

Never perform any of these operations.

  • Modify the content of the Application.desc file.
  • Copy and use a Application.desc file from another version of NintendoSDK.

DESC files can be used when checking a submission, so reference them directly from the package instead of separately copying or renaming each package file.

Preparing the NMETA File

The NMETA file describes the parameters that can be set separately in each application.

An NMETA file will be generated automatically in the same directory as the project file for new projects created using the wizard in Visual Studio with NX Add-On 1.0.0 and later.

When using a version earlier than NX Add-On 1.0.0 or not using Visual Studio, copy and use the following NMETA file as a template for each application.

Type of Application Metadata Source
64-bit applications %NINTENDO_SDK_ROOT%\Resources\SpecFiles\Application.aarch64.lp64.nmeta
32-bit applications %NINTENDO_SDK_ROOT%\Resources\SpecFiles\Application.arm.ilp32.nmeta

Configure the NMETA file as appropriate for each application. See Writing the NMETA File when editing the files directly, and AuthoringEditor when using editing tools to edit the files.

Info

When using NX-VSI, by default, it is set to use the appropriate DESC and NMETA files to perform the build.
You can change the specified paths for each of these files from the DESC file and NMETA file options under Project Configuration Properties > General.

Info

The extension was changed from .meta to .nmeta, but .meta files can still be used.


Building and Debugging

Info

To run an application that has been built on the NX platform, you must first connect the development hardware to the PC and set it up.
For more information about the procedure, see Initializing the SDEV or Initializing the EDEV.

To build and debug on the NX platform, select a solution structure (Debug, Develop, or Release) and set the solution platform to NX32 (for a 32-bit application) or NX64 (for a 64-bit application).
The solution structures differ in the same ways they do for the Windows platform. For more information, see Build Targets and Build Types.

Although you can use the same debugging features for development on the NX platform that you can for the Windows platform, there are some restrictions. For more information, see VSI > Manual.

If no particular settings are configured, the program is run on the development hardware that is specified as the default target by Target Manager.
If you want to run the program on some target other than the default, right-click the toolbar to open the NX Targets menu and select the serial number of the development hardware you want to use.

Info

You can also use distributed builds. You can now also use the free distributed build tool FASTBuild.
By applying the solution file to the converter tool sln2bff, you can generate a BFF file that can be built in FASTBuild.

You must install the FASTBuildUtilizeKit package to use this feature.


In addition, you can change the program execution format as described later by selecting Project Configuration Properties > General > Application Program Format
For more information about the parameters unique to the NX platform that can be set in the configuration properties, see VSI > Project Parameters.

Selecting Raw for Program Application Format (Default)

This option provides the minimum executable build for the NV development hardware. The files are output to an nspd-format directory and can be directly accessed for debugging.
The NCA file or NSP file can be created directly from the code directory (or from the data directory) in this output nspd directory. For more information, see AuthoringTool.
The build time is short because you are only building the executable code. Resources are loaded directly from the PC at run time.
Use this method for normal development.

Selecting NSP for the Program Application Format

An NSP file is created, which is the installation and submission format for applications. This file is used for Installing and Running Applications on the Development Hardware or Writing to NX Development Cards.
The build time is longer than for raw mode because the resources and icon data are also included.

Running your application

Use RunOnTarget to run the exported NSPD directory or NSP file on the development hardware.
You can use both NSP and NSPD_ROOT files either from the Load executable file in Target Manager or by dragging those files to Target Manager.

Use RunOnTarget to dump the log or to run using CLI.

Differentiating Between Formats and Startup Methods

The following is a collection of the different types of formats and startup methods.

Format

Format Features
Raw (NSPD) Faster because some resources are not included.
NSP The resources also include information such as icon data.
This format is required when installing to the console system memory or when writing to a card.
