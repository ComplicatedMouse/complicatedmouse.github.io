Changes to Graphics for Developers

This section describes the changes made to NintendoWare Graphics for developers in NintendoSDK 0.16.0.
“NintendoWare Graphics” refers to the nn::g3d, nn::ui2d, nn::font, and nn::vfx modules.

nn::ui2d and nn::font News

Support for ASTC Texture Formats

Added support for ASTC texture formats. For more information, see ASTC Textures.

Support for Loading Resources Into External Memory Pools

Resources can now be loaded into external memory pools.

This can reduce the number of MemoryPool instances to create and lower processing costs at initialization time.

For more information, see Reducing the Number of Memory Pools by Consolidating MemoryPool Resources in MemoryPool.

Support for Bitmap Fonts and Font Pairs in Composite Fonts (FCPX)

Composite fonts (FCPX) can now handle bitmap fonts and font pairs.

In relation to this, the method used by programs to load composite fonts has been changed.

For more information, see Using Composite Fonts in Layout Programs.

Support for Output of float Color

Monochrome color in each pane now supports output of float color.

Setting the color buffer used to render layout data to the appropriate format, makes it possible for shaders to handle color values greater than 1.0.

This feature has been used to implement glare effects in the LayoutEditor viewer.

For more information, see Using float Color Output.

nn::vfx News

Support for Separate Output of Texture Files

There is now support for separately outputting the texture files (BNTX) when the effect binary is converted.

Related Links

Effect Binary Describes the initialization of the separated textures in the runtime and their binding to the effect binary.

Fixed a Bug Where Effects That Used Particle Shapes Did Not Render Correctly

Fixed a bug where effects did not render correctly when FMDB files with particle shapes were specified.
There was an issue with rendering when there were multiple vtx_attrib_array elements in the FMDB file.

Despite this issue being fixed, FMDB files that are subject to the following restrictions do not render correctly, even in NintendoSDK 0.11.0.

Restrictions on Usable FMDB files

There are restrictions on the settings that can be configured for usable FMDB files.

  • There is a restriction regarding the name attribute of the vtx_attrib element. The usable attributes are limited to _p0, _n0, _t0, _u0, _u1, and _c0.
  • The quantize_type attribute of _u0 and _u1 must be the same.

If you load an FMDB file that clashes with a restriction, a warning will be displayed in the UI in EffectMaker.

If the name attribute is different

If _u0 and _u1 have different quantize_type

Restriction Workaround

Use IntermediateFileBinaryToAscii.exe to convert the unreadable FMDB file into an FMDA file, rewrite the parts that clash with the restrictions, and then use IntermediateFileAsciiToBinary.exe to convert it back into an FMDB file.

We plan to release a script for rewriting the FMDB file in NintendoSDK 1.0.0.

def FuncCompare(x, y):
    #--------------------------------
    if x == "float_32_32":
        return "float_32_32"
    if y == "float_32_32":
        return "float_32_32"
    if x == "float_16_16":
        return "float_16_16"
    if y == "float_16_16":
        return "float_16_16"
    #--------------------------------
    if x == "snorm_16_16":
        return "snorm_16_16"
    if y == "snorm_16_16":
        return "snorm_16_16"
    if x == "snorm_8_8":
        return "snorm_8_8"
    if y == "snorm_8_8":
        return "snorm_8_8"
    #--------------------------------
    if x == "unorm_16_16":
        return "unorm_16_16"
    if y == "unorm_16_16":
        return "unorm_16_16"
    if x == "unorm_8_8":
        return "unorm_8_8"
    if y == "unorm_8_8":
        return "unorm_8_8"
    return x

nn::g3d News

Added a Function for Determining Whether Submesh Boundaries Are Preserved Between LOD Levels

Added the nn::g3d::ResShape::IsSubMeshBoundaryConsistent() function, which determines whether submesh boundaries are preserved between LOD levels. Submesh boundaries are preserved when LOD models are created using the polygon reduction feature of the intermediate file optimizer. However, they are not preserved when LOD models created using other external tools are combined by the intermediate file optimizer.
