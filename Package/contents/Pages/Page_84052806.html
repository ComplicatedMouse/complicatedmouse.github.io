Overview

Information used at run time and by tools can be controlled using command-style annotation when using the g3d shader system. You can edit shaders using any shader authoring tool as long as it is annotated.

The 3D shader converter (3dShaderConverter) creates a shader definition (FSDA or FSDB) file from shader source and shader configuration (FSCA and FSCB) files that include shader annotations.

General

In the following sections, each annotation type is described in detail. For the individual values that can be taken, see the description of supported elements in the shader definition file for Intermediate File Specifications. The shader source used in the demos also use annotations, so they also make a good reference.

Notation Used in This Document

Shader annotations are made using C-style or C++-style comments that start with @@.

For C-style comments, a single annotation is identified as from id to the next id or to the end of the comment. By including the group name and a colon at the beginning of a C-style comment, a group can be specified in the entire C-style comment. The C-style comment format cannot be used for annotations after variables.

Comments must start from the beginning of the line when making annotations using a remote specification.

#define OPTION // @@ id="option" choice="bool"
 
in vec3 pos; // @@ id="position"
uniform sampler2D albedoTex; // @@ id="albedo"
 
layout(std140) uniform Material // @@ id="mat" type="material"
{
    vec4 texsrt[2]; // @@ id="texsrt" type="texsrt"
};
 
// @@ option_id="option" label="Option"
 
/* @@ groupA:
    attrib_id="position" label="Position"
    sampler_id="albedoTex" label="Albedo Texture"
*/


Use the equal sign (=) to assign annotation values to annotations. Do not include spaces or other characters before or after the equal sign.

// OK
#define OPTION // @@ id="option" choice="bool"


// NG
#define OPTION // @@ id = "option" choice ="bool"

id

When using the g3d shader system, symbols are accessed using a special identifier, not using the variable name inside the shader as with OpenGL. The string used for this identifier is called an ID. Variable names are governed by coding conventions and may change depending on how a shader program is being implemented. Using an ID, on the other hand, enables you to minimize the effect of such changes.

These IDs are not only used to get shader symbols from a program but also in assigning models and shaders. If the shader source includes an annotation on a line separate from a variable, this ID is used to identify the corresponding variable. For more information, see Remote Specification of Annotations.

hint

You can use hint information to automate initial assignments of vertex attributes and sampler when assigning a shader to a model when using a tool like 3DEditor. The vertex attributes and samplers within a model that have matching hint information can be assigned as the shader's vertex attributes and samplers when assigning a shader to a model by providing hint information for that shader ahead of time. You can also change assignments manually after automatic assignment.

Remote Specification of Annotations

Excluding the case of being in a block, you can provide an annotation at a point in the code other than immediately after a variable. First, specify an ID immediately after the variable. Then specify that ID in the remotely specified annotation. The following ids can be used. The comment must start at the beginning. Make sure that you include *_id at the beginning.

Name Supported Elements in the Shader Definition File Handling When Omitted Description
option_id <option_var> Cannot be omitted. Represents the option variable ID.
attrib_id <attrib_var> Cannot be omitted. Array of vertex attribute variables.
sampler_id <sampler_var> Cannot be omitted. Sampler variable identifier.
block_id <block_var> Cannot be omitted. The identifier for uniform blocks.
uniform_id <uniform_var> Cannot be omitted. The identifier for uniform variables. 
layout(location = 0)  in vec4   i_Position;     // @@ id="_p0"   hint="position0"

/* @@
    attrib_id="_p0"   label="Position Coordinates" editable="false"
*/

Defining Identifiers With the Same Name

This section describes the behavior when multiple annotations are defined with the same identifier name.

With variables supported by the following remote operations, elements defined by remote operation annotations are overwritten based on the coding of the annotations for the variable declarations.
If multiple remote operations have been coded for the same variable, the one coded last in the source code is overwritten.

  • Option Variables
  • Vertex Attribute Variables
  • Sampler Variables
  • Uniform Block Variables
  • Uniform Variables

The content of other annotations written later in the source code are overwritten.

For example, definitions in the following example are merged this way: renderinfo="ice" type="int" default="4" choice="[0,1]" label="Receive Ice”.

// @@ renderinfo="ice" type="int" default="0" label="Receive Ice"
...
// @@ renderinfo="ice" default="4" choice="[0,1]"

choice Annotation

There are two ways to make choice annotation for option variables and rendering information: with a range or with a list.

To use a range, specify a minimum value and a maximum value, like choice="[0, 4]". To use a list, enumerate a string, like choice="add, sub, mul, div". When creating this string you can specify the individual elements at the UI level, for example, by coding as choice="add:addition, sub:substration".

// @@ renderinfo="culling" type="int" default="0" choice="0:Display Front Face,1:Display Back Face,2:Display Both Faces,3:Do Not Display" order="1" label="Culling Mode"

Option Variables

An option variable is a pre-processor variable that serves as a key to generating shader variations. When controlling shader variations using a shader configuration file (FSC file), the shader variation to use is identified based on the ID specified for the option variable.

The following annotations can be used in option variables.


Name Supported Elements in the Shader Definition File Handling When Omitted Description
enable
true Value of whether the option variable is enabled. When false, not extracted to the shader definition file.
id <option_var> Cannot be omitted. Represents the option variable ID.
type <option_var> static The option variable variation type. Either static or dynamic.
choice <option_var> bool

The range of values for the option variable.

For boolean values, specify bool. For integer values, specify a range such as [0, 4]. For enumerated values, specify ordered pairs such as add:addition, sub:subtraction.
default <option_var> First candidate in choice. Represents the option variable default value.
label <ui_label> Same as id. Label for the UI with a variable to edit.
Comment <ui_comment> Not specified. Comment for the UI with a variable to edit.
group <ui_group> Not specified. The type of the UI group for editing a variable.
order <ui_order> 0 Order priority of the UI with a variable to edit. The lowest number has highest priority.
visible <ui_visible> true Whether to display the UI to edit the variable. 
#define SKINNING                   ( 0 )       // @@ id="skinning" choice="[0, 9]" type="dynamic"

default_max <ui_default_max> Not specified. The default maximum value for the UI for editing a variable.

type specifies the type for a parameter that the library writes, but the parameter can also take values specified as a completely different type on GLSL. As long as the memory layout matches, a single parameter can even take the value of more than one variable or an array. For more information about the memory layout in uniform blocks, see the OpenGL specifications.

If a matrix type is specified for type, then row-major vectors including padding for each line are written by the library. For example, a parameter for which type is float2x3 can take GLSL mat3x2 data.

If the type is srt3d, the library writes a row-major 3×4 matrix. Specifically, the library consecutively writes m00, m01, m02, m03, m10, m11, m12, m13, m20, m21, m22, and m23. When using GLSL, you can retrieve values as mat4x3 and vec4 arrays.

If the type is srt2d or texsrt, the library writes a row-major 2 × 3 matrix. Specifically, the library consecutively writes m00, m10, m01, m11, m02, and m12. When using GLSL, you can retrieve values by coding like the following texsrt0 and texsrt1 examples. Note that if you code using an array of vec2 objects (like with texsrt2), you cannot retrieve the values because padding is inserted between the vec2 objects.

layout(std140) uniform Material // @@ id="mat" type="material"
{
    vec4 texsrt0[2];        // @@ id="texsrt0" type="texsrt"
    vec2 texsrt1_column0;   // @@ id="texsrt1" type="texsrt"
    vec2 texsrt1_column1;
    vec2 texsrt1_column2;
 
    // Cannot retrieve values if you code as an array of vec2 objects, because padding is inserted between the vec2 objects.
    //vec2 texsrt2[3];      // @@ id="texsrt2" type="texsrt"
 
    vec4 texsrt_ex0[3];     // @@ id="texsrt_ex" type="texsrt" converter="texsrt_ex"
};

Shader parameters of type texsrt can be accessed by the CPU using nn::g3d::TexSrt. If default is specified, make sure to specify the members of nn::g3d::TexSrt in order. mode is an integer. The rotation component must be specified in degrees.

struct TexSrt
{
    enum Mode
    {
        MODE_MAYA,
        MODE_3DSMAX,
        MODE_SOFTIMAGE,
        NUM_MODE
    };
 
    Mode mode;
    float sx, sy, r, tx, ty;
};

If the type is float3 or float4, you can use the color controls in 3DEditor. For more information, see Switching Between Color Control and Numeric Control.

Customizing Uniform Variables

You can customize uniform variables by using the depend and converter annotations.

The converter annotation is an identifier that tells the library to use a custom algorithm when converting (calculating) uniform variables. Use the nn::g3d::ResUniformVar::GetConverter() function to get the specified identifier. This identifier specifies which callback functions are added to the library by the application when the library is initialized. Use the nn::g3d::ResShaderParam::SetConvertShaderParamCallback() function to register the callback.

The depend annotation allows you to specify other uniform variables or references to external data when using the custom conversion algorithm specified by converter. To use other uniform variables, specify their IDs. To reference external data, specify the uniform variable's own ID.

layout(std140) uniform Material  // @@ id="mat" type="material"
{
     vec4 color;          // @@ id="color" depend="intensity" converter="mul_intensity"
     float intensity;     // @@ id="intensity"
};

Rendering Information

Rendering information represents material parameters that cannot be directly sent to the GPU. You can freely define data such as rendering priority or flags indicating whether to accept polygon offsets and shadows during shadow rendering.

The following annotations can be used in the rendering information. The comment must start at the beginning.

Name Supported Elements in the Shader Definition File Handling When Omitted Description
renderinfo <render_info_slot> Cannot be omitted. Rendering Information identifier.
type <render_info_slot> string Specify int, float, or string.
choice <render_info_slot> Empty string.

The maximum value allowed for rendering information. For numbers, this range is defined in the form [min, max]. For strings, it is defined in the form [add:addition, sub:subtraction].

You can dynamically specify the rendering information choice from the application when connected to 3DEditor if no choice has been specified. For more information, see Embedding Rendering Information.
default <render_info_slot> Empty string. The default value for rendering information. Selections are listed delimited by spaces. When there are fewer than count selections, the remainder are blank.
count <render_info_slot> 1 The maximum value allowed for rendering information.
optional <render_info_slot> true Boolean value that determines whether there can be a variable number (an arbitrary number) of rendering information values. If false, the number is fixed to the value set for count. Specify false if it is a string and you do not want to add a selection for a blank space to the 3DEditor UI.
label <ui_label> Same as renderinfo. Label for the UI with a variable to edit.
Comment <ui_comment> Not specified. Comment for the UI with a variable to edit.
group <ui_group> Not specified. The type of the UI group for editing a variable.
order <ui_order> 0 Order priority of the UI with a variable to edit. The lowest number has highest priority.
item <ui_item> auto The type of the UI for editing a variable. If type is a number or string, specify check to change the 3DEditor UI to a check box. If a string, 2 enumerated values must be specified for choice. In this case, false and true check box values are assigned to the enumerated values in the order in which they are defined. For example, if choice="A, B" is specified, A will be false and B will be true. 
// @@ renderinfo="clear_color" type="float" count="3" default="0.5 0.5 0.5" choice="[0, 1]" label="Clear Color" optional="false"

Vertex Interleave

You can specify which vertex attributes to interleave in the same vertex buffer by specifying vertex interleave. You cannot specify the order of vertex attributes in the vertex buffer, because binary converters are ordered according to the quantization format.

The following annotations can be used in the vertex interleave. The comment must start at the beginning.

Name Supported Elements in the Shader Definition File Handling When Omitted Description
interleave <interleave> Cannot be omitted. Vertex Interleave information. The id values of the vertex attributes to interleave are specified, separated by spaces. When an empty string is specified, the remaining vertices for which interleave is not specified are shown.
binarize <interleave> true When false is specified, no binary is made. 
layout(location = 0)  in vec4   i_Position;     // @@ id="_p0"   hint="position0"
layout(location = 1)  in ivec4  i_Index0;       // @@ id="_i0"   hint="blendindex0"
layout(location = 2)  in ivec4  i_Index1;       // @@ id="_i1"   hint="blendindex1"
layout(location = 3)  in vec4   i_Weight0;      // @@ id="_w0"   hint="blendweight0"
layout(location = 4)  in vec4   i_Weight1;      // @@ id="_w1"   hint="blendweight1"

/* @@
    attrib_id="_p0"   label="Position Coordinates" editable="false"
    attrib_id="_i0"   label="Blend Indices 1 through 4"     editable="false"
    attrib_id="_i1"   label="Blend Indices 5 through 8"     editable="false"
    attrib_id="_w0"   label="Blend Weights 1 through 4"     editable="false"
    attrib_id="_w1"   label="Blend Weights 5 through 8"     editable="false"
    interleave="_p0 _i0 _i1 _w0 _w1"
*/
// @@ interleave="" binarize="false"

Group

As the name implies, groups define sets of shader parameters. Before you can assign shader parameters to a group, you must define the group. Grouped parameters are displayed as a group in 3DEditor.

The following annotations can be used in a group. The comment must start at the beginning. It cannot be defined in a C-style comment.

The same ID cannot be defined in multiple groups (that is, two or more).

Name Supported Elements in the Shader Definition File Handling When Omitted Description
group <group> Cannot be omitted. When defined as a group id, this is the group name. When defined as an id attribute, this is the name of the parent group to which the group belongs. If the parent group name is specified, the group name on 3DEditor is displayed as a subgroup of the specified parent group.
page <group> Empty string. Page to which the group belongs.
condition <group> Not specified.

You can control whether the group is displayed by using the formula evaluation results. For the formula, you can use alphanumeric characters, various operators (the basic arithmetic operations, bitwise operations, and logical operations), and three types of variables: option, uniform, and renderinfo. To describe a variable, separate the type name and identifier with a period (.) and add a dollar sign symbol ($) as a prefix, as in $renderinfo.mode. If the type name is omitted, the variable is treated as having the option identifier. In addition, true and false can be used as special characters. true is evaluated as 1 and false as 0.

The following restrictions apply to using variables. (1) No support for uniform beyond 2D parameters. (2) No support for more than two sets of rendering information (count > 2) or for variable length renderinfo (optional = true).
label <ui_label> Same as group. UI label. If an empty string is specified, the frame for the corresponding group in 3DEditor is hidden.
Specify an empty string when you want to insert a line break at an arbitrary position within a list of displayed parameters, for example.
Comment <ui_comment> Not specified. UI comment.
order <ui_order> 0 UI order priority. The lowest number has highest priority.

You can use condition to hide parameters that are not used with a particular variation. In the following example, if the option variable corresponding to id="normal_map" is false, parameters that are not used in condition in the group corresponding to group="normal_map_group" are hidden.

#define USE_NORMAL_MAP      ( 0 )       // @@ id="normal_map"
 
// @@ group="normal_map_group" label="Normal Map" order="5" condition="$option.normal_map"
/* @@ normal_map_group:
     option_id="normal_map"          order="1"   label="Normal Map"
     sampler_id="_n0"                order="2"   label="Normal Map"
     uniform_id="texsrt2"            order="3"   label="SRT for Normal Map"
*/

For more information about displaying groups in 3DEditor, see Material Property Window: Shader Tab.

Revision

You can embed revision information in a shader to track which shader revision a model is associated with. You can check the revision and change the intermediate file filter accordingly when the meaning of shader parameters changes because of a shader update.

The following annotations can be used in a group. The comment must start at the beginning.

Name Supported Elements in the Shader Definition File Handling When Omitted Description
revision <revision> Cannot be omitted. Revision information. Specifies the user-defined revision number.


Label Attributes

If you use lists for label, comment, and textblock text and choice annotation, there are label attributes for customizing the display in 3DEditor. You can control the display by appending attributes to the end of the string, as shown in the following examples. You can specify one or multiple attributes.
Label colors and label references are only enabled for label and choice lists.

// @@ uniform_id="diffuse" label="ディフューズ:color{red}:background-color{black}:lang-en{Diffuse}" comment="ディフューズカラー:lang-en{Diffuse Color}"

// @@ id="albedo_color" choice="0::ref{group/texture0}, 1::ref{option_var/const_color0}"


// @@ textblock="dev_group_comment" text="The parameters in this group are for development only. Do not touch this. :lang-en{Developer only}"

Change the color of labels when displayed in 3DEditor.
(Only enabled for label and choice lists.)

Label Reference

Reference another label included in the same shading model. Use this to display the contents of another label instead, or to add a button to the right side of the label view in 3DEditor so you can move to the page with the referenced label. The following example shows how to specify a reference target.
(Only enabled for label and choice lists.)

:ref{Type/The id or name for the type}

For the reference target type, you can specify the following elements that are in the shader definition file.

<option_var>
<attrib_var>
<sampler_var>
<uniform_var>
<render_info_slot>
<group>
<page>

If you omit the string in the label or choice list and instead code :ref{}, the content of the reference target label specified by :ref{} displays in 3DEditor. For example, if you code label=":ref{group/texture0}" in the shader annotation, the content of the <ui_label> element of <group name="texture0"> is displayed.

Restrictions

Because extraction of annotations is performed on the shader source after preprocessing, annotations deleted with the preprocessing are not extracted. The following code fails in extracting mat or texsrt annotations. Be careful that annotations are not accidentally deleted by variations.

#define OPTION // @@ id="option" choice="bool"
#ifndef OPTION
uniform Material // @@ id="mat" type="material"
{
    vec4 texsrt[2]; // @@ id="texsrt" type="texsrt"
};
#endif
