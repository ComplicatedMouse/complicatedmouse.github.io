Information Common to Both

Important Information About Using the Web Applet and Offline Web Applet

Effect of System Updates

System updates may potentially include updates for the web applet and Offline Web Applet. Because of the web browser design, these updates can change both the display and the behavior of webpages.

This document describes the specification and behavior changes caused by different firmware versions from the previous web applet and Offline Web Applet. For more information about the specifications of the web browser included in the web applet and Offline Web Applet, see HTML/CSS/ECMAScript Support List.

Use the web applet and Offline Web Applet with the understanding that they might be affected by system updates in a similar manner in the future.

Effect of Application Processes on Performance

The web applet and Offline Web Applet run on the same CPU that the application that calls them is using. If the application continues to run processes using threads or the like while the web applet and Offline Web Applet are running, the performance of the web applet and Offline Web Applet may be reduced.

Exit Reason Values

Future changes to the web applet and Offline Web Applet specifications may potentially cause the following functions to return exit reasons that are not currently defined. Make sure that your application can continue to run and will not end unexpectedly even if an undefined exit reason is returned.

Applet Function
web applet nn::web::WebPageReturnValue::GetWebExitReason()
Offline Web Applet nn::web::OfflineHtmlPageReturnValue::GetOfflineExitReason()

Disabling the Screen Burn-In Reduction Feature During Video Playback

The web applet and Offline Web Applet include a video player and can play some videos included on websites and offline HTML pages. During such video playback, the Screen Burn-In Reduction feature is disabled.

Enabling JavaScript Extensions

Various JavaScript extensions can be used on webpages displayed in the web applet and Offline Web Applet. Use the following functions to enable use of JavaScript extensions on webpages.

Applet Function
web applet nn::web::ShowWebPageArg::SetJsExtensionEnabled()
Offline Web Applet nn::web::ShowOfflineHtmlPageArg::SetJsExtensionEnabled()

In the web applet, the following conditions must also be satisfied.

  • Connected to the web server via an SSL/TLS connection
  • Displaying a webpage on the same domain as the URL specified when the applet was started

When JavaScript extensions are enabled, a property named nx is added to the window object. As a result, no objects named window.nx can be defined on the webpage.

Passing Parameters From Applications

Webpages displayed in the web applet and Offline Web Applet can include information from applications. Displayed webpages can also be changed based on information from applications by changing the URL (for the web applet) or by changing the path of the offline HTML page (for the Offline Web Applet). In addition, using URL parameters in the URL and path, you can add information from the application and apply it in the same webpage.

The following sample code shows how to pass a player’s positional information to the Offline Web Applet using URL parameters and then display that position on a map on an offline HTML page.

char path[nn::web::ShowOfflineHtmlPageArg::DocumentPathMaxLength];
snprintf(path, sizeof(path), "map.htdocs/index.html?x=%d&y=%d", x, y);

nn::web::ShowOfflineHtmlPageArg showOfflineHtmlPageArg(path);
nn::web::OfflineHtmlPageReturnValue offlineHtmlPageReturnValue;
nn::Result result = nn::web::ShowOfflineHtmlPage(&offlineHtmlPageReturnValue, showOfflineHtmlPageArg);

For usage examples, see the WebDemoQuickHelp sample and the WebDemoRaceResult sample.

Restoring the Applets When Closed by the System

While a webpage is being displayed using an instance of the web applet or Offline Web Applet called from an application, if the user presses the HOME Button and then starts a system application from the HOME Menu, the system will close the instance of the web applet or Offline Web Applet called from the application. Applications can detect this and then call the web applet or Offline Web Applet again to restore them.

Note

The state of the webpage that was displayed is not saved, so the applets cannot be completely restored.

The following describes an example of the system closing an instance of the web applet or Offline Web Applet called from an application.

  1. The web applet or Offline Web Applet is called from the application to display a webpage.
  2. The user presses the HOME Button and transitions to the HOME Menu.
  3. The user starts the My Page applet (a system application).
    1. At this point, the system closes the instance of the web applet or Offline Web Applet called from the application.
  4. The user returns to the HOME Menu.
  5. The user selects the application that is currently being played.
  6. In this example, the instance of the web applet or Offline Web Applet called from the application has been closed.
  7. The application detects that the system has closed the web applet or Offline Web Applet and can restore the applet by calling it again.

The following sample shows how to restore the Offline Web Applet after it has been closed by the system.

nn::Result result;
nn::web::OfflineHtmlPageReturnValue offlineHtmlPageReturnValue;
do
{
    nn::web::ShowOfflineHtmlPageArg showOfflineHtmlPageArg("htmlcontents0.htdocs/index.html");
    result = nn::web::ShowOfflineHtmlPage(&offlineHtmlPageReturnValue, showOfflineHtmlPageArg);
} while ( result.IsSuccess() && offlineHtmlPageReturnValue.GetOfflineExitReason() == nn::web::OfflineExitReason_ExitMessage )

For a usage example, see the WebDemoRaceResult sample.

Using an Application Screenshot in the Background

You can apply a screenshot of the application to the background of the webpage, make the webpage transparent, and display the screenshot layered on top of it. The applied screenshot can also be blurred. This feature can be used to display a game application’s pause screen as an offline HTML page in the Offline Web Applet, for example.

The following sample code shows how to do this.

nn::web::ShowOfflineHtmlPageArg showOfflineHtmlPageArg("htmlcontents0.htdocs/index.html");
showOfflineHtmlPageArg.SetBackgroundKind(nn::web::OfflineBackgroundKind_ApplicationCaptureBlur);
nn::web::OfflinePageReturnValue offlinePageReturnValue;
nn::Result result = nn::web::ShowOfflineHtmlPage(&offlinePageReturnValue, showOfflineHtmlPageArg);

For usage examples, see the WebDemoPauseMenu sample and the WebDemoQuickHelp sample.

Customizing the Spatial Navigation Focus Frame

Performing a spatial navigation operation on a webpage will display the standard focus frame. Instead of using the standard focus frame, you can use CSS to customize focus effects on the webpage to match the design of the application.

Place the following <meta> element in the webpage to hide the standard focus frame.

<meta name="focus-ring-visibility" content="hidden">

The following CSS sample code shows how to display a custom focus frame on button elements.

button:focus {
  outline: solid 4px rgb(0, 128, 255);
}

There are also many other methods of expression. For more information, see the WebDemoSpatialNavigation sample.

The standard focus frame is rendered at the front of the webpage. As a result, except when using full screen, the focus frame is not hidden even if the in-focus element is hidden by a different element. By using an independent focus frame instead of the standard focus frame, you can avoid this issue and correctly hide the frame. If the in-focus element is hidden by an element that is displayed in the full-screen state, the standard focus frame is not displayed.

The footer can be displayed and hidden as necessary as the user selects links or scrolls on the webpage.

There are several features for controlling the footer display state.

  • In the Offline Web Applet, the footer is always hidden.
  • Changing the method for controlling display of the footer.
  • Temporarily locking the footer display state with a JavaScript extension.

Keeping the Footer Hidden

In the Offline Web Applet, you can use the nn::web::ShowOfflineHtmlPageArg::SetFooterEnabled() function when starting the applet to keep the footer hidden so that offline HTML pages can be displayed full-screen. This feature makes it possible to make offline HTML pages displayed in the Offline Web Applet appear to be part of the application. In the web applet, there is no way to keep the footer hidden.

For a usage example, see the WebDemoInstructionManual sample.

Changing the Webpage Style Depending on the Operating Mode

To handle switching webpage operating modes, use the CSS -webkit-nintendo-switch-device-mode media attribute. Use when you want to change the style of a webpage dynamically according to the operating mode. For more information about the operating modes, see Operating Modes.

The following values can be specified for the -webkit-nintendo-switch-device-mode media attribute.

Value Operating Modes
-webkit-nintendo-switch-device-handheld Handheld mode
-webkit-nintendo-switch-device-console Console mode

The following example shows how to apply a style using a CSS @media rule.

@media (-webkit-nintendo-switch-device-mode: -webkit-nintendo-switch-device-console) {
  body {
    font-size: 0.8em;
  }
}

The media attribute in tags such as <link>, and the JavaScript window.matchMedia method also support this attribute.

Language

The web applet and the Offline Web Applet use the application's UI display language and the system language. For more information about the display language and system language, see System Features Guide > Appendixes > Mechanism for Displayed Languages.

The web applet and the Offline Web Applet assume that webpages are written in the system language. The display language determines the priorities of the built-in system fonts for rendering the characters of the webpage. For more information about font priorities, see Font Priority.

The Accept-Language HTTP header sent by the web applet to the web server also uses the display language as the highest-priority language. For more information about the Accept-Language HTTP header, see Accept-Language.

The system language is used for the text in places like footers and dialog boxes. The language code that corresponds to the system language is set for NavigatorLanguage.language. For more information about language codes, see Language Settings.

Fonts

The web applet and the Offline Web Applet use a number of system fonts to display webpages in a variety of languages.

Note

Displaying ligatures is not supported.


Font Priority

The display language determines the priority of the different fonts for different webpages. For example, if the webpage is in Chinese, the Chinese-language fonts have higher priority than Japanese- and Western-language fonts.

Note

The system language determines the priority of fonts for <select> element select boxes for the alert messages and other UI elements of the window.alert() method.

As a result, if the display language and the system language are different, the same character may be presented using different glyphs in webpages and the user interface.

For example, if a webpage that is written in Chinese uses a select box and the system language is set to AmericanEnglish, the glyphs for Japanese characters rather than Chinese characters might get displayed in that select box.

Displaying Characters in a Specified System Font

For situations like webpages featuring a mixture of languages, you can use a system font family name to specify the system fonts for the display of characters.

Type Family Name
For Japan, North America, and Europe nintendo_udsg-r_std_003
For Hangul (Korean) nintendo_udsg-r_ko_003
For Simplified Chinese nintendo_udsg-r_org_zh-cn_003, nintendo_udsg-r_ext_zh-cn_003
For Traditional Chinese nintendo_udjxh-db_zh-tw_003
For the Nintendo Extended Character Set NintendoExt003

The following example shows a CSS file that specifies a system font for each language.

:lang(zh-Hans) {
  font-family: nintendo_udsg-r_ext_zh-cn_003, nintendo_udsg-r_org_zh-cn_003, nintendo_udsg-r_std_003;
}
:lang(ko) {
  font-family: nintendo_udsg-r_ko_003, nintendo_udsg-r_std_003;
}
:lang(zh-Hant) {
  font-family: nintendo_udjxh-db_zh-tw_003, nintendo_udsg-r_std_003;
}

By default, system fonts are used for characters displayed in webpages. However, using web fonts makes it possible to display custom fonts that match the design of the application.

Valid Font Ranges for Traditional Chinese

The glyphs of the alphanumeric characters of the system font for Traditional Chinese differs from those of other fonts. To pair these alphanumeric glyphs with the fonts for Japanese and Western languages, the font for Traditional Chinese is confined to the following ranges.

  • U+2002 to U+2003
  • U+2574 to U+2574
  • U+2641to U+2641

  • U+3100 to U+32FF
  • U+3400 to U+9FFF
  • U+F001 to U+F002
  • U+F900 to U+FAFF

Keyboard

When text needs to be entered on a webpage, a software keyboard applet opens.

Starting the Software Keyboard Applet

The software keyboard applet is started in the following cases.

  • The A Button is pressed while navigating the focus to an input field, such as an <input> element.
  • The A Button is pressed while navigating the focus to a noninput field, and the click event sent to the webpage causes an input field to become in focus.
  • When using the pointer, the A Button is pressed or the screen is touched, and the click or touchend event sent to the webpage causes an input field to become in focus.

Preventing the Software Keyboard Applet From Starting

To prevent the software keyboard applet from starting when an input field becomes in focus from a click or touchend event, you can use setTimeout() as follows.

<!DOCTYPE html>
<meta charset="utf-8">
<button id="button">button</button>
<input type="text" id="input">
<script>
onload = function() {
  document.querySelector('#button').onclick = function() {
    var e = document.querySelector('#input');
    setTimeout(function() {
      e.focus();
    });
  };
};
</script>

Sound

Playing Background Music

By playing background music before calling the web applet or Offline Web Applet, you can continue to play that background music while displaying a webpage. Functions that start the web applet and Offline Web Applet, block until the applet is closed, at which point the thread on which the applet was called is terminated, so background music processes must be executed on a separate thread. Background music cannot be played while the web applet or Offline Web Applet are playing videos.

You can set the volume balance for background music using the applicationVolume parameter of the nn::oe::SetExpectedVolumeBalance() function.

Playing Sound Effects

When operations such as spatial navigation or button presses are performed on a webpage, the web applet and the Offline Web Applet will play the standard system sound effects that have been configured. You can also use JavaScript extensions to play system sound effects other than the standard system sound effects. You can also use Web Audio to play custom sound effects that match the design of applications.

You can set the volume balance for sound effects using the sdkVolume parameter of the nn::oe::SetExpectedVolumeBalance() function.

Add the nx-se-disabled attribute to the HTML elements in the webpage to disable the default system sound effects when the element is in focus or when selecting or clicking the element.

<button nx-se-disabled>

Use the window.nx.footer.unsetAssign() or window.nx.footer.setAssign() JS extensions to disable the standard controller operation features and system sound effects and customize your own. For more information, see Using JavaScript Extensions to Customize Controller Operations.

Playing System Sound Effects With JavaScript Extensions

Using the following JavaScript extension makes it possible to easily play system sound effects on webpages. Up to four sound effects can be played at the same time, including sound effects played by the applet. Sound effects played using this feature stop when the page transitions.

For a usage example, see the WebDemoKeyboardEvents sample.

Using Web Audio to Play Sound Effects

You can use Web Audio to play custom sound effects that match the design of applications. You can use this to play WAVE_FORMAT_PCM type WAVE container format files (WAV) and AAC-LC codec MP4 container format files (MP4 and M4A).

To use Web Audio, you must first use the following functions to enable the feature.

Applet Function
web applet nn::web::ShowWebPageArg::SetWebAudioEnabled()
Offline Web Applet nn::web::ShowOfflineHtmlPageArg::SetWebAudioEnabled()

Use the following functions to set the volume balance for Web Audio. If these functions have not been called, the audio is played at the volume set by the sdkVolume parameter of the nn::oe::SetExpectedVolumeBalance() function.

Applet Function
Web applet nn::web::ShowWebPageArg::OverrideWebAudioVolume()
Offline Web Applet nn::web::ShowOfflineHtmlPageArg::OverrideWebAudioVolume()

The following HTML sample code is for playing the file sample_se.mp4.

<!DOCTYPE html>
<meta charset="utf-8">
<button id='play-se-button' disabled nx-se-disabled>play</button>
<script>
window.AudioContext = window.AudioContext || window.webkitAudioContext;
var audioCtx = new AudioContext();
window.onload = function() {
  var request = new XMLHttpRequest();
  request.open('GET', 'sample_se.mp4', true);
  request.responseType = 'arraybuffer';
  request.onload = function () {
    audioCtx.decodeAudioData(request.response, function (buffer) {
      var element = document.getElementById('play-se-button');
      element.disabled = false;
      element.onclick = function() {
        var source = audioCtx.createBufferSource();
        source.buffer = buffer;
        source.connect(audioCtx.destination);
        source.start(0);
      };
    });
  };
  request.send();
};
</script>

For a usage example, see the WebDemoPauseMenu sample.

The following restrictions apply to the use of Web Audio in the web applet and the Offline Web Applet.

AudioNode can use only AudioBufferSourceNode, AudioDestinationNode, and GainNode.

The AudioBufferSourceNode.detune method does not work correctly.

AudioParam can be used for each AudioNode, but the linearRampToValueAtTime and exponentialRampToValueAtTime methods do not work correctly.

Numerous AudioContext objects can be created on a webpage, but only one can be played. Be careful when using the iframe element to embed a different page.

Although there is no limit on the number of sound effects that can play at the same time, the mixed waveform will be clipped if the amplitude exceeds the maximum value, so noise can be generated if you play loud sound effects at the same time or many sound effects at the same time. If noise is generated, do not play multiple sound effects at the same time.

When you move to another page, the AudioContext objects are destroyed and the playing sound effect stops playing. If you want to play a sound effect when moving to another page, wait for the sound effect to end and then move. You can use the AudioBufferSourceNode.onended event handler to wait for playback to end, but the onended event handler may not be called if the reference to the AudioBufferSourceNode object is lost after the AudioBufferSourceNode.start method starts the playback. To make sure that the onended event handler gets called, keep the AudioBufferSourceNode object and have the onended event handler free the reference.

Adjusting the Output Volume Level

For the web applet and Offline Web Applet, the volume level of each output device is adjusted based on the value configured in the NX system features. For more information about this value, see Sound Design Guide > Support Recommendations > Examples of Adjusting Volume With NX System Features. The configured value is used to adjust the output volume for Web Audio, the video player, and system sound effects.

As a result, depending on the settings in the application, files played using Web Audio or the video player might play at a different volume than files played by the application, even if the files were created to play at the same volume. You can align the volume levels by adjusting the volume balance for Web Audio and the video player. For more information about the volume balance, see Using Web Audio to Play Sound Effects and Playing Video Files.

Customizing Operations

There are several features for customizing controller operations.

  • Disable the zoom feature assigned to the Right Stick and Right Stick presses.
  • Disable touch operations.
  • Limit Left Stick operations to spatial navigation.
  • Use JavaScript extensions to disable standard features and assign custom controller features.
  • Use keyboard events and the Gamepad API to read controller operations.
  • Change the scrolling speed.
  • Change the key repeat frame.

Disabling the Zoom Feature

To display webpages that are always fitted to the screen, add the following <meta> element to disable the zoom feature. 

<meta name="viewport" content="user-scalable=no">

When the zoom feature is disabled, the zoom feature instructions are not displayed in the footer.

For a usage example, see the WebDemoQuickHelp sample.

Disabling Touch Operations

Use the following functions to always use controller operations and disable touch operations for webpages.

Applet Function

Offline Web Applet nn::web::ShowOfflineHtmlPageArg::SetTouchEnabledOnContents()

Write the following JavaScript to disable touch operations for each webpage.

document.addEventListener('touchstart', function(event) {
  event.preventDefault();
});

Note that this code will not disable touch operations in web applet and Offline Web Applet UI elements such as footers and dialog boxes or in applet UIs such as the software keyboard and the error viewer. Because it is impossible to completely disable scrolling using the JavaScript method, you can use these functions for pages that do not scroll.

For a usage example, see the WebDemoSpatialNavigation sample.

Limiting Left Stick Operations to Spatial Navigation

The web applet and Offline Web Applet support two methods of controller input: spatial navigation and a cursor. The user can press the Left Stick to switch between the two input methods at any time.

The cursor input method is different from the input methods used in system feature and most applications and may be unfamiliar to users. As a result, Left Stick operations can be limited to spatial navigation if desired.

Note

Only use this feature when displaying webpages that are designed to support spatial navigation operations.

Be especially careful when enabling this feature in the web applet for general-purpose web services or webpages designed to be displayed on various devices. These types of webpages were not originally designed for display in the web applet and may not necessarily be usable with only spatial navigation. When these types of webpages are among the pages to display using the web applet, do not limit Left Stick operations to spatial navigation, and allow the user to switch to the cursor input method.

Use the following functions to limit Left Stick operations to spatial navigation.

Applet Function
web applet nn::web::ShowWebPageArg::SetPointerEnabled()
Offline Web Applet nn::web::ShowOfflineHtmlPageArg::SetPointerEnabled()

When Left Stick operations are limited to spatial navigation, the instructions for switching the Left Stick input method are not displayed in the footer.

For a usage example, see the WebDemoInstructionManual sample.

Using the following JavaScript extension makes it possible to disable standard controller operation features and assign custom features. Features customized using this feature revert to the default state when the page transitions. If you want to customize multiple pages, call the function on each webpage.

Any scrolling speed that was changed using this feature reverts to the standard speed when you transition the page. Call the function for each webpage to change the speed for multiple pages.

Syntax

Parameters

Return Values
  • None
More Information

Syntax

    String that represents the behavior type.

Return Values
  • None
More Information

String Behavior

step Changes the scroll speed between two levels based on the length of continuous controller input.
exponential Changes the scroll speed, as an exponential function, between two levels based on the length of continuous controller input.
  • options
    • Specifies the properties based on the type of behavior specified for behavior.
    • When constant is specified

        The speed, in pixels per frame. Optional.

Memory

Memory is allocated in the web applet and Offline Web Applet for displaying web pages.

Behavior With Insufficient Memory

Due to limited memory capacity, webpages that consume a large amount of memory may not be able to be displayed. The web applet and Offline Web Applet have different behavior when there is not enough memory.

When there is not enough memory, the web applet works to preserve operation of the included web browser as much as possible. The behavior is as follows.

  • Automatically omits portions of the webpage, such as displaying images and running JavaScript. A log message like the following is displayed in red when parts of a page are omitted.
Parts of this page were omitted
  • If there is not enough memory even after omitting a portion of the page, the web browser restarts and displays the following message. A log message like the following is displayed in red when the web browser is restarted.
Browser engine restarted

When there is not enough memory, the Offline Web Applet works to preserve the page as much as possible. The behavior is as follows.

  • Unlike the web applet, no portion of the page is omitted.
  • The web browser restarts and the page is reloaded.
    • Unlike the web applet, no dialog box message is shown.
    • Like the web applet, a log message is displayed.
  • If memory runs out again immediately after reloading the webpage, the web browser restarts and a dialog box is displayed.

If the dialog box is displayed frequently because there is not enough memory, it may be because the webpage consumes more than the allocated memory. Adjust the content of the webpage so that it consumes less memory.

Displaying Memory Usage Rates

The memory usage rate of the webpage being displayed can be displayed on the screen for purposes of web content development.

For more information about displaying the memory usage rate, see Display Memory Usage Rates.

Cookies

Cookies can be used on webpages displayed in the web applet.

However, third party cookies are blocked.

Offline HTML pages displayed in the Offline Web Applet do not support using cookies.

Web Storage

Webpages displayed in the web applet support use of sessionStorage and localStorage. Offline HTML pages displayed in the Offline Web Applet support use of sessionStorage but do not support using localStorage.

Viewport

The viewport is a virtual window used when laying out pages and can be modified via the name="viewport” attribute in the <meta> element.
The default size is width=1280, height=648. In the Offline Web Applet, when the nn::web::ShowOfflineHtmlPageArg::SetFooterEnabled() function is used while starting the applet to set the footer to be kept hidden, the size of the viewport is set to width=1280, height=720. These same values can also be set by specifying width=device-width, height=device-height.
Although any values can be specified for the viewport size, some values may result in pages no longer being displayed on a pixel-for-pixel basis, or may result in pages no longer fitting on the screen.

Page Cache

The web applet and the Offline Web Applet have a feature called page cache. When the browser opens the links to webpages, the states of those pages are temporarily stored in memory. When the browser navigates back through the history, those saved pages are displayed again. The page cache feature enables the applets to quickly navigate back through the history and to restore the states of pages. However, sometimes you may want to retrieve the page again from the website when you navigate back through the history, or undo the dynamic changes added to webpages.
For these kinds of situations, you can use the following functions to always disable page cache.

Applet Function
web applet nn::web::ShowWebPageArg::SetPageCacheEnabled()
Offline Web Applet nn::web::ShowOfflineHtmlPageArg::SetPageCacheEnabled()

To disable page cache for individual webpages, you can use the following methods.

  • For HTTPS sites, specify no-store for Cache-Control in the HTTP response header so the page will not be cached.
  • Evaluate the persisted property with the pageshow event handler and reload pages if they have been cached.
onpageshow = function(e) {
  if (e.persisted) {
    location.reload();
  }
};

Web Inspector

Web Inspector can be used for developing web content.

The procedure for using Web Inspector is as follows.

  • Web Inspector can be enabled in the Settings Manager.
    • For more information, see Enabling Web Inspector.
  • The web applet and the Offline Web Applet can be started using WebAppletCaller.
  • A dialog showing the IP address is displayed.
  • The web inspector resources include resources for the web applet and resources for the Offline Web Applet.
    • The resources for the web applet are located in $(NINTENDO_SDK_ROOT)/Tools/WebInspector/Web/Resources.
    • The resources for the Offline Web Applet are located in $(NINTENDO_SDK_ROOT)/Tools/WebInspector/OfflineWeb/Resources.
  • Use a PC browser to open Main.html included among these resources.
    • Web Inspector does not operate properly with the default settings of the PC browsers, so make the following changes.
      • For Google Chrome, run the browser with the --allow-file-access-from-files option.
        • In Windows, create a shortcut link, and add the option to the link destination settings.
      • In Mac Safari 10.x, in the Developer menu, add a check mark to Disable cross origin restrictions.
  • In the file path, after Main.html, add “?ws=<IP address shown in the dialog>:9222/devtools/page/1” and then run.
  • Use of Web Inspector starts.


Note

Some of the features are not supported in the web inspector for the Web Applet.

  • Cannot check Call Trees for JavaScript & Events in the Timeline tab
  • Frames is not displayed in the Timeline tab
  • The Protocol, Priority, and IP Address columns are not displayed in the list display in the Network tab
  • The information is incorrect for the Transfer Size column in the list display in the Network tab
  • The Cookie, Bytes Sent, and Bytes Received information is not displayed in the detailed information display for each request in the Network tab.

console

CPU Boost

Under specific conditions, the web applet and the Offline Web Applet can use the CPU boost feature to speed up the display of webpages.

To use this CPU boost, the calling application must specify InFocusOnly as the focus handling mode before the web applet or the Offline Web Applet starts.

When this condition is satisfied, a CPU boost equivalent to specifying nn::oe::CpuBoostMode_FastLoad is enabled under the following circumstances.

  • From after the applet starts until the first page has finished loading
    • If starting in dedicated media mode, it is enabled until video is ready to be played.
  • While loading pages
  • While preparing to play video
    • If an error occurs during these playback preparations, the CPU boost will end at that time.

If these conditions are satisfied, a log like the following is displayed when the applet starts.

[CpuBoost] enabled

Developers of web content can use WebAppletCaller to check their web content with CPU boost enabled.

For more information about CPU boost, see CPU Boost Functionality.

For more information about the focus handling modes, see Notifications For Programs.
