Restrictions in Windows Platforms

You must include nn/nn_Windows.h instead of Windows.h.

Make sure that you include nn/nn_Windows.h instead of Windows.h.

There is a problem of collisions between macros defined in <Windows.h> and the symbols and names in NintendoSDK (for example, CreateFile). This problem is avoided by using nn/nn_Windows.h.

You must define the WIN32_LEAN_AND_MEAN and NOMINMAX macros before including nn/nn_Windows.h.

Define the following macros before including nn/nn_Windows.h.

  • WIN32_LEAN_AND_MEAN
  • NOMINMAX

Operations are not guaranteed if you include nn/nn_Windows.h without first defining these macros.

Building immediately after a proprietary definition of nninitStartup() can result in a LNK2005 error.

When you have a proprietary definition for the nninitStartup() function in your application, the following type of link error can result the first time you attempt a build immediately after defining that function. (Similar link errors can result when redefining default function definitions besides nninitStartup() within the SDK library.)

1> main.obj: error LNK2005: _nninitStartup is already defined in libnn_init.lib(init_Startup.obj).

This error is an effect of the incremental link feature in Visual Studio.

When this link error occurs, either build one more time, or rebuild. The build process should proceed smoothly.
