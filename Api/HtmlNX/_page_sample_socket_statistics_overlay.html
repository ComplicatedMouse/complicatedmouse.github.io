This sample shows a program that has been configured to show the socket statistics overlay tool in the DevMenuOverlay.

Overview

Sends a burst of packets to the client from a server using either TCP or UDP.

File Configuration

This sample program is located in Samples/Sources/Applications/SocketStatisticsOverlay/ .

Required Environment

You must import network connection settings using SettingsManager in advance. For more information, see the nifm library.

How to Use

Run the server first and then run the client. Run the server or client by specifying the following options.

  • (SDEV #1) "-server" to start the server that can host both TCP and UDP.
  • (SDEV #2) "-udpClient server=<ip_addr>:<port>" to run a UDP client.
  • (SDEV #2) "-tcpClient server=<ip_addr>:<port>" to run a TCP client.

After the server or client start running, you can select them by name in the DevMenuOverlay (Features > Controlling the Development Hardware > DevMenuOverlay). You may also select to view a specific socket display.

Note: You may use a windows host machine for SDEV #1, but the socket overlay examples might not perfectly match the comments.

Notes

In the case of network failure, not all control paths are handled.

Procedure

Build and run the sample program with the following arguments.

  • (SDEV #1) "-server"
  • (SDEV #2) "-udpClient server=<ip_addr>:<port>"
  • (SDEV #2) "-tcpClient server=<ip_addr>:<port>"

Program Overview

The program has the following flow for the sample program.

  • Initialize the network interface (NIFM).
  • Initialize the socket library and register for statistics.
  • Server Only
    • Create a thread to run the TCP server that then listens for connections.
    • Create a thread to run the UDP server that then waits for a hello message.
    • Both TCP and UDP servers respond to BurstSend requests using the specified period.
    • Does not shut down automatically and may serve multiple clients sequentially.
  • Client Only (TCP and UDP)
    • Connect to the server. (UDP uses a hello message.)
    • Request BurstSend from the server at a rate to the overlay socket behaviors.
    • The client terminates after completing its overlay behaviors.

  This will show the client's tcp socket gradually filling up until
  either mb_used reaches mb_pool or sb_data reaches sb_size at which point
  the server will slow its send rate to accommodate the client's window size.

  The server app may not see any extra delay on a blocking send call as long as
  the tcp's send buffer mb_used does not reach mb_pool and sb_data does not
  reach sb_size. If the server's send buffer does become full any blocking send
  calls will block until there is room to store the tcp sequence as needed.

Tcp Client:  Healthy receive buffer

  As long as we read from the receive socket buffer faster than the server
  is sending we will not slow down the stream. If you have a large burst that
