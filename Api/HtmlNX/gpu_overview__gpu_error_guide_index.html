1. Overview

This document provides some basic information on GPU crash handling and suggests possible actions to take when a GPU crash happens.

A GPU crash can sometimes be caused by system software, but it can also be caused by problems in user space application code. Here we will focus on the latter, mapping the error information to user space issues. Note that in many cases it is not possible to infer a sequence of NVN calls based on GPU state alone - this document should only be considered a best-effort attempt to assist app developers.

Error information can be grabbed in 2 ways:

  • Automatic log-dump
  • Programmatic queue-error checking

For SDEV, there are some options to control the GPU error handling behavior in the DevMenu:

  • Debug-GPU Crash Automatic Dump (check-box): you must check this option to automatically dump GPU error information.
  • Debug-Terminate Application on GPU Error (drop-menu): when this option is "enabled", the process will be terminated in case of GPU error. As this behavior may leave no time for programmatic error checking and handling at the nvnQueue level(1.2. Programmatic Queue-error checking), one should select "disabled" here.
  • Debug-Gpu Timeout (drop-menu): when this option is "disabled", long draw-calls will be allowed to run without causing a "Timeout error".

1.1. GPU crash log

When a process encounters a gpu error, the gpu core dumper will start to dump error information. The user will see a message in the target-manager which indicates the location of the file.

The message looks like:

[GpuCoreDumper] Start dumping to C:\Users\username/Documents/Nintendo/NXDMP...
[GpuCoreDumper] Dump completed

When the dump is finished, the process will exit.

For SDKs earlier than 3.x.x, the dumped file is in text format. For SDKs later than 4.0.0, the file is in binary format, and the user needs an external tool to convert the file to text format. Currently, the tool is located at NINTENDO_SDK_ROOT%\Tools\Graphics\NxgcdFormatter\NxgcdFormatter.exe (the location and file name are subject to change in future versions).

usage: nxgcdformatter [-o nxgcd_format_file] nxgcd_dump_file
-o : save output to nxgcd_format_file
(default is <nxgcd_dump_file>.txt)

1.2. Programmatic Queue-error checking

Application developer can call nvnQueueGetError() to get the error status of the queue. The return value gives a rough classification of the error that has happened.

The benefit of programmatic error checking is that developer can choose to finalize and re-create a queue instead of terminating the application. It also helps to locate a specific nvnQueueSubmitCommands() that cause the crash.

1.3. About nvnCommandBufferSignalEvent()

While nvnQueueGetError() gives us which nvnQueueSubmitCommands() failed, the detailed error information may not be helpful in pinpointing at the command level which instruction caused the GPU to crash.

nvnCommandBufferSignalEvent() can help the developer to narrow down the actual command that is failing. It writes a value at the specified signal location.

1.3.1. Signal locations

There are three possible locations in the pipeline where the events can be signaled. Below is the table of the signal locations and the conditions.

signal location condition when signaled
TOP immediately after the signal event command is processed at Host
VERTEX_PIPE all of world space processes (VS/TCS/TES/GS stages) have completed the work of previously submitted draw calls
BOTTOM the work for all previously submitted graphics commands have completed

And also there is one flag, NO_WAIT_MEMORY. By default, an event is not signaled until all GPU memory transactions from prior commands have completed. When NO_WAIT_MEMORY is set, the event may be signaled without waiting for all memory transactions, and there may be performance benefit on the GPU call overhead. If the draw call uses transform feedback (XFB), it may be signaled before XFB has completed. There is no effect for TOP signal with NO_WAIT_MEMORY, since it is immediately signaled at Host process.

1.3.2. About nvnCommandBufferReportValue()

nvnCommandBufferReportValue() is the predecessor of nvnCommandBufferSignalEvent(). It writes the given value and also timestamp into the buffer. But the reporting location is always at bottom without NO_WAIT_MEMORY.

1.3.3. GPU call overhead

This is very rough indication about minimum GPU call overhead. The GPU call overhead in real application can be different from this.

signal command GPU call overhead [us] (handheld clock)
SignalEvent(TOP) 0.1
SignalEvent(VERTEX_PIPE|BOTTOM, NO_WAIT_MEMORY) 0.3
SignalEvent(VERTEX_PIPE|BOTTOM) 0.4
ReportValue() 0.4

1.3.4. How to narrow down the failing command

It is difficult to pinpoint the failing command due to the very deeply pipelined nature of the GPU architecture, there are many commands processed at the same time over the many GPU hardware components. But the developer may narrow down the failing command by using 2-3 signal locations (at least, top and bottom).

As described in Signal locations, the narrowing down depends on which location within the graphics pipeline the GPU error happens in. While some of GPU error can specify the error location at the graphics pipeline, others cannot specify the pipeline location. If the error location is at Host, FE or world space pipeline, the error will occur between TOP and VERTEX_PIPE. If the error location is at screen space pipeline, the error will occur between VERTEX_PIPE and BOTTOM. If the error location cannot be defined at any particular location within the graphics pipeline or non-graphics command like compute, copy engine or 2D rasterizer tasks, the error will occur between TOP and BOTTOM.

Below list is the typical case for narrowing down the error:

Host + FE + world space pipeline: between TOP and VERTEX_PIPE

  • Pushbuffer DMA error
  • MMU fault :
    • Faulting unit: Host (host)
    • Faulting unit: Primitive Distributor (pd)
    • Faulting unit: Primitive Engine (pe 0)

screen space pipeline: between VERTEX_PIPE and BOTTOM

  • MMU fault :
    • Faulting unit: Texture Cache (t1 1) (in case there is no texture/SSBO reading/writing except for fragment shader)

others: between TOP and BOTTOM

  • Timeout error
  • Graphics engine exception
  • MMU fault :
    • Faulting unit: Copy Engine 0 (ce0)
    • Faulting unit: Copy Engine 1 (ce1)
    • Faulting unit: 2D Rasterizer (raster twod)
  • all other errors

Below pair of code and figure is an example how the signal locations narrows down the failing command.

for(int n=0; n<num_drawcalls; ++n){
SignalEvent(events[0], WRITE, TOP, 0, n);
DrawArrays(...); // n-th draw call
SignalEvent(events[1], WRITE, VERTEX_PIPE, NO_WAIT_MEMORY, n);
SignalEvent(events[2], WRITE, BOTTOM, NO_WAIT_MEMORY, n);
}
**Figure 2: Example of narrowing down GPU error **


If a GPU error occurred at early timing of pixel pipeline of N-th drawcall, the last signaled value for each locations would be, TOP = N+3, VERTEX_PIPE = N+2, BOTTOM = N-2.

It corresponds to VERTEX_PIPE (N+2) >= failing command (N) > BOTTOM (N-2) for screen space pipeline.

1.3.5. Pinpointing the failing command with IndirectData Barrier

While this is extremely expensive in aspect of performance, there is a way to pinpoint the error call by using IndirectData Barrier for each draw call. IndirectData Barrier is processed at Host. It waits for the completion of previously submitted works. Then signaled value will match to draw call.

For example, with below code sequence, all of GPU errors match to the last value of events[0].

for(int n=0; n<num_drawcalls; ++n){
Barrier(ORDER_INDIRECT_DATA);
SignalEvent(events[0], WRITE, TOP, 0, n);
DrawArrays(...); // n-th draw call
}
Figure 3: Example of Pinpointing GPU error with IndirectData Barrier


1.4. About debug layer

Debug layer is an NVN debugging feature. When enabled, it checks parameters of NVN API calls for illegal API usage before GPU commands are issued. When an illegal call is caught by the debug layer, a notification will be sent through debug layer callback. The debug layer callback notifications can then be mapped to user space errors, so it will be easier to analyze in comparison to a GPU crash log. Therefore, when a GPU crash happens in a debug layer disabled application, it is highly recommended that user should try enabling debug layer first to see if the error can be captured at an earlier stage.

The usage detail of debug layer can be found in the "NVN Programming Guide"

2. GPU Exception Types

In the error log, the "Exception type: xxx" line indicates the type of the exception.

Here, we itemize the 4 basic types of GPU exceptions, the error messages we see when they happen, and the direct causes of the exceptions. In the next sections, we go through the exception types one by one and describe the strategies to triage the exceptions further.

2.1. Pushbuffer DMA error

Error message:

Exception type: pushbuffer DMA (PBDMA) error

This error is usually reported when GPU encounters corrupted or invalid information from command memory.

2.2. MMU fault

Error message:

Exception type: MMU fault

This error is usually reported when there is a corruption in GPU memory management or an invalid memory pointer is sent by the application.

2.3. Timeout error

Error message:

Exception type: scheduler error

Scheduler error code: timeout

This error is usually reported when the GPU is stuck in a single draw call for too long and cannot perform a context switch to another queue. This might be caused by very long running shaders or large amounts of geometry or pixels in a draw call.

2.4. Graphics engine error.

Error message:

Exception type: graphics engine error (<description>)

This error happens during the execution of a GPU command. <description> indicates the subtypes of the error. "class error" is the one most commonly seen, which indicates a error of the command itself of its parameter.

3. Pushbuffer DMA error

When a Pushbuffer DMA error happens, that means something in the submitted command memory has been corrupted.

A well-known cause of a PBDMA error is the unintended overwriting of a command-buffer that is still being transferred to GPU, which in turn is usually due to insufficient fence-syncs before writing to the command-buffer. If the error disappears after inserting a nvnQueueFinish() at frame boundaries, the cause of the exception can be confirmed.

Note:

  • The NVN debug layer has the capability to detect command memory of abnormal state being submitted. When debug layer is on, you will see error message like the following, instead of a GPU crash:

"nvnQueueSubmitCommands: Added command memory is possibly being used by the GPU. Please insert a fence to wait for GPU completion."

  • Sometimes, a PBDMA error is preceeded by a graphics engine error. That typically means that some commands or parameters contents are unintendedly overwritten first, but the pushbuffer structure is not corrupted yet at that moment, and the commands are successfully passed to GPU. In this case, we should treat PBDMA as the main issue.

4. MMU fault

An MMU fault is a GPU memory accessing related fault. A MMU fault will be reported when an invalid memory buffer address is passed from applications side, or resources used in GPU memory management, such as PTE(Page Table Entry), are corrupted. Common root causes of MMU fault from user space are:

  1. An invalid memory buffer address is passed from application
  2. Accessing some resource out of the lifetime of the memory pool or the object (such as a texture)
  3. Failing at the initialization of the memory pool or the object
  4. Accessing before the initialization of the memory pool or the object
  5. Accessing after the finalization of the memory pool or the object
  6. An newly initialized memory pool or the object has overlapping memory range with an old one
  7. Unexpected memory corruption affecting the memory pool or the object being accessed

Therefore, for all MMU faults, it is good practice to check the life-spans of memory pools and objects.

By looking at the succeeding error messages, we can determine some more details of the error. There are "Access type", "Faulting unit" and "Fault type", each of them gives some additional information about the fault. In addition, please also refer to the "Virtual Memory Configuration for Queue" section of the error dump message for further details on mapped ranges in the virtual address space along with each range's associated page kinds.

4.1. Access type

Error message:

Access type: read/write

The "Access type" note gives indication whether the error happens when reading or writing the problematic buffer.

4.2. Faulting unit

Error message:

Faulting unit: <unit>

The "Faulting unit" note tells user which GPU unit has been trying to access memory during the crash. Because different units are used during different parts of the pipeline, and different buffers are involved, from the faulting unit information, we can deduce which part of the pipeline and what kind of buffer are likely to be problematic.

For example:

Faulting unit: Primitive Engine (pe 0)

The above indicates vertex buffer issues

Faulting unit: Texture Cache (t1 1)

The above indicates texture fetching, image or SSBO reading/writing issues

Faulting unit: GPC Cache Controller (gcc)

The above indicates shader memory, uniform memory or shader scratch memory issues.

Faulting unit: Host (host)

The above indicates command memory issues

Faulting unit: Primitive Distributor (pd)

The above indicates index buffer issues

Faulting unit: Copy Engine 0 (ce0)

Faulting unit: Copy Engine 1 (ce1)

The above indicates problem with buffers being copied.

Faulting unit: 2D Rasterizer (raster twod)

The above indicates problem with buffers used for 2D operations (copy or down-sample)

4.3. Faulting type

Faulting type: <type>

The "Faulting type" note gives information about at which step has the memory access failed. The value here can be:

Faulting type: 0 (pde), Page Directory Entry has no valid page tables

Faulting type: 1 (pde size), address references beyond valid Page Directory Entry range

Faulting type: 2 (pte), Page Table Entries needed to translate address are not valid

Faulting type: 3 (va limit viol), address exceeds virtual address limit

Faulting type: 4 (unbound inst), engine is not bound to an instance block

Faulting type: 5 (priv viol), non-privileged request to privileged page

Faulting type: 6 (ro viol), write request to read-only page

Faulting type: 7 (wo viol), read request to write-only page

Faulting type: 8 (pitch mask), request spans 128B boundary and the page is pitch

Faulting type: 9 (work creation), illegal access to page with special work creation kinds

Faulting type: 10 (bad aperture), aperture configuration not supported

Faulting type: 11 (compression failure), Memory Management Unit compression failed

Faulting type: 12 (bad kind), Page Table Entry kind error

Faulting type: 13 (region viol), illegal access to protected region

Faulting type: 14 (poisoned), poisoned request issued by L1

Faulting type: 15 (unknown), fault type unexpected

4.4. Virtual Memory Configuration for Queue

This section lists all of allocated memory pool pages which belongs to the device associated with the queue. As an example:

VA start: 0x0004000000 VA limit 0x2000000000

[0x0004000000-0x00041effff]: pgsize 0x01000 kind 0x00 (PITCH)

...

[0x0500010000-0x050002ffff]: pgsize 0x10000 kind 0xfe (GENERIC_16BX2)

...

[0x05100b0000-0x0513ffffff]: pgsize 0x10000 kind 0xdb (C32_2CRA)

...

User can check if the faulting address is allocated or not, and which page kind the faulting address lies.

The page kind information is related to NVNstorageClass. The "kind" values are numerically equal to the least significant byte of storageClass. When the faulting address does lie in a valid address range, the corresponding "kind" value and name will indicate the storage class of the faulting buffer.

Page kinds can be classified into 3 categories:

  • kind 0x00 (PITCH): used for buffer objects, buffer texture objects, and linear texture objects. (Please refer to the NVN programming guide for further details.)
  • kind 0xfe (GENERIC_16BX2): used for uncompressible block-linear texture objects
  • Other "kinds" are all used for compressible block-linear texture objects

Thus, the kind information can be used to narrow the range of buffer and texture objects that could be related to the fault.

5. Timeout error

A timeout error happens when it takes too long trying to drain a command-queue. (Which is needed in cases like a context switch) The following reasons should be considered for timeout error:

  • Heavy shaders: is there a single shader that takes very long time?
  • Too many primitives/vertices?
  • Too many pixels?
  • Low GPU/Emc frequencies?

When debugging a timeout error, it can be considered to disable the "Debug-Gpu Timeout" option in DevMenu, thus long draw calls will be allowed to continue to run, giving developer a chance to use NVN-GD or LLGD to capture a frame for further debugging.

6. Graphics engine error

A graphics engine error happens insides a GPU command execution process. A tricky case of graphics engine exception is that it also occurs when a command buffer is unintendedly overwritten. In that case, the detail information of the graphic engine exception changes from run to run, and usually the method code can not be decoded to a commonly used hardware method. Another signature of an overwriting issue is immediate followed by an PBDMA error. In that case, the PBDMA issue should be dealt with first.

GPU command execution is inherently complicated. There are works of different natures taking place, and therefore graphics engine error has multiple subtypes, according to which part of the work goes wrong. So we should first have an overview of the possible circumstances.

6.1. Categories

Depending on the expression you see at the <description> part of the error message, the error can be 1 of 8 subtypes of graphics engine error. The possible expressions are:

"class error", "derived state error", "front end error", "memory format error", "semaphore timeout", "illegal method", "illegal class", and "illegal notify".

  • class error: An invalid command or parameter is encountered
  • derived state error: A wrong shader-program-header is detected

We will explain the above 2 cases in detail later.

  • illegal method: A method code that does not mean anything is encountered. The wrong method code and the address will be logged.
  • illegal class: A class code that does not mean anything is encountered. The wrong class code and the address will be logged.

When the above 2 cases happens, user should check for a possible command buffer corruption. The error message itself does not contain much information that can be made use of.

The remaining cases are related to driver internal logic, and it is hard to imagine how userspace could cause those errors. Please report these if you can.

However, please keep in mind that command buffer corruption can theoretically cause any kind of graphic engine errors.

6.2. Class error

In case of a class error, there will be relatively abundant information telling the user what the failed command is and what has gone wrong.

Please inspect the "Method" field to identify the problem. For example:

Method: 0xb197,0x000019d0

It is composed of two hexadecimal numbers.

The first part is the code of the hardware class, which is the engine unit that is executing the command. The most commonly seen hardware class codes are:

  • 0xb197: Maxwell2, 3D engine
  • 0xb12d: Maxwell2, 2D engine
  • 0xb1c0: Maxwell2, Compute engine

The second part is the code of the hardware method. hardware methods are recorded by the driver within the implementations of NVN APIs. The same GPU method may be used by different NVN APIs.

In 6.4. Table 1 Method Codes and Related NVN APIs, we give an incomplete list of method codes and known related NVN APIs.

In the example case, the hardware class code 0xb197 indicates Maxwell2's 3D engine. And according to Table 1, the hardware method code 0x000019d0 can potentially be generated by NVN API functions related to tiled downsample and surface clearing commands such as nvnCommandBufferClearColor or nvnCommandBufferClearDepthStencil.

(Note that code methods in Table 1 is incomplete, errors may occur in methods not listed in the table. In that case please contact support with details.)

After decoding the method codes, the second thing the user should consult is "Class Error Code", which gives the category of the error. For example:

Class Error Code: 0x0000000f

User should look up the code in 6.5. Table 2 Class Error Codes. In this case, we see the Class Error code is decoded to "ErrorInvalidWidth", so we know the issue is that the surface being cleared has an invalid width.

The crash log also contains a "Data" note. It gives information of the data fields of the failed command. For example:

Data: 0x00000000:0x0000003c

In most cases, the data fields of the HW methods that are finally called by the driver are not directly related to the parameters of the NVN API functions called by user, so it generally does not make sense for app developers to decode this information. User can send this information to hardware producer for further analysis.

6.3. Derived State Error

In case of a derived state error, the "error status" line is helpful.

For example:

Error Status: 0xa0000040

It should be decoded as a combination of different bit-masks.

  • The masks 0x80000000 and 0x40000000 are for driver internal use, please ignore.
  • The following bit-masks indicates the pipeline stage where a shader-program-header error is detected:
    • 0x01000000: vertex shader A (cull before fetch shader)
    • 0x02000000: vertex shader B (regular vertex shader)
    • 0x04000000: tessellation control shader
    • 0x08000000: tesselator
    • 0x10000000: geometry shader
    • 0x20000000: pixel shader
  • The lower 24 bits indicates 23 different error types (LSB is not used). Because not all error types are meaningful to the user, here only a subset is explained.
    • 0x00000010: Number of enabled attributes in Imaps and Omaps must fit in CBE and ISBE
    • 0x00000020: For tessellation control shader and geometry shader, number of threads allocated for each primitive is not in 1~32 range
    • 0x00000040: Number of warps allocated for each workgroup is 0. In the case of compute shader, this is related to the parameters set in "layout(local_size_x = ..., local_size_y = ..., local_size_z = ...) in;"
    • 0x00000080: For geometry shader the maximum count of output vertices is illegal (must <=1024, cannot =0)
    • 0x00000200: Non-existent attributes cannot be selected in the Imap or Omap of any enabled shader
    • 0x00000800: POSITION_W is not enabled in the PS Imap
    • 0x00008000: Violation happens when loading new shader-program-header
    • 0x00040000: Shader-program-header has a wrong shader type
    • 0x00800000: Pixel shader producing per-sample colors while target-independent-rasterization is on

As we can see, the error type bits indicate the direct causes of the errors. These variables are in most cases not directly controllable by user. However, they can still be useful hints to user as to what has gone wrong.

6.5.1. Notes about Advanced Blending

Advanced blending is enabled when nvnBlendStateSetAdvancedMode is called with overlap != NVN_BLEND_ADVANCED_MODE_NONE. When advanced blending is enabled, only a single render target is supported. Advanced blending with multiple color targets has undefined behavior and may result in "ErrorIteratedBlendTargetCount". Advanced modes other than NONE are supported only for color target zero. Advanded blending to color target != 0 may result in "ErrorIteratedBlendTarget0Not0".
