About Timeouts

For the OS-API functions that have timeouts, you can specify a nn::TimeSpan type argument (tentatively named timeout) as a timeout value.

The behavior is as follows when timeout > 0.

  • If the target object is signaled before the timeout time has elapsed, the return value invariably indicates that the object has been signaled. (In this case, a return value indicating a timeout is not returned.)
  • If the target object is not signaled before the timeout time has elapsed, a value indicating a timeout is returned. However, there are cases where the target object becomes signaled in the time it takes for the calling thread to confirm that return value.

When timeout == 0, the function acts as a feature that conducts polling as follows.

  • The function returns immediately after it determines the current state of the synchronization object.
    • If the target synchronization object is in a signaled state, a return value that indicates that signaled state is always returned (and is prioritized over return values indicating that they are timeouts).
    • If the target synchronization object is in a non-signaled state, a return value that indicates that it is a timeout is returned.

A negative value cannot be specified for timeout.

When using API functions that have timeouts, take this information into consideration.
