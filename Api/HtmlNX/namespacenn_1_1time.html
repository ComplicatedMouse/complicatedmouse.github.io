Detailed Description

The namespace for the time library.

Feature Overview

The time library provides features involving time, including getting, calculating, and adjusting the time.

Common Preconditions

  • As a rule, you must call the nn::time::Initialize function and initialize the time library before using any of its API resources.

Behavior in Windows

In the Windows operating environment, the PC clock is used in place of the internal counter that increases monotonically at a fixed pace.

Accordingly, if the PC clock is manipulated, the time obtained via the time library will deviate by the amount of that manipulation. There is no way for the time library to detect the occurrence of such deviations.

However, the time obtained via the time library will not be affected by any manipulation in the PC clock that occurs during the period between the initialization and the finalization of the library.

This library uses features of the fs library when running in a Windows environment. Ensure that memory allocation can occur using the allocator set with the nn::fs::SetAllocator() function or with malloc(). For more information, see the fs library documentation.

API For Handling Time Zones

  • The internal specification of the nn::time::TimeZoneRule structure, which is needed for calculating local time and absolute time, might be changed in the future. Do not store it in save data or elsewhere for use at the next startup.
  • The nn::time::LocationName values that can be selected in the system settings and in DevMenu may change, depending on future uses. However, anything that has been selectable in the past continues to be handled by the time library.
  • The nn::time::ToCalendarTime function always works with inputs in the range from nn::time::InputPosixTimeMin to nn::time::InputPosixTimeMax.
  • The nn::time::ToPosixTime function always works with inputs in the range from 2000-01-01 00:00:00 to 2100-01-01 00:00:00.
  • Time difference usage rules for each time zone may be updated with a system update. There are no assurances that the same rules will be used in the future.
  • Future usage rules for time differences will be calculated as the latest available rules being applied, even in the future.
  • The time obtained via the time library is confirmed to fall within the assured operating range. However, operating environments running on Windows are an exception to the rule.

Performs a user clock operation-aware determination of whether a periodic benefit can be claimed.

Parameters
[in,out]pHandoverThe pHandover pointer obtained from the previous determination.
(After the function is called, regardless of whether the call is successful, this pointer must be kept for use for the pHandover parameter in the next call to the function.
On the first call to the function, specify the PeriodicBenefitClaimContext object generated by the constructor as is. On this first call, the determination is performed as if no clock operations have occurred yet.)
[in]currentSnapshotThe current ClockSnapshot object to use to determine whether the benefit can be claimed.
[in]maxPenaltyTimeSpanThe maximum penalty period.
(The specified value is typically less than or equal to the interval at which the benefit can be claimed when there are no clock operations. For example, if the benefit can be claimed again after the date changes, specify a value less than or equal to nn::TimeSpan::FromDays(1).)
[in]acceptableOperationTimeSpanThe amount by which the clock value is allowed to be moved into the future when performing a clock operation.
(Although specifying a small value can be effective in terms of preventing benefits from being claimed at shorter intervals than intended, it can also cause the full maxPenaltyTimeSpan penalty to be applied even for small clock operations. For this reason, in most cases specify a value of approximately nn::TimeSpan::FromMinutes(10). However, if the interval at which benefits can be claimed is short and an acceptableOperationTimeSpan value of this magnitude would be relatively large in comparison, reduce the specified value accordingly.)
[in]pCheckCalendarTimeCallbackCallback class for checking the time.
(For more information, see the CheckCalendarTimeCallback class.)
Returns
The determination result.
Return Values:

nn::time::PeriodicBenefitClaimResult_Success The benefit can be claimed.

nn::time::PeriodicBenefitClaimResult_NotReceivableTiming The benefit cannot be claimed again at this time.

nn::time::PeriodicBenefitClaimResult_Penalty The benefit cannot be claimed because the penalty period is currently in effect. Even after the penalty period is over, the function may return a result indicating another type of failure.

Precondition
  • The time library is initialized.
  • The object specified for currentSnapshot has been initialized using nn::time::ClockSnapshot::CreateWithStandardSystemClock() or nn::time::ClockSnapshot::CreateWithAdjustableSystemClock().
  • pHandover != nullptr.
  • maxPenaltyTimeSpan > nn::TimeSpan(0).
  • acceptableOperationTimeSpan >= nn::TimeSpan(0).
  • pCheckCalendarTimeCallback != nullptr.

Details
This utility function is intended for use in applications that use user clock (nn::time::StandardUserSystemClock) time values to manage events that are periodically triggered on the basis of date and time criteria, such as for benefits that can be claimed every day after the date changes and events that only occur on particular days of the week every week. This function can be implemented to prevent users from intentionally changing the user clock time to attempt to claim benefits at a shorter interval than intended, or conversely, to prevent users from waiting an excessively long time to claim benefits.

Performing a clock operation, depending on the type of operation, can trigger a penalty period during which the benefit cannot be claimed. While this penalty period remains active, the benefit cannot be claimed even if the next date and time at which the benefit is supposed to become eligible again occurs. The penalty period is automatically lifted after the associated amount of wall time (starting from when the operation was performed) elapses.

When the Amount by Which the Clock Was Changed Can Be Determined

The penalty period is determined as follows according to the amount by which the user intentionally changed the clock, which is calculated from two time points: the timing of a previous call to the function, and the time object specified for currentSnapshot. Note that the automatic clock time adjustments performed by the system do not trigger any penalties even if they affect the user clock.

Clock Operation Case How Penalty Period Is Applied
No clock operation No penalty.
Clock value changed to a time in the future Penalty period of maxPenaltyTimeSpan in duration is applied. By applying the same penalty period used for turning back the clock to the past, it would be possible to obtain benefits at an actual interval of 12 hours, for example by advancing the clock by 12 hours in a case where a benefit may only be obtained once a day, and then waiting for the 12 hour penalty period to pass. To prevent this, there is a stricter penalty period than turning back the clock to a time in the past. However, if the amount by which the clock is changed is less than or equal to acceptableOperationTimeSpan, no penalty period is applied.
Clock value changed to a time in the past The amount by which the clock has been changed is used as the penalty period, with maxPenaltyTimeSpan as the maximum duration. In some cases, after the penalty period elapses, the next timing at which the benefit can be claimed may be equal to the same timing as when the benefit was previously claimed. (Assume that the benefit can be claimed again after the date changes. If the benefit is claimed on 07-20 and then the user rolls back the clock by 24 hours and incurs a 24-hour penalty period, after the penalty period elapses, the next date on which the benefit can be claimed will be 07-20.)

When the Amount by Which the Clock Was Changed Cannot Be Determined

The preceding information applies only when the amount by which the user clock is changed can be correctly determined. However, if the two monotonic clock values lack continuity, the amount by which the clock has been changed cannot be determined. (This can happen when the monotonic clock is reset or when save data is transferred to a different device, for example.)
In this case, the penalty period is determined according to whether the time that has elapsed since the benefit was previously claimed (as stored by pHandover) can be successfully obtained, as follows. The elapsed time is obtained using nn::time::CalculateSpanBetween(). For more information about the conditions that affect whether this value can be successfully obtained, see the associated function reference.

Time Elapsed Between Last Benefit Claim and currentSnapshot How Penalty Period Is Applied
Benefit never claimed, and elapsed time cannot be obtained No penalty period.
Elapsed time successfully obtained A penalty period of maxPenaltyTimeSpan in duration starting from when the benefit was previously claimed is applied. In some cases, the penalty period may have already elapsed.
Elapsed time cannot be obtained A penalty of maxPenaltyTimeSpan in duration starting from currentSnapshot is applied.

However, if the Synchronize Clock via Internet setting in system settings was enabled both when the function was last called and when currentSnapshot was obtained, and if the network clock has been corrected, the user clock has not been manipulated. For this reason, no penalty period is incurred.

Additional Information

  • Special Considerations
    Regardless of the result returned by this function, pHandover must be kept for the next call to the function.
  • Saving the Result of Claiming a Benefit
    When this function returns PeriodicBenefitClaimResult_Success, the application should execute the associated benefit claiming process. To prevent inconsistent data states from occurring due to the application being closed or the device being powered off, we strongly recommend that you save the application-side result of claiming the benefit and the pHandover pointer together in save data at this time.
  • Device Time Zone Changes
    Time differences between clock times before and after the device time zone is changed are treated as clock change operations. However, time changes that are not associated with a change of the time zone on the device (such as for daylight saving time) are not treated as clock change operations. For example, if the time difference relative to UTC time is changed from -3 to +1, it is treated as the clock being advanced by 4 hours. However, if the time gets offset by an hour due to a reason not associated with a time zone change (such as for daylight saving time), it is not treated as a clock change operation.
Examples
TimeClaimPeriodicBenefit/TimeClaimPeriodicBenefit.cpp.

◆ CalculateStandardUserSystemClockDifferenceByUser()

nn::TimeSpan nn::time::CalculateStandardUserSystemClockDifferenceByUser ( const ClockSnapshot &  from,
const ClockSnapshot &  to 
)

Gets the amount by which the user intentionally changed the user clock.

Parameters
[in]fromObject representing the start time from which to detect changes.
[in]toObject representing the end time until which to detect changes.
Returns
The amount by which the user intentionally changed the clock.
Precondition
  • The time library is initialized.
  • from and to are both objects initialized with nn::time::ClockSnapshot::CreateWithStandardSystemClock() or objects initialized with nn::time::ClockSnapshot::CreateWithAdjustableSystemClock().

Details
Gets the amount by which the user intentionally changed the nn::time::PosixTime object returned by the user clock (nn::time::StandardUserSystemClock) between from and to. Returns a positive nn::TimeSpan object if the clock was turned forward and a negative nn::TimeSpan object if the clock was turned back.

A return value of nn::TimeSpan(0) indicates that the user has not intentionally changed the clock. It does not guarantee that the user clock nn::time::PosixTime object has not been rolled back or that the clock has not changed by more than the actual amount of time that has elapsed.

Do not convert the user clock time to an nn::time::CalendarTime object to determine whether the date and time have been changed. In addition, because time changes resulting from daylight saving time or the user changing the system time zone do not change the value of the nn::time::PosixTime object itself, this function returns nn::TimeSpan(0) in these cases. To detect changes in the system time zone, you must separately compare the values returned by from.GetLocationName() and to.GetLocationName().

Any of the following operations that occur between from and to are treated as the user having intentionally changed the clock.

  • The user directly changes the clock (such as in System Settings, DevMenu, or DevMenuCommand).
  • The user clock time is changed due to the user enabling the Synchronize Clock via Internet setting in System Settings.

Meanwhile, the following operations are not treated as the user having changed the clock, even if they change the value of the nn::time::PosixTime object.

  • The Synchronize Clock via Internet setting in System Settings remains enabled, and the system automatically corrects the clock.
  • The user clock time is reset to the initial value due to the monotonically increasing clock losing power.
  • Save data is migrated to another system.

Calculates the actual elapsed time between two nn::time::ClockSnapshot objects.

Parameters
[out]pOutA pointer to a buffer storing the elapsed time.
[in]fromObject representing the start time for the elapsed time calculation.
[in]toObject representing the end time for the elapsed time calculation.
Returns
The calculation result.
Return Values:

true Success.

false Failure.

Precondition
  • The time library is initialized.
  • pOut != nullptr.

Details
Returns a negative value for the elapsed number of seconds when the end time is further in the past than the start time.

If there is a discontinuity in the monotonic clock values that store the start time and the end time, which can happen when save data is transferred to a different system or the monotonic clock is reset, you can use the network clock to measure the elapsed time. However, network clock time is not guaranteed to increase monotonically, which means that elapsed time calculations based on network clock time may contain a non-negligible amount of error. As a result, particularly when calculating short elapsed times, elapsed times that are actually positive may be returned as negative values because of factors such as clock error. For more information about the errors, see Network Clock Time Accuracy .

This function will fail to calculate elapsed time and return false if all of the following conditions are satisfied.

  • The monotonically increasing clock values that store the start time and the end time are discontinuous.
  • Either the start time or the end time does not store a network clock time.

nn::Result nn::time::ToPosixTime ( int *  pOutCount,

int  count,
const CalendarTime &  calendarTime,

)

Converts local time to absolute time.

Parameters
[out]pOutCountThe number of elements stored in pOutPosixTimeList.
[out]pOutPosixTimeListAbsolute time.
[in]countThe number of pOutPosixTimeList elements.
[in]calendarTimeThe local time.
[in]ruleThe time zone rule.
Returns
Result of processing.
Return Values:
ResultSuccess Succeeded.
Precondition
  • The time library is initialized.
  • pOutCount != nullptr.
  • pOutPosixTimeList != nullptr.
  • count > 0.
  • 2000 <= calendarTime.year <= 2100.
  • 1 <= calendarTime.month <= 12.
  • 1 <= calendarTime.day <= 31.
  • 0 <= calendarTime.hour <= 23.
  • 0 <= calendarTime.minute <= 59.
  • 0 <= calendarTime.second <= 59
  • The time zone rule has been loaded.
Postcondition
  • 0 <= *pOutCount <= 2.
  • If *pOutCount == 2, pOutPosixTimeList[0] < pOutPosixTimeList[1].

Details
Automatically uses the TimeZoneRule generated from the LocationName obtained by the GetDeviceLocationName function.

Depending on the local time entered, this function returns zero, one, or two absolute time values, as follows.

Entered Local Time Description of Action
The specified date and time comes around only once. *pOutCount == 1.
The clock is rolled back due to the switch in time, so the specified date and time is duplicated and comes around twice. *pOutCount == 2. pOutPosixTimeList[0] and pOutPosixTimeList[1] are the absolute times before and after the switch in time.
The clock is advanced due to the switch in time, so the specified date and time is skipped and does not exist, or the date is specified as something like February 31 that does not exist. *pOutCount == 0.

Duplication and skipping of times occurs when the time-difference rule applied to a location changes.

This happens at the start and end of daylight saving time, but it can also occur when the law is revised in a way that changes the time-difference rule for a location.

Although you can simplify your implementation by coding your application to automatically adopt either only pOutPosixTimeList[0] or only pOutPosixTimeList[1] when getting two absolute time values, be careful that this does not cause problems for the specification of your application.

You can reduce processing times for release builds, by several microseconds, by using a 4 KB alignment for the TimeZoneRule starting address that is passed to this function.
