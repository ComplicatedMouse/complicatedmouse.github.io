Functions for Knowing About Shop Purchases

This group of functions is used for detecting purchases made with the shop app on NX devices and purchases made on the Nintendo website.

These functions can detect purchases even when they are made on a Switch system that has not been registered as the primary console.

These functions do not support the detection of downloadable content purchases. To see when you can begin using a purchased downloadable content item, call the nn::aoc::GetAddOnContentListChangedEvent() function.

Environment Functions for the Shop Service Access Utility Feature

The module's API for setting the shared resources and other resources necessary to use the ShopServiceAccessor class.

Operation of the ShopServiceAccessor class is assured only between the calls to this API's InitializeForShopServiceAccessors() and FinalizeForShopServiceAccessors() functions.

Alias Replacement
For some of the request strings that are specified when requests are made via ShopServiceAccessor, you can use a feature to replace the alias symbol with an appropriate value when the communication request is made. This feature can be used with the following request elements.
  • URL path field strings.
Alias Symbols
Alias symbols are strings represented by keywords in the following prescribed format.
  • Strings enclosed in curly brackets { }.
Available Alias Symbols
  • applicationId : Replaced by the ID of the running application.
  • country : Replaced by the country of residence registered to the Nintendo Account and linked to the specified user account.
Usage Example
const char* path = "/v1/applications/{applicationId}/consumables? country={country}&lang=ja";

◆ InitializePurchasedEvent()

void nn::ec::InitializePurchasedEvent ( )

Initializes the feature for detecting the purchase of items on the shop.

Details
After this function has been called, any purchases made with the shop app on an NX device or via the Nintendo website can be detected.

Depending on the network environment and the state of the server, you might not be able to detect purchases, or purchases might be delayed. Purchases made prior to initialization might also be detected, so avoid processes that assume the purchases were made subsequent to initialization.

Calling this function after it has already been called has no unintended effects.

See also
nn::ec::GetPurchasedEvent(), nn::ec::PopPurchasedItemInfo()
Examples
EcPurchasedEvent/Main.cpp.

◆ GetPurchasedEvent()

nn::os::SystemEvent& nn::ec::GetPurchasedEvent ( )

Gets the event that is signaled every time an item is purchased on the shop.

Precondition
  • nn::ec::InitializePurchasedEvent() has been called already.

Details
Gets the event that is signaled when an item is purchased on the shop.

After the event has been signaled, you can call the nn::ec::PopPurchasedItemInfo() function to get information about the purchased item.

The mode for clearing events is nn::os::ClearMode_ManualClear. Events can be signaled regardless of the application's network usage declaration.

See also
nn::ec::PopPurchasedItemInfo()
Examples
EcPurchasedEvent/Main.cpp.

Parameters

Precondition

  • pOut != nullptr.
Returns
Whether information has been obtained successfully.

Details
Gets the information about the purchased item held by the system, one piece at a time.

We recommend calling this function repeatedly until it returns false so you don't miss out on any information.

const CourseId &  courseId,
const nn::account::UserHandle &  selectedUser 
)

Displays the list of subscription service items (subscription tickets) and the purchase screen for the specified course and specified user.

Parameters
[in]applicationIdParentThe application ID set in the subscription tickets to display in the list.
[in]courseIdID of the subscription service course to display the list for. Specifies the issued course ID in advance.
[in]selectedUserThe handle representing the user selected by the caller. This is specified when the user is selected ahead of time and other cases where you want to have specific users perform actions like purchases.
Returns
None.
Precondition
  • applicationIdParent != nn::ApplicationId::GetInvalidId()
  • courseId is a null-terminated string.
  • The account library is initialized.

Details

  • To display the product information for the application itself, we recommend using the ShowShopSubscriptionList() function that does not specify ApplicationId. Use this function only when you want to display product information for other applications, or display retail version information from the demo version.
  • This function blocks the calling thread because it displays the UI internally.
  • If the user is someone who cannot be used by selectedUser, this function displays the user-selection UI again for a user to be selected.
NX Platform Only:
This function displays Nintendo eShop using the EC applet. For more information, see EC Applet in the System Features Guide.

◆ InitializeForShopServiceAccessors()

Result nn::ec::InitializeForShopServiceAccessors ( void *  pWorkMemoryTop,
size_t  byteSizeOfTotal 
)

Requests initialization of the shop service access utility feature.

Parameters
[in]pWorkMemoryTopThe starting address of the memory space that will be the work buffer. (Must be aligned to nn::os::MemoryPageSize.)
[in]byteSizeOfTotalThe size, in bytes, of this work buffer region. (Must be aligned to nn::os::MemoryPageSize.)
Returns
Returns the execution result.
Return Values:
nn::ResultSuccess The process worked as expected.
Precondition
  • The environment has not been initialized by InitializeForShopServiceAccessors().
  • The starting address is aligned to the memory page size (nn::os::MemoryPageSize).
  • The byte size of the region is aligned to the memory page size (nn::os::MemoryPageSize).
  • The size of the region, in bytes, is greater than the minimum work buffer size (nn::ec::RequiredMinimumWorkMemorySizeForShopServiceAccessor).
  • The address specified for the work buffer does not specify a region for stack variables. This is because it is presumed that the work region will be maintained beyond the local scope. Allocate the work buffer from a static region or a heap.

Details
The specified buffer will be used as the communication work region for the following purposes. When you specify the work buffer, be sure to allocate enough space to hold the data sets for these purposes.

  • Asynchronous retention of input parameters for shop service communication requests. The requests are processed asynchronously, so the service must keep a copy until communication takes place.
  • Asynchronous retention of the responses output for shop service communication requests. The requests are processed asynchronously, so a copy must be retained until the asynchronous handle is destroyed. This is to ensure the survival of the response associated with the actual completion of communication.
Examples
EcCatalog/Main.cpp, and EcConsumableServiceItemManager/EcConsumableServiceItemManager.cpp.
