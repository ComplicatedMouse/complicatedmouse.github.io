◆ DeliveryCacheProgressStatus

enum nn::bcat::DeliveryCacheProgressStatus

The status of data distribution progress.

Details
The statuses are as follows.

  • Queued – Awaiting execution. The system has multiple added synchronized requests and is currently processing a request that was sent earlier.
  • Connect – Connecting to the network.
  • ProcessList – Currently processing the data distribution list. Receiving and analyzing the data distribution list.
  • Download – Downloading files. You can get the file information (such as name and size) for the file currently being downloaded.
  • Commit – Performing commit processing for the files. The downloaded files are moved to the data distribution cache storage for the application. Also deletes the files that are no longer distributed.
  • Done – Process complete. If the process ends correctly, the synchronization is complete. If the process ends abnormally, only directories that were committed have completed synchronization. Use DeliveryCacheProgress::GetResult to determine whether the process completed correctly.

◆ Initialize()

void nn::bcat::Initialize ( )

Initializes the library.

Details
Call this function to allow the system to detect when applications use the data distribution feature. We recommend calling this function immediately after starting the application.

Configuring the settings for the data distribution feature can sometimes make this function take tens of milliseconds to complete.

The call to this function interrupts background synchronization, and subsequently background synchronization will not take place while the application is running. Background synchronization will resume after the application ends.

The file commit process cannot be immediately interrupted and in some cases will take several seconds to stop. (The execution time of this function is not affected.)

The functions of this library automatically initialize the library if the library was not initialized. You can call this function multiple times.

Examples
BcatPractical/BcatPractical.cpp, and BcatSimple/BcatSimple.cpp.

◆ RequestSyncDeliveryCache() [1/2]

nn::Result nn::bcat::RequestSyncDeliveryCache ( DeliveryCacheProgress *  outProgress)

Requests synchronization of the data distribution cache storage.

(Immediate synchronization.)

Parameters
[out]outProgressProgress object.
Returns
The process result.
Return Values:

ResultTooManySyncRequests Too many synchronization requests were sent.

ResultInternetRequestNotAccepted No Internet connection request has been accepted.

ResultInvalidOperation Cannot execute because the parameters required for the data distribution feature are not written in the NMETA file.

Precondition
  • outProgress != nullptr.
  • The data distribution cache storage is not mounted.

Details
Synchronization requests are managed by the execution queue within the system. Synchronization requests that have not yet been run accumulate in the execution queue because the system processes those requests one by one.

ResultTooManySyncRequests is returned when the execution queue is full and no more synchronization requests can be accepted. For this reason, we recommend waiting for the prior synchronization to complete before submitting a subsequent synchronization request.

An Internet connection request must be accepted in advance before calling this function. Returns ResultInternetRequestNotAccepted if the Internet connection request was not accepted.

Note
The parameters that need to be written to the NMETA file are ApplicationId and BcatPassphrase.
Examples
BcatPractical/BcatPractical.cpp, and BcatSimple/BcatSimple.cpp.

◆ RequestSyncDeliveryCache() [2/2]

nn::Result nn::bcat::RequestSyncDeliveryCache ( DeliveryCacheProgress *  outProgress,
const char *  dirName 
)

Requests synchronization of the data distribution cache storage.

(This specification is an immediate synchronization/directory specification.)

Parameters
[out]outProgressProgress object.
[in]dirNameDirectory name.
Returns
The process result.
Return Values:

ResultTooManySyncRequests Too many synchronization requests were sent.

ResultInternetRequestNotAccepted No Internet connection request has been accepted.

ResultInvalidOperation Cannot execute because the parameters required for the data distribution feature are not written in the NMETA file.

Precondition
  • outProgress != nullptr.
  • dirName != nullptr.
  • The data distribution cache storage is not mounted.

Details
This function specifies directories and requests synchronization. When there is an update for non-specified directories, those directories are not synchronized.

If the accessing directories for each application sequence are clearly defined, the target directories can be narrowed down to reduce the time needed for synchronization. However, because this is unrelated to updating non-specified directories, the data distribution cache storage capacity may be insufficient depending on data distribution conditions.

If capacity is insufficient, DeliveryCacheProgress::GetResult returns ResultUsableSpaceNotEnough. When the application detects this error, it must call a function for immediate synchronization that does not specify directories and perform restoration.

If there are no specified directories, DeliveryCacheProgress::GetResult returns ResultNotFound.

Synchronization requests are managed by the execution queue within the system. Synchronization requests that have not yet been run accumulate in the execution queue because the system processes those requests one by one.

ResultTooManySyncRequests is returned when the execution queue is full and no more synchronization requests can be accepted. For this reason, we recommend waiting for the prior synchronization to complete before submitting a subsequent synchronization request.

An Internet connection request must be accepted in advance before calling this function. Returns ResultInternetRequestNotAccepted if the Internet connection request was not accepted.

Note
The parameters that need to be written to the NMETA file are ApplicationId and BcatPassphrase.
Attention
Error handling is complex, so unless there is a specific reason not to, use a function for immediate synchronization that does not specify directories.

◆ CancelSyncDeliveryCacheRequest()

void nn::bcat::CancelSyncDeliveryCacheRequest ( )

Cancels the synchronization request of the data distribution cache.

Precondition
  • The data distribution cache storage is not mounted.

Details
Cancels all synchronization requests triggered by RequestSyncDeliveryCache.

This function returns processing immediately. We recommend that the application call this function and then wait for the progress status to enter the completed state.

The result of a canceled synchronization request is ResultCanceledByUser.

If a request is canceled during commit processing, synchronization continues until that commit processing is completed. This prevents the data distribution cache storage from entering an incomplete state from the forced suspension of file save processing.

If a request is canceled outside of the commit processing, synchronization finishes immediately. Downloaded files are saved to the data distribution cache storage in the commit processing stage. Note that at this time, files that are being downloaded or have finished downloading are not yet saved to the data distribution cache storage.
