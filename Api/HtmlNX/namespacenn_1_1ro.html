Detailed Description

ro library namespace.

Feature Overview

The ro library provides a method for using the dynamic module.

How to Use

The process flow from loading and using one dynamic module until all of the memory is released is as follows.

  1. Call the nn::ro::Initialize() function.
  2. Allocate a heap area for loading the NRR file.
  3. Load the NRR file into the heap area that was allocated.
  4. Call the nn::ro::RegisterModuleInfo() function.
  5. Allocate a heap area for loading the NRO file.
  6. Load the NRO file into the heap area that was allocated.
  7. Call the nn::ro::GetBufferSize() function and investigate the size of the heap area required for the BSS session of the dynamic module.
  8. Allocate the heap area.
  9. Call the nn::ro::LoadModule() function.
  10. Use the nn::ro::LookupSymbol() or nn::ro::LookupModuleSymbol() function to use the features of the loaded dynamic module.
  11. Call the nn::ro::UnloadModule() function.
  12. Release the heap area that was allocated for the BSS session of the dynamic module.
  13. Release the memory that was allocated for the NRO file.
  14. Call the n::ro::UnregisterModuleInfo() function.
  15. Release the memory that was allocated for the NRR file.
  16. Call the nn::ro::Finalize() file.

Terminology

Execution code

This code is the machine language translated and executed by the CPU, and the constants and variables that it directly references.

Module

A meaningful batch of execution code.

Static module

The module that allocates the memory that is loaded when the application starts. This memory will not be released while the application is running.

Dynamic module

The module that can dynamically read and use memory. This memory can be released while the application is running.

Symbol

A meaningful location within the module. While many symbols have names, some do not.

Reference

Using a symbol from another module. This term applies to operations such as calling a function defined in another module or reading/writing variables that are defined in another module.

Resolve

The referenced symbol is in a state where it can be used. Determine the address of the symbol and make it possible to call the function or read/write the variables.

NRO file

Dynamic module file.

NRR file

The file containing registration information for the NRO file. Contains the information for one or more NRO files.

◆ Finalize()

void nn::ro::Finalize ( )

Finalizes the ro library.

Precondition
  • The ro library is in an initialized state.
Postcondition
  • The ro library is in an uninitialized state.
  • The loaded NRO file or NRR file is released.

Details
Deletes all registered module information, and releases all of the dynamic modules that were loaded. The destructors for the static objects of the loaded dynamic modules are called in this function. nn::ro::LoadModule(), nn::ro::UnloadModule(), and nn::ro::Finalize() cannot be called from the destructors of static objects.

This function is not thread-safe. Do not call the ro library functions while this function is running.

Windows Platform Only:
  • This API cannot be used in a Windows environment.
Examples
IndependentBuildSystem/Main.cpp, RoSimple/RoDeadStripStaticApplication/RoDeadStripStaticApplication.cpp, and RoSimple/RoStaticApplication/RoStaticApplication.cpp.

Gets the memory size required to load a Module object.

Parameters
[out]pOutSizeThe address that stores the size of the area used as the BSS section of the Module object.
[in]pImageThe starting address of the memory area where the NRO file was loaded.
Returns
Returns a nn::Result that represents the process result.
  • nn::ro::ResultInvalidNroImage must be handled.
Precondition
  • None.
Postcondition
  • None.

Details
When the nn::ro::LoadModule() function is called, the user must specify the area for the BSS section used by the module. This function gets the memory size that will be required at that time. The obtained memory size is always an integer multiple of nn::os::MemoryPageSize. However, it may not be an integer multiple of nn::os::MemoryBlockUnitSize. Note this fact when directly passing a memory size obtained with the nn::os::AllocateMemoryBlock() function.

This function is thread-safe.

Windows Platform Only:
  • This API cannot be used in a Windows environment.

IndependentBuildSystem/Main.cpp, RoSimple/RoDeadStripStaticApplication/RoDeadStripStaticApplication.cpp, and RoSimple/RoStaticApplication/RoStaticApplication.cpp.

◆ RegisterModuleInfo()

nn::Result nn::ro::RegisterModuleInfo ( RegistrationInfo *  pOutInfo,
const void *  pImage 
)

Registers the module information.

Parameters
[out]pOutInfoThe address that stores the registration object.
[in]pImageThe starting address of the memory to which the NRR file is loaded.
Returns
Returns a nn::Result that represents the process result.
  • nn::ro::ResultInvalidNrrImage must be handled.
  • nn::ro::ResultMaxRegistration must be handled.
  • nn::ro::ResultNotAuthorized must be handled.
Precondition
  • The area indicated by pImage is the memory area allocated by nn::os::AllocateMemoryBlock(), and nn::os::MemoryPermission_ReadWrite has the access rights.
  • pImage is an integer multiplier of nn::os::MemoryPageSize.
  • The state of pOutInfo is uninitialized or Unregistered.
  • The ro library is in an initialized state.
  • The NRR file is in the specified location in the ROM-FS and is not executed in RAW format.
Postcondition
  • The state of pOutInfo is Registered.

Details
Place the NRR file under the management of the ro library, and register the NRO file information. The NRR file that contains the NRO file information must be registered before calling nn::ro::LoadModule(). If the function succeeds, the region specified by pOutInfo and pImage is placed under the management of the ro library and is no longer available for other purposes. To release this area from under the management of the ro library, the nn::ro::UnregisterModuleInfo() function must be called. The restriction that the NRR file be in the specified location in the ROM-FS and not executed in RAW format is disabled if the Ease Nro Restriction option in the DevMenu development support features has been selected.

This function is thread-safe.

Windows Platform Only:
  • This API cannot be used in a Windows environment.
Examples
IndependentBuildSystem/Main.cpp, RoSimple/RoDeadStripStaticApplication/RoDeadStripStaticApplication.cpp, and RoSimple/RoStaticApplication/RoStaticApplication.cpp.

◆ LoadModule() [2/2]

nn::Result nn::ro::LoadModule ( Module *  pOutModule,
const void *  pImage,
void *  buffer,
size_t  bufferSize,
int  flag,
bool  isNotReferenced 
)

Loads the Module object.

When the Module object is not referenced from other modules, nn::ro::UnloadModule() can perform read operations using the option to operate at a faster speed.

Parameters
[out]pOutModuleThe address that stores the Module object.
[in]pImageThe starting address of the memory area where the NRO file was loaded.
[in]bufferThe starting address of the area used as the BSS section.
[in]bufferSizeThe size of the area indicated by buffer.
[in]flagThe flag related to the policy for resolving symbols.
[in]isNotReferencedWhen Module object symbols are not resolved from other modules, specify true to have nn::ro::UnloadModule() operate at a faster speed.
Returns
Returns a nn::Result that represents the process result.
  • nn::ro::ResultInvalidNroImage must be handled.
  • nn::ro::ResultOutOfAddressSpace must be handled.
  • nn::ro::ResultNroAlreadyLoaded must be handled.
  • nn::ro::ResultMaxModule must be handled.
  • nn::ro::ResultNotAuthorized must be handled.
Precondition
  • The state of pOutModule is uninitialized or Unloaded.
  • The area specified by pImage is within the area of memory allocated by the nn::os::AllocateMemoryBlock() function, and the access permission is nn::os::MemoryPermission_ReadWrite.
  • pImage is an integer multiple of nn::os::MemoryPageSize.
  • The area specified by buffer and bufferSize is within the area of memory allocated by the nn::os::AllocateMemoryBlock() function, and the access permission is nn::os::MemoryPermission_ReadWrite.
  • buffer and bufferSize are integer multiples of nn::os::MemoryPageSize.
  • Either nn::ro::BindFlag_Now or nn::ro::BindFlag_Lazy is specified for flag.
  • The NRR file that contains the information for the NRO file to load is registered using nn::ro::RegisterModuleInfo().
  • The ro library is in an initialized state.
Postcondition
  • pOutModule is in the Loaded state.

Details
Relocates the NRO file, resolves references between already loaded modules, and places them under the management of the ro library. If the function succeeds, the area specified by pImage and buffer is placed under the management of the ro library and can no longer be used for other purposes. To release this area from under the management of the ro library, the nn::ro::UnloadModule() function must be called. The constructors for the static objects of the specified module are called automatically when this function is called. nn::ro::LoadModule(), nn::ro::UnloadModule(), and nn::ro::Finalize() cannot be called from the constructors of static objects.

flag must include either nn::ro::BindFlag_Now or nn::ro::BindFlag_Lazy. If you specify nn::ro::BindFlag_Now and use immediate resolution, the symbols will be resolved when the module is loaded. If you specify nn::ro::BindFlag_Lazy and use delayed resolution, the symbols for a given function will be resolved when that function is actually called. However, non-function symbols will be resolved immediately rather than delayed.

If true is specified for isNotReferenced, the processing time will be shorter when nn::ro::UnloadModule() is called on the specified module. However, if true is specified and another module acts to resolve the symbols of this module, the process will abort at that time. Calls via the addresses of symbols that are explicitly obtained through nn::ro::LookupSymbol() and nn::ro::LookupModuleSymbol() are not affected. If false is specified for isNotReferenced, the processing time for nn::ro::UnloadModule() will not be made shorter, but it will be possible to resolve symbols from other modules.

This function is thread-safe.

Windows Platform Only:
  • This API cannot be used in a Windows environment.

◆ UnloadModule()

void nn::ro::UnloadModule ( Module *  pModule)

Releases the Module object.

Parameters
[in]pModuleThe address that stores the Module object.
Precondition
  • pModule is in a Loaded state.
  • The ro library is in an initialized state.
  • The function is not called from a thread where a constructor for a global TLS object in the specified module was called.
Postcondition
  • pModule is in an Unloaded state.
  • The area specified by pImage in the nn::ro::LoadModule() function is nn::os::MemoryPermission_ReadWrite.
  • The area specified by buffer and bufferSize in the nn::ro::LoadModule() function is nn::os::MemoryPermission_ReadWrite.

Details
Releases the specified module. The memory related to this module that is under the management of the ro library is released so it can be used for other purposes. Specifically, this refers to the memory areas specified in the nn::ro::LoadModule() function by the pImage parameter, and by the buffer and bufferSize parameters. The destructors for the static objects of the specified module are called automatically when this function is called. nn::ro::LoadModule(), nn::ro::UnloadModule(), and nn::ro::Finalize() cannot be called from the destructors of static objects. This function will abort if there are any global TLS objects that have been constructed within the module but whose destructors are not called automatically. Never call this function from a thread on which the constructor for a global TLS object of the specified module was called. This function can fail when memory managed by other libraries remains.

This function is thread-safe.

Windows Platform Only:
  • This API cannot be used in a Windows environment.
Examples
IndependentBuildSystem/Main.cpp, RoSimple/RoDeadStripStaticApplication/RoDeadStripStaticApplication.cpp, and RoSimple/RoStaticApplication/RoStaticApplication.cpp.
