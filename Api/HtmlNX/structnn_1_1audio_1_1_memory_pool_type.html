Detailed Description

Structure for representing memory regions used by an audio renderer.

Details
Memory pools define regions of memory that are used (accessed or manipulated) by audio renderers. A memory pool is initialized with a specified address and size and can then be attached to an audio renderer. This causes the memory pool to transition to the attached state. In this state, the audio renderer can use the corresponding memory regions. Conversely, audio renderers cannot use the memory regions corresponding to memory pools that are in the detached state. Resource data (described later) used by audio renderers must be stored in a memory region managed by a memory pool.

Note that the alignment and size of the specified region may be restricted, depending on the runtime environment.

Audio renderers directly access the following types of resource data. All of this data must be contained in the memory space maintained by the memory pool.

  • Buffers specified for WaveBuffer.buffer and WaveBuffer.pContext.
  • Memory storing the data passed to pParameter in the nn::audio::AcquireVoiceSlot() function.
  • The work buffer specified for buffer when the following effects are added.
    • nn::audio::AddDelay()
    • nn::audio::AddReverb()
    • nn::audio::AddI3dl2Reverb()
    • nn::audio::AddLightLimiter()
    • nn::audio::AddCapture()
  • Work buffers specified for the pSendBuffer and pReturnBuffer parameters in the nn::audio::AddAux() function.
  • Buffers specified in the buffer parameter of nn::audio::AddCircularBufferSink().

Memory pools cannot be partially attached or detached. Create separate memory pools according to the granularity you will need when controlling attachment and detachment.

For more information, see Audio Library > Memory Pool.

Directly accessing or changing member variables is prohibited. Use the main structure through the following functions.
