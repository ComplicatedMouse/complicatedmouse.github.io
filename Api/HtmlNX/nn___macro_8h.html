◆ NN_UTF8_TEXT

#define NN_UTF8_TEXT (   str)    (::nn::detail::MakeUtf8String(u8##str))

Gets a reference to the array storing the UTF-8 byte representation of the str string literal.

Parameters
[in]strA string literal.

Details
Gets a reference to the array storing the UTF-8 byte representation of the str string literal. The type of the return value is const char (&)[N]. (N is the byte size + 1 of the UTF-8 representation of str.)

This macro internalizes the difference in the meaning of the u8 string literal prefix between C++17 and earlier and C++20 and later. In C++17 and earlier, string literals with the u8 string literal prefix have the const char[N] type, whereas in C++20 and later, the type is const char8_t[N]. This difference can lead to compatibility problems. By using this macro instead of the u8 string literal prefix, all of the string literals are treated as the const char (&)[N] type, regardless of the C++ version.

Example:

constexpr auto& Name = NN_UTF8_TEXT("Nintendo");
static_assert(sizeof(Name) == 10U);
NN_UTF8_TEXT

Defines local static variables for functions in C++11 and later.

Parameters
[in]typeThe type to define.
[in]nameThe name of the variable to define.
[in]...The variable initialization portion.

Details
In VC++ 2012 and VC++ 2013 environments, emulation occurs with C++11 and later for the following code within the function.

static type name __VA_ARGS__

The initialization of the variable set with name occurs in a thread-safe manner.

In other environments, the code above would be parsed as is.

Example:

class C
{
public:
C();
NN_IMPLICIT C(int);
C (int, int);
};
void F()
{
NN_FUNCTION_LOCAL_STATIC(C, x0); // static C x0;
NN_FUNCTION_LOCAL_STATIC(C, x1a, = 10); // static C x1a = 10;
NN_FUNCTION_LOCAL_STATIC(C, x1b, (10)); // static C x1b(10);
NN_FUNCTION_LOCAL_STATIC(C, x2, (10, 20)); // static C x2(10, 20);
}

◆ NN_PACK

#define NN_PACK

Suppresses implicit padding insertion within the class.

Details
Suppresses implicit padding insertion within the class. The class specified by this macro has 1-byte alignment.

Even if you write this macro in the MSVC environment, it has no effect. Use NN_PACK_BEGIN and NN_PACK_END for the MSVC environment.

We recommend using this macro only in cases where unaligned data members need to be defined, such as when defining data structures for which the format is specified by a standard. Note that if you write processing that accesses unaligned data members, inefficient code may be generated, or an alignment exception may be thrown during execution.

If there are no data members that will be unaligned and you simply want to ensure that unintended padding insertion does not occur, we recommend using NN_ENABLE_WARNING_IMPLICIT_PADDING_WITHIN_CLASS for detection, or using NN_STATIC_ASSERT to ensure the member offsets and class size.

Usage Example (Code Valid for Both MSVC and Non-MSVC)

NN_PACK_BEGIN
struct Example
{
uint8_t m1;
uint16_t m2;
} NN_PACK;
NN_PACK_END
NN_PACK_END
